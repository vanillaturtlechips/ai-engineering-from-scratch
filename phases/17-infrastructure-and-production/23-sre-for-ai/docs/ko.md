# AI를 위한 SRE — 멀티에이전트 인시던트 대응, 런북, 예측적 감지

> AI SRE는 RAG를 통해 인프라 데이터(로그, 런북, 서비스 토폴로지)에 기반한 LLM을 사용하여 조사, 문서화, 조정 단계를 자동화합니다. 2026년 아키텍처 패턴은 멀티에이전트 오케스트레이션입니다. 감독자(supervisor)가 조정하는 전문 에이전트(로그, 메트릭, 런북)로 구성되며, AI가 가설과 쿼리를 제안하고 인간이 판단 사항을 승인합니다. Datadog Bits AI와 Azure SRE Agent는 이를 관리형 제품으로 출시했습니다. 런북은 진화하고 있습니다: NeuBird Hawkeye는 적대적 평가(동일한 인시던트를 두 모델이 분석; 일치 = 신뢰도, 불일치 = 불확실성)를 사용하며, 운영 메모리는 팀 변경 사항에도 지속됩니다. 자동 복구는 신중하게 유지됩니다: AI가 제안하고 인간이 승인합니다. 완전 자율 행동은 좁은 범위(팟 재시작, 특정 배포 롤백)에 제한되며 엄격한 가드레일이 적용됩니다. "설정하고 잊어버리기"를 판매하는 사람은 과장하고 있는 것입니다. 새로운 프론티어: 인시던트 예측. MIT 연구에 따르면 과거 로그 + GPU 온도 + API 오류 패턴으로 훈련된 LLM이 10-15분 전에 89%의 장애를 예측했습니다. 전망: 2026년 말까지 기업 LLM의 95%가 자동화된 장애 조치를 보유할 것입니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, 장난감 멀티에이전트 인시던트 트리아지 시뮬레이터)
**선수 지식:** 17단계 · 13 (관측 가능성), 17단계 · 24 (카오스 엔지니어링)
**소요 시간:** ~60분

## 학습 목표

- 다중 에이전트 AI SRE 아키텍처(감독자 + 전문 에이전트(로그, 메트릭, 런북) + 인간 승인 게이트)를 다이어그램으로 표현.
- 자동 복구(auto-remediation)가 광범위한(서비스 재설계) 대신 좁은 범위(팟 재시작, 배포 롤백)인 이유 설명.
- 적대적 평가 패턴(NeuBird Hawkeye) 명명: 두 모델 일치 = 신뢰도 상승; 불일치 = 에스컬레이션.
- MIT의 89% 조기 발견 결과와 운영 제약 조건(인사이트 없는 예측은 대시보드에 불과) 인용.

## 문제

당직 엔지니어가 새벽 3시에 호출받습니다. "체크아웃에서 높은 오류율 발생." 그들은 Datadog, Loki, 세 가지 런북, 배포 로그를 확인합니다. 30분 후 근본 원인이 vLLM OOM(메모리 부족)이며, 이는 KV 캐시 급증 때문임을 깨닫습니다. 그들은 파드를 재시작하고, 오류가 해결됩니다.

2026년에는 이 조사의 첫 20분을 자동화할 수 있습니다. 서비스별 로그 그룹화, 최근 배포와의 연관성 분석, 런북과의 매칭 — 이 모든 것이 RAG(Retrieval-Augmented Generation) + 도구 사용입니다. 감독형 에이전트가 1차 진단을 수행하고, 인간이 Datadog을 열기 전에 가설을 제시할 수 있습니다.

완전 자율 복구는 다른 문제입니다. 파드 재시작: 안전함. GPU 풀 확장: 정책 허용 시 안전함. 서비스 재설계: 절대 안 됨. 핵심은 좁은 경계를 정확히 그리는 것입니다.

## 개념

### 다중 에이전트 아키텍처

```
          인시던트
             │
             ▼
        감독자(Supervisor)
        /    |    \
       ▼     ▼     ▼
  로그 에이전트  메트릭 에이전트  런북 에이전트
       │     │     │
       └─────┴─────┘
             │
             ▼
        가설 + 증거
             │
             ▼
        인간 승인
             │
             ▼
        액션(좁은 집합)
```

감독자(Supervisor)는 인시던트를 하위 쿼리로 분할합니다. 전문 에이전트들은 도구 접근 권한(로그 검색, PromQL, 문서 검색)을 가집니다. 감독자는 종합한 후 가설 + 증거를 인간에게 제시합니다. 인간은 승인하거나 재지정합니다.

### 자동 복구 범위

**안전함(좁은 범위)**: 파드 재시작, 특정 배포 롤백, 사전 승인된 범위 내 풀 확장, 사전 승인된 기능 플래그 활성화.

**안전하지 않음(넓은 범위)**: 서비스 토폴로지 변경, 리소스 제한 수정, 새 코드 배포, IAM 변경, 데이터베이스 변경.

"설정하고 잊어버리기"를 판매하는 사람은 과장하고 있습니다. AI SRE가 성숙함에 따라 안전한 집합은 커지지만, 경계는 실제로 존재합니다.

### 적대적 평가(NeuBird Hawkeye)

두 모델이 독립적으로 동일한 인시던트를 분석합니다. 근본 원인에 동의하면 신뢰도가 높습니다. 의견이 불일치하면 두 가설을 모두 인간에게 에스컬레이션합니다. 단순한 패턴이지만, 환각된 근본 원인에 대한 효과적인 필터입니다.

### 운영 메모리

팀 교체는 전통적인 SRE의 소리 없는 적입니다 — 부족 지식이 사라집니다. AI SRE는 런북 + 사후 분석을 벡터 DB에 저장합니다. 에이전트들은 모든 새로운 인시던트에서 이를 검색합니다. 새로운 엔지니어가 합류할 때 AI는 전체 기록을 보유합니다.

### 인시던트 예측

MIT 2025 연구: 과거 로그, GPU 온도, API 오류 패턴으로 훈련된 LLM이 테스트 세트에서 10-15분 전에 89%의 장애를 예측했습니다.

현실 점검: 실행(actuation) 없는 예측은 대시보드입니다. 운영 질문은 "예측할 때 무엇을 할 것인가?"입니다. 사전 배수? 페이저? 자동 확장? 답은 정책별로 다릅니다.

### 2026년 제품

- **Datadog Bits AI** — Datadog 내부의 관리형 SRE 코파일럿.
- **Azure SRE Agent** — Azure 네이티브.
- **NeuBird Hawkeye** — 적대적 평가 + 운영 메모리.
- **PagerDuty AIOps** — 트리아지 + 중복 제거.
- **Incident.io Autopilot** — 인시던트 지휘자 + 조정.

### 코드로서의 런북

런북은 Confluence 페이지에서 버전 관리된 마크다운(증상, 가설, 확인, 조치 등 구조화된 섹션)으로 진화합니다. 구조화된 런북은 더 나은 RAG 검색을 지원합니다. 모든 AI-SRE 롤아웃은 비구조화된 런북을 구조화하는 것부터 시작하세요.

### 기억해야 할 숫자

- MIT 조기 감지: 89%의 장애, 10-15분 선행 시간.
- 다중 에이전트 트리아지: 감독자 + (로그, 메트릭, 런북) + 인간.
- 안전한 자동 복구 집합: 파드 재시작, 배포 롤백, 범위 내 확장.
- 적대적 평가: 두 독립 모델; 일치 = 신뢰도.

## 사용 방법

`code/main.py`는 다중 에이전트 트리아지 시스템을 시뮬레이션합니다. 로그 에이전트(log agent)는 오류를 탐지하고, 메트릭 에이전트(metric agent)는 CPU 사용량 급증을 감지하며, 런북 에이전트(runbook agent)는 알려진 문제와 매칭합니다. 슈퍼바이저(supervisor)는 가설(hypotheses)을 순위별로 정렬합니다.

## 연습 문제

1. `code/main.py`를 실행해 보세요. 로그(log) 에이전트와 메트릭(metric) 에이전트가 서로 다른 의견을 보일 경우 어떻게 될까요? 감독자(supervisor)는 어떻게 이를 해결하나요?
2. 서비스에 대한 세 가지 "안전한" 자동 복구(auto-remediation) 액션을 정의하고 각각을 정당화하세요.
3. 구조화된 런북(runbook) 템플릿을 작성하세요: 섹션, 필수 필드, 검증 명령어.
4. 예측 감지(predictive detection)가 12분 선행 시간(lead time)으로 발동됩니다. 페이저(pager) 알림, 사전 드레인(pre-drain), 또는 둘 다를 적용할 정책을 어떻게 수립하시겠습니까?
5. 3인 팀이 2026년에 AI SRE를 도입해야 하는지, 아니면 기다려야 하는지 주장해 보세요. 성숙도(maturity), 작업량(volume), 위험(risk)을 고려하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| AI SRE | "on-call 에이전트" | LLM 기반 사고 조사 + 조정 |
| Supervisor agent | "오케스트레이터" | 사고를 하위 쿼리로 분할하는 최상위 에이전트 |
| Specialized agent | "도메인 에이전트" | 도구(로그, 메트릭, 런북) 접근 권한이 있는 하위 에이전트 |
| Auto-remediation | "AI가 수정한다" | 좁은 범위의 사전 승인된 조치; 광범위한 재설계 아님 |
| Operational memory | "벡터 런북" | RAG를 위한 벡터 DB에 저장된 사후 분석 + 런북 |
| Adversarial eval | "두 모델 검증" | 독립적인 분석; 합의 = 신뢰도 |
| NeuBird Hawkeye | "적대적 평가 모델" | 적대적 평가 + 메모리 패턴을 가진 제품 |
| Bits AI | "Datadog의 SRE 에이전트" | Datadog 관리형 AI SRE |
| Pre-incident prediction | "조기 감지" | 장애 예측 시 10-15분 선행 시간 |
