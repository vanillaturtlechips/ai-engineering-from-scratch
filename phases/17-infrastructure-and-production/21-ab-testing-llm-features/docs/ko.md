# LLM 기능 A/B 테스트 — GrowthBook, Statsig, 그리고 '분위기' 문제

> 전통적인 A/B 테스트는 비결정적 LLM을 위해 설계되지 않았습니다. 중요한 차이점: 평가(eval)는 "모델이 작업을 수행할 수 있는가?"에 답하고, A/B 테스트는 "사용자가 관심을 갖는가?"에 답합니다. 둘 다 필요하며, '분위기' 확인만으로 출시하는 것은 이제 끝났습니다. 2026년에 테스트해야 할 사항: 프롬프트 엔지니어링(문장 구성), 모델 선택(GPT-4 vs GPT-3.5 vs OSS; 정확도 vs 비용 vs 지연 시간), 생성 매개변수(온도, top-p). 실제 사례: 챗봇 보상 모델 변형에서 대화 길이 +70%, 유지율 +30% 달성; Nextdoor AI 제목줄 실험에서 보상 함수 개선 후 CTR +1% 달성; Khan Academy Khanmigo는 지연 시간 대 수학 정확도 축에서 반복 개선. 플랫폼 비교: **Statsig**(2025년 9월 OpenAI에 $1.1B에 인수) — 순차적 테스트, CUPED, 올인원. **GrowthBook** — 오픈소스, 웨어하우스 네이티브, 베이지안 + 빈도주의 + 순차적 엔진, CUPED, SRM 검증, Benjamini-Hochberg + Bonferroni 보정. 선택은 웨어하우스-SQL 선호도와 "OpenAI 인수 여부"가 조직에 중요한지 여부에 따라 달라집니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 간단한 순차적 테스트 시뮬레이터)  
**선수 조건:** 17단계 · 13(관측 가능성), 17단계 · 20(점진적 배포)  
**소요 시간:** ~60분

## 학습 목표

- 평가(evals, "모델이 작업을 수행할 수 있는가")와 A/B 테스트("사용자가 관심을 갖는가")를 구분한다.
- 세 가지 테스트 가능한 축(프롬프트, 모델, 파라미터)을 열거하고 각각에 대한 메트릭을 선택한다.
- CUPED, 순차적 테스트, Benjamini-Hochberg 다중 비교 보정 방법을 설명한다.
- 웨어하우스-SQL 접근 방식과 기업 인수 전략에 따라 Statsig 또는 GrowthBook을 선택한다.

## 문제

시스템 프롬프트를 수동으로 조정했습니다. 느낌이 더 좋아졌습니다. 이를 출시했습니다. 전환율은 노이즈 수준으로만 변했습니다. 당신은 메트릭을 탓합니다. 또는 새 모델을 출시했는데 전환율이 변하지 않았습니다. 모델이 성능이 저하된 것인지, 아니면 변화가 너무 작아 감지할 수 없는 것인지 알 수 없습니다. A/B 테스트 없이 출시했기 때문입니다.

평가(Eval)는 라벨이 지정된 데이터 세트에서 모델이 작업을 수행할 수 있는지 여부를 답합니다. 사용자 선호도를 답하지 않습니다. 통제된 온라인 실험만이 이를 답할 수 있으며, 실험에 충분한 검정력(power)이 있고, 비결정성(non-determinism)을 통제하며, 다중 비교(multiple comparisons)를 보정할 때만 가능합니다.

## 개념

### 평가(Evals) vs A/B 테스트

**평가(Evals)** — 오프라인, 레이블된 데이터 세트, 평가자(평가 기준표 또는 LLM-as-judge 또는 인간). 답변: "고정된 분포에서 출력이 정확/유용/안전한가?"

**A/B 테스트** — 온라인, 실제 사용자, 무작위 할당. 답변: "새로운 변형이 중요한 사용자 수준 지표를 개선하는가?"

둘 다 필요합니다. 평가는 노출 전 회귀를 감지하고, A/B 테스트는 제품 영향을 확인합니다.

### 테스트 대상

1. **프롬프트 엔지니어링** — 문구, 시스템 프롬프트 구조, 예시. 지표: 작업 성공률, 사용자 유지율, 비용/요청.
2. **모델 선택** — GPT-4 vs GPT-3.5-Turbo vs Llama-OSS. 지표: 정확도(작업) + 비용/요청 + 지연 시간 P99. 다중 목표.
3. **생성 파라미터** — 온도(temperature), top-p, max_tokens. 지표: 작업별(출력 다양성 vs 결정성).

### CUPED — 분산 감소

사전 실험 데이터를 활용한 통제 실험. 사후 기간 비교 전 사전 기간 분산을 회귀적으로 제거합니다. 일반적인 분산 감소율: 30-70%. 유효 표본 크기가 무료로 증가합니다.

구현: Statsig와 GrowthBook 모두 구현.

### 순차적 테스트

고전적 A/B 테스트는 고정된 표본 크기를 가정합니다. 순차적 테스트("중간 확인 및 결정")는 반복 검토 시 거짓 양성률을 제어합니다. 항상 유효한 순차적 절차(mSPRT, Howard의 신뢰 구간 시퀀스)는 명확한 승자 발생 시 조기 종료를 허용합니다.

### 다중 비교 보정

95% 신뢰 수준에서 20개의 A/B 테스트를 실행하면 우연히 1개의 거짓 양성이 발생합니다. 본페로니 보정은 테스트당 α를 강화하고, 벤자미니-호흐베르크는 거짓 발견률을 제어합니다. GrowthBook은 둘 다 구현.

### SRM — 표본 비율 불일치

할당 해시는 사용자를 변형에 무작위로 할당합니다. 50/50 분할이 47/53을 제공하면 문제가 발생한 것입니다. SRM 검사가 이를 플래그 처리합니다. 두 플랫폼 모두 구현.

### Statsig vs GrowthBook

**Statsig**:
- OpenAI가 2025년 9월에 11억 달러에 인수. 호스팅, SaaS.
- 순차적 테스트, CUPED, 보류 인구.
- 통합 솔루션: 기능 플래그 + 실험 + 관측 가능성.
- 최적 적합: 이미 번들 제품을 원하는 팀, OpenAI 소유권 무관.

**GrowthBook**:
- 오픈소스(MIT); 웨어하우스 네이티브(Snowflake/BigQuery/Redshift 직접 읽기).
- 다중 엔진: 베이지안, 빈도주의, 순차적.
- CUPED, SRM, 본페로니, BH 보정.
- 자체 호스팅 또는 관리형 클라우드.
- 최적 적합: 웨어하우스-SQL 환경, 데이터 팀이 지표 계층 제어, 오픈소스 선호.

### 비결정성은 검정력 계산을 복잡하게 함

동일한 프롬프트가 다양한 출력을 생성합니다. 전통적 검정력 계산은 IID 관측을 가정합니다. LLM 비결정성으로 인해 유효 표본 크기는 명목상보다 낮습니다. 안전 마진으로 필요한 표본 크기를 ~1.3-1.5배 증가.

### 실제 사례 결과

- 챗봇 보상 모델 변형: 대화 길이 +70%, 유지율 +30%.
- Nextdoor 제목 줄: 보상 함수 개선 후 CTR +1%.
- Khan Academy Khanmigo: 반복적 지연 시간-수학 정확도 트레이드오프.

### 안티패턴: 느낌에 따른 출시

모든 시니어 엔지니어는 "느낌이 좋아서" A/B 없이 출시된 기능을 언급할 수 있습니다. 대부분은 팀이 몇 달간 눈치채지 못한 제품 지표를 악화시켰습니다. A/B는 강제 기능입니다.

### 기억해야 할 숫자

- Statsig OpenAI 인수: 11억 달러, 2025년 9월.
- GrowthBook: 오픈소스 MIT; 베이지안 + 빈도주의 + 순차적.
- CUPED 분산 감소: 30-70%.
- LLM 비결정성 → 표본 크기 버퍼 +30-50%.

## 연습 문제

1. `code/main.py`를 실행하세요. 기준 전환율 3%에서 예상 5% 상승(lift)을 80% 검정력(power)으로 달성하려면 필요한 표본 크기(sample size)는 얼마인가요?  
2. 의료 규제 온프레미스(on-prem) 고객에게 Statsig 또는 GrowthBook 중 어떤 도구를 선택하시겠습니까?  
3. GPT-4와 GPT-3.5를 "해결된 티켓당 비용(cost-per-resolved-ticket)"으로 비교하는 A/B 테스트를 설계하세요. 주요 지표(primary metric), 가드레일 지표(guardrail metric), 보조 지표(secondary metric)는 무엇인가요?  
4. 카나리 배포(canary)는 통과했지만 A/B 테스트에서 전환율이 -1.2% 감소했습니다. 출시하시겠습니까? 에스컬레이션 기준(escalation criteria)을 작성하세요.  
5. 사전 기간(pre-period)의 분산이 사후 기간(post)의 60%일 때 CUPED를 적용하세요. 유효 표본 크기(effective-sample-size) 향상도를 계산하세요.  

> **참고**:  
> - `code/main.py`는 번역하지 않습니다(규칙 3).  
> - "lift", "power", "sample size", "canary", "CUPED" 등 전문 용어는 원문을 유지하되, 문맥에 따라 괄호 안에 한국어 설명을 추가했습니다.  
> - "cost-per-resolved-ticket"은 지표명 그대로 유지하되, 번역 시 "해결된 티켓당 비용"으로 처리했습니다.

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| Eval | "오프라인 테스트" | 모델 능력에 대한 레이블 세트 평가 |
| A/B 테스트 | "실험" | 사용자 대상 실시간 무작위 비교 |
| CUPED | "분산 감소" | 분산 감소를 위한 사전 기간 회귀 분석 |
| 순차적 테스트 | "피킹 허용 테스트" | 조기 중단이 가능한 항상 유효한 절차 |
