# 프롬프트 캐싱과 시맨틱 캐싱 경제학

> **2026-04 기준 가격 스냅샷.** 아래 수치적 주장은 이 문서 게시 시점에 캡처한 공급업체 요금표를 반영합니다. 인용 전 링크된 문서를 통해 최신 정보를 확인하세요.

> 캐싱은 두 계층에서 발생합니다. L2(공급업체 수준) 프롬프트/접두사 캐싱은 반복된 접두사에 대해 어텐션 KV를 재사용합니다 — Anthropic의 프롬프트 캐싱 문서는 긴 프롬프트에서 최대 90% 비용 절감과 85% 지연 시간 감소를 광고합니다; Claude 3.5 Sonnet의 캐시 읽기 비용은 $0.30/M이며, 5분 TTL로 $3.00/M의 신규 생성 대비 2x 쓰기 프리미엄이 적용되는 1시간 TTL 옵션도 있습니다(docs.anthropic.com, 2026-04). OpenAI 프롬프트 캐싱은 1024토큰 이상의 프롬프트에 자동 적용되며, 캐시된 입력은 신규 생성 대비 약 90% 할인된 가격으로 책정됩니다(platform.openai.com, 2026-04); 모델별 정확한 캐시 요금은 실시간 요금표에 따라 다릅니다. L1(애플리케이션 수준) 시맨틱 캐싱은 임베딩 유사도 일치 시 LLM을 완전히 건너뜁니다. 공급업체의 "95% 정확도"는 일치 정확도를 의미하며, 캐시 적중률이 아닙니다 — 보고된 프로덕션 적중률은 개방형 채팅에서 10%에서 구조화된 FAQ에서 70%까지 분포합니다; 두 공급업체 모두 공식 기준을 공개하지 않았으므로 이 수치들은 보장이 아닌 커뮤니티 텔레메트리로 간주해야 합니다. 프로덕션 함정: 병렬화는 캐싱을 무효화합니다(첫 캐시 쓰기 전에 발행된 N개의 병렬 요청은 비용을 수배 증가시킬 수 있음), 접두사 내 동적 콘텐츠는 캐시 적중을 완전히 방해합니다. ProjectDiscovery는 캐시 가능한 접두사에서 동적 텍스트를 분리함으로써 7%에서 74%로 적중률 향상(2025-11)을 보고했습니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 간단한 2계층 캐시 시뮬레이터)  
**선수 지식:** 17단계 · 04(vLLM 서빙 내부), 17단계 · 06(SGLang RadixAttention)  
**소요 시간:** ~60분

## 학습 목표

- L2 프롬프트/접두사 캐싱(공급자에서의 KV 재사용)과 L1 의미 캐싱(유사한 프롬프트에 대한 LLM 우회)을 구분한다.
- Anthropic의 `cache_control` 명시적 표시와 두 가지 TTL 옵션(5분 vs 1시간) 및 가격 승수를 설명한다.
- 적중률, 프롬프트/응답 조합, 토큰 가격을 고려하여 예상 월간 절감액을 계산한다.
- 청구 비용을 5-10배 증가시키는 병렬화 안티패턴과 적중률을 붕괴시키는 동적 콘텐츠 안티패턴의 이름을 말한다.

## 문제

RAG 서비스에 프롬프트 캐싱을 추가했습니다. 하지만 비용은 그대로 유지됩니다. 캐시 적중률을 측정해 보니 7%에 불과합니다. 프롬프트는 정적으로 보이지만 실제로는 그렇지 않습니다 — 시스템 프롬프트에는 분 단위 현재 날짜, 요청 ID, 다양성을 위한 무작위 예제 재정렬이 포함됩니다. 모든 요청이 새로운 캐시 항목을 쓰고, 읽기는 전혀 발생하지 않습니다.

별개로, 에이전트는 사용자 질문당 10개의 병렬 도구 호출을 실행합니다. 첫 번째 캐시 쓰기가 완료되기 전에 10개의 호출이 모두 공급자에 도착합니다. 10번 쓰고, 0번 읽습니다. 비용은 "캐싱 적용 시" 예상 비용의 5~10배가 됩니다.

캐싱은 플래그가 아니라 프로토콜입니다. 두 계층, 두 가지 다른 실패 모드입니다.

## 개념

### L2 — 프로바이더 프롬프트/접두사 캐싱

프로바이더는 캐시 가능한 접두사에 대한 어텐션 KV를 저장하고, 접두사와 일치하는 다음 요청에서 이를 재사용합니다. 쓰기 비용은 한 번만 지불하고, 읽기는 거의 무료입니다.

**Anthropic (Claude 3.5 / 3.7 / 4 시리즈)**: 요청에 명시적 `cache_control` 마커가 있습니다. 어떤 블록을 캐시할 수 있는지 태그합니다. TTL: 5분(쓰기 비용은 기본 1.25배) 또는 1시간(쓰기 비용은 기본 2배). 캐시 읽기: Claude 3.5 Sonnet에서 $0.30/M vs $3.00/M 신선 — 10배 저렴 (docs.anthropic.com, 2026-04 기준). 모델별 요금 차이(Opus/Haiku는 별도 공개); 실시간 요금 페이지를 항상 확인하세요.

**OpenAI**: 1024토큰 이상의 프롬프트에 대해 자동 캐싱(platform.openai.com, 2026-04). 명시적 플래그 없음. 캐시된 입력은 현재 gpt-4o/gpt-5 요금제에서 신선 입력보다 약 10배 저렴합니다. 문서나 릴리스 노트에는 공식 히트율 기준이 없음; 커뮤니티 보고서는 신중한 프롬프트 설계 시 30–60% 집중. `usage.cached_tokens`를 모니터링하여 직접 측정하세요.

**Google (Gemini)**: 명시적 API를 통한 컨텍스트 캐싱; 1M-토큰 컨텍스트는 캐싱이 더 큰 이점을 제공합니다.

**자체 호스팅(vLLM, SGLang)**: Phase 17 · 06에서 RadixAttention — 자체 컴퓨팅에서 동일한 패턴.

### L1 — 애플리케이션 수준 시맨틱 캐싱

LLM을 호출하기 전에 프롬프트를 해시, 임베딩하고 유사한 캐시된 요청을 찾습니다(코사인 유사도 임계값 이상, 일반적으로 0.95+). 히트 시 캐시된 응답을 반환합니다. 미스 시 LLM을 호출하고 결과를 캐시합니다.

오픈소스: Redis Vector Similarity, GPTCache, Qdrant. 상용: Portkey Cache, Helicone Cache.

벤더의 정확도 주장은 반환된 캐시된 응답이 시맨틱적으로 적절했던 빈도를 의미 — 히트율이 아닙니다. 프로덕션 히트율:

- 개방형 채팅: 10-15%.
- 구조화된 FAQ / 지원: 40-70%.
- 코드 질문: 20-30% (작은 변형이 히트 방해).
- 프롬프트를 반복하는 음성 에이전트: 50-80% (음성 정규화된 고정 세트).

### 병렬화 안티패턴

에이전트가 10개의 도구 호출을 병렬로 수행합니다. 10개 모두 동일한 4K-토큰 시스템 프롬프트를 가집니다. Anthropic 캐시 쓰기는 요청당 발생; 첫 번째 캐시-쓰기는 프로바이더가 프롬프트를 본 후 약 300ms에 완료됩니다. 요청 2-10은 동일한 밀리초 창에 도착하고 각각 캐시 미스를 확인합니다. 10개의 쓰기 프리미엄을 지불하고 읽기 할인은 0입니다.

해결: 순차적 배치 — 요청 1만 수행한 후 1의 캐시가 채워지면 2-10을 발사합니다. 첫 번째 도구 호출에 300ms 추가; 청구서를 5-10배 절약.

### 동적 콘텐츠 안티패턴

시스템 프롬프트가 다음과 같습니다:

```
당신은 도움이 되는 어시스턴트입니다. 현재 시간은 14:32:17입니다.
사용자 ID: abc123. 오늘은 화요일입니다...
```

모든 요청이 고유합니다. 모든 요청이 쓰기. 히트율 0%.

해결: 진정으로 정적인 모든 것을 캐시 가능한 접두사로 이동; 캐시 경계 이후에 동적 콘텐츠를 추가:

```
[캐시 가능]
당신은 도움이 되는 어시스턴트입니다. [규칙, 예시, 지침]
[/캐시 가능]
[동적, 캐시되지 않음]
현재 시간: 14:32:17. 사용자: abc123.
```

ProjectDiscovery는 이 방법으로 7%에서 74% 캐시 히트율로 이동했으며 해부학을 공개했습니다.

### 스택 배치 + 캐싱으로 야간 작업 처리

배치 API(Phase 17 · 15)는 24시간 처리 시 50% 할인. 캐시된 입력을 추가하면 그 위에 ~10배 추가. 야간 분류, 라벨링, 보고서 생성 작업은 동기-비캐시 비용의 ~10%로 떨어질 수 있습니다.

### 기억해야 할 숫자

가격 정보는 2026-04에 링크된 벤더 문서에서 캡처했으며 몇 달마다 변동 — 의존하기 전에 재확인하세요.

- Anthropic 캐시 읽기: Claude 3.5 Sonnet에서 $0.30/M, 신선 입력보다 약 10배 저렴 (docs.anthropic.com).
- Anthropic 캐시 쓰기 프리미엄: 1.25배(5분 TTL) 또는 2배(1시간 TTL).
- OpenAI 자동 캐시: 1024토큰 이상 프롬프트에 적용; 캐시된 입력은 현재 요금제에서 신선 입력의 약 10% 가격 (platform.openai.com).
- 시맨틱 캐시 히트율(커뮤니티 보고): 개방형 채팅 ~10%; 구조화된 FAQ 최대 ~70%. 벤더 문서 기준 아님.
- ProjectDiscovery: 접두사에서 동적 콘텐츠 이동으로 7% → 74% 히트율 (프로젝트 블로그, 2025-11).
- 병렬화 안티패턴: N개의 병렬 요청이 첫 캐시 쓰기를 놓칠 때 청구서 5–10배 인플레이션에 대한 일반 보고서.

## 연습 문제

1. `code/main.py`를 실행하세요. 병렬화 플래그를 토글하세요. 요금이 얼마나 변하나요?
2. 시스템 프롬프트에 날짜가 있습니다. 이를 제거하세요. 제거 전후 히트율 계산식을 보여주세요.
3. 요청 도착률을 고려하여 1시간 TTL(2배 쓰기) 대 5분 TTL(1.25배 쓰기)의 손익분기점을 계산하세요.
4. 0.95 임계값에서 시맨틱 캐시는 20% 히트율을 기록합니다. 0.85에서는 50% 히트율이지만 잘못된 캐시 응답이 관찰됩니다. 적절한 임계값을 선택하고 근거를 제시하세요.
5. 사용자 질문당 10개의 병렬 하위 쿼리를 배치 처리합니다. 종단 간 지연 시간을 추가하지 않으면서 캐시 친화적으로 재작성하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| L2 프롬프트 캐시 | "prefix cache" | 공급자가 반복되는 접두사에 대한 KV를 저장 |
| `cache_control` | "Anthropic 캐시 마커" | 캐시 가능한 블록을 표시하는 명시적 속성 |
| 캐시 쓰기 프리미엄 | "write tax" | 첫 번째 캐시 미스 시 추가 비용 (1.25배 또는 2배) |
| L1 시맨틱 캐시 | "임베딩 캐시" | LLM 호출 전 애플리케이션 레벨에서 해시 및 임베딩 수행 |
| GPTCache | "LLM 캐싱 라이브러리" | 인기 있는 오픈소스 OSS L1 캐시 라이브러리 |
| 캐시 히트율 | "hits / total" | 캐시에서 제공되는 요청의 비율 |
| 병렬화 안티패턴 | "the N-write trap" | N개의 병렬 요청이 캐시를 N번 미스 |
| 동적 콘텐츠 함정 | "the time-in-prompt trap" | 접두사의 동적 바이트가 히트율을 저하 |
| RadixAttention | "intra-replica cache" | SGLang의 접두사 캐시 구현 |
