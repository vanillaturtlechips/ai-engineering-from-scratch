# 멀티 리전 LLM 서빙과 KV 캐시 지역성

> 라운드 로빈 부하 분산은 캐시된 LLM 추론에 적극적으로 해롭습니다. 접두사를 보유한 노드에 도착하지 않는 요청은 전체 프리필 비용을 지불합니다 — 긴 프롬프트에서 P50 기준 약 800ms가 소요되며 캐시 적중 시 ~80ms에 비해 훨씬 느립니다. 2026년 프로덕션 패턴은 KV-캐시 이벤트를 소비하고 접두사 해시 일치에 따라 라우팅하는 캐시 인식 라우터(Rust의 vLLM 라우터, llm-d 라우터)입니다. 최근 연구(GORGO)는 크로스 리전 네트워크 지연을 라우팅 목표의 명시적 항목으로 만듭니다. 상용 "크로스 리전 추론" 제품(Bedrock 크로스 리전 추론, GKE 멀티 클러스터 게이트웨이)은 추론을 불투명하게 처리합니다 — 가용성만 처리하고 TTFT(첫 토큰 대기 시간)는 고려하지 않습니다. JPMorgan과 Mayo Clinic은 2024년 11월 us-east-1 장애 조치를 ~22분 만에 수행했습니다. DR(재해 복구) 현실: LLM DR 장애의 32%는 팀이 가중치만 백업하고 토크나이저 파일이나 양자화 설정을 잊은 경우입니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 장난감 접두사-캐시 인식 라우터 시뮬레이터)  
**선수 지식:** 17단계 · 04 (vLLM 서빙), 17단계 · 06 (SGLang RadixAttention)  
**소요 시간:** ~60분

## 학습 목표

- 라운드 로빈 부하 분산이 캐시된 추론을 방해하는 이유를 설명하고 TTFT(첫 토큰 대기 시간) 페널티를 정량화합니다.
- 캐시 인식 라우터 다이어그램: 입력(KV-캐시 이벤트), 알고리즘(접두사-해시 매칭), 타이 브레이커(GPU 활용률).
- LLM의 32% DR(재해 복구) 실패 원인(토큰화기 파일 누락 / 양자화 구성 누락)을 명시하고 3개 파일 DR 체크리스트를 제시합니다.
- 상용 크로스 리전 제공(Bedrock CRI, GKE Multi-Cluster Gateway)과 KV-인식 라우팅을 구분합니다.

## 문제

귀사의 서비스는 us-east-1, us-west-2, eu-west-1에서 실행됩니다. 라운드 로빈 방식의 ALB(Application Load Balancer)를 앞에 배치했지만, 프로덕션 환경의 프리픽스 캐시 적중률이 8%로 급감했습니다. TTFT(시간-첫-토큰) P50은 3배로 증가했습니다. vLLM 로그에는 모든 요청이 전체 프리필 비용을 지불하고 있음이 나타납니다.

라운드 로빈은 상태 비저장 서비스에 최적입니다. 반면 LLM 추론은 설계상 상태 저장 서비스입니다 — KV 캐시는 모델이 본 모든 내용을 인코딩합니다. 블라인드 라우팅은 잘못된 캐시로 라우팅하는 것과 같습니다.

별도로, 귀사 팀에는 DR(재해 복구) 계획이 있습니다. 모델 가중치를 S3 크로스 리전에 백업했지만, 리전 장애가 발생하자 장애 조치를 시도했을 때 복제본이 시작되지 않았습니다. tokenizer.json, 양자화 구성, RoPE 스케일링 구성이 별도의 버킷에 있었고 동기화하지 않았기 때문입니다.

멀티 리전 LLM 서빙은 캐시 문제, 라우팅 문제, DR-위생 문제입니다 — 로드 밸런서 문제가 아닙니다.

## 개념

### 캐시 인식 라우팅

프롬프트와 함께 요청이 도착합니다. 라우터는 접두사(예: 처음 512개 토큰)를 해싱하고, 각 레플리카에 "이 접두사를 캐시에 가지고 있나요?"라고 질의합니다. 레플리카는 블록을 할당하고 제거할 때 pub/sub 채널에 KV-캐시 이벤트를 게시합니다. 라우터는 일치하는 레플리카를 선택하고, 일치하는 것이 없으면 GPU 활용률 기반 타이 브레이커로 폴백합니다.

**vLLM 라우터** (Rust, 2026년 프로덕션 스택): `kv.cache.block_added` 이벤트를 구독하고, 접두사 해시 → 레플리카 인덱스 매핑을 유지하며, O(1) 조회로 라우팅합니다. 일치하는 것이 없으면 최소 큐 깊이로 폴백합니다.

**llm-d 라우터**: 동일한 패턴, Kubernetes 네이티브. ControlPlane API를 통해 이벤트를 게시합니다.

**SGLang RadixAttention** (Phase 17 · 06)은 레플리카 내부의 동등한 구현입니다. 레플리카 간 라우팅은 엄격하게 상위 계층에서 처리됩니다.

### 수치

2K 토큰 프롬프트, Llama 3.3 70B FP8, H100에서 TTFT P50:
- 캐시 히트(동일 레플리카, 접두사 상주): ~80ms.
- 캐시 미스(콜드 프리필): ~800ms.

10배 차이. 라우터가 레플리카 간 접두사 캐시의 60-80%를 히트하면 N-레플리카 용량에서 단일 레플리카 성능을 근사할 수 있습니다. 10% 히트 시 순진 확장(naive scaling)을 근사합니다.

### 크로스 리전에는 새로운 제약 조건이 있습니다 — 네트워크 지연

리전 간 RTT:
- us-east-1 ↔ us-west-2: ~65ms.
- us-east-1 ↔ eu-west-1: ~75ms.
- us-east-1 ↔ ap-southeast-1: ~220ms.

라우팅이 us-east-1의 요청을 ap-southeast-1의 핫 접두사로 보낸다면, 절약된 프리필 시간(800 → 80ms)은 440ms 왕복 지연에 가려집니다. GORGO(2026년 연구)는 이를 명시적으로 — 프리필 시간 + 네트워크 지연을 개별적으로가 아닌 공동으로 최소화합니다. 종종 프리필이 지배적인 대규모 멀티-MB 접두사를 제외하고 지역 라우팅을 유지하는 것이 해결책입니다.

### 상용 "크로스 리전 추론"은 여기서 도움이 되지 않습니다

AWS Bedrock 크로스 리전 추론은 용량 압박 시 요청을 다른 리전으로 자동 라우팅합니다. 이는 가용성을 최적화하며, TTFT가 아닌 추론을 불투명(opaque)하게 처리합니다. GKE Multi-Cluster Gateway도 동일 — 서비스 수준 장애 조치, KV 캐시 인식 없음.

이러한 서비스를 사용하더라도 애플리케이션 계층 캐시 인식 라우터가 여전히 필요합니다. 이들은 "us-east-1이 불타는" 경우를 처리합니다. 캐시 인식 라우팅은 TTFT 경우를 처리합니다.

### DR 위생 — 32% 누락된 파일 문제

널리 인용된 2026년 통계: LLM DR 장애의 32%는 팀이 가중치는 백업했지만 다음을 잊어서 발생합니다:
- `tokenizer.json` 또는 `tokenizer.model`
- 양자화 구성(`quantize_config.json`, AWQ 스케일, GPTQ 제로 포인트)
- 모델별 구성(RoPE 스케일링, 어텐션 마스크, 채팅 템플릿)
- 엔진 구성(`vllm_config.yaml`, 샘플링 기본값, LoRA 어댑터 매니페스트)

해결책은 3개 파일 최소 DR 매니페스트입니다:
1. HF 모델 리포지토리 하의 모든 파일(가중치 + 구성 + 토크나이저).
2. 엔진별 서빙 구성.
3. 배포 매니페스트(K8s YAML, Dockerfile, 종속성 잠금).

추가: 분기별 DR 훈련 실행. JPMorgan의 us-east-1 훈련은 2024년 11월 22분 복구 시간을 기록했는데, 이는 플레이북이 리허설되었기 때문입니다.

### 데이터 거주는 별개의 문제입니다

EU 고객의 PHI는 EU를 떠날 수 없습니다. 캐시 인식 라우터가 접두사 일치를 위해 파리 발 요청을 us-east-1로 보낸다면, TTFT 이득과 무관하게 GDPR을 위반한 것입니다. 캐시 최적화 전에 거주 경계로 라우터를 분할하세요.

### 기억해야 할 수치

- 캐시 히트 vs 미스 TTFT 격차: ~10배(2K 프롬프트에서 80ms vs 800ms).
- 리전 간 RTT US-EU: ~75ms.
- DR 장애: 32% 토크나이저/양자화 구성 누락.
- JPMorgan us-east-1 장애 조치 2024년 11월: 22분(30분 SLA).

## 사용 방법

`code/main.py`는 다중 리전 워크로드에서 세 가지 라우팅 전략(라운드 로빈(round-robin), 캐시 인식 리전별(cache-aware regional), 캐시 인식 글로벌(cache-aware global))을 시뮬레이션합니다. 캐시 적중률(cache hit rate), TTFT(최초 전송 시간) P50/P99, 리전 간 비용(cross-region bill)을 보고합니다.

## Ship It

이 레슨은 `outputs/skill-multi-region-router.md`를 생성합니다. 주어진 리전(region), 레지던시 제약 조건(residency constraints), SLA(서비스 수준 계약)를 기반으로 라우팅 계획을 설계합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 75ms RTT가 주어졌을 때, 지역 간 라우팅이 로컬 전용 라우팅을 능가하는 프롬프트 길이는 얼마인가요?

2. 캐시 적중률이 70%에서 12%로 떨어졌습니다. 가능한 세 가지 원인과 각 원인을 확인할 수 있는 관측 지표를 진단하세요.

3. vLLM에서 5개의 LoRA 어댑터로 서비스되는 70B AWQ-양자화 모델을 위한 DR(Disaster Recovery) 매니페스트를 설계하세요. 모든 파일과 설정을 나열하세요.

4. 엄격한 TTFT(First Token Time) SLO를 가진 핀테크 기업에 Bedrock 지역 간 추론이 "충분한지" 여부를 논증하세요. 구체적인 동작 특성을 인용하세요.

5. 파리(Paris)에서 발생한 요청이 us-east-1의 접두사와 일치합니다. 이 요청을 라우팅하시겠습니까? 정책을 작성하세요.
