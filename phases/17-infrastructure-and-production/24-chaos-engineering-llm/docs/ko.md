# LLM 프로덕션 환경을 위한 카오스 엔지니어링

> 2026년 기준 LLM용 카오스 엔지니어링은 독립적인 학문 분야입니다. 프로덕션 환경에서 실험 전 필수 조건: 정의된 SLI/SLO, 추적+메트릭+로그 관측성, 자동화된 롤백, 런북, 온콜. 아키텍처는 4가지 계층으로 구성: 제어(실험 스케줄러), 대상(서비스, 인프라, 데이터 저장소), 안전(가드+중단+트래픽 필터), 관측성(메트릭+트레이스+로그), 피드백(SLO 조정 반영). 필수 가드레일: 일일 오류 예산 소모율이 예상보다 2배 초과 시 실험 일시정지 알림; 억제 창 + 트레이스 ID 상관관계 분석을 통한 중복 알림 제거. 주기: 주간 소규모 카나리 실험 + SLO 검토; 월간 게임 데이 + 사후 분석; 분기별 크로스팀 복원력 감사 + 의존성 매핑. LLM 특화 실험: 메모리 과부하, 네트워크 장애, 공급자 중단, 잘못된 프롬프트, KV 캐시 제거 폭주. 도구: Harness 카오스 엔지니어링(LLM 기반 권장사항, 영향 범위 축소, MCP 도구 통합); LitmusChaos(CNCF); Chaos Mesh(CNCF 쿠버네티스 네이티브).

**유형:** 학습  
**언어:** Python(표준 라이브러리, 토이 카오스 실험 러너)  
**선수 지식:** 17단계 · 23(SRE for AI), 17단계 · 13(관측성)  
**소요 시간:** ~60분

## 학습 목표

- 5가지 카오스 엔지니어링 사전 요건(SLI/SLO, 관측 가능성, 롤백, 런북, 온콜)을 명명하고, 어떤 것을 생략할 경우 해당 실천이 무너지는 이유를 설명하시오.
- 4가지 평면(제어, 대상, 안전, 관측 가능성)과 SLO로의 피드백 루프를 다이어그램으로 표현하시오.
- LLM 특화 실험 5가지(메모리 과부하, 네트워크 장애, 공급자 중단, 잘못된 프롬프트, KV 제거 폭주)를 열거하시오.
- 스택에 따라 도구(Harness, LitmusChaos, Chaos Mesh)를 선택하시오.

## 문제 정의

기존 스택에서의 카오스 테스트는 잘 정립되어 있습니다. LLM 스택은 새로운 장애 모드를 추가합니다. 4K-토큰 프롬프트에 독(poison) 문자가 포함되면 토크나이저가 12초 동안 정지됩니다. 업스트림 공급자가 429 오류를 반환하면 게이트웨이가 재시도하고, 서비스는 재시도로 증폭된 동시성으로 인해 OOM(메모리 부족) 상태가 됩니다. 버스트 부하에서 KV 캐시 제거 폭풍이 발생하면 재-프리필(re-prefill) 캐스케이드가 발생하여 컴퓨팅 리소스가 포화 상태가 됩니다.

이러한 문제들은 단위 테스트에서는 전혀 나타나지 않습니다. 카오스 엔지니어링은 사용자가 발견하기 전에 이러한 문제들을 사전에 발견하는 방법입니다.

## 개념

### 사전 요구 사항

다음 없이는 프로덕션 환경에서 카오스를 실행하지 마세요:

1. **SLI/SLO** — 정의된 서비스 수준 지표 및 목표.
2. **관측 가능성** — 대시보드에 연결된 트레이스, 메트릭, 로그.
3. **자동 롤백** — 17단계 · 20정책-플래그 롤백.
4. **런북** — 구조화된, 17단계 · 23.
5. **온콜** — 대응할 담당자.

누락된 항목이 있으면 카오스가 실제 인시던트로 변합니다.

### 4가지 평면 + 피드백

**제어 평면** — 실험 스케줄러 (Litmus 워크플로우, Chaos Mesh 스케줄, Harness UI).

**대상 평면** — 서비스, 파드, 노드, 로드 밸런서, 데이터 저장소.

**안전 평면** — 킬 스위치, 억제 창, 영향 범위 제한, 오류 예산 게이트.

**관측 가능성 평면** — 일반 메트릭 + 카오스 유발 실패와 자연 실패 구분을 위한 트레이스 ID 상관 관계.

**피드백 루프** — 실험 결과가 SLO 조정, 런북 업데이트, 코드 수정으로 피드백됩니다.

### 가드레일은 필수

- **소모율 경고**: 일일 오류 예산 소모율이 예상보다 2배 초과하면 실험 일시 중지.
- **억제 창**: 실험 중 영향 범위 내 비실험 경고 무음 처리.
- **트레이스 ID 상관 관계**: 모든 실험 유발 오류에는 태그가 포함되어 온콜 담당자가 중복 제거 가능.

### 5가지 LLM 특화 실험

1. **메모리 과부하** — 높은 동시성으로 긴 컨텍스트 요청을 보내 KV 캐시 선점 폭풍을 유발. 관찰: 서비스가 우아하게 요청을 거부하는지, 아니면 충돌하는지?

2. **네트워크 장애** — 추론 게이트웨이와 공급자 간 연결 차단. 관찰: SLA 내 폴백 작동 여부? (17단계 · 19)

3. **공급자 장애 시뮬레이션** — OpenAI에서 100% 429 오류 발생. 관찰: 라우팅이 Anthropic으로 장애 조치되는지? (17단계 · 16, 19)

4. **잘못된 프롬프트** — 토크나이저 정지 페이로드 주입 (예: 깊이 중첩된 유니코드, 거대한 UTF-8 코드포인트). 관찰: 단일 요청이 작업자를 잠그는가?

5. **KV 제거 폭풍** — vLLM 블록 예산을 포화시켜 제거 강제. 관찰: LMCache가 복구되는지, 아니면 서비스 성능이 저하되는지?

### 주기

- **주간** — 스테이징에서 소규모 카나리 실험, 프로덕션 5% 정도.
- **월간** — 특정 시나리오에 대한 게임 데이 예약; 팀 간 참석; 사후 분석.
- **분기별** — 팀 간 복원력 감사; 의존성 맵 업데이트.

### 도구

- **Harness 카오스 엔지니어링** — 상용; AI 기반 실험 추천; 영향 범위 축소; MCP 도구 통합.
- **LitmusChaos** — CNCF 졸업; Kubernetes 워크플로우 기반.
- **Chaos Mesh** — CNCF 샌드박스; Kubernetes 네이티브 CRD 스타일.
- **Gremlin** — 상용; 광범위한 지원.
- **AWS FIS** / **Azure Chaos Studio** — 관리형 클라우드 제공.

### 작게 시작

첫 실험: 정상 트래픽 하에서 디코딩 레플리카 1개 파드 종료. 재라우팅 및 복구 관찰. 이 작업이 안전하고 작동하면 네트워크 카오스로 확장.

첫 LLM 특화 실험: 5분 동안 공급자 429 오류 1회 주입. 폴백 관찰. 대부분의 팀은 폴백이 완전히 테스트되지 않았음을 발견합니다.

### 기억해야 할 숫자

- 4가지 평면: 제어, 대상, 안전, 관측 가능성.
- 소모율 일시 중지: 예상 일일 예산의 2배 소모 시.
- 주기: 주간 카나리, 월간 게임 데이, 분기별 감사.
- 5가지 LLM 실험: 메모리, 네트워크, 공급자, 잘못된 프롬프트, KV 폭풍.

4. 카오스 실험을 프로덕션에서 실행해야 하는지, 스테이징에서만 실행해야 하는지 주장하세요. 프로덕션이 적절한 경우는 언제인가요?  
5. 일반적인 네트워크 카오스로 재현할 수 없는 LLM 특화 실패 모드 3가지를 제시하세요.  

> **번역 참고사항**  
> - "burn-rate gate"는 도메인 용어로 "버너이트 게이트"로 음차 처리  
> - "RAG"는 "검색 증강 생성(Retrieval-Augmented Generation)"의 약어지만, 기술 문서에서는 일반적으로 영어 약어 그대로 사용되므로 번역하지 않음  
> - "vLLM"은 특정 프레임워크 이름으로 번역하지 않음  
> - "LLM"은 "대형 언어 모델(Large Language Model)"의 약어지만, 기술 문서에서는 일반적으로 영어 약어 그대로 사용되므로 번역하지 않음
