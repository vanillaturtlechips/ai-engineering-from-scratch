# 서버리스 LLM의 콜드 스타트 완화

> 20GB 모델 이미지는 콜드 상태에서 서빙까지 5-10분(7B)에서 20분 이상(70B)이 소요됩니다. 진정한 서버리스 환경에서는 이는 워밍업이 아니라 장애(outage)입니다. 완화 기법은 5개 계층에서 작동합니다: 사전 시드된 노드 이미지(AWS의 Bottlerocket, 듀얼 볼륨 아키텍처), 모델 스트리밍(NVIDIA Run:ai Model Streamer, vLLM의 네이티브 기능), GPU 메모리 스냅샷(Modal 체크포인트, 최대 10배 빠른 재시작), 웜 풀(`min_workers=1`), 계층적 로딩(ServerlessLLM의 NVMe→DRAM→HBM 파이프라인, 10-200배 지연 시간 감소), 그리고 KV 캐시(GB) 대신 입력 토큰(KB)을 이동하는 라이브 마이그레이션. Modal은 2-4초 콜드 스타트를 최소 기준으로 발표했으며, Baseten은 기본 5-10초, 사전 워밍 시 서브초 성능을 제공합니다. 이 강의에서는 5개 계층을 측정, 예산 책정, 조합하는 방법을 배웁니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, 장난감 콜드 스타트 경로 시뮬레이터)
**선수 지식:** 17단계 · 02 (추론 플랫폼 경제학), 17단계 · 03 (GPU 자동 확장)
**소요 시간:** ~60분

## 학습 목표

- 콜드 스타트 완화의 5개 계층을 열거하고 각 계층에서 하나의 도구 또는 패턴을 명명할 것.
- 70B 모델에 대해 (노드 프로비저닝) + (가중치 다운로드) + (HBM에 가중치 로드) + (엔진 초기화)의 합으로 총 콜드 스타트 시간을 계산할 것.
- 라이브 마이그레이션이 KV 캐시(GB)가 아닌 입력 토큰(KB)을 전송하는 이유와 페널티(재계산)가 무엇인지 설명할 것.
- 웜 풀 트레이드오프(유휴 GPU 비용 지불 또는 콜드 스타트 테일 수용)와 `min_workers > 0`이 필수화되는 SLA 임계값을 명명할 것.

## 문제

서버리스 LLM 엔드포인트가 밤새 0으로 확장됩니다. 오전 8시에 트래픽이 급증합니다. 첫 번째 요청은 다음 작업이 진행되는 동안 대기합니다:

1. Karpenter가 GPU 노드를 프로비저닝: 45-60초.
2. 컨테이너가 가중치(weights)가 포함된 30GB 이미지를 풀(pull)함: 120-300초.
3. 엔진이 HBM에 가중치를 로드: 모델 크기와 스토리지 속도에 따라 45-120초.
4. vLLM 또는 TRT-LLM이 CUDA 그래프, KV 캐시 풀, 토크나이저를 초기화: 10-30초.

총 소요 시간: 220-510초(대략 3-8분) 후에 첫 토큰이 반환됩니다. SLA는 2초입니다. 웜 풀(`min_workers=1`)을 배포하면 문제가 사라지는 것처럼 보이지만, 이제 24시간 내내 유휴 상태의 GPU 1대에 대한 비용을 지불해야 합니다. 서비스에 5개의 제품이 있고 각각 웜 복제본 1개를 유지한다면, 단일 사용자가 호출했는지 여부와 관계없이 5 × 24 × 30 = 3,600 GPU-시간/월이 소요됩니다.

콜드 스타트 완화(cold-start mitigation)는 서버리스 경제성을 유지하면서 항상 켜져 있는(always-on) 수준의 지연 시간을 근사화하는 방법입니다.

## 개념

### 레이어 1 — 사전 시드된 노드 이미지 (Bottlerocket)

AWS에서 Bottlerocket의 듀얼 볼륨 아키텍처는 OS와 데이터를 분리합니다. 컨테이너 이미지가 미리 풀된 데이터 볼륨을 스냅샷하고, `EC2NodeClass`에서 스냅샷 ID를 참조합니다. 새 노드는 로컬 NVMe에 이미 가중치가 로드된 상태로 부팅되며, 단계 2와 단계 3의 일부가 사라집니다. Karpenter와 네이티브 호환됩니다. 일반적인 절감 효과: 대형 모델의 경우 콜드 스타트당 2-4분 절약.

GCP에서 동등한 방법: 사전 빌드된 컨테이너 레이어가 포함된 커스텀 VM 이미지. Azure에서: 동일한 패턴의 관리 디스크 스냅샷.

### 레이어 2 — 모델 스트리밍 (Run:ai Model Streamer)

첫 번째 요청에 응답하기 전에 전체 파일을 로드하는 대신, 가중치를 GPU 메모리에 계층별로 스트리밍하고 첫 번째 트랜스포머 블록이 로드되는 즉시 처리를 시작합니다. NVIDIA Run:ai Model Streamer는 vLLM 2026에 네이티브로 탑재됩니다. S3, GCS, 로컬 NVMe와 호환됩니다. I/O와 계산 설정을 오버랩하여 대형 모델의 가중치 로드 시간을 약 절반으로 단축합니다.

### 레이어 3 — GPU 메모리 스냅샷 (Modal)

Modal은 첫 번째 로드 후 GPU 상태(가중치, CUDA 그래프, KV 캐시 영역)의 체크포인트를 생성합니다. 이후 재시작 시 HBM에 직접 역직렬화되어 재초기화보다 10배 빠릅니다. 이는 "2초 내 웜 GPU 부팅"에 가장 가까운 기술입니다. 단점: 스냅샷은 GPU 토폴로지별로 생성되므로, Karpenter가 다른 SKU로 마이그레이션하면 재체크포인트해야 합니다.

### 레이어 4 — 웜 풀 (min_workers=1)

가장 간단한 완화 방법: 항상 1개의 레플리카를 준비 상태로 유지합니다. 비용은 GPU 1대의 시간당 요금이 24시간 발생합니다. 소형 모델에서는 산술적 비용이 큽니다(30초 콜드 스타트를 피하기 위해 시간당 $0.85-$1.50 지불). 대형 모델에서는 비용이 덜 부담됩니다(5분 콜드 스타트를 피하기 위해 시간당 $4 지불). 웜 풀이 필수화되는 SLA 임계값: 일반적으로 70B+ 모델에서 TTFT P99 < 60초.

### 레이어 5 — 계층적 로딩 (ServerlessLLM)

ServerlessLLM은 스토리지를 계층으로 처리합니다: NVMe(빠르지만 용량 큼), DRAM(중간 속도, 계층화됨), HBM(작지만 즉시 사용 가능). 가중치는 DRAM에 미리 로드되고, 필요 시 HBM으로 로드됩니다. 논문에서는 순진한 디스크-대-HBM 방식 대비 콜드 로드 시 10-200배 지연 시간 감소를 보고합니다. 프로덕션 도입은 초기 단계이지만 vLLM과의 통합이 존재합니다.

### 레이어 6 — 라이브 마이그레이션 (보너스 패턴)

노드가 사용 불가능해질 때(스포트 인스턴스 종료, 노드 드레인), 전통적인 패턴은 다른 레플리카를 콜드 스타트하고 요청 큐를 드레인하는 것입니다. 라이브 마이그레이션은 입력 토큰(킬로바이트)을 모델이 로드된 대상으로 이동시키고, KV 캐시를 대상에서 재계산합니다. 네트워크 상에서 GB 단위의 KV 캐시를 전송하는 것보다 재계산이 저렴합니다. 분리된 배포에 적용 가능합니다.

### 웜 풀 수학

P99 TTFT SLA가 2초인 서비스의 경우, "웜 풀 사용 여부"가 아닌 "웜 레플리카 수 및 적용 경로"가 문제입니다.

- 고가치 대화형 경로(라이브 채팅, 음성 에이전트): `min_workers=1-2`.
- 백그라운드 배치 경로(야간 분류): 스케일-투-제로 허용, 5-10분 콜드 스타트 허용.
- 프리미엄 티어: 전용 용량을 가진 테넌트별 `min_workers`.

### 최적화 전 측정

새 노드에서 70B 모델의 콜드 스타트 구성(설명용):

| 단계 | 시간 | 완화 방법 |
|-------|------|-----------|
| 노드 프로비저닝 | 50초 | Bottlerocket + 사전 시드된 이미지, 웜 풀 |
| 이미지 풀 | 180초 | 사전 시드된 데이터 볼륨(제거) |
| HBM으로 가중치 로드 | 75초 | 모델 스트리머(절반 감소); GPU 스냅샷(제거) |
| 엔진 초기화 | 20초 | 지속적 CUDA 그래프 캐시 |
| 첫 번째 순방향 | 3초 | 최소 고유 지연 |
| **총 콜드** | **328초** | |
| **완화 적용 시 총계** | **~15초** | 22배 감소 |

### 기억해야 할 숫자

- Modal 콜드 스타트: 2-4초(GPU 스냅샷 사용 시).
- Baseten 기본 콜드 스타트: 5-10초; 사전 예열 시 초 단위.
- 순수 70B 콜드 스타트: 3-8분.
- Run:ai Model Streamer: 가중치 로드 속도 약 2배 향상.
- ServerlessLLM 계층적 로딩: 10-200배 지연 시간 감소(논문 수치).

## 연습 문제

1. `code/main.py`를 실행하세요. SLO에서 추가 요청 드롭을 통한 콜드 스타트 세금 지불보다 웜 복제본이 더 저렴한 요청률(break-even request rate)을 계산하세요.
2. P99 TTFT SLA가 3초인 13B 모델을 배포합니다. 이를 달성하는 최소 완화 스택(가장 적은 계층)을 선택하세요.
3. Bottlerocket 사전 시딩은 이미지 풀링을 제거하지만 가중치는 여전히 스냅샷에서 HBM으로 로드됩니다. 스냅샷 기반 NVMe 읽기 속도가 7GB/s일 때 70B 모델의 벽시계 시간(wall-clock time)을 계산하세요.
4. 서버리스 제공업체가 GPU 스냅샷(Modal)을 제공하지만, 팀에서 "스냅샷이 PII를 유출한다"는 이유로 거부합니다. 양측의 주장을 제시하세요. 현실적인 위험은 무엇이며, 완화 방안(일회용 스냅샷, 암호화, 네임스페이스 격리)은 무엇인가요?
5. 계층화된 웜 풀 정책을 설계하세요. 유료 사용자, 체험판 사용자, 배치 워크로드에 대해 각각 몇 개의 웜 복제본을 유지할지 결정하세요. 계산 과정을 보여주세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| Cold start | "the big pause" | 새 복제본에서 요청부터 첫 토큰까지의 시간 |
| Warm pool | "always-on minimum" | 최소 하나의 복제본을 준비 상태로 유지하기 위한 `min_workers >= 1` |
| Pre-seeded image | "baked AMI" | 컨테이너 가중치가 미리 로드된 노드 이미지 |
| Bottlerocket | "AWS node OS" | 이중 볼륨 스냅샷을 지원하는 AWS 컨테이너 최적화 OS |
| Model streamer | "streaming load" | 가중치 I/O와 계산 설정의 오버랩 |
| GPU snapshot | "checkpoint to HBM" | 로드 후 GPU 상태 직렬화; 재시작 시 역직렬화 |
| Tiered loading | "NVMe + DRAM + HBM" | 저장 계층 구조; 필요 시 로드 |
| Live migration | "move tokens" | 입력(KB) 전송, 대상 시스템에서 KV 재계산 |
| `min_workers` | "warm replicas" | 서버리스 최소 유지 대기 수 |
| Scale-to-zero | "full serverless" | 유휴 시 비용 없음; 전체 콜드 스타트 비용 수용 |
