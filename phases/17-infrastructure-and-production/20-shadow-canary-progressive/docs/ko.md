# LLM을 위한 섀도우 트래픽, 카나리 롤아웃, 점진적 배포

> LLM 롤아웃은 소프트웨어 배포의 가장 어려운 부분을 결합합니다: 단위 테스트 없음, 확산된 실패 모드, 지연된 신호. 순서는 (1) 섀도우 모드 — 프로덕션 요청을 후보 모델에 중복 전송, 로깅, 비교하며 사용자 영향 없이 진행; 명백한 분포 문제는 잡아내지만 품질 보장은 아님; (2) 카나리 롤아웃 — 10% → 25% → 50% → 75% → 100%로 점진적 트래픽 전환, 각 단계마다 게이트 설치; 지연 시간 백분위수, 요청당 비용, 오류/거부율, 출력 길이 분포, 사용자 피드백률 추적; (3) 안정성 확인 후 대체 모델 A/B 테스트. 비결정성은 감소 불가 — GPU FP 비결합성 및 배치 크기 변동으로 인해 동일한 입력에 대해 실행 간 최대 15% 정확도 변동 발생. 비용은 상수가 아닌 변수 — 20% 더 나은 모델이 호출당 3배 더 비쌀 수 있음. 롤백 속도가 결정적: 롤백에 재배포가 필요하면 너무 느림. 정책은 config/flags에 존재; 모델은 고정된 다이제스트가 있는 레지스트리에 존재; 롤백 = 정책 전환 + 임계값 복원 + 초 단위 이전 모델 고정.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 장난감 카나리 진행 시뮬레이터)  
**선수 지식:** 17단계 · 13 (관측 가능성), 17단계 · 21 (A/B 테스트)  
**소요 시간:** ~60분

## 학습 목표

- 섀도우 모드(영향 없는 비교), 카나리(실시간 트래픽 점진적), A/B(안정성 확인 비교)를 구분한다.
- LLM 특화 카나리 지표 5가지(지연 시간, 요청당 비용, 오류/거부율, 출력 길이 분포, 사용자 피드백)를 열거한다.
- LLM 비결정성(최대 15%)이 롤아웃에서 "안정적"의 의미를 어떻게 바꾸는지 설명한다.
- 시간 단위(재배포)가 아닌 초 단위(정책 전환)로 롤백 경로를 설계한다.

## 문제

새로운 모델을 출시합니다. 오프라인 평가에서는 정확도가 3% 향상되었습니다. 프로덕션에 배포하자마자 24시간 이내에 비용이 40% 증가하고, 사용자 불만(thumb-down)이 8% 늘었으며, "이상한 답변"이라는 고객 티켓 3건이 접수됩니다. 결국 롤백합니다. 재배포에는 3시간이 소요됩니다. 주말이 완전히 망가집니다.

이 모든 문제는 피할 수 있었습니다. 섀도우 모드(shadow mode)라면 비용 40% 급증을 사용자가 인지하기 전에 감지했을 것입니다. 카나리 배포(canary release)라면 사용자 불만(thumb-down)이 증가하는 시점에서 10% 단계에서 중단했을 것입니다. 정책 기반 플래그 롤백(policy-flag rollback)이었다면 30초 만에 롤백이 완료되었을 것입니다. "오프라인 평가 결과가 좋다"와 "실제 사용자가 만족한다" 사이의 간극을 메우는 것이 바로 이러한 프로세스입니다.

## 개념

### 섀도우 모드

후보 모델은 프로덕션과 동일한 요청을 수신하지만, 출력은 로깅되며 사용자에게 반환되지 않습니다. 사용자 영향 없음. 로그:

- 출력 내용 (프로덕션 대비 차이).
- 토큰 수 (비용 차이).
- 지연 시간.
- 거부 및 오류.

감지 항목: 비용 급증, 길이 회귀, 명백한 거부 변경, 하드 오류. 감지하지 못하는 항목: 사용자가 인지할 품질 차이. 섀도우 모드는 스모크 테스트이며, 품질 테스트가 아닙니다.

### 카나리 롤아웃

게이트가 있는 점진적 트래픽 전환. 일반적인 진행 단계: 1% → 10% → 25% → 50% → 75% → 100%. 각 단계에서 5개 메트릭에 게이트 설정:

1. **지연 시간 백분위수** — P50, P95, P99. 위반 조건: 카나리의 P99 > 기준선 1.5배.
2. **요청당 비용** — 혼합 $. 위반 조건: 기준선 대비 20% 초과.
3. **오류/거부율** — 5xx 및 명시적 거부. 위반 조건: 기준선 2배.
4. **출력 길이 분포** — 평균 + P99. 위반 조건: 분포 변화.
5. **사용자 피드백 비율** — 싫어요 / 티켓 제출. 위반 조건: 기준선 1.5배.

### 비결정성은 새로운 변동성

동일한 입력에 대해 비동일한 출력 생성. 원인:

- GPU FP 비결합성 (부동소수점 연산 순서가 배치에 따라 다름).
- 배치 크기 변동성 (128 배치 vs 16 배치 내 동일 프롬프트).
- 샘플링 (온도 > 0).

측정: 동일 평가 세트에서 실행 간 최대 15% 정확도 변동. "안정적" 롤아웃은 메트릭이 예상 변동성 내에 있음을 의미하며, 기준선과 동일함을 의미하지 않습니다. 노이즈 플로어 위에 게이트를 설정하세요.

### 비용은 변수

20% 더 나은 모델이 호출당 3배 더 비쌀 수 있습니다. 요청당 비용은 5개 게이트 중 하나입니다. 단위 경제를 파괴하는 "더 나은" 모델 출시는 롤백 사유입니다.

### 롤백은 무기

- 정책 플래그 (기능 플래그 시스템): 설정에서 비율 전환; 수 초 소요.
- 모델 고정 (레지스트리 다이제스트): 고정된 모델은 자동 업그레이드되지 않음.
- 롤백 = 플래그 되돌리기 + 이전 다이제스트로 고정 설정. 수 초, 시간 단위 아님.

스택에서 롤백에 재배포가 필요하다면, 롤아웃 전에 수정하세요.

### 툴링

**Argo Rollouts** / **Flagger** — 쿠버네티스 점진적 배포 컨트롤러. Istio/Linkerd 가중치 라우팅과 통합.

**Istio 가중치 라우팅** — 서비스 메시 수준 트래픽 분할.

**KServe / Seldon Core** — 카나리 기능이 내장된 모델 서빙.

**기능 플래그** — LaunchDarkly, Flagsmith, Unleash. 정책 수준 전환, 재배포 불필요.

### 메트릭 주기

카나리 게이트는 트래픽 양에 따라 5-15분마다 확인. 1% 트래픽 + 분당 10요청 = 창당 50-150 데이터 포인트 — 지연 시간에는 충분하지만 사용자 피드백은 노이즈 많음. 10%는 약 10배 더 많은 데이터. 각 단계에서 충분한 샘플 수집을 위해 진행 일시정지 필요.

### A/B 단계는 선택적

새 모델이 확연히 다른 경우 (다른 동작, 다른 비용 곡선, 다른 톤), 카나리 통과 후 50%에서 A/B 테스트. 단순 개선판이라면 카나리 게이트 통과 시 100%로 바로 전환.

### 기억해야 할 숫자

- 카나리 진행: 1% → 10% → 25% → 50% → 75% → 100%.
- 비결정성 상한: 동일 입력에서 실행 간 최대 15% 변동.
- 5개 카나리 메트릭: 지연 시간, 비용, 오류/거부, 출력 길이, 사용자 피드백.
- 비용 게이트: 기준선 대비 20% 초과 시 위반.
- 롤백: 수 초, 시간 단위 아님.

## 사용 방법

`code/main.py`는 주입된 회귀(regression)를 포함한 카나리 롤아웃(canary rollout)을 시뮬레이션합니다. 롤아웃이 중단되는 단계와 어떤 게이트(gate)가 트리거(trigger)되었는지 보고합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 25% 비용 회귀를 주입할 때, 카나리아가 중단되는 단계는 어디인가요?
2. 새 모델은 오프라인에서 3% 정확도 향상을 보이지만 비용/요청당 비용이 +18%입니다. 출시할 수 있을까요? 정책에 따라 다릅니다 — 두 가지 경로를 모두 작성하세요.
3. 60초 이내에 완료되는 롤백 방식을 설계하세요. 필요한 인프라를 나열하세요.
4. 평가에서 ±7% 비결정성이 나타납니다. 카나리아 게이트를 설정하여 오탐을 방지하려면 어떤 승수를 사용해야 하나요?
5. 섀도우 모드에서 카나리아보다 먼저 40% 비용 급증을 감지했습니다. 섀도우에서 발동되는 경고 규칙을 작성하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| Shadow mode | "duplicate to new" | 로깅을 위한 영향 없는 후보 전송 |
| Canary | "progressive traffic" | 게이트가 있는 점진적 사용자 노출 롤아웃 |
| Gates | "rollout checks" | 진행을 차단하는 메트릭 임계값 |
| Non-determinism | "LLM variance" | 실행 간 줄일 수 없는 차이 |
| Policy flag | "flag flip rollback" | 구성 수준 롤백, 시간 단위가 아닌 초 단위 |
| Model pin | "registry digest" | 모델 버전에 대한 불변 참조 |
| Argo Rollouts | "K8s progressive" | 쿠버네티스 네이티브 카나리/롤백 컨트롤러 |
| KServe | "inference K8s" | 카나리 기본 요소를 갖춘 모델 서빙 |
| Istio weighted | "mesh split" | 서비스 메시 트래픽 분배기 |
