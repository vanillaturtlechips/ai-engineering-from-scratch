# SGLang과 RadixAttention을 활용한 접두사 중심 워크로드 최적화

> SGLang은 KV 캐시를 라디컬 트리에 저장된 재사용 가능한 1급 리소스로 취급합니다. vLLM이 요청을 FCFS(선입선출) 방식으로 스케줄링하는 반면, SGLang의 캐시 인식 스케줄러는 더 긴 공유 접두사를 가진 요청을 우선시합니다. 이는 핫 브랜치가 HBM에 상주하도록 깊이 우선 라디컬 탐색과 같은 효과를 냅니다. ShareGPT 스타일의 1K 프롬프트에서 Llama 3.1 8B 기준 SGLang은 초당 ~16,200 토큰을 처리하는 반면 vLLM은 ~12,500 토큰으로, 약 29% 성능 우위를 보입니다. 접두사 중심 RAG 워크로드에서는 6.4배 성능 차이가 발생합니다. 음성 복제 형태의 워크로드에서는 캐시 적중률이 86%를 넘었습니다. 2026년 xAI, LinkedIn, Cursor, Oracle, GCP, Azure, AWS를 통해 400,000개 이상의 GPU에 배포되었습니다. 단, 접두사 순서가 불규칙할 경우 6.4배 성능 차이는 사라지며, 순서 조정이 엔지니어의 주요 제어 수단입니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, 토이 라디컬 트리 캐시 + 캐시 인식 스케줄러)
**선수 지식:** 17단계 · 04 (vLLM 서빙 내부 구조), 14단계 (에이전틱 RAG)
**소요 시간:** ~75분

## 학습 목표

- **RadixAttention 다이어그램**: 접두사가 라디스 트리에 저장되는 방식과 동일한 브랜치에 루팅된 시퀀스 간 KV 블록 공유 메커니즘 설명.
- **캐시 인식 스케줄링**과 **FCFS(선입선출)**가 접두사 중심 트래픽에 부적합한 이유 설명.
- **접두사 캐시 적중률**과 **프롬프트 길이 분포**가 주어졌을 때 작업 부하의 예상 속도 향상 계산.
- **6.4배 성능 향상**을 실현하는 프롬프트 순서 결정 정책(디스시플린) 이름 제시 (미실현된 잠재적 이점 대비).

## 문제 정의

클래식 서빙은 각 요청의 프롬프트를 불투명한 데이터로 취급합니다. 5,000개의 RAG 요청이 모두 동일한 2,000토큰 시스템 프롬프트와 동일한 검색 프리앰블로 시작하더라도, vLLM은 해당 2,000토큰 접두사를 5,000번 프리필합니다. GPU는 동일한 작업을 반복합니다.

관찰: 에이전트 및 RAG 워크로드의 프롬프트는 거의 항상 긴 접두사를 공유합니다. 시스템 프롬프트, 도구 스키마, 몇 샷 예시, 검색 헤더, 대화 기록 등이 요청 간에 반복됩니다. 해당 접두사에 대한 KV 캐시를 한 번 저장하고 재사용하면, 다시 프리필할 필요가 없습니다.

RadixAttention은 정확히 이 작업을 수행합니다. 토큰은 기수 트리(radix tree)에 인덱싱되며, 각 노드는 루트부터 해당 경로까지의 토큰 시퀀스에 대한 KV 블록을 소유합니다. 새로운 요청은 트리를 탐색하며, 토큰이 일치하는 노드의 KV 블록을 재사용합니다. 프리필 비용은 전체 프롬프트가 아닌 "새로운" 접미사에 비례하게 됩니다.

과제는 스케줄링입니다. 두 요청이 2,000토큰 접두사를 공유하고 세 번째 요청이 동일한 접두사 중 200토큰만 공유하는 경우, 긴 접두사를 공유하는 두 요청을 함께 처리하여 해당 접두사가 HBM에 유지되도록 하고 싶습니다. FCFS(선입선출) 방식은 반대로 동작합니다. 먼저 도착한 요청을 처리하므로, 다음 긴 접두사 요청이 도착하기 전에 핫 브랜치가 제거될 수 있습니다.

## 개념

### KV 인덱스로서의 라딕스 트리

라딕스 트리(압축 트라이)는 토큰 시퀀스를 저장합니다. 각 노드는 토큰 범위와 해당 범위에 대해 계산된 KV 블록을 소유합니다. 자식 노드는 시퀀스를 하나 이상의 토큰으로 확장합니다.

```
root
 |- "You are a helpful assistant..."  (2,000 토큰, 124 KV 블록)
      |- "Context: <doc A>..."        (500 토큰, 31 블록)
           |- "Question: Alice..."    (80 토큰, 5 블록)
           |- "Question: Bob..."      (95 토큰, 6 블록)
      |- "Context: <doc B>..."        (520 토큰, 33 블록)
```

새로운 요청이 시스템 프롬프트 + "Context: <doc A>" + "Question: Carol"로 들어옵니다. 스케줄러는 다음과 같이 동작합니다: 시스템 접두사 일치 (124 블록 재사용), doc-A 브랜치 일치 (31 블록 재사용), 이후 "Question: Carol"에 대해서만 새 블록 할당 (4 블록). 프리필 비용: 새 토큰 4블록. 트리 없을 경우: 160 블록. 프리필 비용 ~40배 절감.

### 캐시 인식 스케줄링

캐시 교체가 빈번하면 라딕스 트리 기반 재사용은 무의미합니다. 두 가지 핵심 정책:

1. **깊이 우선 디스패치**. 큐에서 다음 요청을 선택할 때 현재 실행 중인 세트와 동일한 브랜치에 루팅된 요청을 우선합니다. 이는 핫 브랜치를 고정시킵니다.
2. **블록 레벨이 아닌 브랜치 레벨의 LRU**. 개별 블록이 아닌 전체 브랜치를 교체합니다(가장 최근에 사용된 짧은 리프부터 시작). 이렇게 하면 캐시 형태가 라딕스 형태와 일치합니다.

FCFS(선입선출)는 두 정책을 모두 위반합니다. 2,000 토큰을 공유하는 요청이 50 토큰을 공유하는 요청 뒤에 대기하면, 2,000 토큰 브랜치가 교체되어 50 토큰 요청을 수용합니다.

### 암기해야 할 벤치마크 수치

- Llama 3.1 8B, H100, ShareGPT 1K 프롬프트: SGLang ~16,200 tok/s vs vLLM ~12,500 (~29% 우위).
- 접두사 중심 RAG(동일 시스템 + 동일 문서, 질문 변동): SGLang에서 최대 6.4배 성능 향상.
- 음성 복제 워크로드: 86.4% 접두사 캐시 히트율.
- SGLang 고객사 전체 프로덕션 히트율: 프롬프트 규율에 따라 50-99%.
- 2026년 기준 400,000+ GPU에 배포.

### 순서 주의 사항

6.4배 수치는 일관된 프롬프트 템플릿 순서에 의존합니다. 클라이언트가 일부 요청에서는 `[시스템, 도구, 컨텍스트, 히스토리, 질문]`으로, 다른 요청에서는 `[시스템, 컨텍스트, 도구, 히스토리, 질문]`으로 프롬프트를 구성하면 라딕스 트리는 공유 접두사를 찾을 수 없습니다. 인간에게는 공유 접두사로 보이는 것이 라딕스 트리에는 두 개의 별도 시퀀스로 인식됩니다.

엔지니어의 조치: 프롬프트 템플릿은 캐시 키입니다. 순서를 고정하세요. 변경 불가능한 항목(시스템, 도구, 스키마)을 먼저 배치합니다. 검색 컨텍스트를 다음으로 배치합니다. 사용자 질문을 마지막에 배치합니다. 동적 콘텐츠를 접두사에 교차 배치하지 마세요.

연구 사례: 캐시 가능한 접두사에서 동적 콘텐츠를 제거한 한 배포에서는 캐시 히트율이 7%에서 74%로 향상되었습니다.

### 라딕스 어텐션의 강점과 약점

강점:
- RAG(동일 검색 서문, 변동 질문).
- 에이전트(동일 도구 스키마, 변동 쿼리).
- 긴 시스템 프롬프트를 사용하는 채팅.
- 반복되는 서문이 있는 음성/비전 워크로드.

약점(vLLM 수준 처리량으로 회귀):
- 고유 프롬프트를 사용하는 단일 생성(코드 완성, 시스템 프롬프트 없는 개방형 채팅).
- 모든 요청이 접두사에 고유 콘텐츠를 교차 배치하는 동적 프롬프트.

### 커널 문제가 아닌 스케줄러 문제인 이유

KV 재사용을 커널 트릭으로 구현할 수 있습니다. SGLang의 통찰은 스케줄러가 핫 브랜치를 상주시킬 때만 재사용이 효과적이라는 점입니다. 혼합 부하에서 캐시를 교체하는 순진한 "사용 가능한 경우 재사용" 정책은 캐시 교체를 유발합니다. 라딕스 트리 인덱스 스케줄러가 커널 트릭을 29% 프로덕션 우위로 전환합니다.

### vLLM과의 상호작용

두 시스템은 엄격한 경쟁자가 아닙니다. 2026년 vLLM은 접두사 캐싱(`--enable-prefix-caching`)과 캐시 인식 라우터(Rust로 작성된 vLLM 라우터)를 추가했습니다. 격차는 줄었지만 완전히 사라지지는 않았습니다. SGLang은 전체 스택이 라딕스 우선인 반면, vLLM은 이를 접목했습니다. 접두사 재사용이 지배적인 워크로드에서는 SGLang이 여전히 기본 선택입니다. 강력한 접두사 패턴이 없는 범용 서빙에서는 vLLM이 동등하거나 더 나은 성능을 유지합니다.

## 사용 방법

`code/main.py`는 장난감 수준의 기수 트리(radix-tree) KV 캐시 구현과 FCFS(First-Come-First-Served) 및 캐시 인식(cache-aware) 두 가지 정책을 가진 스케줄러를 포함합니다. 동일한 워크로드를 두 정책 모두에 실행한 후, 접두사 캐시 히트율(prefix-cache hit rate)과 처리량 변화(throughput delta)를 보고합니다. 이후 "스크램블된 순서(scrambled ordering)" 워크로드를 실행하여 6.4배의 성능 저하를 보여줍니다.

## Ship It

이 레슨은 `outputs/skill-radix-scheduler-advisor.md`를 생성합니다. 워크로드 설명(프롬프트-템플릿 형태, 검색 패턴, 동시 테넌트 수)이 주어지면, 프롬프트-오더링 처방과 SGLang 채택에 대한 go/no-go 결정을 제공합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 동일한 워크로드에서 FCFS와 캐시 인식(cache-aware) 방식을 비교하세요. 성능 차이(delta)는 어디에서 발생하나요? — 프리필(prefill) 절약, 디코드(decode) 절약, 또는 큐 지연(queue delay) 중 무엇인가요?
2. 워크로드를 수정하여 프롬프트가 `[system, tools, context]`를 무작위로 섞도록 변경하세요. 다시 실행한 후 캐시 적중률(hit rate)이 어떻게 변하는지 관찰하세요. 그 이유는 무엇인가요?
3. Llama 3.1 8B에서 2,000토큰 시스템 프롬프트를 하나의 라디스(radix) 브랜치로 상주시킬 때 HBM 비용을 계산하세요. 프리픽스 재사용 없이 16-시퀀스 배치(batch)의 비용과 비교하세요.
4. SGLang RadixAttention 논문을 읽고, 프리픽스 중심 부하 하에서 트리 형태의 LRU 교체(tree-shaped LRU eviction)가 블록 형태 LRU(block-shaped LRU)보다 우수한 이유를 3문장으로 설명하세요.
5. 고객이 8%의 낮은 캐시 적중률을 보고했습니다. 가능한 원인 3가지와 각 원인에 대해 실행할 진단 방법을 제시하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| RadixAttention | "the SGLang thing" | KV 캐시를 라디스 트리로 인덱싱하여 공유 접두사가 블록을 재사용하는 방식 |
| Radix tree | "compact trie" | 각 노드가 토큰 범위와 해당 KV 블록을 소유하는 트리 |
| Cache-aware scheduler | "hot-branch-first" | 상주하는 브랜치를 공유하는 요청을 우선하는 스케줄러 |
| Prefix-cache hit rate | "how much of your prompt was free" | 재사용된 KV 블록에서 제공된 프롬프트 토큰의 비율 |
| FCFS | "first-come first-served" | 접두사 지역성을 파괴하는 기본 스케줄링 방식 |
| Branch-level LRU | "evict the leaf" | 라디스 트리 구조에 맞춘 교체 정책 |
| Prompt template ordering | "the cache key" | 프롬프트 구성 요소 순서가 트리가 공유할 수 있는 내용을 결정 |
| System prompt pinning | "resident prefix" | 변경되지 않는 시스템 부분을 고정하여 교체 충돌을 방지하는 방식
