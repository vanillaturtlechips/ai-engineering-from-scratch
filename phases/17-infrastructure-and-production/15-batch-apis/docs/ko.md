# 배치 API — 업계 표준인 50% 할인

> 모든 주요 공급자는 50% 할인과 약 24시간 처리 시간을 제공하는 비동기 배치 API를 제공합니다. OpenAI, Anthropic, Google 및 대부분의 추론 플랫폼(Fireworks 배치 티어, Together 배치)이 동일한 패턴을 구현합니다. 프롬프트 캐싱과 결합하면 야간 파이프라인의 비용이 동기식 비캐시 비용의 ~10%로 감소합니다. 규칙은 매우 간단합니다: 상호작용이 필요하지 않다면 배치에 속해야 합니다. 콘텐츠 생성 파이프라인, 문서 분류, 데이터 추출, 보고서 생성, 대량 라벨링, 카탈로그 태깅 등 24시간 지연 시간을 허용할 수 있는 모든 작업은 배치로 전환하기 전까지 낭비되는 비용입니다. 2026년 프로덕션 패턴은 모든 새로운 LLM 워크로드를 세 가지 레인(대화형(캐싱된 동기식), 반대화형(대체 기능이 있는 비동기 큐), 배치(야간, 캐시된 입력 스택))으로 분류하는 것입니다. 대화형처럼 보이지만 몇 분의 지연 시간을 허용하는 워크로드는 대부분 비용을 낭비합니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 장난감 배치-대-동기 비용 시뮬레이터)  
**선수 조건:** 17단계 · 14단계 (프롬프트 및 시맨틱 캐싱)  
**소요 시간:** ~45분

## 학습 목표

- 세 가지 공급자 배치 API(OpenAI, Anthropic, Google)의 이름과 공통 50% 할인 + 24시간 처리 보장 조건을 설명할 수 있다.
- 야간 분류 작업에서 배치 + 캐시된 입력(stacking batch + cached-input) 비용을 계산하고 동기식-미캐시(synchronous-uncached) 기준과 비교할 수 있다.
- 작업 부하(workload)를 대화형(interactive) / 반대화형(semi-interactive) / 배치(batch)로 분류하고 해당 레인(lane) 선택을 정당화할 수 있다.
- 두 가지 함정: 부분 상호작용성(partial interactivity, 사용자가 24시간보다 빠른 응답 기대)과 출력 스키마 드리프트(output-schema drift, 공급자별 배치 파일 형식 불일치)를 설명할 수 있다.

## 문제

귀하의 팀은 야간 보고서 생성 파이프라인을 운영합니다. 50,000개의 문서를 요약하고, 요약본을 클러스터링한 후 경영진 요약본을 작성합니다. 동기식으로 실행할 경우 4시간이 소요되며 비용은 $2,000/밤입니다. 이때 배치 API에 대해 듣게 됩니다.

배치 API를 사용하면 비용을 50% 절감할 수 있습니다. 또한 시스템 프롬프트(50,000개 호출에 공유됨)에 프롬프트 캐싱을 활성화할 수 있습니다. 이 두 방법을 함께 적용하면 비용이 $180/밤으로 떨어집니다. 이는 기존 비용의 약 9%에 해당하며, 동일한 파이프라인에서 3가지 구성 변경만으로 달성됩니다.

배치 API는 LLM 비용 최적화 도구 중 가장 저렴하지만 가장 활용되지 않는 방법입니다. 그 이유는 주로 조직 문화 때문입니다. 팀들은 "실시간" 처리를 생각하지만, 실제 SLA(서비스 수준 계약)는 "아침까지"입니다. 이 교훈은 청구서의 90%를 테이블에 남겨두지 않는 방법에 관한 것입니다.

## 개념

### 세 가지 배치 API

**OpenAI 배치 API**: 요청 목록이 포함된 JSONL 파일 업로드. 24시간 내 처리 약속(실제로는 ~2-8시간). 입력 및 출력 토큰에 대해 50% 할인. `/v1/batches` 엔드포인트. 캐시 가능한 입력은 캐시 입력 가격 정책도 추가 적용.

**Anthropic 메시지 배치**: JSONL 업로드. 24시간 내 처리. 50% 할인. `cache_control` 지원 — 캐시 쓰기는 명시적, 배치 내에서 자동 읽기 발생.

**Google Vertex AI 배치 예측**: BigQuery 또는 GCS 입력. Gemini에 대해 유사한 50% 할인. Vertex 파이프라인과 통합.

### 의미: 비동기식, 느린 것이 아님

배치는 "24시간 내 반환 약속" — "24시간 소요"가 아님. 일반적인 P50은 2-6시간. 공급자는 GPU 재고가 미활용되는 비피크 시간대에 배치를 예약.

### 캐싱과 스택

동일한 4K-토큰 시스템 프롬프트로 50k 문서 요약:

- 동기식 비캐시: 50000 × ($입력 × 4000 + $출력 × 200) 정가.
- 동기식 캐시: 첫 쓰기 후 시스템 프롬프트 캐시; 나머지 49999개는 입력 비용 10배 저렴.
- 배치 캐시: 위 모든 혜택 + 읽기/쓰기 모두 50% 할인.

스택: 배치 + 캐시 = 동기식 비캐시 비용의 ~10%. 밤새 실행되고 공유 시스템 프롬프트가 있는 모든 워크로드는 이 조합 사용.

### 워크로드 분류

**인터랙티브** — 사용자가 응답을 기다림. TTFT(첫 응답 시간) 중요. 프롬프트 캐싱이 있는 동기식 호출. 배치 불가.

**반인터랙티브** — 사용자가 작업 제출 후 몇 분 후 확인. 비동기 큐에 배치, 배치 불가 시 동기식 폴백. 중간 규모 RAG 인덱싱 사례.

**배치** — 사용자가 "아침까지" 또는 "다음 시간까지" 결과 기대. 콘텐츠 파이프라인, 대규모 분류, 오프라인 분석. 항상 배치, 항상 캐싱 스택.

흔한 실수: 파이프라인이 프로덕션이라 모든 것을 인터랙티브로 분류. 프로덕션은 지연 시간 명세가 아님 — SLA가 중요.

### 부분적 인터랙티비티 함정

일부 기능은 인터랙티브처럼 보이지만 5-10분 지연 허용. 예: "새로고침" 버튼이 있는 야간 고객 건강 보고서. 사용자가 새로고침 클릭; 10분 대기 가능. 팀은 동기식으로 배포. 50개의 동시 새로고침은 배치 후 이메일 전송 비용의 10배.

질문: "이 사용자에게 24시간은 어떤 의미인가?" 답이 "알아채지 못함"이면 배치 처리.

### 출력 스키마 함정

배치 파일 형식은 공급자마다 다름:

- OpenAI: JSONL, 한 줄에 하나의 요청.
- Anthropic: JSONL, 한 줄에 하나의 메시지; 응답 형식 내장.
- Vertex: BigQuery 테이블 또는 TFRecord가 있는 GCS 접두사.

모든 공급자를 아우르는 "단일 배치 클라이언트" 작성은 공급자별 어댑터 코드 필요. 멀티 공급자 배치 광고 게이트웨이(Portkey, LiteLLM 일부 티어)도 원시 형식을 얇게 래핑.

### 기억해야 할 숫자

- 공급자 간 배치 할인: 입력 + 출력에 50% 고정 할인.
- 처리 SLA: 24시간 보장, 일반적인 P50은 2-6시간.
- 배치 + 캐시 입력 스택: 동기식 비캐시 비용의 ~10%.
- 워크로드 분류 규칙: 24시간 지연 허용 시 항상 배치.

## 사용 방법

`code/main.py`는 50,000개 문서 작업 부하에 대해 동기식(sync), 동기식+캐시(sync+cache), 배치(batch), 배치+캐시(batch+cache) 방식의 비용을 계산합니다. 절감 효과를 달러($)와 백분율(%)로 보고합니다.

## Ship It

이 레슨은 `outputs/skill-batch-triager.md`를 생성합니다. 작업 부하 특성을 고려하여 인터랙티브/세미/배치 작업으로 분류하고 절감 효과를 추정합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 100k-doc 파이프라인에서 3K-토큰 시스템 프롬프트와 500-토큰 출력 시, 풀 스택(배치 + 캐시)과 동기식 기준선 간 비용 절감 효과를 계산하세요.
2. 알고 있는 실제 제품에서 세 가지 기능을 선택하세요. 각각을 대화형/반자동/배치 유형으로 분류하세요.
3. 사용자가 보고서 생성에 3시간이 걸렸다고 불평합니다. 이는 배치 처리 오류 분류 때문인가요, 아니면 정당한 대화형 요청인가요? 판단 기준을 작성하세요.
4. 배치 API의 SLA(서비스 수준 계약)는 24시간이지만 P99(99번째 백분위 수)는 20시간입니다. 이 정보를 사용자에게 어떻게 전달하겠습니까? 극단적인 경우(edge case)에서 다운스트림 시스템의 동작은 어떻게 되나요?
5. 손익분기점 계산: 공유 접두사(shared-prefix) 길이가 어느 정도일 때 배치 + 캐시 방식이 자체 예약 GPU에서 밤새 실행하는 것보다 저렴해지는지 계산하세요.
