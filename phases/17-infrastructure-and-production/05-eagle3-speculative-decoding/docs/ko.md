# EAGLE-3 프로덕션 환경에서의 추측 디코딩

> 추측 디코딩은 빠른 초안 모델과 목표 모델을 결합합니다. 초안 모델이 K개의 토큰을 제안하면, 목표 모델이 단일 순방향 패스로 검증합니다. 수락된 토큰은 비용이 들지 않습니다. 2026년 기준, EAGLE-3는 프로덕션급 변종으로 — 원시 토큰이 아닌 목표 모델의 은닉 상태에 초안 헤드를 훈련시켜, 일반 채팅에서 수락률 알파(alpha)를 0.6-0.8 범위로 끌어올립니다. 올바른 질문은 "초안이 얼마나 빠른가?"가 아니라 "내 트래픽에서 알파는 얼마인가?"입니다. 알파가 ~0.55 미만으로 떨어지면, 높은 동시성 환경에서 추측 디코딩은 순비용이 발생합니다. 거부된 초안마다 추가적인 목표 모델 순방향 패스가 필요하기 때문입니다. 이 레슨에서는 먼저 알파를 측정한 후 플래그를 전환하는 방법을 배웁니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 장난감 수락률 시뮬레이터)  
**선수 지식:** Phase 17 · 04 (vLLM 서빙 내부 구조), Phase 10 · 18 (다중 토큰 예측)  
**소요 시간:** ~60분

## 학습 목표

- 추론적 디코딩의 세 가지 세대를 나열하고, EAGLE-3가 EAGLE-2 및 클래식 드래프트 모델과 비교해 어떤 변경 사항을 도입했는지 설명하시오.
- 수용률(acceptance rate) α를 정의하고, α와 K(드래프트 길이)로부터 예상 속도 향상치를 계산하며, 목표 동시성(concurrency)에 대한 손익분기점 α를 식별하시오.
- vLLM 2026에서 추론적 디코딩이 기본값이 아닌 선택적 기능(opt-in)인 이유와, α 측정 없이 활성화하는 것이 프로덕션 환경에서 반패턴(anti-pattern)인 이유를 설명하시오.
- 측정 계획 수립: 어떤 벤치마크, 어떤 프롬프트 분포, 어떤 동시성 지점, 어떤 메트릭을 기준으로 할지 작성하시오.

## 문제

디코딩은 메모리 바운드(memory-bound)입니다. H100에서 Llama 3.3 70B FP8을 실행할 때, 각 디코딩된 토큰은 약 140GB/s의 가중치를 읽고 하나의 토큰을 출력합니다. 디코딩 중에는 GPU 연산이 거의 유휴 상태 — 병목 현상은 행렬 곱셈(matmul) 처리량이 아닌 HBM 대역폭입니다.

스펙큘레이티브 디코딩(speculative decoding)은 이 격차를 활용합니다. 저렴한 초안 모델(draft model)로 K개의 후보 토큰을 생성한 후, 대상 모델(target model)이 단일 순전파(forward pass)에서 K개 모두를 검증하도록 합니다. 검증된 각 토큰은 사실상 무료입니다(대상이 어차피 수행해야 했던 배치-오브-K 순전파에 상각됨).

기존의 초안 모델 접근법은 동일한 패밀리(예: Llama 3.3 70B를 위한 Llama 3.2 1B 초안)의 더 작은 모델을 사용합니다. 작동은 하지만 수용률(acceptance rate)이 보통 수준입니다 — 작은 모델의 분포가 대상과 달라지기 때문입니다. EAGLE, 이후 EAGLE-2, EAGLE-3은 대상 모델의 내부 상태에 직접 경량 초안 헤드(draft head)를 학습시켜 초안 분포가 대상을 훨씬 더 정확히 추적하도록 합니다. 따라서 초안 모델 사용 시 알파(alpha)가 0.4에서 EAGLE-3 사용 시 0.6-0.8로 향상됩니다.

주의 사항: EAGLE-3은 vLLM 2026에서 선택적 기능입니다. `speculative_config`를 명시적으로 설정해야 합니다. 플래그가 없으면 가속화가 적용되지 않습니다. 실제 트래픽에서 알파를 측정하지 않고 활성화한 팀은 꼬리 지연(tail latency)이 개선되지 않고 오히려 악화되는 경우가 많습니다.

## 개념

### 추측 디코딩이 실제로 제공하는 이점

추측 디코딩(spec decode)을 사용하지 않을 때 토큰당 비용은 하나의 타겟 순전파(target forward)입니다. 초안 길이 K와 수용률(acceptance rate) alpha로 추측 디코딩을 사용할 때, 타겟 순전파당 예상 토큰 수는 `1 + K * alpha`입니다. 속도 향상은 `(1 + K * alpha) / (1 + epsilon)`로 계산되며, 여기서 epsilon은 초안-검증 오버헤드입니다. K=5, alpha=0.7인 경우: `(1 + 5*0.7) / (1 + 0.1) = 4.5 / 1.1 = 4.1x`입니다. 실제 수치는 2-3배 범위에 머무르는데, 이는 프로덕션 트래픽에서 alpha가 그렇게 높지 않고, 높은 배치 크기에서 epsilon이 증가하기 때문입니다.

### 수용률(alpha)이 유일한 지표인 이유

거부된 토큰은 사라지지 않습니다 — 첫 번째 거부된 토큰에 대해 두 번째 타겟 순전파를 강제합니다. alpha가 0.4로 떨어지는 워크로드에서는 초안 오버헤드, 검증, 재시도 비용을 지불하게 됩니다. 높은 동시성(예: 256 동시 요청)에서는 디코딩 배치가 이미 충분히 커서 "타겟 단독"과 "검증 포함 타겟" 간의 메모리 대역폭 격차가 줄어듭니다. 대부분의 2026 하드웨어에서 alpha 0.55 미만일 경우 추측 디코딩은 순손실이 됩니다.

alpha는 워크로드에 따라 다릅니다. ShareGPT 스타일의 일반 채팅에서는 ShareGPT로 훈련된 EAGLE-3가 0.6-0.8을 기록합니다. 도메인 특화 트래픽(코드, 의료, 법률)에서는 일반 데이터로 훈련된 초안 헤드가 0.4-0.6으로 떨어집니다. 도메인 특화 초안 헤드를 훈련하면 alpha를 회복할 수 있습니다 — 이는 타겟 파인튜닝에 비해 가볍고 빠른 훈련 작업입니다.

### EAGLE 생성 방식 요약

- **클래식 초안 모델**: 동일 패밀리의 소형 모델. Alpha 0.3-0.5. 인프라는 단순 — 두 모델을 로드하고, 초안 모델은 타겟 순전파당 K번 순전파를 실행합니다.
- **EAGLE-1 (2024)**: 타겟 은닉 상태(마지막 레이어)로 훈련된 단일 초안 헤드. Alpha ~0.5-0.6. 타겟 위에 작은 파라미터 오버헤드가 있습니다.
- **EAGLE-2 (2025)**: 적응형 초안 길이와 트리 기반 초안(한 번의 타겟 패스에서 여러 브랜치 검증). Alpha ~0.6-0.7. 더 복잡한 초안 스케줄러.
- **EAGLE-3 (2025-2026)**: 여러 타겟 레이어(마지막 레이어 뿐만 아니라)로 훈련된 초안 헤드, 더 나은 정렬. 일반 채팅에서 Alpha ~0.6-0.8.

### 2026년 프로덕션 레시피

1. 타겟 모델을 일반 모드로 출시. 타겟 동시성에서 기준 TTFT(첫 토큰 대기 시간), ITL(인터 토큰 대기 시간), 처리량을 측정합니다.
2. vLLM `speculative_config`를 통해 EAGLE-3 초안을 활성화. 벤치마크를 다시 실행합니다.
3. 수용률 alpha를 로깅. vLLM V1은 이를 `spec_decode_metrics.accepted_tokens_per_request`로 보고합니다. 요청된 초안 길이로 나누어 alpha를 계산합니다.
4. 프로덕션 트래픽 분포에서 alpha < 0.55인 경우, 추측 디코딩을 비활성화하거나 도메인 특화 EAGLE-3 초안을 훈련합니다.
5. 프로덕션 동시성에서 다시 실행. P99 ITL이 악화되지 않았는지 확인합니다.

### 프로덕션 함정: P99 꼬리

추측 디코딩으로 평균 ITL은 감소합니다. 조정하지 않으면 P99는 악화될 수 있습니다. 거부된 초안은 두 번의 패스(초안 + 검증 실패 + 재시도)를 트리거합니다. 전체 배치 하에서 이 두 패스는 직렬화됩니다. P50이 아닌 P99 ITL을 모니터링해야 합니다.

### EAGLE-3의 현재 배포 현황

Google은 2025년 AI 개요에 추측 디코딩을 배포했습니다(동일 품질, 더 빠른 응답). vLLM V1은 `speculative_config`를 문서화된 인터페이스로 제공합니다. V1의 N-gram GPU 추측 디코딩은 청크 기반 프리필과 호환되는 변형입니다. SGLang은 접두사 중심 워크로드에 대해 EAGLE-3를 권장 초안 경로로 지원합니다.

### 손익분기점 수학 한 줄 요약

예상 속도 향상: `S(alpha, K) = (1 + K*alpha) / (1 + verify_overhead)`. `S = 1`로 설정하면 alpha는 `alpha_breakeven = verify_overhead / K`가 됩니다. 일반적인 verify_overhead ~0.15, K=5인 경우: `alpha_breakeven = 0.03`입니다. 하지만 이는 원시 디코딩 수학입니다. 높은 동시성에서는 검증 오버헤드가 증가하고 디코딩 배치가 이미 시퀀스 간 메모리 읽기를 분산시키므로, 실제 alpha_breakeven은 ~0.45-0.55로 상승합니다.

### 추측 디코딩을 사용하지 말아야 할 경우

- 배치-1 오프라인 생성(지연 시간이 중요하지 않은 경우). 일반 타겟을 사용합니다.
- 매우 짧은 출력(50토큰 미만). 초안 오버헤드와 검증 비용이 지배적입니다.
- 도메인 훈련 초안 헤드가 없는 특수 도메인. Alpha가 너무 낮습니다.
- vLLM v0.18.0 + 초안 모델 추측 디코딩 + `--enable-chunked-prefill`. 이 조합은 컴파일되지 않습니다. 문서화된 예외는 V1의 N-gram GPU 추측 디코딩입니다.

## 사용 방법

`code/main.py`는 다양한 알파(alpha) 값과 초안 길이 K에 걸쳐 추측 디코딩(speculative decoding)을 적용하거나 적용하지 않은 디코딩 루프를 시뮬레이션합니다. 이 스크립트는 손익분기점 알파(break-even alpha), 측정된 속도 향상(speedup), 꼬리 동작(tail behavior)을 출력합니다. 여러 (알파, K) 조합에서 실행하여 추측 디코딩이 더 이상 유리하지 않은 지점을 정확히 확인할 수 있습니다.

## Ship It

이 레슨은 `outputs/skill-eagle3-rollout.md`를 생성합니다. 대상 모델, 트래픽 분배 설명, 동시성 목표를 입력으로 받아 단계별 EAGLE-3 롤아웃 계획을 생성합니다 — 벤치마크 기준선, 활성화 구성, 알파 측정, 알파 ≥ 0.55 게이트, P99 ITL 모니터링.

## 연습 문제

1. `code/main.py`를 실행하세요. K=5에서 2배 속도 향상을 위해 필요한 알파(alpha)는 얼마인가요? 3배 속도 향상을 위해서는? 이 값들이 `verify_overhead`에 얼마나 민감한가요?

2. 프로덕션 트래픽이 70% 일반 채팅, 30% 코드로 분할된다고 가정합니다. 일반 채팅은 ShareGPT로 학습된 EAGLE-3에서 알파 0.7을 달성하고, 코드는 알파 0.4를 달성합니다. 혼합 알파(blended alpha)는 얼마이며, 스펙큘레이티브 디코딩(spec decode)이 순수익(net-positive)인가요?

3. vLLM의 `speculative_config` 문서를 읽으세요. 세 가지 모드(드래프트 모델, EAGLE, N-gram)의 이름을 말하고, 어떤 모드가 청크 기반 프리필(chunked prefill)과 호환되는지 설명하세요.

4. EAGLE-3 활성화 후 평균 ITL(Inference Time Latency)이 25% 감소했지만 P99 ITL은 15% 증가했습니다. 원인을 진단하고 완화 방안을 제안하세요.

5. Llama 3.3 70B에 대한 EAGLE-3 드래프트 헤드의 메모리 비용을 계산하세요. 이를 Llama 3.2 1B를 클래식 드래프트 모델로 실행할 때의 메모리 비용과 비교하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| Speculative decoding | "draft plus verify" | 저렴한 모델로 K개의 토큰을 제안하고, 대상 모델의 한 번의 순전파(forward)로 K개 전체를 검증 |
| Acceptance rate alpha | "spec accept rate" | 대상 모델이 수락한 초안 토큰의 비율; 유일하게 중요한 지표 |
| Draft length K | "spec k" | 대상 모델의 한 번의 순전파(forward)당 초안이 제안하는 토큰 수; 일반적으로 4-8 |
| Verify overhead epsilon | "spec overhead" | 검증-재시도(verify-and-reroll)와 일반 대상 순전파(forward) 간의 추가 비용; 배치 크기에 따라 증가 |
| EAGGE-3 | "latest EAGLE" | 2025-2026년 변형 모델; 여러 대상 레이어에 대해 초안 헤드(draft head)를 학습; 일반 채팅에서 alpha 0.6-0.8 달성 |
| `speculative_config` | "vLLM spec config" | vLLM V1의 명시적 활성화 옵션; 기본값이 없으므로 가속화를 원하면 반드시 활성화해야 함 |
| N-gram spec decode | "N-gram draft" | 프롬프트 내 N-gram 조회를 활용한 GPU 측 초안 생성; 청크 기반 사전 채우기(chunked-prefill)와 호환 |
| Break-even alpha | "no-op alpha" | 추론 속도 향상이 0이 되는 alpha 값; 프로덕션 동시성 환경에서 주의 깊게 모니터링 필요 |
| Rejected-draft two-pass | "reroll cost" | 초안이 거부될 때 발생하는 두 번의 대상 순전파(forward); P99 꼬리 지연(tail latency) 유발 요인 |
