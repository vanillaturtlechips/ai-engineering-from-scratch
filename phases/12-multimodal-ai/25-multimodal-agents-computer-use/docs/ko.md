# 멀티모달 에이전트와 컴퓨터 사용 (캡스톤)

> 2026년 프론티어 제품은 스크린샷을 읽고, 버튼을 클릭하며, 웹 UI를 탐색하고, 양식을 채우고, 워크플로우를 종단간(end-to-end)으로 완료하는 멀티모달 에이전트입니다. SeeClick과 CogAgent(2024)는 GUI-접지(grounding) 기본 요소를 증명했습니다. Ferret-UI는 모바일을 추가했고, ChartAgent는 차트를 위한 시각적 도구 사용을 도입했습니다. VisualWebArena과 AgentVista(2026)는 프론티어가 추구하는 벤치마크이며, 심지어 Gemini 3 Pro와 Claude Opus 4.7도 AgentVista의 어려운 작업에서 ~30% 점수를 기록합니다. 이 캡스톤은 12단계의 모든 요소(인식(고해상도 VLM), 추론(도구 사용 LLM), 접지(좌표 출력), 장기 메모리, 평가)를 통합합니다.

**유형:** 캡스톤  
**언어:** Python (표준 라이브러리, 액션 스키마 + 에이전트 루프 스켈레톤)  
**선수 과목:** 12단계 · 05 (LLaVA), 12단계 · 09 (Qwen-VL JSON), 14단계 (에이전트 엔지니어링)  
**소요 시간:** ~240분

## 학습 목표

- 멀티모달 에이전트 루프 설계: 인지(perceive) → 추론(reason) → 실행(act) → 관찰(observe) → 반복(repeat).
- GUI 기반 출력 스키마(클릭 좌표, 텍스트 입력, 스크롤, 드래그)를 VLM이 JSON 형식으로 생성할 수 있도록 구축.
- 스크린샷 전용 에이전트 vs 접근성 트리 에이전트 vs 하이브리드 에이전트 비교.
- 소규모 VisualWebArena 데이터셋에서 멀티모달 에이전트 벤치마크 평가 설정.

## 문제 정의

예약 사이트 워크플로우: "4월 15일 도쿄행 비행기 찾아줘. 복도 좌석이고 $800 미만, 예약해줘."

멀티모달 에이전트는 다음을 수행해야 합니다:

1. 브라우저 스크린샷 캡처.
2. 스크린샷 + URL + 목표를 파싱하여 계획 수립.
3. 구조화된 액션 출력: 클릭 (x,y 좌표), "도쿄" 입력 (요소 E), 스크롤 다운, 선택 (라디오 버튼).
4. 브라우저에 액션 적용.
5. 새로운 상태(다음 스크린샷) 관찰.
6. 작업 완료 시까지 반복.

각 단계는 멀티모달 VLM 호출입니다. VLM 출력은 파싱 가능한 JSON이어야 합니다. 오류가 단계별로 누적되므로 복구가 중요합니다.

## 개념

### GUI 그라운딩 — 기본 요소

GUI 그라운딩은: 스크린샷과 자연어 지시문이 주어졌을 때, 클릭할 (x, y) 좌표(또는 다른 동작)를 출력하는 것입니다.

SeeClick (arXiv:2401.10935)은 대규모 첫 번째 오픈 결과였습니다: 합성 + 실제 GUI 데이터로 VLM(Visual Language Model)을 파인튜닝(fine-tuning)하고, 좌표를 일반 텍스트 토큰으로 출력합니다. 작동합니다.

CogAgent (arXiv:2312.08914)는 밀집 UI를 위한 1120x1120 고해상도 인코딩을 추가했습니다. 웹 내비게이션에서 점수: ~84%.

Ferret-UI (arXiv:2404.05719)는 모바일 UI에 집중하며, iOS 접근성 데이터와 통합됩니다.

출력 형식은 일반적으로 JSON입니다:

```json
{"action": "click", "x": 384, "y": 220, "element_desc": "검색 버튼"}
```

`element_desc`는 복구에 도움을 줍니다: 스크린샷 간 좌표가 어긋나면 의미적 힌트가 시스템이 재그라운딩(re-grounding)하도록 합니다.

### 액션 스키마

일반적인 액션 스키마는 6-10개의 액션 유형을 가집니다:

- `click`: (x, y)
- `type`: (텍스트, x?, y?)
- `scroll`: (방향, 양)
- `drag`: (x0, y0, x1, y1)
- `select`: (옵션 인덱스)
- `hover`: (x, y)
- `navigate`: (URL)
- `wait`: (밀리초)
- `done`: (성공 여부, 설명)

에이전트는 단계당 하나의 액션을 출력합니다. 브라우저 래퍼는 실행하고 새로운 상태를 반환합니다.

### 스크린샷 전용 vs 접근성 트리

두 가지 입력 모드:

- 스크린샷 전용: 전체 이미지, 구조적 정보 없음. 가장 일반적; 모든 앱에서 작동.
- 접근성 트리: 구조화된 DOM / iOS 접근성 정보. 그라운딩에 훨씬 더 안정적; 트리가 사용 가능한 곳에서 작동.
- 하이브리드: 둘 다, 트리는 원자적 액션의 안정적 그라운더로, 스크린샷은 의미적 컨텍스트로 사용.

프로덕션 에이전트는 가능한 경우 하이브리드를 사용합니다. 브라우저 자동화(Selenium + 접근성)는 항상 트리를 가짐; 데스크톱 앱은 때때로 가짐.

### 장기 기억(Long-horizon memory)

20단계 워크플로우는 20개의 스크린샷을 생성합니다. VLM의 컨텍스트는 빠르게 채워집니다. 세 가지 압축 전략:

- 요약 체인(Summary-chain): 5단계마다 발생한 일을 요약하고, 오래된 스크린샷을 삭제.
- 프레임 건너뛰기(Skip-frame): 첫 번째, 마지막, 그리고 3번째마다의 스크린샷 유지.
- 도구 기록 로그(Tool-recorded log): 액션을 실행하고, 수행된 작업의 텍스트 로그를 유지; 오래된 스크린샷을 다시 보지 않음.

Claude의 컴퓨터 사용 API는 로그 패턴을 사용합니다. 더 간단하고 안정적입니다.

### 시각적 도구 사용

ChartAgent (arXiv:2510.04514)는 차트 이해를 위한 시각적 도구 사용을 도입했습니다: 자르기, 확대, OCR, 외부 감지 호출. 에이전트는 "영역 (100, 200, 300, 400)으로 자르기 후 OCR 호출"을 도구 호출로 출력할 수 있습니다. 도구는 텍스트를 반환하고, VLM은 추론을 계속합니다.

이 패턴은 일반화됩니다: 마크 프롬프트 집합(set-of-mark prompting), 영역 주석(region annotation), 외부 감지 도구는 모두 "도구 호출 출력, 구조화된 응답 수신" 스키마에 맞습니다.

### 2026 벤치마크

- ScreenSpot-Pro. ~1k 웹 스크린샷에서의 GUI 그라운딩. 오픈 SOTA Qwen2.5-VL-72B ~85%. 프론티어 ~90%.
- VisualWebArena. 엔드투엔드 웹 작업(쇼핑, 포럼, 중고거래). 오픈 SOTA ~20%. Gemini 3 Pro ~27%.
- AgentVista (arXiv:2602.23166). 가장 어려운 2026 벤치마크. 12개 도메인에 걸친 현실적인 워크플로우. 프론티어 모델 점수 27-40%; 오픈 모델 10-20%.
- WebArena / WebShop. 오래된 벤치마크; 프론티어 모델에 의해 포화 상태.

### 여전히 어려운 이유

에이전트 성능 병목 현상:

1. 세밀한 시각적 그라운딩. "작은 X 클릭"은 모바일 해상도에서 자주 실패합니다.
2. 장기 계획. 10개 액션 후 에이전트는 목표에서 벗어납니다.
3. 오류 복구. 클릭 실패(잘못된 버튼) 시 감지 + 복구는 거의 훈련 데이터가 없습니다.
4. 페이지 간 컨텍스트. 탭 또는 긴 양식 간 이동 시 상태 손실.

연구 방향: 메모리 아키텍처, 명시적 재계획, 멀티모달 검증(액션 성공 여부를 위한 스크린샷 매칭).

### 캡스톤 빌드

캡스톤 작업: 다음 작업을 수행하는 컴퓨터 사용 에이전트를 구축합니다:

1. 예약 사이트 모의 페이지의 HTML + 스크린샷을 읽습니다.
2. 다단계 시퀀스를 계획합니다: 검색 → 선택 → 양식 작성 → 제출.
3. 액션 스키마에 맞는 JSON 액션을 출력합니다.
4. 고정된 10개 작업 슬라이스에서 평가합니다.

이 레슨은 실제 브라우저로 쉽게 확장할 수 있는 스캐폴드 코드를 제공합니다.

## 사용 방법

`code/main.py`는 캡스톤 스캐폴드입니다:

- 액션 스키마 JSON 정의 (10가지 액션).
- 딕트 형태의 모의 브라우저 상태.
- 에이전트 루프 골격: 상태 수신, 액션 발행, 적용, 반복.
- 10개 작업 미니 벤치마크 (합성 페이지)를 통한 종단간 성공률 측정.
- 액션 실패 시 오류 복구 훅.

## 연습 문제

1. `screenshot_region` 도구(크롭 + 확대/축소)를 추가하여 액션 스키마를 확장하세요. 어떤 작업들이 이점을 얻을 수 있나요?

2. AgentVista(arXiv:2602.23166)를 읽고, 가장 어려운 작업 범주와 프론티어 모델들이 여전히 실패하는 이유를 설명하세요.

3. 장기 메모리 압축: ≤4개의 스크린샷을 실시간으로 유지하고, 임의의 수를 로깅하는 요약 체인(summary-chain)을 설계하세요.

4. 오류 복구 훅 구축: 액션 실패(버튼 미발견) 시 에이전트는 다음에 어떤 행동을 해야 하나요?

5. 10개의 웹 작업에서 스크린샷 전용 Claude 4.7과 하이브리드(스크린샷 + 접근성 트리) Qwen2.5-VL을 비교하세요. 어떤 모델이 어떤 작업에서 승리하나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|---------------------|-----------|
| GUI 그라운딩(GUI grounding) | "클릭 좌표" | 스크린샷에서 지시 대상의 (x,y) 좌표를 모델 출력 |
| 액션 스키마(Action schema) | "툴 정의" | 유효한 액션(클릭, 입력, 스크롤, 드래그)에 대한 JSON 설명 |
| 접근성 트리(Accessibility tree) | "구조화된 DOM" | 브라우저/iOS API에서 제공하는 기계 판독 가능 UI 계층 구조 |
| 하이브리드 에이전트(Hybrid agent) | "스크린샷 + 트리" | 이미지와 구조화된 정보를 모두 사용; 단독 사용보다 더 신뢰성 높음 |
| 시각적 툴사용(Visual tool use) | "확대/자르기/감지" | 에이전트가 계획 중간에 외부 비전 툴(OCR, 감지)을 호출 |
| 요약 체인(Summary-chain) | "메모리 압축" | 주기적인 텍스트 요약이 긴 스크린샷 기록을 대체 |
| 비주얼웹아레나(VisualWebArena) | "E2E 웹 벤치" | 2024년 엔드투엔드 웹 작업 벤치마크 |
| 에이전트비스타(AgentVista) | "2026년 어려운 벤치" | 12개 도메인 실제 워크플로우; 제미니 3 프로도 ~30% 점수 |
