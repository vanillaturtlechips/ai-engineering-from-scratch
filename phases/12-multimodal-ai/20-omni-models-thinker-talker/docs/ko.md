# Omni 모델: Qwen2.5-Omni과 Thinker-Talker 분할

> 2024년 5월 GPT-4o의 제품 데모는 기반 모델 때문이 아니라 제품 형태 — 음성으로 말하고, 모델이 카메라로 본 것을 인식하며, 250ms 이내에 응답하는 음성 인터페이스 — 때문에 파괴적이었습니다. 오픈 생태계는 2024년 나머지 기간과 2025년 내내 그 제품 수준에 도달하기 위해 경쟁했습니다. Qwen2.5-Omni(2025년 3월)은 참조용 오픈 디자인입니다: Thinker(대형 텍스트 생성 트랜스포머)와 Talker(병렬 음성 생성 트랜스포머)가 스트리밍 음성 토큰으로 연결된 구조입니다. Mini-Omni은 이를 단순화했고, Moshi는 지연 시간을 맞췄으며, GLM-4-Voice는 중국어로 확장했습니다. 이 레슨은 Thinker-Talker 아키텍처와 실시간 대화를 가능하게 하는 지연 시간 예산을 분석합니다.

**유형:** 구축(Build)  
**언어:** Python (표준 라이브러리, 스트리밍 파이프라인 지연 시간 시뮬레이터 + VAD 루프)  
**사전 요구 사항:** 12단계 · 19(오디오-LLM), 12단계 · 16(모든 언어 간 변환)  
**소요 시간:** ~180분

## 학습 목표

- 추론 파이프라인(Thinker: 텍스트 추론, Talker: 음성 합성)을 분할하고 병렬 스트리밍이 작동하는 이유를 설명.
- 대화형 상호작용에서 구성 요소별 시간-첫-오디오-바이트(TTFAB) 예산을 계산.
- Thinker 내에서 비전, 오디오, 텍스트를 아우르는 TMRoPE의 시간 정렬 위치 인코딩 설명.
- 실시간 대화 패턴 3가지(반-이중, 턴-테이킹, 전이중) 명칭.

## 문제 정의

실시간 음성 비서는 많은 작업을 빠르게 수행해야 합니다:

1. **사용자 음성 인식**. 실시간 음성 토큰화, 음성 활동 감지(VAD)를 통해 사용자가 말을 마쳤는지 확인.
2. **선택적 시각 입력**. 2-4 FPS의 카메라 입력을 오디오와 함께 Thinker로 스트리밍.
3. **사고 처리**. 대화 기록을 기반으로 응답 생성.
4. **음성 출력**. 오디오 토큰 합성, 파형으로 디코딩, 사용자 스피커로 스트리밍.

각 단계는 지연 시간을 추가합니다. 대화 경험을 위해서는 전체 왕복 시간이 **500ms 미만**이어야 합니다. 이 임계값을 넘으면 사용자는 지연을 인지하기 시작합니다. GPT-4o는 약 250ms, Moshi는 약 160ms, Qwen2.5-Omni는 약 350-500ms를 주장합니다.

모든 구성 요소는 스트리밍 방식으로 작동해야 합니다. "모든 것을 배치 처리한 후 디코딩"하는 방식은 허용되지 않습니다.

## 개념

### 사고자(Thinker)와 화자(Talker)

Qwen2.5-Omni의 구성 요소:

- 사고자(Thinker): 7B-80B 규모의 텍스트 생성 트랜스포머. 텍스트, 이미지, 오디오 토큰을 인터리빙하여 처리. 말할 내용을 나타내는 텍스트 토큰을 출력.
- 화자(Talker): 더 작은 음성 생성 트랜스포머(200M-1B). 사고자의 텍스트 출력 토큰과 최근 음성 컨텍스트 토큰을 소비. 이산 음성 토큰(잔차-VQ 인덱스)을 출력.
- 음성 디코더: 스트리밍 웨이브폼 디코더(SNAC, MoVQGAN 계열). 음성 토큰을 실시간으로 오디오 샘플로 변환.

분리가 중요한 이유. 사고자는 추론을 위해 커야 함. 화자는 텍스트를 음성 토큰으로 변환하는 지역적 작업이므로 작아도 됨. 더 큰 화자는 표현력이 증가하지 않으며, 오히려 느려짐.

병렬 실행 방식:

1. 사고자가 텍스트 토큰 t_i를 출력.
2. 화자가 t_i를 소비(스트리밍) 후 음성 토큰 s_i, s_{i+1}, ..., s_{i+k}를 출력.
3. 음성 디코더가 음성 토큰을 수신 즉시 오디오 샘플을 출력.
4. 사고자가 t_{i+3}에 도달할 때, 화자는 이미 t_0..t_{i+2}에 대한 오디오를 스트리밍 완료.

### TMRoPE — 시간 정렬된 멀티모달 위치

사고자는 이미지 프레임(예: 4FPS), 오디오 프레임(50프레임/초), 대화 기록의 텍스트를 통합해야 함. 순진한 시퀀스 순서(모든 이미지 → 모든 오디오 → 텍스트)는 시간적 정렬을 잃음.

TMRoPE는 모든 토큰에 절대 타임스탬프를 할당. t=2.3초의 비전 토큰, t=2.32초의 오디오 토큰, t=2.35초의 사용자 텍스트 토큰 "stop". RoPE는 타임스탬프로 어텐션을 회전시켜 모델이 이들을 시간적으로 동시에 인식.

이는 "그는 손을 흔들며 '안녕'이라고 말했다"와 같은 시나리오가 작동하게 하는 인프라. 모델은 동일한 개념적 순간에 비디오 프레임과 오디오를 인식.

### 스트리밍 음성 합성

음성 토큰은 스트리밍되어야 함. Mini-Omni(Xie & Wu, 2024)는 "언어 모델이 스트리밍 중 듣고 말할 수 있음"을 도입: 사고자 출력 토큰과 화자 출력 토큰이 동일 시퀀스에서 인터리빙. 사고자가 다음 텍스트 토큰을 확정하면 즉시 화자가 발화. 배치 경계 없음.

Moshi(Défossez et al., 2024년 10월)는 가장 빠른 오픈소스 구현. 단일 A100에서 160ms TTFAB. 아키텍처: 텍스트와 음성 토큰을 번갈아 출력하는 단일 7B 트랜스포머. "내부 독백"을 통해 사고 스트림과 발화 스트림을 분리. 이는 신중한 훈련으로 사고자 + 화자를 하나의 모델로 융합한 효과.

### VAD 및 대화 차례

입력 측에서 음성 활동 감지(VAD)가 실행. 두 가지 패턴:

- 반이중(Half-duplex): 사용자가 말하면 모델이 듣고, 모델이 말하면 사용자가 들음. VAD 무음 감지(~200ms)를 통한 명확한 핸드오프.
- 전이중(Full-duplex): 동시에 말할 수 있음. 모델이 백채널("응")이나 중단 가능. 훨씬 어려움. Moshi는 이를 지원.

Qwen2.5-Omni는 기본적으로 반이중 모드를 지원하며, 무음 임계값을 통한 대화 차례 관리. 전이중 모드는 애플리케이션 계층 처리 필요.

### Qwen3-Omni(2025년 11월)

후속 모델. Qwen3-80B 사고자, 더 큰 화자, 개선된 TMRoPE-v2. GPT-4o의 250ms에 근접한 지연 시간. 오픈 웨이트. OmniBench에서 Gemini 2.0 Live와 경쟁력 있는 벤치마크.

### 프로덕션 지연 시간 예산

일반적인 스트리밍 상호작용 시:

- 마이크 → 오디오 토큰: 40-80ms.
- 프리필(프롬프트 + 기록): 7B에서 100-200ms, 70B에서 훨씬 더 오래.
- 첫 사고자 텍스트 토큰: 40ms.
- 화자가 첫 텍스트 토큰 처리: 20ms.
- 첫 음성 토큰 확정: 40ms.
- 잔차-VQ 디코딩: 30ms.
- 음성 웨이브폼 디코딩: 50-80ms.

총 TTFAB: 7B에서 320-510ms, 70B에서 600-900ms. 프론티어 품질은 일반적으로 70B+를 의미하므로, 프론티어 지연 시간 격차가 발생.

### 토큰 생성률 계산

16kHz 음성에서 50Hz 기본 음성 토큰을 사용할 때, 출력 1초당 50개의 음성 토큰이 필요. 화자는 초당 ≥50토큰을 생성해야 함. H100에서 일반적인 LLM 처리량(30-80토큰/초)을 고려할 때, 작은(200-300M) 화자는 충분히 빠름. 7B 화자는 뒤처질 수 있음.

이것이 "주 모델을 사용하는 것" 대신 작은 전용 화자 모델이 존재하는 이유.

## 사용 방법

`code/main.py`:

- 모의 토큰 방출 속도를 사용한 Thinker-Talker 파이프라인 시뮬레이션.
- 구성 가능한 모델 크기 및 마이크 샘플 레이트에 대한 TTFAB(Thinker-to-Talker First Audio Byte) 계산.
- VAD(Voice Activity Detection) 무음 임계값을 활용한 반이중(half-duplex) 턴테이킹(turn-taking) 시연.

## Ship It

이 레슨은 `outputs/skill-omni-streaming-budget.md`를 생성합니다. 실시간 음성 제품의 목표 TTFAB(Target Time to First Actionable Byte)와 기능 세트(시각 입력, 이중 언어, 전이중 통신)를 고려하여 Qwen2.5-Omni, Qwen3-Omni, Moshi, Mini-Omni 중 적절한 모델을 선택하고 Thinker/Talker 크기를 조정합니다.

## 연습 문제

1. 목표 TTFAB(Time To First Audio Byte)는 300ms입니다. 7B Thinker와 300M Talker에서 각 구성 요소의 지연 시간을 작성하세요.

2. Qwen2.5-Omni는 TMRoPE를 사용합니다. 사용자가 t=1초에 말을 시작하고 카메라가 t=1.2초에 제스처를 포착하는 프롬프트에서 모델이 인식하는 내용을 설명하세요.

3. 전이중(full-duplex) 지원을 위해 모델이 청취 중에 오디오를 출력해야 합니다. 이를 학습하는 훈련 데이터 형식을 제안하세요.

4. Moshi 논문 4장을 읽고 "내적 독백(inner monologue)" 분리 방식을 설명하세요. 그리고 왜 이 방식이 Thinker-Talker 분리를 피하는지 설명하세요.

5. 처리량 예산(throughput budget)을 계산하세요: 16kHz 음성에서 50 base-layer tokens/sec를 유지하려면 Talker가 얼마나 빠르게 토큰을 출력해야 하나요?  

> **참고**:  
> - TTFAB(Time To First Audio Byte): 첫 오디오 바이트 생성까지의 시간  
> - TMRoPE: 회전 위치 임베딩(rotary position embedding) 변형  
> - 전이중(full-duplex): 동시 송수신 지원  
> - 내적 독백(inner monologue): 모델의 자기 반성 과정  
> - Base-layer tokens: 음성 인코딩의 기본 계층 토큰

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|-------------------|-----------|
| Thinker | "추론 뇌" | 무엇을 말할지 생성하는 대규모 텍스트 생성 트랜스포머 |
| Talker | "음성 생성 입" | Thinker의 텍스트를 기반으로 이산 음성 토큰을 생성하는 소형 트랜스포머 |
| TTFAB | "지연 예산" | 첫 오디오 바이트까지 시간(Time-to-first-audio-byte): 사용자 음성 종료부터 첫 오디오 샘플 출력까지 |
| TMRoPE | "시간 정렬 RoPE" | 영상, 오디오, 텍스트 전반에 걸친 절대 타임스탬프 사용 위치 인코딩 |
| 반이중(Half-duplex) | "턴 테이킹" | 사용자와 모델이 번갈아 수행; VAD 무음 감지로 사용자 종료 확인 |
| 전이중(Full-duplex) | "동시 진행" | 모델이 동시에 말하고 들을 수 있음; 백채널 가능 |
| 내적 독백(Inner monologue) | "모시 분리" | 사고 스트림과 발화 스트림이 교차하는 단일 모델 설계 |
