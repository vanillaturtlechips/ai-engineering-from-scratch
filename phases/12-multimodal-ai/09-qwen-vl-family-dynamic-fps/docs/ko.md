# Qwen-VL 패밀리와 Dynamic-FPS 비디오

> Qwen-VL 패밀리 — Qwen-VL (2023), Qwen2-VL (2024), Qwen2.5-VL (2025), Qwen3-VL (2025) — 는 2026년 가장 영향력 있는 오픈 비전-언어 모델 계보입니다. 각 세대는 오픈 생태계 전체가 12개월 이내에 복제한 결정적인 단일 아키텍처 베팅을 했습니다: M-RoPE를 통한 네이티브 동적 해상도, 절대 시간 정렬이 있는 dynamic-FPS 샘플링, ViT의 윈도우 어텐션, 구조화된 에이전트 출력 형식. Qwen3-VL에 이르러 레시피는 안정화되었습니다: 네이티브 종횡비 입력을 갖춘 2D-RoPE-ViT 인코더, 대규모 Qwen3 언어 기반으로의 MLP 프로젝터, OCR, 그라운딩, 에이전트 행동을 1급 목표로 강조한 훈련 단계. 이 레슨은 모든 조절 장치가 왜 현재 위치에 있는지 이해할 수 있도록 연대순으로 패밀리를 읽습니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, M-RoPE 인코더 + dynamic-FPS 샘플러)
**선수 지식:** Phase 12 · 06 (패치 앤 팩)
**소요 시간:** ~120분

## 학습 목표

- M-RoPE의 세 축(시간적(temporal), 높이(height), 너비(width)) 회전을 계산하고, 세 축이 모두 필요한 이유를 설명할 수 있다.
- 비디오에 대한 동적-FPS 샘플링 전략을 선택하고, 초당 토큰 수(tokens-per-second) 대 이벤트 감지 정확도(event-detection accuracy)를 고려하여 근거를 제시할 수 있다.
- Qwen-VL의 4가지 세대별 개선 사항을 순서대로 나열하고, 각각이 가능하게 한 기능을 설명할 수 있다.
- Qwen2.5-VL 스타일의 JSON 에이전트 출력 형식을 구성하고, VLM 응답에서 구조화된 도구 호출(tool calls)을 파싱할 수 있다.

## 문제 정의

Qwen-VL은 2023년 8월에 LLaVA-1.5와 BLIP-2에 대한 직접적인 대응으로 출시되었습니다. Qwen 팀이 목표로 한 격차는 세 가지였습니다: 해상도, 비디오, 구조화된 출력.

해상도: LLaVA-1.5는 336x336에서 실행되었습니다. 사진에는 적합했지만, 중문 인보이스나 빽빽한 스프레드시트 스크린샷에는 쓸모없었습니다. Qwen-VL의 첫 번째 혁신은 448x448 해상도와 바운딩 박스(bounding box) 출력으로, 모델이 사물을 가리킬 수 있게 했습니다.

비디오: Video-LLaMA는 프레임별 인코더를 쌓고 이를 LLM에 공급했습니다. 짧은 클립에는 작동했지만, 시간 축이 신호인 여러 분 길이의 비디오에는 적합하지 않았습니다. Qwen 팀은 시간을 이해하는 단일 인코더를 원했습니다.

구조화된 출력: LLaVA는 자유 형식의 텍스트를 출력했습니다. 에이전트는 JSON이 필요합니다. Qwen-VL은 바운딩 박스 좌표를 텍스트로 포함하는 명시적인 JSON 출력 형식으로 학습되었습니다.

모든 Qwen-VL 생성은 이 세 가지 축 중 하나를 확장합니다.

## 개념

### Qwen-VL (2023년 8월)

1세대: 인코더로 OpenCLIP ViT-bigG/14 (2.5B 파라미터), LLama 호환 Q-Former (256개 쿼리로 1단계 처리), Qwen-7B 기반. 주요 기여:

- 448x448 해상도 (당시 오픈소스 VLM 최고 성능).
- 그라운딩: 명시적 좌표-토큰 출력으로 이미지-텍스트 쌍 학습. "고양이는 <box>(112, 204), (280, 344)</box>에 있습니다".
- 처음부터 중문+영문 다국어 학습.

당시 벤치마크: 영어에서는 GPT-4V와 경쟁, 중문에서는 압도적 성능. 그라운딩 감독 학습이 주요 혁신.

### Qwen2-VL (2024년 9월) — M-RoPE 및 네이티브 해상도

Qwen2-VL은 고정 해상도 + Q-Former 스택을 네이티브 동적 해상도 ViT 인코더로 대체. 주요 변경점:

- 네이티브 동적 해상도. ViT는 28로 나누어지는 모든 HxW(패치 14에 2x 공간 병합)를 수용. 1120x672(40x24 병합 패치) 이미지는 960개 시각 토큰 생성. 리사이즈, 타일링, 썸네일 불필요.
- M-RoPE(Multimodal RoPE). 각 토큰은 1D 대신 3D 위치(t, h, w) 보유. 이미지의 경우 t=0, 비디오의 경우 t=프레임 인덱스. RoPE는 축별 주파수로 쿼리/키 벡터 회전. 위치 임베딩 테이블 불필요.
- MLP 프로젝터. Q-Former 제거; 병합된 패치 토큰에 2층 MLP 사용.
- 동적 FPS 비디오. 기본 1-2 FPS 샘플링이지만 임의의 프레임 수 수용.

결과: Qwen2-VL-7B는 여러 멀티모달 벤치마크에서 GPT-4o와 동급, DocVQA에서 94.5 vs 88.4로 승리. 아키텍처 변경이 결정적.

### Qwen2.5-VL (2025년 2월) — 동적 FPS + 절대 시간

Qwen2.5-VL의 큰 변화는 비디오 처리: 동적 FPS는 단순히 "필요 시 프레임 증가"가 아님. 논문에서 공식화:

- 절대 시간 토큰. 위치 인덱스(프레임 0, 1, 2...) 대신 실제 타임스탬프 사용. "0:04에 고양이가 점프합니다." 모델은 프레임 토큰과 `<time>0.04</time>` 토큰을 함께 처리.
- 동적 FPS. 느린 영상은 1 FPS, 액션 장면은 4+ FPS 샘플링. 사용자 또는 트레이너가 선택; M-RoPE가 적응.
- ViT의 윈도우 어텐션. 처리량 향상을 위해 공간 어텐션은 블록 내 로컬, 몇 층마다 글로벌 어텐션.
- 명시적 JSON 출력 형식. 툴 호출 데이터 학습: "{\"tool\": \"click\", \"coords\": [380, 220]}". 즉시 에이전트 준비 완료.
- MRoPE-v2 확장. 위치가 최대 입력 크기에 비례해 10분 비디오도 주파수 범위 초과 없음.

벤치마크: Qwen2.5-VL-72B는 대부분의 비디오 벤치마크에서 GPT-4o를 능가, 문서 처리에서 Gemini 2.0과 동급, GUI 그라운딩(ScreenSpot: 84% 정확도 vs GPT-4o 38%)에서 오픈소스 모델 SOTA.

### Qwen3-VL (2025년 11월)

Qwen3-VL은 재설계보다 통합에 중점을 둔 점진적 업그레이드: 더 큰 LLM 백본(Qwen3-72B), 확장된 학습 데이터, 개선된 OCR, Qwen3 "사고 모드"를 통한 강화된 추론. ViT와 M-RoPE는 유지. 논문은 아키텍처 대신 데이터 및 학습 개선에 집중.

계보 교훈: 2025년까지 Qwen-VL 아키텍처는 안정화. 이후 세대는 계산 자원과 데이터 규모를 확장할 뿐, 기본 요소는 변경되지 않음.

### M-RoPE 수학적 설명

기존 RoPE는 차원 `d`의 쿼리 `q`를 위치 `m`으로 회전하며 좌표 쌍을 사용:

```
q_rot[2i]   = q[2i]   * cos(m * theta_i) - q[2i+1] * sin(m * theta_i)
q_rot[2i+1] = q[2i]   * sin(m * theta_i) + q[2i+1] * cos(m * theta_i)
theta_i     = 10000^(-2i/d)
```

M-RoPE는 은닉 차원을 3개 대역으로 분할. 예를 들어 `d = 96`일 때 시간, 높이, 너비에 각각 32차원 할당. 각 대역은 자체 축 위치로 회전. (t=5, h=10, w=20) 패치는 3개 대역에 `R_t(5)`, `R_h(10)`, `R_w(20)` 적용.

텍스트 토큰은 `t = 텍스트 인덱스, h = 0, w = 0`(또는 정규화 선택) 사용, 호환성 유지. 비디오 프레임은 `t = 프레임 시간, h = 행, w = 열` 사용. 단일 이미지는 `t = 0` 사용.

장점: 하나의 위치 인코딩으로 텍스트, 이미지, 비디오를 분기 코드나 별도 위치 테이블 없이 처리.

### 동적-FPS 샘플링 로직

총 길이 `T`초 비디오와 목표 토큰 예산 `B`가 주어졌을 때:

1. 최대 FPS 계산: `fps_max = B / (T * 프레임당 토큰 수)`.
2. `fps <= fps_max`를 만족하는 `{1, 2, 4, 8}` 중 목표 FPS 선택.
3. 모션이 높은 경우(광학 흐름 휴리스틱 또는 명시적 사용자 요청) 높은 FPS 선택. 모션이 낮으면 낮은 FPS 선택.
4. 선택한 FPS로 균일 샘플링; 프레임 사이에 `<time>t</time>` 토큰 삽입.

Qwen2.5-VL은 이 로직을 암묵적으로 학습. 추론 시 사용자는 `fps` 파라미터로 제어. 60초 액션 시퀀스를 4 FPS로 샘플링하고 프레임당 81토큰 = 19440토큰, 32k 컨텍스트에서 처리 가능.

### 구조화된 에이전트 출력

Qwen2.5-VL의 에이전트 학습은 구조화된 툴 호출을 명시적으로 대상:

```
{
  "tool": "mouse_click",
  "coords": [1024, 512],
  "button": "left",
  "modifier": null
}
```

파싱은 결정적: 모델 출력에 JSON.parse 적용. "1024, 512 좌표 클릭"과 같은 자유 형식 출력은 정규식 및 모호성 처리 필요. 이 변화로 Qwen2.5-VL의 ScreenSpot 점수는 Qwen2-VL의 55%에서 84%로 급상승.

## 사용 방법

`code/main.py`는 다음을 구현합니다:

- 텍스트, 이미지 패치, 비디오 프레임을 혼합한 패킹된 시퀀스에 대한 M-RoPE 위치 계산
- 동적-FPS 샘플러: (duration, budget, motion_level)이 주어졌을 때 FPS를 선택하고 프레임 타임스탬프를 출력
- 좌표 필드가 포함된 도구 호출 응답을 처리하는 장난감 Qwen2.5-VL JSON 출력 파서

실행한 후, 5분짜리 비디오에서 고정-FPS와 동적-FPS를 교체했을 때 차이를 느껴보세요.

## 연습 문제

1. (t=3, h=5, w=7) 위치의 패치에 대해 M-RoPE(Multi-Resolution Rotary Position Embedding) 회전을 계산하시오. 히든 차원 48(밴드당 16, 기본 세타 10000)을 사용할 때, 각 밴드에서 첫 세 쌍의 회전 각도를 제시하시오.

2. 1FPS로 촬영된 10분 분량의 보안 카메라 영상은 몇 프레임인가? 해상도 384에 3x 풀링을 적용할 때 총 토큰 수는? Qwen2.5-VL의 기본 32k 컨텍스트는 이를 처리할 수 있는가?

3. 30초 테니스 경기, 30초 요리 데모, 30초 UI-에이전트 녹화에 적합한 FPS를 선택하시오. 동적 FPS 논리로 각각을 설명하시오.

4. Qwen2.5-VL은 Q-Former를 완전히 제거했다. 2023년에는 불가능했지만 2025년에는 간단한 MLP가 작동하는 이유는? (힌트: 데이터 규모와 인코더 품질)

5. Qwen2.5-VL JSON 툴콜 출력 3개를 파이썬 딕셔너리로 파싱하시오. 잘못된 JSON에서 실패하는 경우와 Qwen 쿡북이 권장하는 복구 전략은 무엇인가?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|---------------------|----------|
| M-RoPE | "멀티모달 RoPE" | 시간, 높이, 너비 대역이 숨겨진 차원에 있는 3D 회전 위치 임베딩 |
| Dynamic FPS | "스마트 샘플링" | 움직임, 지속 시간, 토큰 예산을 기반으로 비디오마다 선택된 프레임 샘플링 속도 |
| Absolute time token | "타임스탬프 토큰" | `<time>t</time>`가 시퀀스에 인터리빙되어 모델이 프레임 인덱스가 아닌 실제 초를 인식하도록 함 |
| Window attention | "로컬 어텐션" | 속도를 위해 작은 창으로 제한된 공간적 자기 어텐션; 주기적으로 글로벌 어텐션 추가 |
| Structured agent output | "JSON 모드" | VLM이 좌표와 도구 이름을 포함한 파싱 가능한 JSON을 출력하도록 가르치는 훈련 데이터 감독 |
| min_pixels / max_pixels | "해상도 경계" | 요청당 Qwen2.5-VL이 총 픽셀 수 및 따라서 토큰 수를 제어 |
| Grounding | "포인트-앳-잇" | 텍스트 토큰으로 바운딩 박스 좌표 출력; Qwen-VL v1부터 사용됨 |
