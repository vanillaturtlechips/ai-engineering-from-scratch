# 문서 및 다이어그램 이해

> 문서는 사진이 아닙니다. PDF, 과학 논문, 송장, 또는 손으로 작성한 양식에는 레이아웃, 표, 다이어그램, 각주, 헤더, 그리고 일반 이미지 이해로는 포착할 수 없는 의미 구조가 있습니다. VLM(Visual Language Model) 이전 스택은 파이프라인이었습니다: Tesseract OCR + LayoutLMv3 + 표 추출 휴리스틱. VLM 물결은 OCR이 없는 모델(Donut(2022), Nougat(2023), DocLLM(2023))로 이를 대체했으며, 이 모델들은 구조화된 마크업을 직접 출력합니다. 2026년에는 "2576px 네이티브 해상도의 Claude Opus 4.7에 페이지 이미지를 입력하면" 구조화된 마크업 출력이 무료로 제공되는 것이 최전선입니다. 이 레슨은 문서 AI의 3시대 흐름을 읽습니다.

**유형:** 구축(Build)  
**언어:** Python (표준 라이브러리, 레이아웃 인식 문서 파서 스켈레톤)  
**선수 지식:** 12단계 · 05 (LLaVA), 5단계 (NLP)  
**소요 시간:** ~180분

## 학습 목표

- 문서 AI의 세 가지 시대(OCR 파이프라인, OCR-Free, VLM-Native)를 설명할 수 있다.
- LayoutLMv3의 세 가지 입력 스트림(텍스트, 레이아웃(bbox), 이미지 패치)과 통합 마스킹을 설명할 수 있다.
- Donut(OCR-Free, 이미지 → 마크업), Nougat(과학 논문 → LaTeX), DocLLM(레이아웃 인식 생성), PaliGemma 2(VLM-Native)를 비교할 수 있다.
- 새로운 작업(인보이스, 과학 논문, 손글씨 양식, 중국어 영수증)에 적합한 문서 모델을 선택할 수 있다.

## 문제 정의

"PDF 이해"는 겉보기보다 훨씬 어렵습니다. 정보는 다음 요소들에 분산되어 있습니다:

- 텍스트 내용(신호의 90%).
- 레이아웃(헤더, 각주, 사이드바, 두 열 형식).
- 표(행, 열, 병합된 셀).
- 그림 및 다이어그램.
- 필기 주석.
- 폰트 및 타이포그래피(제목 vs 본문).

원시 OCR은 텍스트만 추출하고 나머지는 손실됩니다. 인보이스에 중점을 둔 시스템은 "Total: $1,245"가 각주가 아닌 오른쪽 하단에서 추출되었음을 알아야 합니다.

## 개념

### 시대 1 — OCR 파이프라인 (2021년 이전)

클래식 스택:

1. PDF → 페이지당 이미지.
2. Tesseract(또는 상용 OCR)이 단어별 경계 상자와 함께 텍스트를 추출.
3. 레이아웃 분석기가 블록(헤더, 표, 단락)을 식별.
4. 표 구조 인식기가 표를 파싱.
5. 도메인 규칙 + 정규 표현식이 필드를 추출.

깨끗한 인쇄 텍스트에는 작동. 필기, 기울어진 스캔, 복잡한 표, 비영어 문자에서는 실패. 모든 실패 모드에는 사용자 정의 예외 경로가 필요.

### TrOCR (2021)

TrOCR (Li et al., arXiv:2109.10282)은 Tesseract의 클래식 CNN-CTC를 합성 + 실제 텍스트 이미지로 훈련된 트랜스포머 인코더-디코더로 대체. 필기 및 다국어 텍스트에서 깨끗한 승리. 여전히 파이프라인(검출기 → TrOCR → 레이아웃)이지만 OCR 단계가 크게 개선.

### 시대 2 — OCR-프리 (2022-2023)

첫 OCR-프리 모델: 검출을 완전히 건너뛰고 이미지 픽셀을 구조화된 출력에 직접 매핑.

Donut (Kim et al., arXiv:2111.15664):
- 인코더-디코더 트랜스포머, 인코더는 Swin-B.
- 출력은 양식 이해를 위한 JSON, 요약을 위한 마크다운 또는 작업별 스키마.
- OCR, 레이아웃, 검출 없음.

Nougat (Blecher et al., arXiv:2308.13418):
- 과학 논문에 특화되어 훈련.
- 출력은 LaTeX/마크다운.
- 방정식, 다단 레이아웃, 그림 처리.
- 모든 arXiv-파서가 호출하는 모델.

이들은 전문가이지 일반주의자가 아님. 과학 논문에서 Donut은 실패, 송장에서 Nougat은 실패.

### LayoutLMv3 (2022)

다른 접근법. LayoutLMv3 (Huang et al., arXiv:2204.08387)는 OCR을 유지하지만 레이아웃 이해를 추가:

- 세 가지 입력 스트림: OCR 텍스트 토큰, 토큰별 2D 경계 상자, 이미지 패치.
- 세 가지 모달리티(마스킹된 텍스트, 마스킹된 패치, 마스킹된 레이아웃)에 걸친 마스킹 훈련 목표.
- 다운스트림: 분류, 엔티티 추출, 표 QA.

LayoutLMv3는 OCR 기반 문서 이해의 정점. 양식 및 송장에서 강력. 업스트림 OCR 필요. 표준화된 문서 벤치마크에서 VLM 이전 최고 정확도.

### DocLLM (2023)

DocLLM (Wang et al., arXiv:2401.00908)은 LayoutLM의 생성형 형제. 레이아웃 토큰에 조건화된 자유 형식 답변 생성. 문서 QA에 더 적합; 여전히 OCR 입력에 의존.

### 시대 3 — VLM-네이티브 (2024+)

2024 VLM은 파이프라인 전체를 대체할 만큼 충분히 좋아짐. 고해상도 전체 페이지 이미지를 VLM에 입력하고 질문을 하면 답변을 얻음.

- LLaVA-NeXT 336-tile AnyRes는 작은 문서에 작동.
- Qwen2.5-VL 동적 해상도는 2048+ 픽셀을 네이티브로 처리.
- Claude Opus 4.7은 2576px 문서를 지원.
- PaliGemma 2 (2025년 4월)는 문서 + 필기에 특화 훈련.

VLM-네이티브와 OCR-파이프라인 간 격차는 빠르게 좁혀짐. 2026년까지 VLM-네이티브가 승리하는 분야:

- 장면 텍스트(필기 + 인쇄, 혼합 문자).
- 병합 셀이 있는 복잡한 표.
- 텍스트에 포함된 수학 방정식.
- 텍스트 주석이 있는 그림.

OCR 파이프라인이 여전히 승리하는 분야:

- 페이지당 지연 시간이 중요한 대규모 순수 스캔 작업.
- 파이프라인 신뢰성(결정적 실패 vs VLM 환각).
- 감사 가능한 OCR 출력이 필요한 규제 환경.

### Claude 4.7 / GPT-5 프론티어

2576픽셀 네이티브 입력에서 프론티어 VLM은 인간 수준의 정확도로 문서 이해. 2026년 초 벤치마크:

- DocVQA: Claude 4.7 ~95.1, PaliGemma 2 ~88.4, Nougat ~77.3, 파이프라인 LayoutLMv3 ~83.
- ChartQA: Claude 4.7 ~92.2, GPT-4V ~78.
- VisualMRC: Claude 4.7 ~94.

폐쇄형 모델 격차는 주로 해상도와 기본 LLM 규모. 7B 오픈 모델은 몇 점 뒤지만 따라잡는 중.

### 수학 방정식 및 LaTeX 출력

과학 논문은 방정식에 정확한 LaTeX 출력이 필요. Nougat은 이에 훈련됨. LaTeX 목표(Qwen2.5-VL-Math, Nougat 파생)로 훈련된 VLM은 사용 가능한 LaTeX를 생성. 명시적 LaTeX 훈련 없이는 VLM이 읽기 가능하지만 부정확한 전사본을 생성.

2026년 과학 논문 파이프라인: PDF에 Nougat을 연결한 후 까다로운 페이지에 VLM 적용.

### 필기

여전히 가장 어려운 하위 작업. 혼합 인쇄 + 필기(의사 노트, 작성 양식)는 OCR 파이프라인이 비용 면에서 VLM을 앞섬. 필기 전용 VLM은 개선 중(Claude 4.7, PaliGemma 2).

### 2026년 레시피

새로운 문서-AI 프로젝트:

- 대규모 순수 인쇄 송장: LayoutLMv3 + 규칙, 비용 효율적.
- 혼합 문서(과학 + 필기 + 양식): VLM-네이티브(PaliGemma 2 또는 Qwen2.5-VL).
- 전체 arXiv 수집: 수학용 Nougat, 그림용 VLM.
- 규제: OCR 파이프라인 + VLM 검증기로 교차 확인.

## 사용 방법

`code/main.py`:

- 장난감 레이아웃 인식 토크나이저: (텍스트, 바운딩 박스) 쌍을 입력으로 받아 LayoutLMv3 스타일의 입력을 생성합니다.
- Donut 스타일의 작업 스키마 생성기: 폼을 위한 JSON 템플릿입니다.
- OCR 파이프라인, Donut, Nougat, VLM 네이티브 간 페이지당 토큰 예산 비교 분석.

## Ship It

이 레슨은 `outputs/skill-document-ai-stack-picker.md`를 생성합니다. 문서-AI 프로젝트(도메인, 규모, 품질, 규제 요건)가 주어졌을 때, OCR 파이프라인, OCR-프리 전문가 모델, VLM-네이티브 방식 중 적절한 접근 방식을 선택하는 방법을 다룹니다.

## 연습 문제

1. 프로젝트에 하루 10M(1,000만) 건의 인보이스(송장)가 있습니다. 정확도를 잃지 않으면서 페이지당 비용을 최소화하는 기술 스택은 무엇인가요?

2. LayoutLMv3가 양식 QA에서는 순수-CLIP-VLM(Visual-Language Model)을 능가하지만 장면 텍스트(scene-text)에서는 성능이 낮은 이유는 무엇인가요? 바운딩 박스(bbox) 스트림이 포기하는 것은 무엇인가요?

3. Nougat은 LaTeX을 생성합니다. VLM-네이티브 출력이 LaTeX 충실도(fidelity)에서 Nougat을 능가하는 테스트 케이스와, Nougat이 승리하는 케이스를 제안하세요.

4. PaliGemma 2 논문(Google, 2024)을 읽으세요. PaliGemma 1 대비 문서 정확도를 끌어올린 핵심 훈련 데이터 추가는 무엇이었나요?

5. 규제 준수 하이브리드 시스템을 설계하세요: OCR 파이프라인을 주력으로, VLM을 보조 검증 수단으로 활용할 때, 의견 불일치(disagreement)를 어떻게 해결할 것인가요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|---------------------|-----------|
| OCR 파이프라인 | "Tesseract-style" | 단계별 스택: 감지 -> OCR -> 레이아웃 -> 규칙; 결정론적, 취약함 |
| OCR-프리 | "Donut-style" | 명시적 OCR을 생략하는 이미지-출력 변환기; 단일 모델 |
| 레이아웃 인식 | "LayoutLM" | 입력에 토큰별 바운딩 박스 좌표 포함; 모달리티 간 통합 마스킹 |
| VLM-네이티브 | "Frontier VLM" | 고해상도 페이지 이미지를 Claude/GPT/Qwen VLM에 직접 입력; 파이프라인 없음 |
| DocVQA | "Doc 벤치마크" | 문서 VQA 표준; 가장 많이 인용된 점수 |
| 마크업 출력 | "LaTeX / MD" | 자유 형식 텍스트 대신 구조화된 출력 형식; 다운스트림 자동화 가능 |
