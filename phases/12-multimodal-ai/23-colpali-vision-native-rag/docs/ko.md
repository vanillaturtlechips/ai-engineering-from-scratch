# ColPali와 Vision-Native 문서 RAG

> 전통적인 RAG는 PDF를 텍스트로 파싱하고, 청크로 분할한 후 임베딩하고 벡터 저장소를 구축합니다. 모든 단계에서 신호 손실이 발생합니다: OCR은 차트 데이터를 누락시키고, 청킹(chunking)은 표 행을 분리하며, 텍스트 임베딩은 그림을 무시합니다. ColPali(Faysse et al., 2024년 7월)는 더 간단한 질문을 던졌습니다: "왜 텍스트를 추출해야 하는가?" PaliGemma를 통해 페이지 이미지를 직접 임베딩하고, ColBERT 스타일의 후기 상호작용(late interaction)을 검색에 활용하며, 문서가 지닌 레이아웃, 그림, 폰트, 서식 신호를 모두 보존합니다. 공개된 벤치마크 결과: 시각적으로 풍부한 문서에서 텍스트 기반 RAG보다 20-40% 더 높은 종단간(end-to-end) 정확도를 달성했습니다. ColQwen2, ColSmol, VisRAG가 이 패턴을 확장했습니다. 이 레슨은 vision-native RAG 논문을 읽고 작은 ColPali 유사 인덱서를 구축합니다.

**유형:** 구축(Build)  
**언어:** Python (표준 라이브러리, 멀티 벡터 인덱서 + MaxSim 스코어러)  
**선수 지식:** 11단계(LLM 엔지니어링 — RAG 기초), 12단계·05(LLaVA)  
**소요 시간:** ~180분

## 학습 목표

- 양방향 인코더 검색(문서당 하나의 벡터)과 지연 상호작용 검색(문서당 여러 벡터)의 차이점을 설명합니다.
- ColBERT의 MaxSim 연산과 ColPali가 이를 텍스트 토큰에서 이미지 패치로 일반화하는 방식을 설명합니다.
- 소규모 ColPali 유사 인덱서 구축: 페이지 → 패치 임베딩 → 쿼리-용어 임베딩에 대한 MaxSim → 상위-k 페이지.
- 인보이스/재무 보고서 사용 사례에서 ColPali + Qwen2.5-VL 생성기 vs 텍스트-RAG + GPT-4를 비교합니다.

## 문제

PDF에 대한 텍스트 기반 RAG(Retrieval-Augmented Generation)는 문서의 대부분을 버립니다. 재무 보고서의 3분기 매출 성장률은 일반적으로 차트에 표시되고, 의료 보고서의 결과는 주석이 달린 이미지에 있으며, 법적 계약의 서명 블록은 텍스트 사실이 아닌 레이아웃 사실입니다.

텍스트 기반 RAG 파이프라인:

1. PDF → OCR/ pdftotext를 통한 텍스트 추출.
2. 텍스트 → 300-500 토큰 청크 분할.
3. 청크 → 바이-인코더 임베딩(하나의 벡터).
4. 사용자 쿼리 → 임베딩 → 코사인 유사도 → 상위-k 청크.
5. 청크 + 쿼리 → LLM.

5단계의 손실 발생 과정. 차트는 포착되지 않음. 테이블은 청크 간에 분할됨. 다단 레이아웃은 평탄화됨. 그림 주석은 사라짐.

ColPali의 해결책: OCR을 건너뛰고 페이지 이미지를 직접 임베딩. ColBERT 스타일의 지연 상호작용을 검색에 사용하여 모델이 쿼리 시 세부 패치에 주의를 기울일 수 있도록 함.

## 개념

### ColBERT (2020)

ColBERT (Khattab & Zaharia, arXiv:2004.12832)는 텍스트 검색 방법입니다. 문서당 하나의 벡터 대신 토큰당 하나의 벡터를 생성합니다. 쿼리 시:

- 쿼리 토큰은 자체 임베딩을 얻습니다 (N_q 벡터).
- 문서 토큰은 임베딩을 얻습니다 (일반적으로 캐시되는 N_d 벡터).
- 점수 = 쿼리 토큰에 대한 문서 토큰의 코사인 유사도 최대값 합계: Σ_i max_j cos(q_i, d_j).

이것은 MaxSim 연산입니다. 각 쿼리 토큰은 가장 잘 일치하는 문서 토큰을 "선택"합니다. 최종 점수는 합계입니다.

장점: 강력한 재현율, 용어 수준 의미 처리. 단점: 문서당 N_d 벡터, 저장 비용 고가.

### ColPali

ColPali (Faysse et al., arXiv:2407.01449)는 ColBERT 패턴을 이미지에 적용합니다.

- 각 페이지는 PaliGemma (ViT + 언어)로 패치 임베딩으로 인코딩됩니다: 페이지당 N_p 벡터.
- 각 사용자 쿼리(텍스트)는 쿼리 토큰 임베딩으로 인코딩됩니다: N_q 벡터.
- 점수 = Σ_i max_j cos(q_i, p_j), 즉 쿼리 텍스트 토큰과 페이지 이미지 패치 간 MaxSim.
- 총점 기준으로 상위 k개 페이지를 검색합니다.

문서 수집 시: PaliGemma로 모든 페이지를 임베딩하고 모든 패치 임베딩을 저장합니다. 쿼리 시: 쿼리 토큰을 임베딩하고 저장된 모든 페이지 임베딩에 대해 MaxSim을 계산한 후 상위 k개 페이지를 반환합니다.

장점: 시각적으로 풍부한 문서에서 텍스트 RAG보다 20-40% 향상된 종단간 성능. 각 패치 벡터는 로컬 레이아웃과 콘텐츠를 캡처합니다.

단점: 페이지당 N_p 패치 × 4바이트 부동소수점 × D차원 벡터 = 저장 공간이 빠르게 증가. PQ/OPQ 양자화로 완화.

### ColQwen2 및 ColSmol

ColQwen2 (illuin-tech, 2024-2025)는 PaliGemma를 Qwen2-VL로 교체합니다. 더 나은 기본 인코더, 더 나은 검색.

ColSmol은 로컬/엣지 사용을 위한 소규모 변형입니다. ~1B 파라미터의 ColSmol 검색기는 소비자 GPU에서 실행됩니다.

### VisRAG

VisRAG (Yu et al., arXiv:2410.10594)는 다른 변형입니다: 패치에 대한 MaxSim 대신 VLM으로 각 페이지를 단일 벡터로 풀링한 후 바이인코더 검색. 더 빠른 인덱싱 + 더 작은 저장 공간, 더 약한 재현율.

품질 대 비용 트레이드오프: 품질은 ColPali, 규모는 VisRAG.

### M3DocRAG

M3DocRAG (Cho et al., arXiv:2411.04952)는 멀티모달 검색을 다중 페이지 다중 문서 추론으로 확장합니다. 문서 간 페이지를 검색하고 VLM을 위한 다중 페이지 컨텍스트를 구성합니다.

### ViDoRe — 벤치마크

ColPali의 동반 벤치마크. 시각적 문서 검색 평가. 작업에는 재무 보고서, 과학 논문, 행정 문서, 의료 기록, 매뉴얼이 포함됩니다. 메트릭: nDCG@5.

ColPali-v1은 ViDoRe에서 ~80% nDCG@5를 기록하며, 동일 문서에 대한 텍스트 RAG는 ~50-60%를 기록합니다.

### 종단간 RAG 파이프라인

비전 네이티브 RAG의 경우:

1. 수집: PDF → 페이지 이미지 → PaliGemma 인코딩 → 모든 패치 임베딩 저장.
2. 쿼리: 사용자 텍스트 → 쿼리 토큰 임베딩 → 모든 인덱싱된 페이지에 대한 MaxSim → 상위 k개 페이지.
3. 생성: 상위 k개 페이지 이미지 + 쿼리 → VLM (Qwen2.5-VL 또는 Claude) → 답변.

어디에도 OCR이 없습니다. 그림, 차트, 글꼴, 레이아웃이 모두 답변에 반영됩니다.

### 저장 공간 계산

페이지당 729개 패치와 128차원 임베딩을 가진 50페이지 재무 보고서:

- ColPali: 50 * 729 * 128 * 4바이트 = ~18MB 원시, PQ 후 ~4MB.
- 텍스트 RAG: 50개 청크 * 768차원 * 4바이트 = ~150kB.

ColPali는 문서당 저장 공간이 ~30배 더 큽니다. 대규모에서는 OPQ/PQ로 ~5-10배까지 줄일 수 있으며, 일반적으로 허용 가능한 수준입니다.

### 텍스트 RAG가 여전히 우세한 경우

- 레이아웃 신호가 없는 순수 텍스트 문서 (위키 문서, 채팅 로그). 텍스트 RAG가 더 간단하고 저장 비용이 저렴합니다.
- 저장 공간이 비용을 지배하는 수백만 페이지 아카이브.
- 검색과 함께 추출 가능한 OCR 텍스트를 요구하는 엄격한 규제 요구사항.

2026년 기준 재무 보고서, 과학 논문, 법률 계약서, 의료 기록, UX 문서 등 그 외 모든 경우 — 비전 네이티브 RAG가 우세합니다.

## 사용 방법

`code/main.py`:

- Toy 패치 인코더: "페이지"(작은 특징 벡터 그리드)를 패치 임베딩 배열로 매핑합니다.
- MaxSim 점수 계산기: 쿼리 토큰 임베딩 집합과 페이지 패치 집합 간의 ColBERT 스타일 점수를 계산합니다.
- 5개의 장난감 페이지를 인덱싱하고, 3개의 쿼리를 실행한 후 점수가 포함된 상위 k개 결과를 반환합니다.

## 연습 문제

1. 200페이지 연간 보고서, 페이지당 729패치, 128차원 임베딩, 4바이트 부동소수점. 원시 저장 공간과 PQ-압축(8배) 저장 공간을 계산하시오.

2. MaxSim은 Σ_i max_j cos(q_i, p_j)입니다. 이 합이 단순한 평균 유사도가 포착하지 못하는 것은 무엇인가?

3. ColPali는 페이지를 패치 집합으로 인덱싱합니다. 대신 단어 수준(ColBERT 방식)으로 인덱싱하면 무엇이 달라지는가? 트레이드오프는?

4. 100만 페이지 코퍼스에 대해 쿼리당 500ms 지연 시간 예산으로 엔드투엔드 파이프라인을 설계하시오. ColQwen2/VisRAG 중 선택하고 근거를 제시하시오.

5. M3DocRAG(arXiv:2411.04952)를 읽고, 다중 페이지 어텐션 패턴과 단일 페이지 ColPali 검색과의 차이점을 설명하시오.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|---------------------|-----------|
| 늦은 상호작용 | "ColBERT-style" | 단일 문서 벡터가 아닌 토큰/패치 단위 임베딩 + MaxSim을 활용한 검색 |
| MaxSim | "Max-over-patches" | 각 쿼리 토큰에 대해 가장 유사도가 높은 문서 토큰을 선택; 쿼리 전체에 대해 합산 |
| 바이-인코더 | "Single-vector" | 문서당 하나의 벡터; 빠르지만 세부 정보 손실 발생 |
| 멀티-벡터 | "Many-vectors-per-doc" | 문서/페이지당 N_p개의 벡터 저장; 저장 비용은 증가하지만 재현율 향상 |
| 패치 임베딩 | "Page feature" | VLM 인코더에서 추출한 이미지 패치당 하나의 벡터; 페이지당 캐싱 |
| ViDoRe | "Vision doc bench" | 시각적 문서 검색을 위한 ColPali의 벤치마크 스위트 |
| PQ 양자화 | "Product quantization" | 벡터 유사도 유지하면서 저장 공간을 ~8배 축소하는 압축 기술 |
