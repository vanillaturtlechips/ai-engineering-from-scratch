# Janus-Pro: 통합 멀티모달 모델을 위한 분리된 인코더

> 통합 멀티모달 모델은 피할 수 없는 긴장 관계를 가지고 있습니다. 이해는 개념 수준의 풍부한 정보를 담은 SigLIP 또는 DINOv2 출력 벡터와 같은 의미론적 특징을 원합니다. 생성은 선명한 픽셀로 재구성할 수 있는 VQ 토큰과 같은 재구성 친화적인 코드를 원합니다. 단일 인코더에서는 이 두 목표가 호환되지 않습니다. Janus(DeepSeek, 2024년 10월)와 Janus-Pro(DeepSeek, 2025년 1월)는 해결책은 단념하는 것이라고 주장합니다: 두 인코더를 분리하세요. 태스크 간에 트랜스포머 본체를 공유하되, 이해는 SigLIP을 통해, 생성은 VQ 토크나이저를 통해 라우팅합니다. 7B 규모에서 Janus-Pro는 GenEval에서 DALL-E 3를 능가하면서 MMMU에서 LLaVA와 동등한 성능을 보입니다. 이 레슨은 왜 단일 인코더가 실패하는 곳에서 두 인코더가 작동하는지 설명합니다.

**유형:** 구축(Build)
**언어:** Python (표준 라이브러리, 이중 인코더 라우팅 + 공유 본체 신호)
**사전 요구 사항:** 12단계 · 13단계(Transfusion), 12단계 · 14단계(Show-o)
**소요 시간:** 약 120분

## 학습 목표

- 단일 공유 인코더가 이해 또는 생성 품질 중 하나를 희생하는 이유를 설명.
- Janus-Pro의 라우팅 설명: 이해를 위한 입력 측면의 SigLIP 특징, 생성을 위한 입력 및 출력의 VQ 토큰.
- Janus가 성공하지 못한 환경에서 Janus-Pro가 성공하는 데이터 혼합 확장 추적.
- 디커플드(Janus-Pro), 커플드-연속(Transfusion), 커플드-이산(Show-o) 아키텍처 비교.

## 문제 정의

통합 모델은 이해(understanding)와 생성(generation) 작업 간에 트랜스포머 본체(transformer body)를 공유합니다. 이전 시도(Chameleon, Show-o, Transfusion)는 모두 양방향 작업에 단일 시각 토크나이저를 사용했습니다. 이 토크나이저는 다음과 같은 절충안입니다:

- 재구성(생성) 최적화: VQ-VAE는 세부적인 픽셀 정보를 포착하지만 의미적 일관성이 약한 토큰을 생성합니다.
- 의미(이해) 최적화: SigLIP 임베딩은 "고양이" 이미지를 "고양이" 토큰 근처에 그룹화하지만 우수한 재구성을 허용하지 않습니다.

Show-o와 Transfusion은 이 절충안으로 인해 한 방향(understanding 또는 generation)에서 눈에 띄는 품질 저하를 겪습니다. Janus-Pro는 다음과 같은 질문을 던집니다: 작업마다 요구사항이 다른데 왜 단일 토크나이저를 강제해야 하는가?

## 개념

### 분리된 시각 인코딩

Janus-Pro의 아키텍처는 두 인코더를 분리합니다:

- 이해 경로. 입력 이미지 → SigLIP-SO400m → 2층 MLP → 트랜스포머 본체.
- 생성 경로. 입력 이미지(기존 이미지에 조건 부여 시) → VQ 토크나이저 → 토큰 ID → 트랜스포머 본체.
- 출력 생성. 트랜스포머가 예측한 이미지 토큰 → VQ 디코더 → 픽셀.

트랜스포머 본체는 공유됩니다. 본체의 상류와 하류 모든 요소는 작업별로 특화됩니다.

입력은 프롬프트 형식으로 구분됩니다: `<understand>` 태그는 SigLIP 경로로 라우팅되고, `<generate>`는 VQ 경로로 라우팅됩니다. 또는 라우팅은 작업에 따라 암시적으로 결정됩니다.

### 이 방식이 작동하는 이유

이해 손실은 SigLIP 특징을 사용하며, CLIP 스타일 사전 훈련은 의미적 유사성에 맞게 조정되었습니다. 모델의 인식 벤치마크는 Show-o / Transfusion보다 개선되었습니다. 입력 특징이 작업에 더 적합하기 때문입니다.

생성 손실은 VQ 토큰을 사용하며, 토크나이저는 재구성을 위해 조정되었습니다. 이미지 품질은 Show-o보다 개선되었습니다. VQ 코드가 픽셀로 깔끔하게 재구성되기 때문입니다.

공유된 트랜스포머 본체는 두 입력 분포(SigLIP 및 VQ)를 보고 둘 모두와 작동하도록 학습합니다. 주장: 충분한 데이터 + 충분한 파라미터가 있다면 본체는 전환을 흡수합니다.

### 데이터 확장 — Janus vs Janus-Pro

Janus(원본, arXiv 2410.13848)는 분리를 도입했지만 소규모(1.3B 파라미터, 제한된 데이터)였습니다. Janus-Pro(arXiv 2501.17811)는 확장했습니다:

- 7B 파라미터(기존 1.3B 대비).
- 1단계(정렬) 이미지-텍스트 쌍 90M(기존 72M 대비).
- 2단계(통합) 72M(기존 26M 대비).
- 3단계 이미지 생성 지시 샘플 200k 추가.

결과: Janus-Pro-7B는 MMMU에서 LLaVA와 동등(60.3 vs ~58)하며, GenEval에서 DALL-E 3을 능가(0.80 vs 0.67)합니다. 하나의 오픈 모델이 통합 스펙트럼 양쪽에서 경쟁력을 입증했습니다.

### JanusFlow — 수정된 흐름 변형

JanusFlow(arXiv 2411.07975)는 VQ 생성 경로를 수정된 흐름 생성 경로(연속적)로 교체합니다. 분리는 SigLIP-이해 + 수정된 흐름-생성으로 변경됩니다. 품질 상한이 더 높아집니다. 아키텍처는 분리된 인코더-공유 본체 구조를 유지합니다.

### 공유 본체의 역할

트랜스포머 본체는 통합 시퀀스를 처리하지만 두 입력 분포를 가집니다. 그 역할은 다음과 같습니다:

- 이해: SigLIP 특징 + 텍스트 토큰 소비 → 텍스트 자동회귀 생성.
- 생성: 텍스트 토큰 + (선택적 이미지 VQ 토큰) 소비 → 이미지 VQ 토큰 자동회귀 생성.

본체는 블록별 모달리티 특화 가중치가 없습니다. Qwen 또는 Llama 내부에서 찾을 수 있는 텍스트 스타일 트랜스포머에 두 입력 어댑터가 추가된 형태입니다.

흥미롭게도, 이는 Janus-Pro의 본체가 사전 훈련된 LLM에서 초기화될 수 있음을 의미합니다. Janus-Pro는 DeepSeek-MoE-7B에서 초기화됩니다. 이 선택은 중요합니다: LLM은 순수 통합 모델이 도달하기 어려운 추론 능력을 제공합니다.

### InternVL-U와 비교

InternVL-U(레슨 12.10)는 2026년 후속 모델입니다. 다음을 결합합니다:

- 네이티브 멀티모달 사전 훈련(InternVL3 백본).
- 분리된 인코더 라우팅(SigLIP 입력, VQ + 확산 헤드 출력).
- 통합 이해 + 생성 + 편집.

InternVL-U는 Janus-Pro의 아키텍처 선택을 더 큰 프레임워크에 통합합니다. 분리된 인코더 아이디어는 이제 대규모 통합 모델의 기본값입니다.

### 한계

분리된 인코더는 아키텍처 복잡성을 추가합니다. 두 토크나이저 훈련, 두 입력 경로 유지, 두 실패 모드 집합이 필요합니다. 생성이 필요 없는 제품의 경우 Janus-Pro는 과도한 설계입니다 — LLaVA 계열 이해 모델을 선택하세요.

이해가 필요 없는 제품의 경우 Janus-Pro는 과도한 사양입니다 — Stable Diffusion 3 / Flux 모델을 선택하세요.

둘 다 필요한 제품의 경우 Janus-Pro는 현재 참조 오픈 아키텍처입니다.

## 사용 방법

`code/main.py`는 Janus-Pro 라우팅을 시뮬레이션합니다:

- 두 개의 모의 인코더: SigLIP 유사(256차원 의미 벡터 생성) 및 VQ 유사(정수 코드 생성).
- 작업 태그에 따라 인코더를 선택하는 프롬프트 라우터.
- 어떤 인코더가 생성했는지와 무관하게 토큰 시퀀스를 처리하는 공유 본체(대용).
- 1단계(정렬)에서 3단계(지시 튜닝)로 전환하는 가중치 샘플링 일정.

이미지 QA, T2I, 이미지 편집에 대한 3가지 예시 라우팅 경로를 출력합니다.

## Ship It

이 레슨은 `outputs/skill-decoupled-encoder-picker.md`를 생성합니다. 통합 생성 + 이해 기능을 프론티어 수준의 품질로 원하는 제품에 대해, 구체적인 데이터 규모 권장 사항과 함께 Janus-Pro, JanusFlow 또는 InternVL-U를 선택합니다.

## 연습 문제

1. Janus-Pro-7B가 GenEval에서 DALL-E 3을 능가한다. 7B 오픈 모델이 생성(generation)에서는 최첨단 프로프라이어터리 모델과 대등할 수 있지만 이해(understanding)에서는 그렇지 않은 이유를 설명하시오.

2. 라우터 함수 구현: 프롬프트 텍스트를 입력받아 `understand` 또는 `generate`로 분류한다. "설명하고 스케치하시오"와 같은 모호한 프롬프트는 어떻게 처리하겠는가?

3. JanusFlow는 VQ 경로를 정류 흐름(rectified flow)으로 대체한다. 이제 트랜스포머 본체는 무엇을 출력하며, 손실 함수(loss function)에는 어떤 변화가 발생하는가?

4. Janus-Pro 아키텍처가 추가 디커플드 인코더(one more decoupled encoder)로 처리할 수 있는 네 번째 태스크를 제안하시오. 예시: 이미지 분할(DINO-style), 깊이 추정(MiDaS-style).

5. Janus-Pro 4.2절의 데이터 확장(data scaling) 내용을 읽고, Janus 대비 T2I 품질 향상에 가장 크게 기여한 데이터 단계는 무엇인가?

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|-------------------|-----------|
| Decoupled encoding | "두 개의 시각 인코더" | 방향별(separate) 토크나이저 또는 인코더: 의미 이해를 위한 인코더, 생성을 위한 재구성 인코더 |
| Shared body | "하나의 트랜스포머" | 단일 트랜스포머가 인코더 출력을 처리; 모달리티 특화 가중치 없음 |
| SigLIP for understanding | "의미적 특징" | CLIP 계열 시각 타워로 풍부한 개념적 특징을 제공하지만 재구성 성능은 낮음 |
| VQ for generation | "재구성 코드" | 픽셀로 깨끗하게 디코딩되는 벡터 양자화 토큰 |
| JanusFlow | "수정된 흐름 변형" | VQ 대신 연속 흐름 매칭 생성 헤드를 사용하는 Janus-Pro |
| Routing tag | "태스크 태그" | 입력 인코더를 선택하는 프롬프트 마커 (`<understand>` / `<generate>`)
