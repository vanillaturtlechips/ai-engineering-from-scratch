# 캡스톤 10 — 멀티에이전트 소프트웨어 엔지니어링 팀

> SWE-AF의 팩토리 아키텍처, MetaGPT의 역할 기반 프롬프팅, AutoGen 0.4의 타입드 액터 그래프, Cognition의 Devin, Factory의 Droids는 모두 2026년 동일한 형태로 수렴했습니다: 아키텍트가 계획하고, N명의 코더가 병렬 워크트리에서 작업하며, 리뷰어가 게이트를 통과시키고, 테스터가 검증합니다. 병렬 워크트리는 벽시계 시간을 처리량으로 변환합니다. 공유 상태와 핸드오프 프로토콜이 실패 지점이 됩니다. 캡스톤 과제는 팀을 구축하고, SWE-bench Pro에서 평가하며, 어떤 핸드오프가 얼마나 자주 실패하는지 보고하는 것입니다.

**유형:** 캡스톤  
**언어:** Python / TypeScript (에이전트), Shell (워크트리 스크립트)  
**선수 과목:** 11단계(LLM 엔지니어링), 13단계(툴), 14단계(에이전트), 15단계(자율성), 16단계(멀티에이전트), 17단계(인프라)  
**적용 단계:** P11 · P13 · P14 · P15 · P16 · P17  
**소요 시간:** 40시간

## 문제

단일 에이전트 코딩은 대규모 작업에서 한계에 부딪힙니다. 개별 에이전트가 약해서가 아니라, 200k 토큰 컨텍스트에 아키텍처 계획, 4개의 병렬 코드베이스 조각, 리뷰어 코멘트, 테스트 출력을 모두 담을 수 없기 때문입니다. 다중 에이전트 팩토리는 문제를 분할합니다: 아키텍트가 계획을 소유하고, 코더들이 병렬 워크트리에서 구현을 담당하며, 리뷰어가 승인하고, 테스터가 검증합니다. SWE-AF의 "팩토리" 아키텍처, MetaGPT의 역할 분담, AutoGen의 타입화된 액터 그래프 — 이 세 가지 프레임워크는 모두 동일한 구조를 설명합니다.

실패 지점은 핸드오프 과정에 있습니다. 아키텍트가 계획한 내용을 코더들이 구현할 수 없습니다. 코더들이 충돌하는 변경 사항을 생성합니다. 리뷰어가 환각(hallucinated)된 수정을 승인합니다. 테스터가 아직 작성 중인 코더와 경쟁합니다. 여러분은 이러한 팀 중 하나를 구축하고, 50개의 SWE-bench Pro 이슈에 적용한 뒤 모든 핸드오프 과정을 추적하고 사후 분석을 공개하게 됩니다.

## 개념

역할(Role)은 타입이 지정된 에이전트입니다. **아키텍트**(Claude Opus 4.7)는 이슈를 읽고 계획을 작성한 후 명시적 인터페이스를 가진 하위 작업으로 분할합니다. **코더**(Claude Sonnet 4.7, N개의 병렬 인스턴스, 각각 `git worktree` + Daytona 샌드박스 내 작업)는 독립적으로 하위 작업을 구현합니다. **리뷰어**(GPT-5.4)는 병합된 diff를 읽고 승인하거나 특정 변경 사항을 요청합니다. **테스터**(Gemini 2.5 Pro)는 테스트 스위트를 격리된 환경에서 실행하고 아티팩트와 함께 통과/실패 결과를 보고합니다.

통신은 공유 작업 보드(파일 기반 또는 Redis)를 통해 이루어집니다. 각 역할은 처리할 수 있는 권한을 가진 작업만 소비합니다. 핸드오프는 A2A 프로토콜 타입 메시지로 이루어집니다. 조정 관련 고려 사항: 병합 충돌 해결(코디네이터 역할 또는 자동 3-way 병합), 공유 상태 동기화(코더가 작업을 시작하면 계획이 고정되며 재계획은 별도의 이벤트로 처리), 리뷰어 게이트키핑(리뷰어는 자신이 제안한 변경 사항이나 자신의 변경 사항을 승인할 수 없음).

토큰 증폭은 숨겨진 비용입니다. 모든 역할 경계마다 요약 프롬프트와 핸드오프 컨텍스트가 추가됩니다. 단일 에이전트 40턴 실행은 4개 역할에서 총 160턴으로 증가합니다. 평가 기준은 토큰 효율성과 단일 에이전트 기준치를 특별히 고려합니다. 왜냐하면 "다중 에이전트가 작동하는가"가 아닌 "비용 대비 효과가 있는가"가 핵심 질문이기 때문입니다.

## 스택

- 오케스트레이션: 공유 상태 + 에이전트별 서브 그래프를 활용한 LangGraph
- 메시징: 타입화된 에이전트 간 메시지를 위한 A2A 프로토콜(Google 2025)
- 모델: Opus 4.7(아키텍트), Sonnet 4.7(코더), GPT-5.4(리뷰어), Gemini 2.5 Pro(테스터)
- 작업 트리 격리: 코더별 `git worktree add` + 데이토나 샌드박스
- 병합 조정자: 커스텀 3-way 병합 + LLM 기반 충돌 해결
- 평가: SWE-bench Pro(50개 이슈), SWE-AF 시나리오, 단위 테스트를 위한 HumanEval++
- 관측 가능성: 역할 태그가 지정된 Langfuse 스팬, 에이전트별 토큰 회계
- 배포: 각 역할을 별도의 Deployment로 구성한 K8s + 백로그 기반 HPA

## 빌드 방법

1. **태스크 보드.** 파일 기반 JSONL에 타입이 지정된 메시지: `plan_request`, `subtask`, `diff_ready`, `review_needed`, `test_needed`, `approved`, `rejected`, `replan_needed`. 에이전트는 태그에 구독한다.

2. **아키텍트.** GitHub 이슈를 읽고, Opus 4.7을 실행하여 계획 템플릿을 사용한다. 이 템플릿은 명시적 하위 작업 인터페이스(변경된 파일, 공개 함수, 테스트 영향)를 요구한다. 하위 작업 DAG가 포함된 하나의 `plan_request`를 발행한다.

3. **코더.** N개의 병렬 작업자로, 각각 보드에서 하나의 하위 작업을 할당받는다. 각 코더는 새로운 `git worktree add` 브랜치와 Daytona 샌드박스를 생성한다. 하위 작업을 구현한 후, 패치 + 테스트 델타를 포함한 `diff_ready`를 발행한다.

4. **병합 조정자.** 모든 코더가 작업을 완료하면, N개의 브랜치를 스테이징 브랜치로 3-way 병합한다. 파일 수준 충돌이 존재할 경우에만 LLM 기반 충돌 해결을 수행한다.

5. **리뷰어.** GPT-5.4가 병합된 diff를 읽는다. 자신이 작성한 diff는 승인할 수 없다. `approved`(no-op) 또는 특정 변경 요청을 포함한 `review_feedback`을 발행하며, 이는 관련 코더에게 다시 전달된다.

6. **테스터.** Gemini 2.5 Pro가 깨끗한 샌드박스에서 테스트 스위트를 실행한다. 아티팩트를 캡처한다. `test_passed` 또는 스택 트레이스를 포함한 `test_failed`를 발행한다. 실패한 테스트는 실패한 하위 작업을 소유한 코더에게 다시 전달된다.

7. **인계 회계.** 역할 경계를 넘는 모든 메시지는 Langfuse에 스팬으로 기록되며, 페이로드 크기와 사용된 모델이 포함된다. 하위 작업당 토큰 증폭 계수(coder_tokens + reviewer_tokens + tester_tokens + architect_share / coder_tokens)를 계산한다.

8. **평가.** 50개의 SWE-bench Pro 이슈에서 실행한다. 단일 에이전트 베이스라인(단일 작업 트리의 Sonnet 4.7)과 pass@1 및 $-per-solved-issue를 비교한다.

9. **사후 분석.** 각 실패한 이슈에 대해 실패한 인계 지점(계획이 너무 모호함, 병합 충돌, 리뷰어의 잘못된 승인, 테스터 불안정)을 식별한다. 인계 실패 히스토그램을 생성한다.

## 연습 문제

1. 실행 중인 diff 중간에 명백한 버그(예: 메인 본문 앞에 `return None` 추가)를 주입합니다. 리뷰어의 잘못된 승인률을 측정합니다. 잘못된 승인률이 5% 미만이 될 때까지 리뷰어 프롬프트를 조정합니다.

2. 두 명의 코더(architect + coder + reviewer + tester, 코더는 두 개의 하위 작업을 순차적으로 실행)로 축소합니다. 벽시계 시간과 통과율을 비교합니다.

3. 병합 조정자(merge coordinator)를 단일 작성자 제약 조건(서브태스크가 서로 겹치지 않는 파일 집합을 처리)으로 대체합니다. 아키텍트(architect)의 계획 부담을 측정합니다.

4. 리뷰어를 GPT-5.4에서 Claude Opus 4.7로 교체합니다. 잘못된 승인률과 토큰 비용 변화량을 측정합니다.

5. 다섯 번째 역할인 문서 작성자(documenter, Haiku 4.5)를 추가합니다. 리뷰 후 변경 로그(changelog) 항목을 생성하게 합니다. 문서 품질이 추가 토큰 비용을 정당화하는지 측정합니다.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |

| 병렬 워크트리 | "고립된 브랜치" | `git worktree add`가 개발자마다 새로운 작업 트리를 생성하는 것 |
| 태스크 보드 | "공유 메시지 버스" | 에이전트들이 구독하는 파일 또는 Redis 저장소의 타입이 지정된 메시지 |
| 핸드오프 | "역할 경계" | 한 역할의 컨텍스트에서 다른 역할의 컨텍스트로 넘어가는 모든 메시지 |

| A2A 프로토콜 | "에이전트 간" | 2025년 Google의 타입이 지정된 에이전트 간 메시지 규격 |

| 거짓 승인 | "리뷰어 환각" | 리뷰어가 알려진 버그가 있는 diff를 승인하는 것 |
