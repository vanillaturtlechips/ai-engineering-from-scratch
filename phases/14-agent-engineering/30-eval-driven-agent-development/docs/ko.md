# 평가 주도 에이전트 개발

> Anthropic의 지침: "간단한 프롬프트로 시작하고, 포괄적인 평가를 통해 최적화한 후, 필요한 경우에만 다단계 에이전트 시스템을 추가하라." 평가는 마지막 단계가 아니다. 이는 14단계의 다른 모든 선택을 주도하는 외부 루프이다.

**유형:** 학습 + 구축  
**언어:** Python (표준 라이브러리)  
**사전 요구 사항:** 14단계 전체  
**소요 시간:** ~60분

## 학습 목표

- 세 가지 평가 계층(정적 벤치마크, 커스텀 오프라인, 온라인 프로덕션)의 이름을 말하고 각각이 어떤 용도인지 설명할 수 있다.
- 평가자-최적화기 간 긴밀한 피드백 루프(evaluator-optimizer tight loop)를 설명할 수 있다.
- 2026년 모범 사례: 평가(evals)는 코드 옆에 위치하며, CI에서 실행되고, PR을 차단(gate)하는 방식을 설명할 수 있다.
- 모든 Phase 14 레슨을 해당 평가 사례와 연결할 수 있다.

## 문제

에이전트는 데모를 통과합니다. 그러나 데모로 예측할 수 없는 방식으로 프로덕션 환경에서 실패합니다. 벤치마크는 "이 모델이 광범위하게 능력이 있는가?"라는 질문에는 답하지만, "이 에이전트가 내 제품에 적합한 패치를 제공하고 있는가?"라는 질문에는 답하지 못합니다. 해결책: 모든 가드레일과 학습된 규칙이 평가 사례에 매핑된 상태에서, 3개 계층에서 지속적으로 실행되는 평가입니다.

## 개념

### 세 가지 평가 계층

1. **정적 벤치마크** — 코드용 SWE-bench Verified(레슨 19), 브라우징/데스크톱용 WebArena/OSWorld(레슨 20), 일반형 GAIA(레슨 19), 도구 사용용 BFCL V4(레슨 06). 모델 간 비교 및 회귀 검증에 사용. 오염 문제 존재: SWE-bench+에서 32.67% 솔루션 유출 발견. 항상 Verified / +-감사 점수를 보고.

2. **맞춤형 오프라인 평가** — 제품 형태에 맞춤:
   - LLM-as-judge(Langfuse, Phoenix, Opik — 레슨 24).
   - 실행 기반(패치 실행, 테스트 확인).
   - 궤적 기반(골드 표준과 행동 시퀀스 비교; OSWorld-Human에서 상위 에이전트가 골드 대비 1.4-2.7배 우수).

3. **온라인 평가** — 프로덕션 환경:
   - 세션 재생(Langfuse).
   - 가드레일 트리거 경고(레슨 16, 21).
   - 단계별 비용/지연 추적(레슨 23 OTel 스팬).

### 평가자-최적화기(Anthropic)

밀접한 루프:

1. 제안자가 출력 생성.
2. 평가자가 판단.
3. 평가자가 통과할 때까지 개선.

이것은 Self-Refine(레슨 05)의 일반화 버전. 신뢰성을 위해 모든 에이전트 흐름을 평가자-최적화기로 래핑 가능.

### 2026년 모범 사례

- 평가 코드는 코드와 함께 위치.
- 모든 PR에서 CI 실행.
- 평가 점수로 병합 게이트(예: "메인 대비 5% 이상 회귀 없음").
- 모든 가드레일은 평가 사례와 매핑.
- 모든 학습된 규칙(Reflexion, 프로 워크플로우 학습 규칙)은 실패 사례와 매핑.

### 14단계 통합

14단계의 모든 레슨은 평가 사례를 생성:

| 레슨 | 생성되는 평가 사례 |
|--------|------------------------|
| 01 에이전트 루프 | 예산 소진, 무한 루프 가드레일 |
| 02 ReWOO | 도구 실패 시 플래너 재계획 |
| 03 Reflexion | 재시도 시 학습된 반성 적용 |
| 05 Self-Refine/CRITIC | 평가자가 개선된 출력 통과 |
| 06 도구 사용 | 인수 강제 적용; 알 수 없는 도구 거부 |
| 07-10 메모리 | 검색 인용이 소스와 일치; 오래된 사실 무효화 |
| 12 워크플로우 패턴 | 각 패턴이 올바른 출력 생성 |
| 13 LangGraph | 재개 시 상태 정확히 재현 |
| 14 AutoGen 액터 | DLQ가 충돌 핸들러 감지 |
| 16 OpenAI 에이전트 SDK | 가드레일이 올바른 입력에서 트리거 |
| 17 Claude 에이전트 SDK | 서브에이전트 결과가 오케스트레이터로 반환 |
| 19-20 벤치마크 | SWE-bench Verified 점수, WebArena 성공률, OSWorld 효율성 |
| 21 컴퓨터 사용 | 단계별 안전성 검사 시 주입된 DOM 감지 |
| 23 OTel | 스팬이 필수 속성 방출 |
| 26 실패 모드 | 감지기가 알려진 실패 태깅 |
| 27 프롬프트 인젝션 | PVE가 중독된 검색 거부 |
| 28 오케스트레이션 | 감독자가 올바른 전문가에게 라우팅 |
| 29 런타임 형태 | DLQ가 N% 실패 처리 |

평가 스위트에 각 사례가 포함되면 14단계를 모두 커버한 것.

### 평가 주도 개발이 실패하는 경우

- **기준선 없음.** 마지막 정상 상태 없는 평가는 해석 불가. 기준선 저장.
- **근거 없는 LLM 평가자.** 평가자도 환각 발생. CRITIC 패턴(레슨 05) — 외부 도구에 근거.
- **평가에 과적합.** 평가 최적화가 프로덕션 유용성과 동떨어짐. 사례 교체.
- **불안정한 평가.** 비결정적 사례가 오탐 유발. 시드 고정, 상태 스냅샷.

## 빌드 및 실행

`code/main.py`는 표준 라이브러리 평가 하네스입니다:

- 범주(benchmark, custom, online)를 가진 케이스 레지스트리.
- 테스트 대상 스크립트 에이전트.
- 평가자-최적화기 루프: 제안(propose), 판단(judge), 개선(refine)을 통과 또는 최대 라운드까지 반복.
- CI 게이트: 통과율 집계 + 기준선 대비 회귀(regression) 분석.

실행 방법:

```
python3 code/main.py
```

출력: 케이스별 통과/실패, 회귀 플래그, CI 게이트 판정.

## 사용 방법

- 에이전트 코드와 동일한 리포지토리에 평가 케이스를 작성하세요.
- CI를 통해 모든 PR(풀 리퀘스트) 시 실행하세요.
- 회귀(regression) 발생 시 빌드를 실패 처리하세요.
- 시간에 따른 통과율(pass rate)을 추적하세요.
- 모든 프로덕션 실패를 새로운 케이스와 연결하세요.

## Ship It

`outputs/skill-eval-suite.md`는 CI 게이트와 회귀 추적을 갖춘 에이전트 제품을 위한 3계층 평가 스위트를 구축합니다.

## 연습 문제

1. 프로덕션 실패 사례 중 하나를 선택하세요. 이를 재현하는 평가 케이스를 작성하세요. 현재 에이전트가 이 케이스를 통과하나요?
2. 도메인에 맞는 LLM-판단자 평가 기준(rubric)을 3가지 차원(사실성(factual), 어조(tone), 범위(scope))으로 구성하세요. 50개의 세션을 평가하세요.
3. 평가 스위트를 CI에 통합하세요. 5% 이상의 성능 저하(regression) 발생 시 빌드를 실패 처리하세요.
4. 궤적 효율성(trajectory-efficiency) 메트릭을 추가하세요: 에이전트가 수행한 단계 수와 골드 궤적(gold trajectory) 단계 수를 비교하세요.
5. 모든 Phase 14 레슨을 평가 스위트의 평가 케이스에 매핑하세요. 누락된 부분이 있나요? 그렇다면 그 부분을 보완해야 합니다.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 정적 벤치마크(Static benchmark) | "Off-the-shelf eval" | SWE-bench, GAIA, AgentBench, WebArena, OSWorld |
| 커스텀 오프라인 평가(Custom offline eval) | "Domain eval" | LLM-as-judge / exec / 제품 형태에서의 trajectory |
| 온라인 평가(Online eval) | "Production eval" | 세션 재생, 가드레일 경고, 비용/지연 시간 추적 |
| 평가자-최적화기(Evaluator-optimizer) | "Propose-judge-refine" | 평가자가 통과할 때까지 반복 |
| CI 게이트(CI gate) | "Merge blocker" | 평가 성능 저하 시 빌드 실패 처리 |
| 베이스라인(Baseline) | "Last-known-good" | 성능 저하 감지를 위한 기준 점수 |
| 궤적 효율성(Trajectory efficiency) | "Steps over gold" | 에이전트 단계 수 / 인간 전문가 최소 단계 수 |
