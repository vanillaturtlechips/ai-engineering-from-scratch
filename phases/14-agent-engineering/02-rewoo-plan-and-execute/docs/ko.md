# ReWOO와 Plan-and-Execute: 분리된 계획 수립

> ReAct는 사고와 행동을 하나의 스트림에서 교차시킵니다. ReWOO는 이를 분리합니다: 먼저 큰 계획을 수립한 후 실행합니다. HotpotQA에서 토큰 사용량을 5배 줄이고 정확도를 +4% 향상시켰으며, 계획자(planner)를 7B 모델로 증류할 수 있습니다. Plan-and-Execute는 이를 일반화했고, Plan-and-Act는 웹 탐색에 확장 적용했습니다.

**유형:** 구축(Build)
**언어:** Python (표준 라이브러리)
**사전 요구 사항:** 14단계 · 01 (에이전트 루프)
**소요 시간:** ~60분

## 학습 목표

- ReWOO의 **플래너(Planner) / 워커(Worker) / 솔버(Solver)** 분할이 ReAct의 인터리빙 루프(interleaved loop) 대비 토큰을 절약하고 강건성을 향상시키는 이유를 설명.
- **계획 DAG(plan DAG)**, **의존성 순서 실행기(dependency-ordered executor)**, **워커 출력을 조합하는 솔버(solver)** 구현 — 모두 표준 라이브러리(stdlib) 사용.
- 2026년 "다섯 가지 워크플로우 패턴(five workflow patterns)" 프레임워크(Anthropic)를 활용해 **계획 후 실행(plan-then-execute)** vs **인터리빙 ReAct** 중 어떤 방식으로 작업을 실행할지 결정.
- **Plan-and-Act**의 합성 계획 데이터(synthetic plan data)가 장기 웹/모바일 작업에 필요한 경우를 식별.

## 문제

ReAct의 생각-행동-관찰 교차 루프는 단순하고 유연하지만, 각 도구 호출은 이전 모든 생각을 포함한 전체 이전 컨텍스트를 전달해야 합니다. 토큰 사용량은 깊이에 따라 제곱으로 증가합니다. 더 나쁜 점은: 도구 호출이 루프 중간에 실패하면 모델이 오류 관찰로부터 전체 계획을 다시 유도해야 한다는 것입니다.

ReWOO(Xu et al., arXiv:2305.18323, 2023년 5월)는 이 문제를 인식하고 다음과 같은 전략을 제안했습니다: 전체 계획을 미리 수립하고, 증거를 병렬로 수집한 후 마지막에 답변을 조합합니다. 계획 수립을 위한 1회의 LLM 호출, 증거 수집을 위한 N회의 도구 호출(병렬 가능), 문제 해결을 위한 1회의 LLM 호출로 구성됩니다. 이 접근 방식은 유연성 감소(정적 계획)가 단점이지만, 토큰 효율성이 크게 향상되고 실패 모드가 더 명확해지는 장점이 있습니다.

## 개념

### 세 가지 역할

```
Planner:  user_question -> [plan_dag]
Workers:  [plan_dag]     -> [evidence]        (tool calls, possibly parallel)
Solver:   user_question, plan_dag, evidence -> final_answer
```

Planner는 DAG를 생성합니다. 각 노드는 도구 이름, 인수, 그리고 의존하는 이전 노드(예: `#E1`, `#E2` 참조)를 명시합니다. Workers는 위상 정렬 순서로 노드를 실행합니다. Solver는 모든 것을 종합하여 최종 답변을 생성합니다.

### 토큰 5배 감소 이유

ReAct는 단계 수에 비례하여 프롬프트 길이가 선형적으로 증가합니다. 10단계에서 프롬프트는 생각 1 + 행동 1 + 관찰 1 + 생각 2 + 행동 2 + 관찰 2 등을 포함하며, 각 중간 단계에는 원본 프롬프트가 중복됩니다.

ReWOO는 하나의 플래너 프롬프트(대형), N개의 작은 워커 프롬프트(각각 도구 호출만 포함, 체인 없음), 하나의 솔버 프롬프트를 사용합니다. HotpotQA에서 논문에서는 토큰 사용량이 약 5배 감소하면서 정확도가 +4 절대점 상승했다고 측정합니다.

### 더 강건한 이유

ReAct에서 워커 3이 실패하면 루프는 중간 단계에서 오류를 추론해야 합니다. ReWOO에서는 워커 3이 오류 문자열을 반환하고, 솔버는 원본 계획과 함께 컨텍스트에서 이를 확인하여 우아하게 성능을 저하시킬 수 있습니다. 실패 지역화는 단계 단위가 아닌 노드 단위입니다.

### 플래너 증류

논문의 두 번째 결과: 플래너는 관찰 결과를 보지 않으므로 175B 교사 모델의 플래너 출력을 7B 모델에 파인튜닝할 수 있습니다. 소형 모델은 계획 수립을 처리하고, 대형 모델은 추론 시 필요하지 않습니다. 이는 현재 표준 접근 방식으로, 많은 2026년 프로덕션 에이전트가 소형 플래너와 대형 실행기 또는 그 반대 구성을 사용합니다.

### 계획-실행(Plan-and-Execute, LangChain, 2023)

LangChain 팀의 2023년 8월 게시물은 ReWOO를 일반화해서 "계획-실행" 패턴으로 명명했습니다. 사전 플래너가 단계 목록을 생성하고, 실행기가 각 단계를 실행하며, 선택적 재플래너는 결과를 관찰한 후 수정할 수 있습니다. 이는 ReWOO보다 ReAct에 가깝지만(재플래너가 관찰 결과를 계획에 반영), 토큰 절감 효과는 유지합니다.

### 계획-행동(Plan-and-Act, Erdogan et al., arXiv:2503.09572, ICML 2025)

계획-행동은 장기 웹/모바일 에이전트로 패턴을 확장합니다. 핵심 기여는 합성 계획 데이터입니다: 레이블된 궤적 생성기가 명시적인 계획을 포함한 훈련 데이터를 생성합니다. WebArena 유사 작업에서 단일 ReAct 궤적이 일관성을 잃는 30–50단계를 넘어 계속 작동하는 플래너 모델 파인튜닝에 사용됩니다.

### 어떤 패턴을 선택할지

| 패턴 | 선택 시기 |
|---------|------|
| ReAct | 짧은 작업, 미지의 환경, 반응형 예외 처리 필요 시 |
| ReWOO | 알려진 도구로 구조화된 작업, 토큰 민감도, 병렬화 가능한 증거 수집 |
| 계획-실행 | ReWOO와 유사하지만 부분 실행 후 재계획 필요 시 |
| 계획-행동 | 장기 작업(>30단계), 웹/모바일/컴퓨터 사용 |
| 사고 트리(Tree of Thoughts) | 검색이 비용 대비 가치 있을 때(레슨 04) |

Anthropic의 2024년 12월 지침: 가장 단순한 것부터 시작하세요. 작업이 도구 호출 1회 + 요약이라면 ReWOO를 구축하지 마세요. 40단계 연구 과제라면 ReAct만 사용하지 마세요.

## 구축 방법

`code/main.py`는 장난감 ReWOO를 구현합니다:

- `Planner` — 프롬프트로부터 계획 DAG를 생성하는 스크립트 정책입니다.
- `Worker` — 레지스트리를 통해 각 노드의 도구 호출을 디스패치합니다.
- `Solver` — 증거를 읽고 최종 답변을 생성하는 스크립트 합성입니다.
- 종속성 해결 — `#E1`과 같은 참조는 이전 작업자 출력으로 대체됩니다.

데모는 "프랑스의 수도 인구는 수백만 단위로 반올림하면 얼마인가요?"라는 질문에 두 단계 계획으로 답변합니다: (1) 수도 조회, (2) 인구 조회, 이후 해결.

실행 방법:

```
python3 code/main.py
```

트레이스에는 전체 계획, 작업자 결과, 솔버 합성 순서가 표시됩니다. 토큰 수(대략적인 문자 수를 출력함)를 ReAct 스타일의 인터리브 실행과 비교해 보세요 — ReWOO는 이런 구조화된 작업에서 우수합니다.

## 사용 방법

LangGraph는 Plan-and-Execute를 레시피 형태로 제공합니다(ReAct의 경우 `create_react_agent`, 계획-실행의 경우 커스텀 그래프). CrewAI의 Flows는 해당 패턴을 직접 인코딩합니다: 작업을 미리 정의하면 Flow DAG가 이를 실행합니다. Plan-and-Act의 합성 데이터 접근 방식은 여전히 연구 단계에 머물러 있으며, 런타임 패턴(명시적 계획 DAG)은 LangGraph와 CrewAI Flows를 통해 프로덕션에 배포됩니다.

## Ship It

`outputs/skill-rewoo-planner.md`는 사용자 요청과 도구 카탈로그를 기반으로 ReWOO 계획 DAG를 생성합니다. 이 문서는 계획을 검증한 후(사이클 없음, 모든 참조 해결됨, 모든 도구 존재함) 실행기로 전달합니다.

## 연습 문제

1. 독립적인 계획 노드(plan node)에 대해 워커 실행을 병렬화하세요. 6-노드 DAG와 2개의 병렬 그룹(parallel group)에서 어떤 이점을 얻을 수 있나요?  
2. 워커 중 하나라도 오류를 반환할 때 실행되는 리플래너 노드(replanner node)를 추가하세요. ReWOO에 가장 작은 변경 사항을 적용하여 Plan-and-Execute로 만드는 방법은 무엇인가요?  
3. `Planner`를 소형 모델(7B 클래스)로 대체하고 `Solver`는 프론티어 모델(frontier model)에 유지하세요. 종단 간(end-to-end) 품질을 비교할 때, 분할(split)은 어디에서 실패하나요?  
4. ReWOO 논문의 4장에서 플래너 증류(planner distillation)에 대해 읽으세요. 개념적으로 175B -> 7B 결과를 재현하려면 어떤 훈련 데이터가 필요하며, 계획 품질(plan quality)을 어떻게 평가하나요?  
5. 토이(toy)를 Plan-and-Act의 궤적 형태(trajectory shape)로 이식하세요: 계획은 DAG가 아닌 시퀀스(sequence)입니다. 어떤 트레이드오프가 변경되나요?

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| ReWOO | "관찰 없이 추론" | 계획 수립, 증거 병렬 수집, 문제 해결 — 계획 프롬프트에 관찰 없음 |
| Plan-and-Execute | "LangChain의 계획-실행 패턴" | 실행 후 선택적 재계획자 노드를 포함한 ReWOO |
| Plan-and-Act | "확장된 계획-실행" | 장기 작업을 위한 합성 계획 훈련 데이터와 명시적 계획자/실행자 분리 |
| 증거 참조 | "#E1, #E2, ..." | 디스패치 시 이전 작업자 출력으로 대체되는 계획 노드 플레이스홀더 |
| 계획자 증류 | "소형 계획자, 대형 실행자" | 대형 교사 모델의 계획 트레이스로 소형 모델 파인튜닝 |
| 토큰 효율성 | "적은 왕복 통신" | 논문에서 ReAct 대비 HotpotQA에서 5배 적은 토큰 사용 |
| DAG 실행기 | "위상 정렬 디스패처" | 종속성 순서로 계획 노드 실행; 각 레벨에서 병렬 처리 |
