# 메모리 블록과 수면 시간 계산 (레타)

> MemGPT는 2024년에 레타로 변경되었습니다. 2026년 진화에는 두 가지 아이디어가 추가되었습니다: 모델이 직접 편집할 수 있는 이산적 기능 메모리 블록과, 주 에이전트가 유휴 상태일 때 비동기적으로 메모리를 통합하는 수면 시간 에이전트입니다. 이것이 하나의 대화를 넘어 메모리를 확장하는 방법입니다.

**유형:** 구축
**언어:** Python (표준 라이브러리)
**사전 요구 사항:** 14단계 · 07 (MemGPT)
**소요 시간:** ~75분

## 학습 목표

- Letta가 사용하는 세 가지 메모리 계층(코어(core), 리콜(recall), 아카이브(archival))의 이름과 각각의 역할을 설명할 수 있다.
- 메모리-블록 패턴: 인간 블록(Human block), 페르소나 블록(Persona block), 그리고 1급 타입 객체로서의 사용자 정의 블록(user-defined blocks)을 설명할 수 있다.
- 수면 시간 계산(sleep-time compute)이 무엇인지, 왜 크리티컬 패스(critical path) 밖에 위치하는지, 그리고 왜 주 에이전트(primary agent)보다 더 강력한 모델을 실행할 수 있는지 설명할 수 있다.
- 주 에이전트(primary agent)가 응답을 제공하고, 수면 시간 에이전트(sleep-time agent)가 턴 사이에 블록을 통합하는 스크립트 기반 2-에이전트 루프를 구현할 수 있다.

## 문제

MemGPT(레슨 07)는 가상 메모리 제어 흐름을 해결했습니다. 하지만 3가지 프로덕션 문제가 발생했습니다:

1. **지연 시간.** 모든 메모리 작업은 크리티컬 패스에 위치합니다. 사용자가 기다리는 동안 에이전트가 가지치기, 요약 또는 충돌 해결을 수행해야 한다면 꼬리 지연 시간이 급증합니다.
2. **메모리 부패.** 쓰기 작업이 누적됩니다. 모순된 사실이 그대로 유지됩니다. 검색은 오래된 콘텐츠에 의해 방해받습니다.
3. **구조 손실.** 평면적인 아카이브 저장소는 "Human 블록은 항상 프롬프트에 포함; Persona 블록은 항상 프롬프트에 포함; Task 블록은 세션마다 교체됨"과 같은 구조를 표현할 수 없습니다.

Letta(letta.com)는 2026년 리팩토링 버전입니다. 메모리 블록은 구조를 명시적으로 표현하며, 유휴 시간 컴퓨팅을 통해 통합 작업을 크리티컬 패스에서 분리합니다.

## 개념

### 세 가지 계층

| 계층 | 범위 | 위치 | 작성자 |
|------|-------|----------------|------------|
| 코어 | 항상 표시됨 | 메인 프롬프트 내부 | 에이전트 도구 호출 + 유휴 시간 재작성 |
| 리콜 | 대화 기록 | 검색 가능 | 자동 턴 로깅 |
| 아카이브 | 임의의 사실 | 벡터 + KV + 그래프 | 에이전트 도구 호출 + 유휴 시간 수집 |

코어는 MemGPT의 핵심입니다. 리콜은 대화 버퍼와 제거된 꼬리 부분입니다. 아카이브는 외부 저장소입니다. 이 분할은 MemGPT의 두 계층 과부하를 정리합니다.

### 메모리 블록

블록은 코어 계층의 유형화된, 지속적, 편집 가능한 섹션입니다. 원래 MemGPT 논문에서는 두 가지를 정의했습니다:

- **인간 블록** — 사용자에 대한 사실(이름, 역할, 선호도, 목표).
- **페르소나 블록** — 에이전트의 자기 개념(정체성, 어조, 제약 조건).

Letta는 임의의 사용자 정의 블록으로 일반화합니다: 현재 목표를 위한 `Task` 블록, 코드베이스 사실을 위한 `Project` 블록, 하드 제약 조건을 위한 `Safety` 블록. 각 블록에는 `id`, `label`, `value`, `limit`(문자 제한), `description`(모델이 편집 시기를 알 수 있도록)이 있습니다.

블록은 도구 표면을 통해 편집할 수 있습니다:

- `block_append(label, text)`
- `block_replace(label, old, new)`
- `block_read(label)`
- `block_summarize(label)` — 제한에 가까운 블록을 압축합니다.

### 유휴 시간 계산

2025년 Letta 추가 기능: 두 번째 에이전트를 백그라운드에서 실행하며, 크리티컬 경로에서 벗어납니다. 유휴 시간 에이전트는 대화 기록과 코드베이스 컨텍스트를 처리하고, 공유 블록에 `learned_context`를 작성하며, 아카이브 레코드를 통합하거나 무효화합니다.

파생되는 속성:

- **지연 비용 없음.** 기본 응답은 메모리 작업을 기다리지 않습니다.
- **더 강력한 모델 허용.** 유휴 시간 에이전트는 지연 제약이 없기 때문에 더 비싸고 느린 모델일 수 있습니다.
- **자연스러운 통합 창.** 사용자가 기다리지 않을 때 중복 제거, 요약, 모순된 사실 무효화를 수행합니다.

이 형태는 인간의 작업 방식과 일치합니다: 작업을 수행하고, 숙고한 후, 장기 기억이 밤새 정착됩니다.

### Letta V1과 네이티브 추론

Letta V1(`letta_v1_agent`, 2026)은 `send_message`/하트비트를 폐기하고 인라인 `Thought:` 토큰 대신 네이티브 추론을 채택합니다. Responses API(OpenAI)와 확장된 사고 기능이 있는 Messages API(Anthropic)는 별도의 채널로 추론을 방출하며, 턴을 통해 전달됩니다(실제 환경에서는 공급자 간 암호화됨). 제어 루프는 여전히 ReAct입니다. 사고 추적은 프롬프트 형태가 아닌 구조적 형태입니다.

### 이 패턴이 실패하는 경우

- **블록 팽창.** 무한한 `block_append`는 제한을 빠르게 초과합니다. 제한을 초과하는 쓰기 전에 블록 요약기를 연결합니다.
- **조용한 드리프트.** 유휴 시간 에이전트가 블록을 재작성하고 기본 에이전트가 이를 인지하지 못합니다. 블록 버전을 관리하고 추적에서 차이를 표시합니다.
- **오염된 통합.** 유휴 시간 에이전트가 공격자 접근 가능 콘텐츠를 코어로 처리합니다. 레슨 27은 유휴 시간 표면에도 적용됩니다.

## 빌드하기

`code/main.py`는 다음을 구현합니다:

- `Block` — id, label, value, limit, description.
- `BlockStore` — CRUD + `near_limit(label)` 헬퍼 함수.
- 두 개의 스크립트 에이전트 — `PrimaryAgent`는 턴을 제공하고, `SleepTimeAgent`는 턴 사이에 통합 작업을 수행합니다.
- 블록 쓰기와 함께 3턴 대화를 보여주는 트레이스, 그리고 블록을 요약하고 오래된 사실을 무효화하는 슬립 타임 패스가 포함됩니다.

실행 방법:

```
python3 code/main.py
```

트랜스크립트는 다음과 같은 분할을 보여줍니다: 기본 턴은 빠르고 원시 쓰기를 생성하는 반면, 슬립 패스는 압축하고 정리합니다.

## 사용 방법

- **Letta** (letta.com)에서 참조 구현을 확인하세요. 자체 호스팅 또는 관리형 클라우드 서비스를 이용할 수 있습니다.
- **Claude Agent SDK 스킬**을 블록 형태의 지식으로 활용 — 스킬은 에이전트가 필요 시 로드하는 이름, 버전, 검색 가능한 명령어 블록입니다.
- **커스텀 빌드**를 통해 스토리지 백엔드를 직접 제어하려는 팀을 위한 옵션. Letta API 계약을 사용하면 나중에 마이그레이션이 가능합니다.

## Ship It

`outputs/skill-memory-blocks.md`는 모든 런타임(안전 규칙 및 인용 연결 포함)을 위한 Letta 형태의 블록 시스템을 생성하며, 수면 시간 훅을 제공합니다.

## 연습 문제

1. `near_limit`가 true를 반환할 때 블록 값을 모델 생성 요약으로 대체하는 `block_summarize` 도구를 추가하세요. 어떤 트리거 임계값이 요약 호출과 블록 오버플로우를 모두 최소화하나요?
2. 아카이빙 시 수면 시간 중복 제거(sleep-time dedup)를 구현하세요: 텍스트의 토큰 중복도가 90%를 초과하는 두 레코드는 하나로 병합합니다. 이는 수면 단계에서 수행하며, 절대 크리티컬 경로에서는 실행하지 마세요.
3. 블록 버전 관리. 모든 쓰기 작업 시 이전 값과 차이(diff)를 기록합니다. `block_history(label)`을 노출시켜 운영자가 "에이전트가 X를 왜 잊었는가?"를 디버깅할 수 있게 합니다.
4. 수면 시간 에이전트를 신뢰할 수 없는 작성자로 처리합니다. Persona 또는 Safety 블록을 수정할 경우, 커밋 전 두 번째 에이전트의 검토를 요구하세요.
5. 예제를 Letta API(`letta_v1_agent`)를 사용하도록 포팅하세요. 블록 스키마에서 어떤 변경이 발생하며, 네이티브 추론이 트레이스 형태를 어떻게 바꾸나요?
