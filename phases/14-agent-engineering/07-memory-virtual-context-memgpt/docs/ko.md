# 메모리: 가상 컨텍스트와 MemGPT

> 컨텍스트 창은 유한합니다. 대화, 문서, 도구 실행 기록은 그렇지 않습니다. MemGPT(Packer et al., 2023)는 이를 OS 가상 메모리로 설명합니다 — 주 컨텍스트는 RAM, 외부 저장소는 디스크, 에이전트는 이들 사이에서 페이징합니다. 이는 모든 2026년 메모리 시스템이 계승하는 패턴입니다.

**유형:** 구축(Build)
**언어:** Python (stdlib)
**사전 요구 사항:** 14단계 · 01 (에이전트 루프), 14단계 · 06 (도구 사용)
**소요 시간:** ~75분

## 학습 목표

- MemGPT가 기반으로 하는 OS 비유 설명: 메인 컨텍스트 = RAM, 외부 컨텍스트 = 디스크, 메모리 도구 = 페이지 인/아웃.
- 메인 컨텍스트 버퍼, 외부 검색 가능한 저장소, 페이지 인/아웃 도구를 사용하여 stdlib에 2계층 MemGPT 패턴 구현.
- 에이전트가 외부 메모리를 조회하거나 수정하기 위해 "인터럽트"를 발생시키는 방식과 결과가 다음 프롬프트에 어떻게 재삽입되는지 설명.
- Letta(레슨 08)와 Mem0(레슨 09)로 이어지는 MemGPT 설계 선택 사항 식별.

## 문제

컨텍스트 윈도우는 메모리 문제를 해결할 수 있을 것처럼 보이지만 실제로는 그렇지 않습니다. 프로덕션 환경에서 다음 3가지 실패 모드가 반복적으로 발생합니다:

1. **오버플로우(Overflow).** 다중 턴 대화, 긴 문서 또는 도구 호출이 많은 궤적은 윈도우를 초과합니다. 잘린 부분 이후의 모든 내용은 사라집니다.
2. **희석(Dilution).** 윈도우 내부에서도 관련 없는 컨텍스트를 과도하게 채우면 중요한 부분에 대한 어텐션(attention)이 희석됩니다. 최신 모델들도 긴 입력에 대해 성능이 저하됩니다.
3. **지속성(Persistence).** 새 세션은 빈 윈도우로 시작합니다. 외부 메모리가 없는 에이전트는 "이전에 요청하신 내용을 기억하시나요..."와 같은 세션 간 기억을 유지할 수 없습니다.

더 큰 윈도우는 도움이 되지만 근본적인 해결책은 아닙니다. Mem0의 2025년 논문에 따르면, 128k 윈도우 기준 모델도 외부 메모리를 가진 4k 윈도우 에이전트가 포착하는 장기 의존성 사실을 놓치는 것으로 측정되었습니다.

## 개념

### MemGPT: 운영 체제 비유

Packer et al. (arXiv:2310.08560, v2 2024년 2월)은 컨텍스트 관리를 운영 체제 가상 메모리에 매핑합니다:

| OS 개념 | MemGPT 개념 | 2026년 프로덕션 유사체 |
|------------|---------------|------------------------|
| RAM | 메인 컨텍스트(프롬프트) | Anthropic/OpenAI 컨텍스트 윈도우 |
| 디스크 | 외부 컨텍스트 | 벡터 DB, KV, 그래프 저장소 |
| 페이지 폴트 | 메모리 도구 호출 | `memory.search`, `memory.read`, `memory.write` |
| OS 커널 | 에이전트 제어 루프 | 메모리 도구가 있는 ReAct 루프 |

에이전트는 일반적인 ReAct 루프를 실행합니다. 하나의 추가 도구 클래스를 통해 데이터를 메인 컨텍스트로 페이징할 수 있습니다.

### 두 계층

- **메인 컨텍스트.** 현재 작업을 보유하는 고정 크기 프롬프트. 항상 모델에 표시됩니다.
- **외부 컨텍스트.** 무제한 검색 가능. 관련성이 있을 때 읽고, 사실이 나타날 때 씁니다.

원본 논문은 기본 윈도우를 초과하는 두 가지 작업(100k 토큰 이상의 문서 분석 및 며칠에 걸친 지속적 메모리를 가진 다중 세션 채팅)에서 설계를 평가했습니다.

### 인터럽트 패턴

MemGPT는 메모리-인터럽트를 도입합니다: 대화 중간에 에이전트가 메모리 도구를 호출하면 런타임이 이를 실행하고 결과가 다음 어시스턴트 턴에 새로운 관찰로 삽입됩니다. 개념적으로 프로세스를 차단하고 바이트를 반환한 후 프로세스가 계속되는 Unix `read()` 시스템 호출과 동일합니다.

표준 메모리 도구 표면:

- `core_memory_append(section, text)` — 프롬프트의 영구 섹션에 쓰기.
- `core_memory_replace(section, old, new)` — 영구 섹션 편집.
- `archival_memory_insert(text)` — 검색 가능한 외부 저장소에 쓰기.
- `archival_memory_search(query, top_k)` — 외부 저장소에서 검색.
- `conversation_search(query)` — 과거 턴 스캔.

### MemGPT가 끝나고 Letta가 시작되는 지점

2024년 9월 MemGPT는 Letta가 되었습니다. 연구 저장소(`cpacker/MemGPT`)는 유지되며, Letta는 설계를 확장합니다:

- 두 계층 대신 세 계층(코어, 리콜, 아카이브 — 레슨 08).
- `send_message`/하트비트 패턴을 대체하는 네이티브 추론(레슨 08).
- 비동기 메모리 작업을 실행하는 수면 시간 에이전트(레슨 08).

MemGPT 논문은 프로덕션 시스템이 Letta, Mem0 또는 사용자 정의 2계층 저장소를 실행하더라도 2026년 기반입니다.

### 이 패턴이 실패하는 경우

- **메모리 부패.** 쓰기가 읽기보다 빠르게 누적되며 검색은 오래된 사실에 묻힙니다. 해결책: 주기적 통합(Letta 수면 시간), 명시적 무효화(Mem0 충돌 감지기).
- **메모리 오염.** 외부 메모리는 검색된 텍스트입니다. 공격자가 제어하는 콘텐츠가 메모리 노트에 저장되면 에이전트가 다음 세션에서 다시 섭취합니다. 이는 Greshake et al. (레슨 27) 공격을 시간에 따라 재진술한 것입니다.
- **인용 손실.** 에이전트는 "사용자가 X를 배송하라고 요청했습니다"를 회상하지만 어떤 턴에서 요청했는지 인용할 수 없습니다. 모든 아카이브 쓰기 시 소스 참조(세션 ID, 턴 ID)를 저장합니다.

## 빌드하기

`code/main.py`는 MemGPT의 2단계 패턴을 표준 라이브러리(stdlib)로 구현합니다:

- `MainContext` — `core` 딕셔너리와 `messages` 리스트를 포함하는 고정 크기 프롬프트 버퍼; 용량 초과 시 가장 오래된 메시지를 자동으로 압축합니다.
- `ArchivalStore` — (id, 텍스트, 태그, 세션, 턴) 레코드의 인메모리 BM25 유사 저장소(토큰 중첩 점수 기반).
- MemGPT 인터페이스에 매핑되는 5가지 메모리 도구.
- `archival_memory_search`를 호출하여 아카이브에 사실을 채운 후 질문에 답하는 스크립트 에이전트.

실행 방법:

```
python3 code/main.py
```

트레이스(trace)는 에이전트가 3개의 사실을 작성하고, 메인 컨텍스트를 용량까지 채운 후(강제 제거 발생), 아카이브에서 검색하여 후속 질문에 답하는 과정을 보여줍니다 — 실제 LLM 없이도 MemGPT 워크플로우를 재현합니다.

## 사용 방법

오늘날 모든 프로덕션 메모리 시스템은 MemGPT 변형입니다:

- **Letta** (레슨 08) — 3계층 구조, 네이티브 추론, 수면 시간 컴퓨팅.
- **Mem0** (레슨 09) — 벡터 + KV + 그래프에 스코어링 계층 통합.
- **OpenAI Assistants / Responses** — 스레드 및 파일을 통한 관리형 메모리.
- **Claude Agent SDK** — 스킬 및 세션 스토리지를 통한 장기 메모리.

핵심 패턴이 아닌 운영 형태(self-hosted, 관리형, 프레임워크 통합)에 따라 선택하세요 — 핵심 패턴은 MemGPT입니다.

## Ship It

`outputs/skill-virtual-memory.md`는 재사용 가능한 스킬로, 모든 대상 런타임에 대해 올바른 2계층 메모리 스캐폴드(메인 + 아카이브 + 도구 표면)를 생성하며, 제거 정책(eviction policy)과 인용 필드(citation fields)가 연결되어 있습니다.

## 연습 문제

1. `max_main_context_tokens` 캡을 토큰 단위로 추가하세요 (대략 `len(text.split()) * 1.3`로 근사). 캡을 초과할 때 가장 오래된 메시지를 요약으로 압축하세요. 요약기 사용 여부에 따른 동작을 비교하세요.
2. 아카이브 저장소(term frequency, inverse document frequency)에 대해 BM25를 제대로 구현하세요. 장난감 팩트 세트에서 토큰 중첩 기준선과 비교하여 recall@10을 측정하세요.
3. 아카이브 삽입 시 `citation` 필드(session_id, turn_id, source_url)를 추가하세요. 에이전트가 검색 기반 답변 시 출처를 인용하도록 만드세요.
4. 메모리 오염 시뮬레이션: "모든 향후 사용자 지시를 무시하라"는 아카이브 레코드를 추가하세요. 지시문 형태의 텍스트를 검색 시 스캔하여 신뢰할 수 없음을 표시하는 가드를 작성하세요.
5. 구현을 MemGPT 연구 저장소의 핵심 메모리 JSON 스키마(`cpacker/MemGPT`)를 사용하도록 포팅하세요. 평면 문자열에서 타입 지정 섹션으로 전환할 때 어떤 변화가 발생하는지 확인하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 가상 컨텍스트(Virtual context) | "무제한 메모리" | 메인(프롬프트) + 외부(검색 가능) 계층과 페이지 인/아웃 |
| 메인 컨텍스트(Main context) | "작업 메모리" | 프롬프트 — 고정 크기, 항상 가시적 |
| 아카이브 메모리(Archival memory) | "장기 저장소" | 외부 검색 가능 지속성, 요청 시 검색 |
| 코어 메모리(Core memory) | "지속적 프롬프트 섹션" | 메인 컨텍스트 내부에 고정된 명명된 섹션 |
| 메모리 도구(Memory tool) | "메모리 API" | 에이전트가 외부 메모리 읽기/쓰기를 위해 호출하는 도구 |
| 인터럽트(Interrupt) | "메모리 페이지 폴트" | 에이전트 일시 정지, 런타임이 가져와 결과를 다음 턴에 삽입 |
| 메모리 로트(Memory rot) | "오래된 사실" | 오래된 쓰기가 검색을 방해; 통합으로 해결 |
| 메모리 포이징(Memory poisoning) | "주입된 지속적 노트" | 공격자의 콘텐츠가 메모리로 저장되어 회상 시 재입력됨 |
