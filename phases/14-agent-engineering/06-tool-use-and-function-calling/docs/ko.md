# 도구 사용 및 함수 호출

> Toolformer (Schick et al., 2023)는 자기 지도 도구 주석을 시작했습니다. Berkeley Function Calling Leaderboard V4 (Patil et al., 2025)는 2026년 기준: 40% 에이전트형, 30% 다중 턴, 10% 실시간, 10% 비실시간, 10% 환각. 단일 턴은 해결됨. 메모리, 동적 의사 결정, 장기 도구 체인은 미해결.

**유형:** 구축
**언어:** Python (표준 라이브러리)
**선수 지식:** 14단계 · 01 (에이전트 루프), 13단계 · 01 (함수 호출 심층 분석)
**소요 시간:** ~60분

## 학습 목표

- Toolformer의 자기 지도 학습 신호 설명: 실행 시 다음 토큰 손실(next-token loss)이 감소할 때만 도구 주석(tool annotations)을 유지.
- BFCL V4의 5가지 평가 범주와 각 측정 항목 명명:
  - **정확성(Accuracy)**: 도구 사용 시 작업 완료의 정확도
  - **효율성(Efficiency)**: 목표 달성에 필요한 도구 호출 횟수
  - **강건성(Robustness)**: 불완전한/노이즈 있는 입력에서의 성능
  - **일반화(Generalization)**: 훈련 데이터 분포 외 작업 적응력
  - **안전성(Safety)**: 유해/위험한 도구 사용 방지 능력
- 스키마 검증(schema validation), 인수 강제 변환(argument coercion), 실행 샌드박싱(execution sandboxing)을 갖춘 stdlib 도구 레지스트리 구현.
- 2026년 미해결 과제 3가지 진단:
  - **장거리 도구 체이닝(long-horizon tool chaining)**: 다중 단계 도구 상호작용 최적화
  - **동적 의사 결정(dynamic decision-making)**: 실시간 환경 변화에 따른 도구 선택 적응
  - **메모리(memory)**: 장기적 컨텍스트 유지를 위한 도구 사용 기록 관리

## 문제 정의

초기 도구 사용은 "모델이 올바른 함수 호출을 예측할 수 있는가?"라는 질문을 던졌습니다. 현대 도구 사용은 "모델이 40단계에 걸쳐 도구를 연결할 수 있는가? 메모리, 부분 관측 가능성, 도구 실패 복구 기능을 갖추면서 존재하지 않는 도구를 환각(hallucination)하지 않고 작업을 수행할 수 있는가?"라는 질문을 던집니다.

Toolformer는 기준선을 확립했습니다: 모델은 자기 지도 학습(self-supervision)을 통해 도구를 호출할 시점을 학습할 수 있습니다. BFCL V4는 2026년 평가 목표를 정의합니다. 이 둘 사이의 격차가 바로 프로덕션 에이전트(production agents)가 활동하는 공간입니다.

## 개념

### Toolformer (Schick et al., NeurIPS 2023)

아이디어: 모델이 사전 학습 코퍼스에 후보 API 호출을 직접 주석 처리하도록 합니다. 각 후보에 대해 실행한 후, 다음 토큰의 손실을 줄이는 경우에만 주석을 유지합니다. 필터링된 코퍼스로 미세 조정합니다.

지원 도구: 계산기, QA 시스템, 검색 엔진, 번역기, 캘린더. 자기 지도 학습 신호는 도구 사용이 텍스트 예측에 도움이 되는지 여부만 고려합니다 — 인간 라벨은 필요 없습니다.

규모 결과: 대규모 모델에서 도구 사용이 자연스럽게 나타납니다. 작은 모델은 도구 주석으로 인해 성능이 저하되지만, 큰 모델은 이점을 얻습니다. 이것이 2026년 프론티어 모델에는 강력한 도구 사용이 내장된 반면, 대부분의 7B 모델은 신뢰할 수 있는 도구 사용을 위해 명시적인 미세 조정이 필요한 이유입니다.

### Berkeley Function Calling Leaderboard V4 (Patil et al., ICML 2025)

BFCL은 2026년 사실상의 평가 기준입니다. V4 구성:

- **에이전틱(40%)** — 전체 에이전트 궤적: 메모리, 다중 턴, 동적 결정.
- **멀티턴(30%)** — 도구 체인을 사용한 대화형 대화.
- **라이브(10%)** — 사용자 제출 실제 프롬프트(더 어려운 분포).
- **논라이브(10%)** — 합성 테스트 케이스.
- **환각(10%)** — 도구를 호출하지 말아야 할 경우 감지.

V3는 상태 기반 평가를 도입했습니다: 도구 시퀀스 후 API의 실제 상태(예: "파일이 생성되었는가?")를 확인하는 대신 도구 호출의 AST를 일치시킵니다. V4는 웹 검색, 메모리, 형식 민감도 범주를 추가했습니다.

2026년 주요 발견: 단일 턴 함수 호출은 거의 해결되었습니다. 실패는 메모리(턴 간 컨텍스트 유지), 동적 의사 결정(이전 결과에 기반한 도구 선택), 장기 체인(20+ 단계 후 드리프트), 환각 감지(적합한 도구가 없을 때 호출 거부)에 집중됩니다.

### 도구 스키마

모든 공급자는 스키마를 가지고 있습니다. 세부 사항은 다르지만 구조는 동일합니다:

```
name: 문자열
description: 문자열 (기능, 사용 시기)
input_schema: JSON 스키마 (속성, 필수, 유형, 열거형)
```

Anthropic은 `input_schema`를 직접 사용합니다. OpenAI는 `function.parameters`를 사용합니다. 둘 다 JSON 스키마를 허용합니다. 설명은 매우 중요합니다 — 모델이 올바른 도구를 선택하기 위해 이를 읽습니다. 나쁜 도구 설명은 잘못된 도구 선택 실패의 1순위 근본 원인입니다.

### 인수 검증

어떤 도구 호출도 신뢰하지 마세요. 다음을 검증하세요:

1. **형식 강제.** 모델이 스키마에서 `int`로 지정된 곳에 문자열 `"5"`를 반환할 수 있습니다. 모호하지 않으면 강제 변환하고, 그렇지 않으면 거부합니다.
2. **열거형 검증.** 스키마에서 `status in {"open", "closed"}`로 지정되었는데 모델이 `"in_progress"`를 반환하면, 설명적 오류와 함께 거부합니다.
3. **필수 필드.** 필수 필드 누락 → 모델에게 즉시 오류 관찰 전달, 충돌이 아님.
4. **형식 검증.** 날짜, 이메일, URL — 정규식이 아닌 구체적인 파서로 검증합니다.

모든 검증 실패는 모델이 올바른 형태로 재시도할 수 있도록 구조화된 관찰을 반환해야 합니다.

### 병렬 도구 호출

현대 공급자는 한 어시스턴트 턴에서 병렬 도구 호출을 지원합니다. 루프:

1. 모델이 고유한 `tool_use_id`를 가진 3개의 도구 호출을 방출합니다.
2. 런타임이 이를 실행합니다(독립적인 경우 병렬로).
3. 각 결과는 `tool_use_id`로 연관된 `tool_result` 블록으로 반환됩니다.

엔지니어링 규칙: 상관 ID를 매우 중요하게 취급합니다. 이를 교환하면 잘못된 도구가 잘못된 결과를 수신하게 됩니다.

### 샌드박싱

도구 실행은 샌드박스 경계입니다. 자세한 내용은 레슨 09를 참조하세요. 간단한 버전: 모든 도구는 읽기/쓰기 표면, 네트워크 접근, 타임아웃, 메모리 제한을 명시해야 합니다. 일반적인 `run_shell(cmd)`는 위험 신호이며, 특정 `git_status()`가 더 안전합니다.

## 빌드하기

`code/main.py`는 프로덕션 형태의 도구 레지스트리를 구현합니다:

- JSON Schema 부분 집합 검증기(표준 라이브러리만 사용).
- 설명, 입력 스키마, 타임아웃, 실행기를 포함한 도구 등록.
- 인수 강제 변환 및 열거형(enum) 검증.
- 상관 ID를 사용한 병렬 도구 디스패치.
- 구조화된 문자열로 된 오류 관찰.

실행 방법:

```
python3 code/main.py
```

트레이스에는 미니 에이전트가 한 턴에 세 개의 도구를 호출하는 과정이 표시되며, 의도적으로 잘못된 형식의 호출 하나가 포함되어 있습니다. 이 호출은 모델이 처리할 수 있는 설명적 오류와 함께 거부됩니다.

## 사용 방법

모든 공급자(Anthropic, OpenAI, Gemini, Bedrock)는 자체 도구 스키마를 가지고 있습니다. 여러 공급자를 사용해야 할 경우 번역 계층(OpenAI Agents SDK, Vercel AI SDK, LangChain 도구 어댑터)을 활용하세요. BFCL은 참조 벤치마크입니다 — 도구 사용이 제품의 핵심이라면 출시 전에 에이전트에 대해 BFCL을 실행해 보세요.

## Ship It

`outputs/skill-tool-registry.md`는 주어진 작업 도메인에 대한 도구 카탈로그, 스키마 및 레지스트리를 생성합니다. 각 도구의 설명 품질 검사(도구 설명이 모델에게 언제 해당 도구를 사용해야 하는지 알려주는지 여부)를 포함합니다.

## 연습 문제

1. 모델이 다른 도구를 명시적으로 거부하도록 하는 "no-op" 도구를 추가하세요. BFCL과 유사한 환각 테스트(hallucination test)에서 측정하세요.  
2. 문자열로 표현된 정수(int-as-string) 및 부동소수점(float-as-string)에 대한 인수 강제 변환(argument coercion)을 구현하세요. 강제 변환이 실제 버그를 숨기기 시작하는 지점은 어디인가요?  
3. 도구별 타임아웃과 서킷 브레이커(3회 연속 실패 후 60초간 도구 사용 거부)를 추가하세요. 이 기능이 모델의 복구 방식에 어떤 변화를 가져오나요?  
4. BFCL V4 설명을 읽고 하나의 범주(예: "multi-turn")를 선택하세요. 10개의 예시 프롬프트를 에이전트에 실행시키고 통과율(pass rate)을 보고하세요.  
5. stdlib 검증기를 Pydantic 또는 Zod로 이식하세요. Pydantic/Zod가 기존 간단한 검증기가 놓친 어떤 문제를 잡아내나요?

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| 함수 호출(Function calling) | "툴 사용(Tool use)" | 검증된 스키마를 통한 구조화된 출력 툴호출 |
| 툴포머(Toolformer) | "자기 지도식 툴 주석(Self-supervised tool annotation)" | Schick 2023 — 다음 토큰 손실을 줄이는 결과를 가진 툴호출 유지 |
| BFCL(Berkeley Function Calling Leaderboard) | "버클리 함수 호출 리더보드" | 2026 벤치마크: 40% 에이전트형, 30% 다중 턴, 10% 실시간, 10% 비실시간, 10% 환각 |
| 툴스키마(Tool schema) | "모델을 위한 함수 시그니처" | 이름, 설명, 인수의 JSON 스키마 |
| 툴사용ID(tool_use_id) | "상관 ID(Correlation ID)" | 툴호출과 결과를 연결; 병렬 디스패치에 필수적 |
| 환각 탐지(Hallucination detection) | "호출하지 말아야 할 때 알기" | V4 범주: 어떤 툴도 적합하지 않을 때 호출 거부 |
| 인수 강제 변환(Argument coercion) | "문자열-정수 복구(String-to-int repair)" | 예측 가능한 스키마 불일치에 대한 좁은 수정; 모호할 경우 거부 |
| 샌드박싱(Sandboxing) | "툴 실행 경계" | 툴별 읽기/쓰기 표면, 네트워크, 타임아웃, 메모리 제한
