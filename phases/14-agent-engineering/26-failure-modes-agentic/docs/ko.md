# 실패 모드: 에이전트가 실패하는 이유

> MASFT(버클리, 2025)는 3개 범주에 걸쳐 14가지 다중 에이전트 실패 모드를 분류합니다. Microsoft의 Taxonomy 문서는 기존 AI 실패가 에이전트 환경에서 어떻게 증폭되는지 설명합니다. 산업 현장 데이터는 5가지 반복되는 모드(환각적 행동, 범위 확장, 연쇄적 오류, 컨텍스트 손실, 도구 오용)로 수렴됩니다.

**유형:** 학습 + 실습  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 14단계 · 05(자기 개선 및 CRITIC), 14단계 · 24(관측 가능성)  
**소요 시간:** ~60분

## 학습 목표

- MASFT의 세 가지 실패 범주와 각 범주별 최소 네 가지 구체적인 실패 모드를 설명할 수 있다.
- 에이전트 실패(agentic failure)가 기존 AI 실패 모드(편향, 환각)를 증폭시키는 이유를 설명할 수 있다.
- 산업계에서 반복적으로 발생하는 다섯 가지 실패 모드와 그 완화 방안을 설명할 수 있다.
- 실패 모드 레이블을 에이전트 추적(trace)에 태그하는 stdlib 탐지기를 구현할 수 있다.

## 문제 정의

팀들은 90%의 트레이스에서 작동하는 에이전트를 배포합니다. 10%의 실패는 무작위 노이즈가 아닙니다 — 이는 소수의 반복되는 범주에 속합니다. 이러한 실패 유형을 명명할 수 있게 되면, 이를 모니터링하고 수정할 수 있습니다.

## 개념

### MASFT (Berkeley, arXiv:2503.13657)

다중 에이전트 시스템 장애 분류 체계. 14가지 장애 유형을 3개 범주로 클러스터링. 평가자 간 코헨 카파(Cohen's Kappa) 0.88 — 범주가 신뢰성 있게 구분 가능.

주요 주장: 장애는 다중 에이전트 시스템의 근본적 설계 결함이며, 더 나은 기본 모델로 해결할 수 있는 LLM 한계가 아님.

### Microsoft 에이전트 AI 시스템 장애 유형 분류

- 기존 AI 장애(편향, 환각, 데이터 유출)가 에이전트 환경에서 증폭됨.
- 자율성으로부터 새로운 장애 발생: 대규모 의도치 않은 행동, 도구 오용, 임무 이탈.
- 백서는 에이전트 제품의 위험 등록부(risk register) 역할.

### 에이전트 AI 결함 특성 분석 (arXiv:2603.06847)

- 장애는 오케스트레이션, 내부 상태 진화, 환경 상호작용에서 발생.
- 단순한 "나쁜 코드"나 "나쁜 모델 출력"이 아님.

### LLM 에이전트 환각 현상 조사 (arXiv:2509.18970)

두 가지 주요 현상:

1. **지시 사항 이탈** — 에이전트가 시스템 프롬프트를 따르지 않음.
2. **장거리 컨텍스트 오용** — 에이전트가 이전 턴의 컨텍스트를 잊거나 잘못 적용.

하위 의도 오류: 생략(누락된 단계), 중복(반복된 단계), 무질서(순서가 뒤바뀐 단계).

### 산업에서 반복되는 5가지 장애 유형

Arize, Galileo, NimbleBrain의 2024-2026 현장 분석 결과:

1. **환각된 행동.** 에이전트가 존재하지 않는 도구를 호출하거나 주장을 날조.
2. **범위 확장.** 에이전트가 사용자의 요청을 넘어 작업 확장(추가 PR 생성, 추가 이메일 전송).
3. **연쇄적 오류.** 하나의 잘못된 호출이 다운스트림 효과 유발. 유령 SKU 환각으로 4개의 API 호출 발생 — 다중 시스템 장애.
4. **컨텍스트 손실.** 장기 작업에서 초기 턴 제약 조건 잊음.
5. **도구 오용.** 올바른 도구에 잘못된 인수로 호출하거나 완전히 잘못된 도구 선택.

연쇄적 오류가 가장 치명적. 에이전트는 "실패"와 "작업 불가능"을 구분하지 못하며, 400 오류 시 성공 메시지를 환각하여 루프를 종료하는 경우가 많음.

### 완화 방안: 단계별 검증 게이트

추론 체인의 모든 단계에 자동화된 검증 게이트 설치, 환경 상태와 사실적 근거 확인. 구체적 방법:

- 단계별 안전 분류기(레슨 21).
- 도구 호출 인수 검증(레슨 06).
- 검색된 콘텐츠를 알려진 사실과 교차 확인(레슨 05, CRITIC).
- 상태 재확인으로 성공 환각 탐지(파일 실제 생성 여부 확인).

### 장애 모니터링이 실패하는 경우

- **크래시만 태깅.** 대부분의 에이전트 장애는 유효한 출력 생성. 콘텐츠 수준 검증 필요.
- **기준선 부재.** 드리프트 감지에는 마지막 정상 상태 기준 필요. 없으면 "악화 중"이라고 판단할 수 없음.
- **과도한 알림.** 모든 장애가 페이지 생성. 클러스터링 및 빈도 제한 필요.

## 빌드하기

`code/main.py`는 표준 라이브러리 실패 모드 태거를 구현합니다:

- 5가지 모드를 포함하는 합성 트레이스 데이터셋.
- 모드별 감지기 함수(툴 호출, 출력, 반복 액션에 대한 시그니처 패턴).
- 각 트레이스에 레이블을 지정하고 모드 분포를 보고하는 태거.

실행 방법:

```
python3 code/main.py
```

출력: 트레이스별 레이블 + 집계된 분포, 피닉스의 트레이스 클러스터링이 표면화하는 결과를 저비용으로 재현한 것.

## 사용 방법

- **Phoenix**를 프로덕션 드리프트 클러스터링에 사용 (레슨 24).
- **Langfuse**를 세션 재생 + 주석에 사용.
- **Custom**을 관측성 플랫폼이 감지할 수 없는 도메인 특화 시그니처에 사용.

## Ship It

`outputs/skill-failure-detector.md`는 도메인에 맞춤화된 실패 모드 감지기(failure-mode detector)를 생성하며, 트레이스 저장소(trace store)에 연결됩니다.

## 연습 문제

1. "성공 환각(success hallucination)" 감지기 추가: 에이전트가 성공을 반환했지만 목표 상태가 변경되지 않은 경우.
2. 구축한 제품의 실제 트레이스 100개에 태그 지정. 어떤 모드가 지배적인가? 이를 수정하는 데 드는 비용은 얼마인가?
3. "계단식 반경(cascade radius)" 메트릭 구현: N단계에서 발생한 실패가 몇 개의 후속 단계에 영향을 미쳤는지 계산.
4. MASFT의 14가지 실패 모드 읽기. 제품에 적용 가능한 3가지 선택. 감지기 작성.
5. CI 작업에 감지기 연결: 트레이스의 5% 이상이 특정 모드를 태그할 경우 빌드 실패 처리.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| MASFT | "다중 에이전트 실패 분류 체계" | 버클리 14-모드 분류 체계 |
| Cascading error | "연쇄적 실패" | 초기 실수가 N단계로 전파됨 |
| Context loss | "제약 조건 망각" | 장기 계획에서 초기 턴 정보 소실 |
| Tool misuse | "잘못된 도구/잘못된 인자" | 유효한 호출이지만 잘못된 실행 |
| Success hallucination | "가짜 완료" | 에이전트가 400 오류 상태에서 성공 주장; 상태 변화 없음 |
| Scope creep | "과도한 확장" | 요청받은 작업 이상의 추가 수행 |
| Instruction-following deviation | "지시 불이행" | 시스템 프롬프트 또는 사용자 제약 조건 무시 |
| Sub-intention errors | "계획 오류" | 계획 실행 시 누락, 중복, 순서 오류 발생 |
