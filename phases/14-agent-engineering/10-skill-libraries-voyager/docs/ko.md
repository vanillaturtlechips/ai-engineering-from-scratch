# 스킬 라이브러리와 평생 학습 (Voyager)

> Voyager (Wang et al., TMLR 2024)는 실행 가능한 코드를 스킬로 취급합니다. 스킬은 이름이 지정되며, 검색 가능하고, 조합 가능하며, 환경 피드백에 의해 개선됩니다. 이는 Claude Agent SDK 스킬, 스킬킷 및 2026년 스킬 라이브러리 패턴의 참조 아키텍처입니다.

**유형:** 구축(Build)
**언어:** Python (stdlib)
**선수 조건:** 14단계 · 07 (MemGPT), 14단계 · 08 (Letta Blocks)
**소요 시간:** ~75분

## 학습 목표

- Voyager의 세 가지 구성 요소인 **자동 커리큘럼(automatic curriculum)**, **스킬 라이브러리(skill library)**, **반복적 프롬프팅(iterative prompting)**을 각각 설명하고 각 구성 요소의 역할을 기술할 수 있다.
- Voyager가 **원시 명령어(primitive commands)**가 아닌 **액션 공간 코드(action space code)**를 사용하는 이유를 설명할 수 있다.
- **등록(registration)**, **검색(retrieval)**, **조합(composition)**, **실패 기반 개선(failure-driven refinement)** 기능을 갖춘 **stdlib 스킬 라이브러리**를 구현할 수 있다.
- Voyager의 패턴을 **2026 Claude Agent SDK 스킬** 및 **스킬킷(skillkit) 생태계**에 매핑할 수 있다.

## 문제

모든 세션에서 모든 기능을 처음부터 다시 구축하는 에이전트는 다음 세 가지 잘못된 방식을 보입니다:

1. **토큰 낭비.** 모든 작업에서 동일한 추론을 다시 유도합니다.
2. **진행 상황 손실.** 세션 A에서 학습한 수정 사항이 세션 B로 이전되지 않습니다.
3. **장기적 구성 실패.** 복잡한 작업에는 기능 계층 구조가 필요하지만, 일회성 프롬프트로는 이를 표현할 수 없습니다.

Voyager의 해결책: 재사용 가능한 각 기능을 라이브러리에 저장된 이름 있는 코드 조각으로 취급하고, 유사도 기반으로 검색하며, 다른 기술과 조합 가능하고, 실행 피드백을 통해 개선할 수 있도록 합니다.

## 개념

### 세 가지 구성 요소

Voyager (arXiv:2305.16291)는 에이전트를 다음 세 가지 구성 요소로 구조화합니다:

1. **자동 커리큘럼.** 호기심 기반 제안자(proposer)가 에이전트의 현재 기술 세트와 환경 상태를 기반으로 다음 작업을 선택합니다. 탐색은 상향식(bottom-up)으로 이루어집니다.
2. **기술 라이브러리.** 각 기술은 실행 가능한 코드입니다. 작업 성공 시 새로운 기술이 추가됩니다. 기술은 쿼리-설명 유사도를 기반으로 검색됩니다.
3. **반복적 프롬프팅 메커니즘.** 실패 시 에이전트는 실행 오류, 환경 피드백, 자체 검증 출력을 받은 후 기술을 개선합니다.

Minecraft 평가(Wang et al., 2024): 기준 모델 대비 3.3배 더 많은 고유 아이템, 8.5배 빠른 돌 도구 제작, 6.4배 빠른 철 도구 제작, 2.3배 긴 맵 탐색. 이 수치는 Minecraft에 특화되어 있지만 패턴은 전이 가능합니다.

### 액션 공간 = 코드

대부분의 에이전트는 원시 명령을 출력합니다. Voyager는 JavaScript 함수를 출력합니다. 기술(skill)은 다음과 같습니다:

```
async function craftIronPickaxe(bot) {
  await mineIron(bot, 3);
  await mineStick(bot, 2);
  await placeCraftingTable(bot);
  await craft(bot, 'iron_pickaxe');
}
```

하위 기술(sub-skills)로 구성됩니다. 설명과 임베딩을 키로 저장됩니다. 프롬프트가 아닌 프로그램으로 검색됩니다.

이것은 2026년 Claude Agent SDK 기술입니다: 이름이 지정된 검색 가능한 코드 청크 + 에이전트가 필요 시 로드하는 지침.

### 기술 검색

새 작업 "다이아몬드 곡괭이 만들기." 에이전트는:

1. 작업 설명을 임베딩합니다.
2. 기술 라이브러리에서 유사도 상위 k개 기술을 쿼리합니다.
3. `craftIronPickaxe`, `mineDiamond`, `placeCraftingTable` 등을 검색합니다.
4. 검색된 기본 기술 + 새로운 논리로 새 기술을 구성합니다.

이것은 MCP 리소스(Phase 13)와 Agent SDK 기술이 구현하는 패턴입니다: 현재 작업에 스코프된 지식/코드 표면 위의 검색.

### 반복적 개선

Voyager의 피드백 루프:

1. 에이전트가 기술을 작성합니다.
2. 기술이 환경에 대해 실행됩니다.
3. 세 가지 신호 중 하나가 반환됩니다: `성공`, `오류`(스택 트레이스 포함), `자체 검증 실패`.
4. 에이전트는 신호를 컨텍스트로 사용해 기술을 재작성합니다.
5. 성공 또는 최대 라운드까지 반복합니다.

이것은 Self-Refine(레슨 05)을 환경 기반 검증이 있는 코드 생성에 적용한 것입니다. CRITIC(레슨 05)은 외부 도구를 검증자로 사용하는 동일한 패턴입니다.

### 커리큘럼과 탐색

Voyager의 커리큘럼 모듈은 에이전트가 보유한 것과 아직 수행하지 않은 것을 기반으로 "호수 근처에 대피소 건설"과 같은 작업을 제안합니다. 제안자는 환경 상태 + 기술 인벤토리를 사용해 현재 능력보다 약간 높은 작업을 선택합니다 — 탐색의 최적 지점입니다.

생산 에이전트의 경우 이는 "누락된 것" 연산자로 변환됩니다: 현재 기술 라이브러리와 도메인이 주어졌을 때, 아직 다루지 않은 기술은 무엇인가? 팀은 일반적으로 이를 수동으로 커리큘럼 검토로 구현합니다.

### 이 패턴이 실패하는 경우

- **기술 라이브러리 부패.** 약간 다른 설명으로 동일한 기술이 10번 추가됩니다. 쓰기 시 중복 제거 추가; 검색은 하나만 반환합니다.
- **구성된 기술 드리프트.** 부모 기술이 개선된 자식에 의존합니다. 기술 버전 관리; v1에 고정된 부모는 v3을 자동으로 가져오지 않습니다.
- **검색 품질.** 기술 설명에 대한 벡터 검색은 라이브러리가 수백 개를 넘어가면 저하됩니다. 태그 필터와 하드 제약 조건("`category=tooling`이 있는 기술만")으로 보완합니다.

## 빌드하기

`code/main.py`는 표준 라이브러리 스킬 구현을 제공합니다:

- `Skill` — 이름, 설명, 코드(문자열 형식), 버전, 태그, 의존성.
- `SkillLibrary` — 등록, 검색(토큰 겹침 기준), 구성(의존성 토폴로지 정렬), 개선(업데이트 시 버전 업데이트).
- 3개의 기본 스킬을 등록하고, 4번째 스킬을 구성하며, 실패를 유발하고 개선하는 스크립트 기반 에이전트.

실행 방법:

```
python3 code/main.py
```

트레이스에는 라이브러리 쓰기, 검색, 구성, 실패한 실행, v2 개선 과정이 표시됩니다 — Voyager의 전체 루프를 확인할 수 있습니다.

## 사용 방법

- **Claude Agent SDK 스킬** (Anthropic) — 2026년 참조: 각 스킬에는 설명, 코드, 지침이 포함되며, 에이전트 세션 중 필요 시 동적으로 로드됩니다.
- **skillkit** (npm: skillkit) — 32개 이상의 AI 코딩 에이전트를 위한 크로스 에이전트 스킬 관리 도구.
- **커스텀 스킬 라이브러리** — 도메인 특화형 (데이터 에이전트용 SQL 스킬, 인프라 에이전트용 Terraform 스킬). Voyager 패턴은 축소 적용 가능합니다.
- **OpenAI Agents SDK `tools`** — 기본 수준; 각 툴은 경량 스킬로 구현됩니다.

## Ship It

`outputs/skill-skill-library.md`는 Voyager 형태의 스킬 라이브러리를 생성하며, 등록, 검색, 버전 관리 및 개선 기능이 모든 대상 런타임에 대해 연결되어 있습니다.

## 연습 문제

1. `compose()`에 의존성 사이클 감지기를 추가하세요. 기술 A가 B에 의존하고 B가 A에 의존할 때 어떤 일이 발생하나요? 오류 vs 경고?
2. 기술별 버전 고정 기능을 구현하세요. 부모 기술이 자식 기술 `crafting@1`을 조합할 때, `crafting@2`로의 개선이 부모 기술을 자동으로 업그레이드하지 않아야 합니다.
3. 토큰 중첩 검색 방식을 문장 임베딩(sentence-transformers) 또는 BM25 표준 라이브러리 구현으로 대체하세요. 50개 기술로 구성된 소규모 라이브러리에서 검색@5 성능을 측정하세요.
4. "커리큘럼" 에이전트를 추가하세요: 현재 라이브러리와 도메인 설명을 기반으로 5개의 누락된 기술을 제안하세요. 매주 실행하도록 합니다.
5. Anthropic의 Claude Agent SDK 기술 문서를 읽고, 소규모 라이브러리를 SDK의 기술 스키마로 이식하세요. 발견 가능성에 어떤 변화가 생기나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 스킬(Skill) | "재사용 가능한 능력" | 유사성 기반 검색이 가능한 이름 붙은 코드 조각 + 설명 |
| 스킬 라이브러리(Skill library) | "에이전트의 방법 메모리" | 검색 및 조합 가능한 스킬의 영구 저장소 |
| 커리큘럼(Curriculum) | "작업 제안자" | 현재 능력 격차에 의해 구동되는 상향식 목표 생성기 |
| 컴포지션(Composition) | "스킬 DAG" | 스킬을 호출하는 스킬; 실행 시 위상 정렬됨 |
| 반복적 개선(Iterative refinement) | "자기 수정 루프" | 환경 피드백 + 오류 + 자기 검증이 다음 버전으로 반영됨 |
| 액션-스페이스-애즈-코드(Action-space-as-code) | "프로그래밍 액션" | 시간적으로 확장된 동작을 위한 원시 명령이 아닌 함수 출력 |
| 쓰기 시 중복 제거(Dedup on write) | "스킬 축소" | 거의 중복된 설명이 하나의 표준 스킬로 축소됨>
