# 오케스트레이션 패턴: 슈퍼바이저, 스웜, 계층적

> 2026개 프레임워크에서 반복되는 4가지 오케스트레이션 패턴: 슈퍼바이저-워커, 스웜/피어-투-피어, 계층적, 토론. Anthropic의 조언: "요구사항에 맞는 시스템을 구축하는 것이 중요합니다." 간단하게 시작하고, 단일 에이전트 + 5가지 워크플로우 패턴으로는 부족할 때만 토폴로지를 추가하세요.

**유형:** 학습 + 구축  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 14단계 · 12 (워크플로우 패턴), 14단계 · 25 (멀티에이전트 토론)  
**소요 시간:** ~60분

## 학습 목표

- 네 가지 반복 오케스트레이션 패턴의 이름과 각 패턴이 적합한 경우를 설명한다.
- 2026년 LangChain 권장 사항인 "툴 호출 기반 감독(tool-call-based supervision)"과 "감독자 라이브러리(supervisor libraries)"를 비교한다.
- Anthropic의 "올바른 시스템 구축" 규칙과 이것이 토폴로지 선택을 어떻게 제한하는지 설명한다.
- 공통 스크립트 LLM에 대해 네 가지 패턴을 stdlib로 구현한다.

## 문제 정의

팀들은 "다중 에이전트" 접근법을 필요로 하기 전에 이미 손을 뻗습니다. 프레임워크 전반에 걸쳐 네 가지 패턴이 반복됩니다. 이 패턴들을 명명할 수 있게 되면, 적절한 접근법을 선택하거나 토폴로지를 완전히 생략할 수 있습니다.

## 개념

### 감독자-작업자(Supervisor-worker)

- 중앙 라우팅 LLM이 전문 에이전트에게 작업을 할당합니다.
- 결정 사항: 자기 자신으로 루프백, 전문 에이전트에게 전달, 종료.
- 전문 에이전트는 서로 직접 통신하지 않으며, 모든 라우팅은 감독자를 통해 이루어집니다.

프레임워크: LangGraph `create_supervisor`, Anthropic 오케스트레이터-작업자, CrewAI 계층적 프로세스.

**2026 LangChain 권장 사항:** `create_supervisor` 대신 직접 도구 호출을 통해 감독자를 구현하세요. 더 세밀한 컨텍스트 엔지니어링 제어가 가능하며, 각 전문 에이전트가 볼 내용을 정확히 결정할 수 있습니다.

### 스웜/피어-투-피어(Swarm / peer-to-peer)

- 에이전트들이 공유 도구 표면을 통해 직접 작업을 전달합니다.
- 중앙 라우터가 없습니다.
- 감독자보다 낮은 지연 시간(적은 홉 수).
- 추론이 더 어려움(중앙 제어 지점 없음).

프레임워크: LangGraph 스웜 토폴로지, OpenAI 에이전트 SDK 전달(모든 에이전트가 서로 전달 가능한 경우).

### 계층적(Hierarchical)

- 감독자가 하위 감독자를 관리하고, 하위 감독자가 작업자를 관리합니다.
- LangGraph에서는 중첩된 서브그래프로, CrewAI에서는 중첩된 크루로 구현됩니다.
- 운영 복잡성이 증가하지만 대규모 에이전트 군집에 확장 가능합니다.

필요한 경우: 단일 감독자의 컨텍스트 예산으로 모든 전문 에이전트 설명을 담을 수 없을 때.

### 토론(Debate)

- 병렬 제안자 + 반복적 상호 비판(레슨 25).
- 진정한 오케스트레이션은 아니지만 검증 기능 수행 — 프레임워크에서 토폴로지 선택으로 나타남.

### CrewAI 크루 vs 플로우

CrewAI는 두 가지 배포 모드를 공식화합니다:

- **플로우(Flow):** 결정적 이벤트 기반 자동화(프로덕션 시작 시 권장).
- **크루(Crew):** 자율 역할 기반 협업.

이는 위의 네 가지 패턴과 직교하지만 토폴로지에 매핑됩니다: 플로우는 일반적으로 감독자 또는 계층적 구조이며, 크루는 일반적으로 LLM 라우터가 있는 감독자 구조입니다.

### Anthropic의 지침

"LLM 공간에서의 성공은 가장 정교한 시스템을 구축하는 것이 아닙니다. 당신의 필요에 맞는 올바른 시스템을 구축하는 것입니다."

결정 순서:

1. 단일 에이전트 + 워크플로우 패턴(레슨 12) — 여기서 시작하세요.
2. 감독자-작업자 — 2-4개의 전문 에이전트가 있을 때.
3. 스웜 — 추론 명확성보다 지연 시간이 더 중요할 때.
4. 계층적 — 감독자 컨텍스트 예산이 부족할 때만.
5. 토론 — 비용보다 정확도가 더 중요할 때.

### 이 패턴이 실패하는 경우

- **토폴로지 우선 사고.** "다중 에이전트가 필요하다"고 말하기 전에 다중 에이전트가 해결할 문제를 먼저 식별하지 않음.
- **스웜에서의 불필요한 전달 반복.** A -> B -> A -> B. 홉 카운터를 사용하세요.
- **가짜 계층 구조.** "엔터프라이즈"라서 3계층 구성; 실제 팀은 2개. 축소하세요.

## 빌드하기

`code/main.py`는 스크립팅된 LLM에 대해 stdlib의 네 가지 패턴을 모두 구현합니다:

- `Supervisor` — 중앙 라우터.
- `Swarm` — 직접 핸드오프를 사용하는 P2P(Peer-to-Peer).
- `Hierarchical` — 감독자의 감독자.
- `Debate` — 병렬 제안자 + 비평.

각 패턴은 동일한 3가지 의도 작업(환불/버그/판매)을 처리합니다. 추적 형태는 다릅니다.

실행 방법:

```
python3 code/main.py
```

출력: 패턴별 추적 결과 + 연산 횟수. `Supervisor`가 가장 깔끔하고, `Swarm`이 가장 짧으며, `Hierarchical`가 가장 깊고, `Debate`가 가장 비용이 많이 듭니다.

## 사용 방법

- **LangGraph**를 감독자(supervisor) 및 계층적(중첩된 서브그래프) 구현에 사용.
- **OpenAI Agents SDK**를 핸드오프-도구(handoffs-as-tools) (감독자 형태) 구현에 사용.
- **CrewAI Flow**를 프로덕션 결정론적(deterministic) 구현에 사용.
- **커스텀** 구현을 토론(debate) 시나리오 또는 정확한 제어가 필요한 경우 사용.

## 연습 문제

1. 라우터(router)를 제거하여 감독자-작업자(supervisor-worker)를 스웜(swarm)으로 변환해 보세요. 어떤 부분이 고장나고, 어떤 부분이 개선되나요?
2. 스웜에 홉 카운터(hop counter)를 추가하고 3번의 핸드오프(handoff) 이후에는 거부하도록 설정해 보세요. A→B→A로 튕기는 현상을 포착할 수 있나요?
3. 12명의 전문가(specialist)로 구성된 도메인을 위한 2단계 계층적 시스템을 구축해 보세요. 중첩(nesting) 없이 컨텍스트 예산(context budget)이 어디에서 실패하나요?
4. 프로덕션 형태의 워크로드(workload)에서 4가지 패턴(pattern)을 프로파일링해 보세요. 어떤 패턴이 어떤 지표(지연 시간(latency), 비용(cost), 정확도(accuracy), 디버깅 가능성(debuggability))에서 우수한 성능을 보이나요?
5. Anthropic의 "Building Effective Agents" 포스트를 읽고, 프로덕션 워크플로우(production flow) 각각을 4가지 패턴 중 하나에 매핑해 보세요. 깔끔하게 매핑되지 않는 경우가 있나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 감독자-작업자(Supervisor-worker) | "라우터 + 전문가" | 중앙 LLM이 전문가에게 작업 할당; 전문가 간 직접 통신 없음 |
| 스웜(Swarm) | "P2P(Peer-to-Peer)" | 공유 도구를 통한 직접 작업 인계; 중앙 라우터 없음 |
| 계층적(Hierarchical) | "감독자의 감독자" | 대규모 집단을 위한 중첩된 하위 그래프 |
| 토론(Debate) | "제안자 + 비평자" | 병렬 제안자, 상호 비평(레슨 25) |
| 도구 호출 기반 감독(Tool-call-based supervision) | "라이브러리 없는 감독자" | 컨텍스트 제어를 위한 직접 도구 호출로 감독자 구현 |
| 크루(Crew) | "자율 팀" | CrewAI의 역할 기반 협업 모드 |
| 플로우(Flow) | "결정적 워크플로우" | CrewAI의 이벤트 기반 프로덕션 모드 |
