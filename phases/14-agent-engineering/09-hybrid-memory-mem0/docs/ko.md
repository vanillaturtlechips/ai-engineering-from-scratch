# 하이브리드 메모리: 벡터 + 그래프 + KV (Mem0)

> Mem0 (Chhikara et al., 2025)은 메모리를 세 가지 저장소로 병렬 처리합니다 — 의미적 유사성을 위한 벡터, 빠른 사실 조회를 위한 KV, 엔티티-관계 추론을 위한 그래프. 검색 시 스코어링 레이어가 이 세 가지를 융합합니다. 이는 2026년 외부 메모리의 프로덕션 표준입니다.

**유형:** 구축
**언어:** Python (stdlib)
**사전 요구 사항:** 14단계 · 07 (MemGPT), 14단계 · 08 (Letta Blocks)
**소요 시간:** ~75분

## 학습 목표

- 단일 저장소(벡터만, 그래프만, KV만)가 에이전트 메모리에 불충분한 이유를 설명.
- Mem0의 세 가지 병렬 저장소와 각각이 최적화하는 대상을 명명.
- Mem0의 융합 점수(relevance, importance, recency) 설명 및 계층 구조가 아닌 가중 합(weighted sum)인 이유 설명.
- `add()`로 세 저장소에 동시 기록하고 `search()`로 결과를 융합하는 stdlib 기반 장난감 메모리 구현.

## 추가 설명 (번역 규칙 상 제외)
- **전문 용어 처리**: "KV"는 "키-값(Key-Value)"으로 풀었으나, 기술 문서 컨텍스트에서 "KV"도 널리 쓰이므로 원어 유지.
- **코드 블록**: 구현 예시 요청 시 실제 코드 작성 시 ```python``` 블록 사용 필요 (현재 규칙상 번역 불가).  
  (예: `add()` 구현 시 `vector_store.append(embedding)` 등)

## 문제 정의

세 가지 쿼리 클래스 중 하나에 대해 하나의 저장소가 잘못되었습니다:

- **의미적 유사성(Semantic similarity)** — "지난주에 에이전트 드리프트(agent drift)에 대해 어떤 내용을 논의했나요?" 벡터(vector)가 승리; KV와 그래프(graph)는 놓침.
- **사실 조회(Fact lookup)** — "사용자의 전화번호는 무엇인가요?" KV가 승리; 벡터는 낭비, 그래프는 과도한 기능.
- **관계 추론(Relationship reasoning)** — "어떤 고객들이 동일한 청구 엔티티(billing entity)를 공유하나요?" 그래프(graph)가 승리; 벡터와 KV는 답변 불가.

프로덕션 에이전트(production agents)는 한 세션에서 세 가지 모두를 발행합니다. 단일 저장소(single-store) 메모리는 항상 두 가지에 대해 잘못됩니다. Mem0의 기여는 세 가지를 단일 `add`/`search` 표면 뒤에 연결하고, 이를 융합하는 점수 함수(scoring function)를 제공하는 것입니다.

## 개념

## 병렬로 작동하는 세 개의 저장소

Mem0 (arXiv:2504.19413, 2025년 4월)에서 `add(text, user_id, metadata)`에 대해:

1. 텍스트에서 후보 사실 추출 (LLM 기반 단계).
2. 각 사실을 벡터 저장소(임베딩)에 기록하여 의미 검색 지원.
3. 각 사실을 (user_id, fact_type, entity)를 키로 하는 KV 저장소에 기록하여 O(1) 조회 지원.
4. 각 사실을 관계형 쿼리를 위한 타입화된 엣지로 그래프 저장소(Mem0g)에 기록.

`search(query, user_id)`에 대해:

1. 벡터 저장소는 임베딩 코사인 유사도 기준 상위 k개 반환.
2. KV 저장소는 쿼리 파생 (user_id, type, entity) 키로 직접 일치 항목 반환.
3. 그래프 저장소는 쿼리 엔티티에서 도달 가능한 서브그래프 반환.
4. 스코어링 레이어가 세 결과를 통합.

## 통합 스코어링

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **Relevance** — 벡터 코사인, KV 정확 일치, 그래프 경로 가중치.
- **Importance** — 기록 시 태그 지정 또는 학습 (일부 사실은 더 중요: 이름, ID, 정책).
- **Recency** — 마지막 기록/조회 이후 시간에 따른 지수 감쇠.

가중치는 제품별로 조정. 채팅 에이전트는 `w_recency` 높게, 컴플라이언스 에이전트는 `w_importance` 높게, 검색 에이전트는 `w_relevance` 높게 설정.

## Mem0g과 시간적 추론

Mem0g은 충돌 감지기를 추가. 새 사실이 기존 엣지와 모순될 경우 기존 엣지는 무효 표시되지만 삭제되지 않음. 시간 기반 쿼리("3월 사용자 도시는?")는 해당 시점 유효한 서브그래프를 탐색.

이는 Letta의 무효화 패턴을 일반화한 컴플라이언스 등급 동작.

## 벤치마크 수치

Mem0 논문(2025) 보고:

- **LoCoMo** (장문 대화 메모리): 91.6
- **LongMemEval** (장기 에피소드 메모리): 93.4
- **BEAM 1M** (1M-토큰 메모리 벤치마크): 64.1

비교 기준(풀 컨텍스트 128k LLM, 평면 벡터 저장소, 평면 KV)은 모두 10점 이상 차이. 벤치마크만으로 선택은 불가능하지만, 운영 형태가 중요 — 수치는 통합 설계가 반올림 오차가 아님을 보여줌.

## 범위 분류

Mem0은 메모리 범위를 다음과 같이 분류:

- **사용자 메모리** — 세션 간 유지, `user_id`로 키 관리.
- **세션 메모리** — 단일 스레드 내 유지.
- **에이전트 메모리** — 에이전트 인스턴스별 상태.

모든 기록은 하나의 범위를 선택. 검색은 범위별 가중치로 다중 범위 쿼리 가능. 무분별한 범위 혼합은 "보조자가 Alice에게 Bob의 프로젝트를 언급" 같은 사고를 유발.

## 이 패턴의 실패 사례

- **임베딩 드리프트.** 처음 100개 쿼리에서는 정상적이던 벡터 결과가 코퍼스 확장 시 저하. 상위 N개 사용 기록의 주기적 재임베딩 추가.
- **KV 스키마 확장.** `(user_id, type, entity)`는 단순해 보이지만 모든 팀이 자체 `type`을 추가할 때 문제 발생. 분기별로 타입 세트 감사.
- **그래프 폭발.** 하나의 노이즈 추출기가 메시지당 50개 엣지 추가. `add` 호출당 그래프 기록 제한; 저신뢰도 엣지 삭제.

## 구축 방법

`code/main.py`는 표준 라이브러리(stdlib)에서 세 가지 저장소 패턴을 구현합니다:

- `VectorStore` — 임베딩 대체 수단으로 사용되는 순진한 토큰 중첩 유사도.
- `KVStore` — `(user_id, fact_type, entity)`를 키로 하는 딕셔너리.
- `GraphStore` — 타입이 지정된 엣지 (subject, relation, object, valid).
- `Mem0` — `add()`, `search()`, 퓨전 스코어링, 스코프 인식 검색을 제공하는 최상위 퍼사드.
- 다중 사용자, 다중 세션 대화에 대한 작업된 트레이스.

실행 방법:

```
python3 code/main.py
```

출력은 세 가지 별도의 리콜 경로와 퓨전된 상위 k개 결과를 보여줍니다. `main()` 함수 상단의 스코어링 가중치를 변경하면 순위가 어떻게 변하는지 확인할 수 있습니다.

## 사용 방법

- **Mem0 (Apache 2.0)** — 프로덕션 환경 준비 완료. Postgres + Qdrant + Neo4j로 자체 호스팅하거나 관리형 클라우드 서비스를 사용할 수 있습니다.
- **Letta** — 3계층 코어/리콜/아카이빙 구조; 벡터 및 그래프 백엔드를 직접 제공.
- **Zep** — 상용 대안 솔루션으로 시간 기반 KG(Knowledge Graph) 및 사실 추출 기능 제공.
- **커스텀 빌드** — 추출기(규정 준수) 또는 융합 가중치(최신성 우선인 음성 에이전트)에 대한 정확한 제어가 필요할 때 사용.

## Ship It

`outputs/skill-hybrid-memory.md`는 퓨전 스코어러, 스코프 분류 체계, 시간 기반 무효화가 연결된 3단계 메모리 스캐폴드를 생성합니다.

## 연습 문제

1. 장난감 벡터 유사도 모델을 실제 임베딩 모델(sentence-transformers, Ollama, OpenAI embeddings)로 교체하세요. 합성 장기 대화에서 recall@10을 측정하세요. 1000회 쓰기 작업 후 순위 변동이 발생하나요?
2. 시간 기반 쿼리 추가: `search(query, as_of=timestamp)`. 해당 시간 이전에 유효한 레코드만 반환하세요. 어떤 저장소가 가장 많은 작업이 필요할까요?
3. 충돌 감지기 구현: 들어오는 사실이 그래프 간선과 모순될 경우 기존 간선을 무효화하고 둘 다 로깅하세요. "사용자가 베를린에 거주" → "사용자가 리스본에 거주" 시나리오로 테스트하세요.
4. 퓨전 스코어러에 `user_feedback` 차원(검색된 레코드에 대한 좋아요) 추가 구현. 게임화 방지(에이전트가 이미 좋아한 레코드만 반환하는 문제)는 어떻게 해결할 수 있나요?
5. Mem0 문서(`docs.mem0.ai`)를 읽고 장난감 모델을 `mem0` 클라이언트 호출로 포팅하세요. 동일한 20개 테스트 쿼리에서 검색 품질을 비교하세요.
