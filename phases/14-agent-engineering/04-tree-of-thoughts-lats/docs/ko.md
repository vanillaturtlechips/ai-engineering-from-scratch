# 사고 과정 트리(ToT)와 LATS: 의도적 탐색

> 단일 사고 과정 체인은 되돌아갈 여지가 없습니다. ToT(Yao et al., 2023)는 추론을 각 노드에 대한 자기 평가와 함께 트리 구조로 변환합니다. LATS(Zhou et al., 2024)는 몬테카를로 트리 탐색(Monte Carlo Tree Search) 하에서 ToT, ReAct, Reflexion을 통합합니다. 게임 오브 24(Game of 24)는 4%(CoT)에서 74%(ToT)로 향상되었으며, LATS는 HumanEval에서 pass@1 92.7%를 기록했습니다.

**유형:** 구축(Build)
**언어:** Python (표준 라이브러리)
**선수 지식:** 14단계 · 01 (에이전트 루프), 14단계 · 03 (리플렉션)
**소요 시간:** ~75분

## 학습 목표

- 추론을 검색(search)으로 구성: 노드는 "생각", 엣지는 "확장", 값은 "얼마나 유망한지"로 표현.
- 자기 평가 점수(self-evaluation scoring)를 활용한 stdlib ToT(Chain-of-Thought) 스타일 BFS 트리 검색 구현.
- 선택(select) / 확장(expand) / 시뮬레이션(simulate) / 역전파(backpropagate) 단계를 포함한 장난감 LATS MCTS(Monte Carlo Tree Search) 루프로 확장.
- 검색이 토큰 증가 비용을 정당화할 수 있는 경우(24 게임, 코드 생성)와 단일 경로만으로 충분한 경우(간단한 Q&A)를 판단.

## 문제 정의

연쇄적 사고(Chain-of-thought)는 선형적 접근 방식입니다. 첫 단계에서 오류가 발생하면 이후 모든 단계가 잘못된 전제 하에 진행됩니다. "24 게임"(네 개의 숫자를 + − × ÷ 연산으로 조합해 24를 만드는 문제)에서 GPT-4의 연쇄적 사고 정확도는 4%에 그칩니다. 모델이 초기에 잘못된 부분 표현식을 선택하면 이후 복구가 불가능합니다.

추론에 필요한 것은 여러 후보를 제안하고, 이를 평가한 뒤 유망한 후보를 선택하며, 막다른 골목에 도달하면 되돌아가는 능력입니다. 즉, 탐색(search)이 필요합니다. 사고 트리(Tree of Thoughts)와 LATS가 이 분야의 두 가지 대표적인 접근 방식입니다.

## 개념

### 생각의 나무(Tree of Thoughts, Yao et al., NeurIPS 2023)

각 노드는 일관된 중간 단계("생각")입니다. 각 노드는 K개의 자식 생각으로 확장될 수 있습니다. LLM은 점수 매기기 프롬프트를 사용해 각 노드를 자체 평가합니다. 탐색은 트리를 탐색합니다 — BFS, DFS 또는 빔 서치.

```
                     (루트: "4, 6, 4, 1로 24 만들기")
                    /               |            \
           ("6 - 4 = 2")    ("4 + 1 = 5")    ("4 * 6 = 24")  <- 점수: 높음
              /   \              |                  |
          ...    ...          ...                완료
```

자체 평가는 핵심 요소입니다. 논문에서는 세 가지 변형을 보여줍니다: `확실함 / 가능성 있음 / 불가능` 분류, `1..10` 숫자 점수, 후보 간 투표. 세 방법 모두 24 게임(Game of 24)에서 CoT를 크게 능가했습니다(4% → GPT-4 사용 시 74%).

### LATS(Zhou et al., ICML 2024)

LATS는 MCTS 하에서 ToT, ReAct, Reflexion을 통합합니다. LLM은 세 가지 역할을 수행합니다:

- **정책(Policy)**: 후보 다음 행동 제안(ReAct 스타일).
- **가치 함수(Value function)**: 부분 궤적 점수 매기기(ToT 스타일 자체 평가).
- **자기 반성기(Self-reflector)**: 실패 시 자연어 반성문 작성(Reflexion 스타일)하고 이를 사용해 향후 롤아웃을 재시도.

환경 피드백(관측값)은 가치 함수에 혼합되어 탐색이 모델 의견이 아닌 실제 도구 결과에 의해 정보를 받습니다. 논문 당시 결과: GPT-4로 HumanEval pass@1 92.7%(SOTA), GPT-3.5로 WebShop 평균 75.9(경사 기반 파인튜닝에 근접).

### MCTS, 최소로

반복당 4단계:

1. **선택(Select)**: UCT(트리를 위한 상한 신뢰 구간)를 사용해 루트에서 리프까지 이동.
2. **확장(Expand)**: 정책을 통해 K개의 자식 생성.
3. **시뮬레이션(Simulate)**: 자식에서 정책을 사용해 롤아웃, 가치 함수(또는 환경 보상)로 리프 점수 매기기.
4. **전파(Backpropagate)**: 경로를 따라 방문 횟수와 가치 추정치 업데이트.

UCT 공식: `Q(s, a) + c * sqrt(ln N(s) / N(s, a))`. 첫 번째 항은 활용(exploitation), 두 번째는 탐색(exploration). 작업별로 `c`를 조정.

### 비용 현실

탐색은 토큰을 폭발적으로 증가시킵니다. 24 게임의 ToT는 CoT의 100–1000배 토큰을 사용합니다. LATS도 유사합니다. 이는 무료가 아닙니다. 다음을 위해 탐색을 예약하세요:

- 단일 궤적로 명백히 불충분한 작업(24 게임, 복잡한 코드).
- 벽시계 시간보다 정확성이 더 중요한 작업.
- 저렴하고 신뢰할 수 있는 가치 함수가 있는 작업(코드 단위 테스트, 수학 명시적 목표).

작업에 단일 정답이 있고 평가기가 노이즈가 많다면 탐색은 종종 상황을 악화시킵니다 — "높은 점수"의 오답을 찾기 때문입니다.

### 2026년 위치

대부분의 프로덕션 에이전트는 LATS를 실행하지 않습니다. 도구 기반 검증(CRITIC, 레슨 05)과 함께 ReAct를 실행합니다. 탐색은 특수 분야에서 나타납니다:

- 테스트를 가치 함수로 실행하는 코딩 에이전트(HumanEval 스타일).
- 여러 쿼리 경로를 탐색하는 심층 연구 에이전트.
- LangGraph 서브그래프 내부의 계획 중심 워크플로우.

AlphaEvolve(레슨 11)는 2025년 극한: 코드 진화 탐색, 기계 검증 가능한 적합성, 프론티어 성과(56년 만에 첫 4x4 행렬 곱셈 개선).

## 빌드하기

`code/main.py`는 다음을 구현합니다:

- 스타일화된 "산술 연산 선택" 작업에 대한 소규모 ToT BFS(너비 우선 탐색).
- 동일한 작업에서 장난감 LATS MCTS 루프(선택/확장/시뮬레이션/역전파)를 UCT 선택과 함께 구현.
- 기호 점수(symbolic score)와 자체 평가 점수(self-eval score)를 조합하는 가치 함수(value function).

실행 방법:

```
python3 code/main.py
```

트레이스(trace)는 ToT가 BFS로 노드당 세 개의 후보를 확장하는 반면, LATS는 MCTS를 통해 최적의 롤아웃(rollout)으로 수렴하는 과정을 보여줍니다. 두 방법에 대한 토큰 수(token count)도 출력됩니다.

## 사용 방법

LangGraph는 ToT(Tree of Thoughts) 스타일의 탐색을 서브그래프 패턴으로 제공합니다. LATS에 대한 LangChain 팀의 블로그(2024년 5월)가 참조 튜토리얼입니다. LlamaIndex는 `TreeOfThoughts` 에이전트를 제공합니다. 2026년 대부분의 프로덕션 에이전트에서는 이 패턴이 `if task_complexity > threshold: use_search()` 게이트 뒤에 구현됩니다. Lesson 05의 평가자-최적화자 패턴을 참조하세요.

## Ship It

`outputs/skill-search-policy.md`는 작업 형태(task shape), 예산(budget), 평가자 신뢰도(evaluator fidelity)를 고려하여 선형 ReAct, ToT, LATS, 진화 검색(evolutionary search) 중 적절한 방법을 선택합니다.

2. 값 함수를 더 노이즈가 많은 평가기(랜덤 지터 추가)로 교체해 보세요. MCTS가 여전히 최적의 리프 노드를 찾을 수 있나요? 허용되는 최소 신호 대 잡음비는 얼마인가요?
3. 빔 서치 ToT(각 레벨에서 상위 k개 유지)를 구현하고 BFS와 비교해 보세요. 제한된 토큰 예산에서 어떤 방식이 더 효과적일까요?
4. LATS 5.1절을 읽고 HumanEval 통과율(trajectory count)을 재현해 보세요: 보고된 pass@1을 달성하기 위해 몇 번의 롤아웃이 필요한가요?
5. LATS 논문의 "LATS가 덜 도움이 되는 경우" 논의를 읽고, 작업 형태에 따른 검색 전략 매핑 결정 규칙을 한 단락으로 작성해 보세요.  

> **번역 참고사항**:  
> - "pass@1"은 코드 생성 평가에서 일반적으로 사용되는 지표로, 1회 시도 시 정답 코드 생성 확률을 의미하므로 번역하지 않고 그대로 유지했습니다.  
> - "beam-search ToT", "BFS", "UCT", "MCTS", "LATS" 등의 알고리즘/기법 이름은 고유명사로 간주해 번역하지 않았습니다.
