# Anthropic's Workflow Patterns: Simple Over Complex

> Schluntz와 Zhang(Anthropic, 2024년 12월)은 워크플로우(사전 정의된 경로)와 에이전트(동적 도구 사용)를 구분합니다. 5가지 워크플로우 패턴이 대부분의 사례를 커버합니다. 직접 API 호출로 시작하세요. 단계를 예측할 수 없을 때만 에이전트를 추가하세요.

**유형:** 학습 + 구축  
**언어:** Python (표준 라이브러리)  
**사전 요구 사항:** 14단계 · 01(에이전트 루프)  
**소요 시간:** ~60분

## 학습 목표

- Anthropic의 다섯 가지 워크플로우 패턴 이름 말하기: 프롬프트 체이닝(prompt chaining), 라우팅(routing), 병렬화(parallelization), 오케스트레이터-워커(orchestrator-workers), 평가자-최적화기(evaluator-optimizer).
- 에이전트(agent) vs 워크플로우(workflow) 차이점과 각각의 엔지니어링 비용 설명하기.
- 워크플로우를 선택할 시점(에이전트를 선택할 시점) 식별하기.
- 스크립티드 LLM(scripted LLM)에 대해 stdlib에서 다섯 가지 패턴 모두 구현하기.

## 문제

팀들은 단일 함수 호출을 원하는 문제에 대해 다중 에이전트 프레임워크를 선택합니다. 그러나 실제 비용이 발생합니다: 프레임워크는 프롬프트를 흐리게 하고, 제어 흐름을 숨기며, 조기 복잡성을 유발하는 레이어를 추가합니다. Schluntz와 Zhang의 2024년 12월 게시물은 가장 많이 인용된 업계 반작용 사례입니다: 간단하게 시작하고, 복잡성은 비용을 상쇄할 때만 추가해야 합니다.

## 개념

### 워크플로우 vs 에이전트

- **워크플로우.** 미리 정의된 코드 경로를 통해 조율되는 LLM과 도구. 엔지니어가 그래프를 소유합니다.
- **에이전트.** LLM이 동적으로 자체 도구를 지시하고 자체 단계를 수행합니다. 모델이 그래프를 소유합니다.

둘 다 각자의 위치가 있습니다. 워크플로우는 더 저렴하고 빠르며 디버깅이 쉽습니다. 에이전트는 열린 문제를 해결하지만 실패 모드를 추론하기 어렵게 만듭니다.

### 증강된 LLM

다섯 가지 패턴의 기반: 검색(검색), 도구(액션), 메모리(지속성)라는 세 가지 기능이 연결된 하나의 LLM. 모든 API 호출에서 이를 사용할 수 있습니다.

### 다섯 가지 패턴

1. **프롬프트 체이닝.** 호출 1의 출력이 호출 2의 입력이 됩니다. 작업이 깨끗한 선형 분해를 가질 때 사용합니다. 단계 사이에 선택적 프로그래밍 게이트를 사용할 수 있습니다.

2. **라우팅.** 분류기 LLM이 어떤 다운스트림 LLM 또는 도구를 호출할지 선택합니다. 범주적으로 다른 입력이 다른 처리를 필요로 할 때 사용합니다(예: 1차 지원 vs 환불 vs 버그 vs 영업).

3. **병렬화.** N개의 LLM 호출을 동시에 실행하고 결과를 집계합니다. 두 가지 형태: 섹션 분할(다른 청크)과 투표(동일 프롬프트, N회 실행, 다수결/종합).

4. **오케스트레이터-워커.** 오케스트레이터 LLM이 동적으로 실행할 워커(또한 LLM)를 결정하고 그 출력을 종합합니다. 에이전트 루프와 유사하지만 오케스트레이터가 무한히 반복하지는 않습니다.

5. **평가자-최적화기.** 하나의 LLM이 답변을 제안하고 다른 LLM이 이를 평가합니다. 평가자가 통과할 때까지 반복합니다. 이는 Self-Refine(레슨 05)을 일반화한 것입니다.

### 워크플로우가 에이전트를 능가하는 경우

- **예측 가능한 작업.** 단계를 열거할 수 있다면 그렇게 해야 합니다.
- **비용 제약 작업.** 워크플로우는 단계 수가 제한되어 있지만 에이전트는 무한 루프에 빠질 수 있습니다.
- **규정 준수 작업.** 감사자는 그래프를 읽고 싶어하며, 궤적에서 추론하고 싶어하지 않습니다.

### 에이전트가 워크플로우를 능가하는 경우

- **열린 연구.** 다음 단계가 마지막 단계의 반환 값에 따라 달라질 때.
- **가변 길이 작업.** 단계 수가 알려지지 않은 몇 분에서 몇 시간 분량의 작업.
- **새로운 도메인.** 아직 올바른 워크플로우를 모를 때 — 먼저 탐색, 나중에 체계화.

### 컨텍스트 엔지니어링 동반자

"Effective context engineering for AI agents"(Anthropic 2025)는 인접 학문을 공식화합니다: 200k 윈도우는 예산이 아니라 컨테이너입니다. 무엇을 포함할지, 언제 압축할지, 언제 컨텍스트를 성장시킬지. 이 커리큘럼의 14단계 레슨(리넘버링 전 14단계 레슨 06)에서 컨텍스트 압축을 자세히 다룹니다.

## 빌드하기

`code/main.py`는 `ScriptedLLM`에 대한 5가지 워크플로우 패턴을 구현합니다:

- `prompt_chain(input, steps)` — 순차적(sequential) 실행.
- `route(input, classifier, handlers)` — 분류(classification) + 디스패치(dispatch).
- `parallel_vote(prompt, n, aggregator)` — N회 실행 후 집계(aggregate).
- `orchestrator_workers(task, workers)` — 오케스트레이터(orchestrator)가 작업자(workers) 선택.
- `evaluator_optimizer(task, proposer, evaluator, max_iter)` — 통과(pass)할 때까지 반복(loop).

실행 방법:

```
python3 code/main.py
```

각 패턴은 실행 추적(trace)을 출력합니다. 패턴당 총 코드 라인은 ~10-15줄이며, 프레임워크의 비용은 수천 줄로 측정됩니다.

## 사용 방법

- 대부분의 작업에는 직접 API 호출을 사용하세요.
- 프레임워크는 패턴이 **지속 가능한 상태**(LangGraph), **액터 모델 동시성**(AutoGen v0.4), 또는 **역할 템플릿**(CrewAI)이 진정으로 필요한 경우에만 사용하세요.
- Claude Code 하네스 형태를 재구축 없이 사용할 때는 **Claude Agent SDK**를 활용하세요.

## 연습 문제

1. 신뢰도 임계값을 사용한 라우팅 구현. 임계값 미만 -> 인간 개입으로 에스컬레이션. 티어-1 지원 사용 사례에서 임계값은 어디에 위치해야 하는가?
2. `parallel_vote`에 타임아웃 추가. 한 호출이 지연될 때 발생하는 현상? 누락된 투표로 어떻게 집계하는가?
3. `evaluator_optimizer`를 밴딧(bandit)으로 전환: 반복 과정에서 상위 2개 출력을 유지하여, 늦게 나타난 좋은 결과가 늦게 나타난 나쁜 결과에 의해 덮어쓰여지지 않도록 함.
4. 프롬프트 체이닝과 라우팅 결합: 라우터가 3개의 체인 중 하나를 선택. 토큰 비용 측정 vs 단일 대형 프롬프트 대안.
5. 프로덕션 기능 중 하나를 선택. 워크플로우 그래프 작성. 단계 수 계산. 여기서 에이전트가 실제로 더 나을까?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 워크플로우(Workflow) | "사전 정의된 흐름" | 엔지니어가 소유한 LLM 및 도구 호출 그래프 |
| 에이전트(Agent) | "자율적 AI" | 모델 소유 그래프; 동적 도구 방향 설정 |
| 증강 LLM(Augmented LLM) | "도구가 있는 LLM" | LLM + 검색 + 도구 + 메모리; 원자적 단위 |
| 프롬프트 체이닝(Prompt chaining) | "순차적 호출" | 호출 N의 출력이 호출 N+1의 입력이 됨 |
| 라우팅(Routing) | "분류기 디스패치" | 어떤 체인/모델이 입력을 처리할지 선택 |
| 병렬화(Parallelization) | "팬 아웃" | N개의 동시 호출; 섹션 분할 또는 투표로 집계 |
| 오케스트레이터-워커(Orchestrator-workers) | "디스패처 에이전트" | 오케스트레이터 LLM이 전문가 LLM을 동적으로 선택 |
| 평가자-최적화기(Evaluator-optimizer) | "제안자 + 판단자" | 평가자가 통과할 때까지 반복; Self-Refine 일반화 |
