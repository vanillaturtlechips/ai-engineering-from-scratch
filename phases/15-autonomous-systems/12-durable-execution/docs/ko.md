# 장기 실행 백그라운드 에이전트: 내구성 있는 실행

> 프로덕션 장기 호라이즌 에이전트는 `while True`에서 실행되지 않습니다. 모든 LLM 호출은 체크포인트, 재시도, 재생 기능을 갖춘 액티비티가 됩니다. Temporal의 OpenAI 에이전트 SDK 통합은 2026년 3월에 GA(일반 공급)되었습니다. Claude Code 루틴스(Anthropic)는 지속적인 로컬 프로세스 없이 예약된 Claude Code 호출을 실행합니다. 세션은 인간 입력 시 일시 중지되고, 배포 후에도 생존하며, `thread_id`로 식별되는 최신 체크포인트에서 재개됩니다. 새로운 사용성 뒤에는 오래된 패턴인 워크플로우 오케스트레이션이 있으며, 여기에는 하나의 새로운 입력이 추가되었습니다: 복구 시 결정적으로 재생되어야 하는 비결정적 액티비티인 LLM 호출.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 최소 내구성 실행 상태 머신)  
**선수 지식:** 15단계 · 10 (권한 모드), 15단계 · 01 (장기 호라이즌 에이전트)  
**소요 시간:** ~60분

## 문제 설명

4시간 동안 실행되는 에이전트를 고려해 보자. 이 에이전트는 세 개의 도구를 호출하고, 사용자에게 두 번 프롬프트를 표시하며, 40번의 LLM 호출을 수행한다. 중간에 호스트가 재부팅된다면 어떤 일이 발생할까?

- 순진한 `while True` 루프의 경우: 모든 것이 손실된다. 실행이 처음부터 다시 시작된다. 실제 부작용을 일으키는 세 번의 도구 호출이 다시 실행된다. 사용자는 이미 승인한 내용을 다시 요청받는다. 40번의 LLM 호출이 다시 과금된다.
- 지속적 실행(durable execution)의 경우: 가장 최근의 체크포인트부터 실행이 재개된다. 이미 완료된 활동은 다시 실행되지 않으며, 그 결과는 지속적 로그(durable log)에서 재생된다. 사용자는 이미 승인한 내용을 다시 승인하지 않는다. 이미 수행된 LLM 호출은 다시 과금되지 않는다.

이것은 워크플로우 엔진(Temporal, Cadence, Uber의 Cherami 등)이 10년 동안 제공해 온 것과 동일한 패턴이다. 새로운 점은 LLM 호출이 이제 일종의 활동(non-deterministic, 비용이 많이 들고 부작용이 있는)으로 간주되며, 이 패턴에 깔끔하게 들어맞는다는 것이다.

이 강의의 주요 주제: 장기 실행 신뢰성은 감소한다(METR은 "35분 감소"를 관찰함 — 성공률은 지평선에 따라 대략 제곱으로 감소). 지속적 실행은 신뢰성 프로파일이 지원하는 것보다 더 긴 실행을 가능하게 하며, 설계가 올바르면 안전하게 실패할 수 있는 새로운 방법이지만 설계가 잘못되면 안전하지 않게 실패할 수 있다.

## 개념

### 활동, 워크플로우, 리플레이

- **워크플로우**: 결정적 오케스트레이션 코드. 활동 순서, 분기, 대기 시간을 정의합니다. 이벤트 로그에서 리플레이할 때 예상치 못한 분기 없이 재생될 수 있도록 결정적이어야 합니다.
- **활동**: 비결정적이며 실패할 수 있는 작업 단위. LLM 호출, 도구 호출, 파일 쓰기, HTTP 요청 등이 포함됩니다. 각 활동은 입력값과 (완료 시) 출력값과 함께 로깅됩니다.
- **이벤트 로그**: 내구성 있는 백엔드 저장소. 모든 활동 시작, 완료, 실패, 재시도 및 워크플로우 결정이 기록됩니다.
- **리플레이**: 복구 시 워크플로우 코드는 처음부터 재실행됩니다. 이미 완료된 활동은 재실행 없이 로그된 결과를 반환합니다. 완료되지 않은 활동만 실제로 실행됩니다.

이는 React가 가상 DOM에 대해 재렌더링하거나 Git이 커밋으로부터 작업 트리를 재구성하는 것과 같은 형태입니다. 오케스트레이터의 결정성이 내구성을 저렴하게 만듭니다.

### LLM 호출이 패턴에 적합한 이유

LLM 호출은 다음과 같은 특징을 가집니다:
- 비결정적 (온도 > 0; 온도 0도 모델 버전에 따라 드리프트 발생).
- 비용이 큼 (금전적 비용 및 지연 시간).
- 실패 가능성 (요청 제한, 타임아웃).
- 부작용 발생 가능 (도구 호출 시).

이는 정확히 활동의 프로필과 일치합니다. 모든 LLM 호출을 활동으로 감싸면 지수 백오프를 통한 재시도, 재시작 간 체크포인팅, 디버깅을 위한 리플레이 가능한 트레이스가 제공됩니다.

### `thread_id`로 키된 체크포인트

LangGraph, Microsoft Agent Framework, Cloudflare Durable Objects, Claude Code Routines는 모두 동일한 API 형태로 수렴했습니다: `thread_id`(또는 이에 상응하는 식별자)가 세션을 식별하며, 각 상태 전환은 백엔드에 지속됩니다 (기본값은 PostgreSQL, 개발용은 SQLite, 캐시는 Redis). 재개 시 최신 체크포인트를 읽습니다.

백엔드 선택은 중요합니다:

- **PostgreSQL**: 내구성 있고 쿼리 가능, 배포 간 생존. LangGraph의 기본값.
- **SQLite**: 로컬 개발 전용; 호스트 간 데이터 손실 발생.
- **Redis**: 빠르지만 AOF/스냅샷 구성 없이는 일시적.
- **Cloudflare Durable Objects**: 투명하게 분산; 고유 키로 범위 지정; 수시간에서 수주 간 생존.

### 인간 입력을 1급 상태로

제안 후 커밋(Lesson 15)은 내구성 있는 "인간 대기 중" 상태를 요구합니다. 워크플로우가 일시 중지되고, 외부 큐가 대기 중인 요청을 보관하며, 승인 시 정확히 해당 지점에서 재개됩니다. 내구성이 없으면 최선의 노력만 가능하지만, 내구성이 있으면 밤새 도착한 승인에 대해 아침에 워크플로우가 재개됩니다.

### 35분 성능 저하

METR은 모든 에이전트 클래스가 연속 작동 35분 이후 신뢰성 감소를 관찰했습니다. 작업 지속 시간을 두 배로 늘리면 실패율이 대략 네 배가 됩니다. 내구성 있는 실행은 이를 수정하지 않지만, 신뢰성 프로필이 지원하는 것보다 더 오래 실행할 수 있게 합니다. 안전한 패턴은 내구성과 재진입 시 신선한 HITL(인간 개입)을 요구하는 체크포인트, 그리고 총 계산량을 벽시계 시간과 무관하게 제한하는 예산 킬 스위치(Lesson 13)를 결합하는 것입니다.

### 내구성 있는 실행이 부적절한 경우

- 인간 입력 없이 몇 분 미만으로 실행되는 작업. 오버헤드가 이점보다 큼.
- 엄격하게 읽기 전용 정보 검색.
- 하나의 컨텍스트 창 내에서 종단 간 정확성이 요구되는 작업 (일부 추론 작업; 일부 일회성 생성).

## 사용 방법

`code/main.py`는 표준 라이브러리 Python으로 구현된 최소한의 지속 실행(durable-execution) 엔진을 포함합니다. 다음 기능을 지원합니다:

- 입력과 출력을 JSON 이벤트 로그에 기록하는 `@activity` 데코레이터
- 활동을 순차적으로 실행하는 워크플로우 함수
- 완료된 활동을 다시 실행하지 않고 재생(replay)하는 `run_or_replay(workflow, event_log)` 함수

드라이버는 3단계 활동 워크플로우를 시뮬레이션하고, 중간에 크래시를 발생시킨 후 (a) 모든 것을 재실행하는 순진한 재시도 방식과 (b) 누락된 활동만 실행하는 재생 방식을 비교합니다.

## Ship It

`outputs/skill-durable-execution-review.md`는 제안된 장기 실행 에이전트 배포에 대한 검토 문서로, 올바른 durable-execution 형태(활동(activity), 결정성(determinism), 체크포인트 백엔드(checkpoint backend), 인간 입력 상태(human-input state), HITL-on-resume 정책)를 평가합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 순차적 재시도(naive retry)와 리플레이(replay) 간의 활동-실행 횟수 차이를 관찰하세요. 충돌 지점(crash point)을 변경하고, 리플레이 횟수가 어떻게 변하는지 확인하세요.

2. 토이 엔진(toy engine)을 `thread_id`를 명시적으로 사용하도록 변환하세요. 엔진을 공유하는 두 개의 동시 세션(sessions)을 시뮬레이션하고, 이벤트 로그가 충돌하지 않는지 확인하세요.

3. 토이 엔진의 활동(activity) 하나를 선택하세요. 비결정성(non-determinism)을 도입하세요(예: 워크플로우 결정 내부의 벽시계 타임스탬프). 리플레이 시 발생하는 분기(divergence)를 증명하세요. 실제 엔진들이 이를 어떻게 처리하는지 설명하세요(부작용 등록, `Workflow.now()` API 등).

4. LangChain의 "Runtime behind production deep agents" 포스트를 읽으세요. 런타임이 지속하는 모든 상태(state)를 나열하고, 각 상태가 어떤 실패 모드(failure mode)를 커버하는지 이름을 붙이세요.

5. 6시간 자율 코딩 작업을 위한 체크포인트 정책(checkpoint policy)을 설계하세요. 어디에 체크포인트를 둘 것인가요? 충돌 시 재개(resume-on-crash)는 어떤 모습일까요? 어떤 경우에 새로운 HITL(인간 개입)이 필요할까요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|---|---|---|
| 워크플로우(Workflow) | "에이전트의 스크립트" | 결정적 오케스트레이션 코드; 이벤트 로그로부터 재실행 가능 |
| 액티비티(Activity) | "한 단계" | 비결정적 단위(LLM 호출, 툴 호출); 실행 전후 로깅됨 |
| 이벤트 로그(Event log) | "백킹 스토어" | 모든 상태 전이에 대한 지속적 기록 |
| 리플레이(Replay) | "재개" | 워크플로우 재실행; 완료된 액티비티는 재실행 없이 로그된 결과 반환 |
| 체크포인트(Checkpoint) | "저장 지점" | thread_id로 키된 지속적 상태; 재개 시 최신 상태 우선 적용 |
| thread_id | "세션 키" | 지속적 상태를 범위화하는 식별자 |
| 35분 성능 저하(35-minute degradation) | "신뢰성 감소" | METR: 성공률이 시간 지평에 따라 ~제곱근 형태로 감소 |
| 비결정성(Non-determinism) | "리플레이 시 드리프트" | 벽시계, 난수, LLM 출력; 부작용(side effect)으로 등록 필요 |
