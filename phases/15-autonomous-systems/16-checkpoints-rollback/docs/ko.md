# 체크포인트와 롤백

> 모든 그래프 상태 전환은 지속됩니다. 워커가 충돌하면 해당 워커의 임대(lease)가 만료되고 다른 워커가 가장 최신 체크포인트부터 작업을 이어받습니다. Cloudflare Durable Objects는 수시간 또는 수주에 걸쳐 상태를 유지합니다. 제안-후-커밋(Propose-then-commit, 레슨 15)은 각 액션에 대한 롤백 계획을 정의합니다. 액션 후 검증은 이 과정을 완료합니다. EU AI Act 제14조는 고위험 시스템에 대해 효과적인 인간 감독을 의무화합니다. 이는 실제로 체크포인트를 조회할 수 있어야 하고, 롤백을 리허설할 수 있어야 하며, 감사 추적이 배포 후에도 유지되어야 함을 의미합니다. 주요 실패 모드: 멱등성 키(idempotency key)와 사전 조건 검사가 없으면 일시적 장애 후 재시도가 이미 승인된 액션을 중복 실행할 수 있습니다. 액션 후 검증이 이를 감지합니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 체크포인트 및 롤백 상태 머신)  
**선수 지식:** 15단계 · 12 (내구성 있는 실행), 15단계 · 15 (제안-후-커밋)  
**소요 시간:** ~60분

## 문제 정의

내구성 있는 실행(Durable execution, 레슨 12)은 충돌한 에이전트를 재개할 수 있게 합니다. 제안-후-커밋(Propose-then-commit, 레슨 15)은 승인된 액션을 감사 가능하게 합니다. 이 레슨은 두 개념을 결합합니다: 승인된 액션이 부분적으로 실행되다가 충돌하고 재개될 때 어떤 일이 발생하는가? 롤백은 언제 실행되며, 어떤 상태를 기준으로 실행되는가?

실제 시스템들은 이를 다르게 구현합니다:

- **LangGraph**는 PostgreSQL에 모든 그래프 상태 전이를 체크포인트합니다. 워커 충돌 시 임대가 해제되고 다른 워커가 최신 체크포인트부터 재개합니다. 워크플로우는 `interrupt()`에서 일시 중지되며, 이 자체도 지속됩니다.
- **Cloudflare Durable Objects**는 시간 또는 주 단위로 키별 상태를 유지합니다. 승인된 액션에 대한 계산과 스토리지를 동일 위치에 배치합니다.
- **Microsoft Agent Framework**는 워크플로우 API에서 `Checkpoint` 프리미티브를 노출합니다; 재실행과 멱등성(idempotency)이 재시도를 처리합니다.

모든 사례에서 실제로 작동하는 조합은 다음과 같습니다:  
- 멱등성 키(idempotency key, 중복 실행 방지)  
+ 사전 조건 검사(precondition check, 상태가 여전히 승인 시와 동일한지 확인)  
+ 사후 액션 검증(post-action verify, 부작용이 실제로 발생했는지 확인)  
+ 검증 실패 시 롤백(rollback on verify-fail)

## 개념

### 모든 전환은 지속되어야 한다

그래프 상태 전환은 워크플로우를 하나의 명명된 상태에서 다른 상태로 이동시키는 모든 단계를 의미한다. 순진한 구현은 특정 커밋 지점에서만 지속성을 가지지만, 프로덕션 구현은 모든 전환에서 지속성을 가진다. 비용(몇 번의 추가 쓰기)은 신뢰성 향상(재생이 어디서나 가능, 임대 복구가 정확함)에 비해 작다.

### 임대 복구

워커가 충돌하더라도 워크플로우는 손실되지 않는다. 임대(이 워커가 이 실행을 수행 중이라는 단기간의 주장)는 단순히 만료된다. 다른 워커가 최신 체크포인트를 가져와 재개한다. 임대 메커니즘은 프로덕션 시스템이 롤링 배포 중에도 진행 중인 작업을 잃지 않도록 한다.

### 멱등성 + 사전 조건

멱등성만으로는 충분하지 않다. 예를 들어, "잔액이 $1000 초과 시 A에서 B로 $100 이체" 워크플로우가 승인되었다. 워크플로우가 커밋된 후 실행 중 충돌하고 재개된다. 멱등성 키만 확인하고 실행을 재개하면 이체는 한 번만 실행된다(정답). 하지만 충돌과 재개 사이에 다른 워크플로우로 A의 잔액이 $500으로 감소했다고 가정하자. 멱등성 검사는 여전히 통과하지만, 사전 조건은 통과하지 못한다. 사전 조건 검사가 없으면 초과 인출이 발생한다.

모든 중요한 작업에는 다음이 필요하다:

- **멱등성 키**: 중복 실행을 방지한다.
- **사전 조건 검사**: 상태가 승인된 내용과 여전히 일치하는지 확인한다.

### 작업 후 검증

"툴이 200을 반환했다"는 검증이 아니다. 실제 검증은 대상 상태를 다시 읽고 부작용이 실제로 발생했는지 확인한다. 패턴:

- 데이터베이스 업데이트: `UPDATE ... RETURNING *` 후 반환된 행이 의도한 상태와 일치하는지 확인.
- 이메일 전송: 제출 후 보낸 편지함에서 메시지 ID 확인.
- 파일 쓰기: 파일을 다시 읽고 해시 확인.
- API 호출: 대상 리소스에 대한 후속 `GET` 수행.

검증이 실패하면 워크플로우는 알려진 불량 상태에 있다. 롤백이 시작된다.

### 롤백 계획

제안 후 커밋(레슨 15)의 모든 중요한 작업은 롤백 계획을 수반한다. 유형:

- **인밴드 롤백**: 부작용을 직접 역전(`INSERT` 후 `DELETE`, 전송 후 `수정 이메일 전송`).
- **보상 트랜잭션**: 원본을 무효화하는 새로운 작업(표준 SAGA 패턴).
- **아웃오브밴드 롤백**: 인간에게 알림, 워크플로우 일시 중지, 조사를 위해 불량 상태 유지.

무작업 롤백("이것을 되돌릴 수 없음")은 제안 시 명시되어야 한다. 롤백이 없는 작업은 커밋 시 더 강력한 HITL(레슨 15의 도전-응답)이 필요하다.

### EU AI 법 제14조 운영 해석

제14조는 고위험 시스템에 대해 "효과적인 인간 감독"을 요구한다. 운영 측면에서 구현자는 다음과 같이 해석한다:

- 체크포인트는 감사자가 조회할 수 있어야 한다.
- 롤백은 리허설(최소 한 번 종단간 테스트)되어야 한다.
- 감사 추적은 배포 후에도 유지되어야 한다(체크포인트 백엔드는 일시적이지 않음).
- 실패한 검증은 알림이 발생해야 하며, 조용히 로깅되지 않아야 한다.

커밋 중 충돌한 후 재개하고 검증 + 롤백 경로 없이 부작용을 완료하는 워크플로우는 제14조 테스트를 통과하지 못한다.

### 날카로운 실패 모드: 중복 실행

이 분야에서 가장 흔한 프로덕션 사고:

1. 작업 승인, 멱등성 키 k.
2. 커밋 시작, 실행, 200 반환.
3. 워크플로우가 "커밋됨" 상태를 지속하기 전에 충돌.
4. 워크플로우 재개; "승인되었지만 커밋되지 않음"을 확인; 재실행.
5. 부작용이 두 번 발생.

완화: 실행 전 "진행 중" 의도를 지속하고, 멱등성 키로 실행한 후, 작업 후 검증이 성공해야만 "커밋됨"을 표시한다. 작업이 실행되고 상태 쓰기가 실패하면 검증 후 (필요시) 재실행한다. 상태 쓰기가 성공하고 작업이 실패하면 복구 경로를 통해 정확히 한 번 검증 및 실행한다.

## 사용 방법

`code/main.py`는 체크포인트(checkpoint) 기능, 멱등성(idempotency), 선행 조건(precondition), 검증(verify), 롤백(rollback)을 구현한 워크플로우를 포함합니다. 드라이버는 다음 네 가지 시나리오를 시뮬레이션합니다:  
- 깨끗한 실행(clean run)  
- 충돌 후 재시도(멱등성 작동)  
- 선행 조건 실패(워크플로우 중단, 실행 없음)  
- 검증 실패(롤백 실행)

## Ship It

`outputs/skill-rollback-rehearsal.md`는 제안된 워크플로우에 대한 롤백-리허설 테스트를 설계하고 체크포인트 백엔드의 감사 추적 지속성을 감사합니다.

## 연습 문제

1. `code/main.py`를 실행합니다. 네 가지 시나리오를 검증하세요. 커밋 중 충돌(crash-during-commit) 시나리오의 경우 재시도 전체에서 액션이 정확히 한 번 실행되는지 확인하세요.

2. "완료 표시 후 작업 수행(mark as done first, then do it)" 패턴을 수정하여 상태 쓰기(status write)가 액션 이후에 발생하도록 변경합니다. 충돌 시나리오를 다시 실행합니다. 중복 액션이 몇 번 실행되는지 측정하세요.

3. 특정 프로덕션 액션(예: "Slack 채널에 게시")에 대한 롤백 계획을 설계하세요. 인밴드(in-band), 보상형(compensating), 아웃오브밴드(out-of-band)로 분류하고 선택 이유를 설명하세요.

4. 알고 있는 워크플로우 하나를 선택합니다. 모든 상태 전이를 식별하고 각각에 내구성 요구사항(persist / do not persist)을 표시합니다. 현재 지속하지 않고 있는 항목을 세어보세요.

5. 리허설 롤백 테스트: 실제 워크플로우를 실행한 후 충돌을 유발하고 롤백 경로가 실행되는지 확인하는 엔드투엔드 테스트를 설계하세요. 테스트는 무엇을 검증해야 하나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|---|---|---|
| 체크포인트(Checkpoint) | "저장 지점" | 모든 그래프-상태 전환은 내구성 있는 저장소에 지속됨 |
| 리스(Lease) | "작업자 클레임" | 작업자가 실행을 진행 중임을 나타내는 단기 클레임; 충돌 시 만료 |
| 선행 조건(Precondition) | "상태 게이트" | 승인된 작업과 상태가 여전히 일관됨을 확인하는 단언 |
| 사후 검증(Post-action verify) | "재확인 검사" | 대상 시스템에서 부작용이 실제로 발생했는지 확인 |
| 인밴드 롤백(In-band rollback) | "직접 실행 취소" | 역연산으로 부작용을 되돌림 |
| 보상 트랜잭션(Compensating transaction) | "SAGA 취소" | 원래 작업을 무효화하는 새로운 작업 |
| 완료 표시 우선(Mark-as-done-first) | "상태 쓰기 순서" | 커밋에서 반환하기 전에 커밋된 상태를 지속 |
| 제14조(Article 14) | "EU AI법 인간 감독" | 운영적: 조회 가능한 체크포인트, 리허설된 롤백, 감사 가능한 추적 기록 |
