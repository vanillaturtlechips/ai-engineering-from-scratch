# 킬 스위치, 서킷 브레이커, 카나리 토큰

> 킬 스위치(kill switch)는 에이전트의 편집 영역 외부에 보관된 부울 값입니다. Redis 키, 기능 플래그, 서명된 구성 파일 등으로 에이전트를 완전히 비활성화합니다. 서킷 브레이커(circuit breaker)는 더 세분화된 제어를 제공합니다. 특정 패턴(예: 연속으로 5번 동일한 도구 호출)에 반응하여 해당 경로를 일시 중지하고 인간에게 알립니다. 카나리 토큰(canary token)은 고전적인 기만 기술에서 유래했습니다. 에이전트가 정당한 이유로 접근할 수 없는 가짜 자격 증명이나 허니팟 레코드를 두며, 이 레코드에 접근하면 경고가 트리거됩니다. eBPF 기반 데이터 경로(예: Cilium)는 격리된 파드의 이그레스 트래픽을 커널 계층에서 포렌식 허니팟으로 재작성할 수 있습니다. 공개된 Cilium 벤치마크에 따르면 부하 시 P99 데이터 경로 지연 시간이 1밀리초 미만입니다(정책 업데이트가 노드에 전달되는 방식에 따라 전파 예산이 결정되며, 데이터 경로 자체와는 무관함). 이동 평균에 적응하는 통계적 감지기(EWMA, CUSUM)는 드리프트를 조용히 수용합니다. 이를 굽히지 않는 엄격한 헌법적 제한과 함께 계층화하세요.

**유형:** 학습
**언어:** Python (표준 라이브러리, 3가지 감지기 시뮬레이터: 킬 스위치, 서킷 브레이커, 카나리)
**선수 지식:** 15단계 · 13 (비용 조정기), 15단계 · 10 (권한 모드)
**소요 시간:** ~60분

## 문제 정의

비용 제어기(레슨 13)는 에이전트가 지출할 수 있는 금액을 제한합니다. 그러나 예산 내에서 에이전트가 수행할 수 있는 행동을 제한하지는 않습니다. $50의 속도 제한을 가진 에이전트라도 여전히 비밀을 유출하거나, 잘못된 게시물을 게시하거나, 리소스를 삭제할 수 있습니다. 토큰 측면에서 비용이 많이 드는 행동이 실제로는 저렴한 경우가 많기 때문입니다.

이 레슨에서는 비용 계층 옆에 위치한 세 가지 감지기를 다룹니다:

1. **킬 스위치(kill switch)**: 에이전트가 접근할 수 없는 외부에 위치한 부울(boolean) 방식의 종료 버튼.
2. **서킷 브레이커(circuit breaker)**: 특정 경로를 일시 중지시키는 행동 패턴 감지기.
3. **캐너리 토큰(canary token)**: 합법적 이유가 없는 에이전트가 접촉 시 스스로를 노출시키는 유인 장치.

이 세 가지 모두 LLM 이전의 기술입니다. 고전적인 속임수, 속도 제한 우회, 기능 플래그 종료는 자율 에이전트보다 먼저 등장했습니다. 새로운 점은 공격 표면입니다: 에이전트는 신뢰할 수 없는 콘텐츠를 읽고(레슨 11), 자신의 메모리를 편집하며, 여러 안전한 것처럼 보이는 행동을 조합해 위험한 행동을 수행할 수 있습니다. 여기서 언급된 감지기들은 에이전트의 자체 보고를 신뢰하지 않기 때문에 작동합니다.

## 개념

### 킬 스위치

킬 스위치는 에이전트가 읽을 수는 있지만 쓸 수는 없는 부울 값입니다. 구현 방식:

- **관리형 서비스의 기능 플래그.** LaunchDarkly, Statsig, Unleash. 업데이트는 수초 내 전파됩니다.
- **에이전트가 폴링하는 Redis 키.** 단순함; 에이전트 프로세스가 매 턴마다 확인해야 합니다.
- **객체 저장소의 서명된 설정.** 에이전트는 부팅 시 서명을 검증하며, 서명되지 않은 상태는 거부합니다.
- **OS 수준 신호 또는 컨테이너 생명주기 종료.** Docker `kill`, Kubernetes `kubectl delete pod`, systemd stop.

올바른 킬 스위치의 속성:

- 에이전트가 이를 `off`로 설정할 수 없습니다. (에이전트 자격 증명이 쓰기 권한이 없는 시스템에 위치)
- 시작 시뿐만 아니라 모든 중요한 동작 시 확인됩니다.
- 꺼져 있을 때 에이전트는 외부적으로 관찰 가능한 동작(접근 가능한 시스템에 로깅 포함)을 하지 않습니다.
- 다시 활성화하는 것은 자동 타임아웃이 아닌 명시적인 인간 동작입니다.

### 서킷 브레이커

서킷 브레이커는 전체 에이전트가 아닌 특정 패턴을 일시 중지합니다. 고전적인 구조(2007년 Nygard 저서, 현재 기준):

- **닫힘(Closed):** 동작 허용.
- **열림(Open):** 동작 차단.
- **반개(Half-open):** 쿨다운 후 1–3회 프로브 시도(기본값 1회); 성공 시 브레이커 닫힘, 실패 시 다시 열림.

에이전트 관련 트리거:

- 동일한 툴 호출 5회 연속(반복 루프).
- 다른 입력으로 동일 툴에 연속 5회 실패(시스템적 장애).
- 임계값을 초과하는 툴 호출 속도(Lesson 13 속도).
- 신뢰 영역 외 콘텐츠 읽기 후 특정 툴 호출(예: `file.delete`)(Lesson 11).

### 캐니리 토큰

캐니리 토큰(허니토큰)은 에이전트가 절대 접근해서는 안 되는 입력입니다. 접근 시 경보가 발생합니다.

에이전트 예시:

- 실제 권한이 없는 환경의 가짜 `AWS_SECRET_ACCESS_KEY`. 유출 시도는 즉시 추적 가능합니다.
- 알려진 센티넬 값으로 표시된 가짜 데이터베이스 레코드. 이 레코드를 읽거나 업데이트하면 경보가 발생합니다.
- 작업 공간의 가짜 파일. 이 파일을 읽거나 수정하면 의심스럽습니다.
- 세션 시작 시 주입된 가짜 메모리 항목. 이후 출력에 나타나면 메모리가 유출된 것입니다.

캐니리 설계는 워크플로우에 따라 다릅니다. 에이전트가 합법적 이유로 접근할 수 있는 캐니리는 노이즈입니다.

### 통계적 한계와 하드 한계 계층화 이유

통계적 감지기(EWMA, CUSUM, 툴 호출 속도의 z-점수)는 이동 기준선에 적응합니다. 이는 합법적 드리프트에는 장점이지만 적대적 드리프트에는 취약점입니다. 인내심 있는 공격자는 기준선을 천천히 이동시켜 경보를 회피할 수 있습니다.

하드 헌법적 한계(Lesson 17)는 굽히지 않습니다. "절대 `secrets@`로 이메일 보내지 않기"를 헌법적 규칙으로 정하면 더 간단하고 감사 가능하며 느린 드리프트로 게임화할 수 없습니다. 계층화 패턴: 통계적 감지기가 대부분의 노이즈를 잡고, 하드 한계가 감지기를 우회한 공격을 잡습니다.

### eBPF 데이터 경로 리다이렉트를 통한 격리

Cilium의 eBPF 기반 네트워크 정책은 격리된 파드의 이그레스 트래픽을 커널 계층에서 포렌식 허니팟으로 재작성할 수 있습니다. Cilium의 공개 벤치마크에 따르면 TCP_RR에서 P99 데이터 경로 지연 시간이 서브 밀리초 수준이며, 클러스터에서 격리를 *트리거*하는 데 걸리는 벽시계 시간은 정책 업데이트가 제어 평면에서 노드로 전파되는 방식(운영자가 미리 로드된 Kubernetes에서는 수백 밀리초가 현실적; 테일 지연은 클러스터 크기와 조정 주기에 따라 다름)에 의해 결정됩니다. 이는 두 가지를 제공합니다: 에이전트가 실제 데이터를 유출하지 않으며, 허니팟 내 에이전트 동작이 완전히 관찰됩니다. 캐니리 토큰과 동일한 패턴, 다른 계층: 에이전트는 제한된 상태에서 시도하는 행동으로 의도를 드러냅니다.

### 어떤 감지기에도 잡히지 않는 것

- **허용된 프리미티브로 구성된 시맨틱 공격.** 모든 개별 동작은 모든 감지기를 통과하지만, 조합이 문제입니다. 이는 Lesson 10의 Auto Mode 복합 실패 사례입니다.
- **페이로드의 일부로 감지기를 포함하는 공격.** 에이전트가 "행동 전 캐니리 비활성화"라는 페이지를 읽고 해당 기능이 있다면 감지기가 무력화됩니다. 캐니리 토큰은 에이전트가 수정할 수 없는 시스템에 위치해야 합니다.

## 사용 방법

`code/main.py`는 세 개의 감지기(킬 스위치, 서킷 브레이커, 캐니리 파일)를 통한 짧은 에이전트 궤적을 시뮬레이션합니다. 외부 딕셔너리에 보관된 킬 스위치, 동일한 툴 호출 5회 시 작동하는 서킷 브레이커, 읽기 시 경고를 트리거하는 캐니리 파일이 포함됩니다. 합성 궤적을 입력으로 사용하며, 이는 합법적인 작업, 반복 루프, 캐니리 프로브, 그리고 킬 스위치 작동 시나리오(에이전트 작업 중단)로 구성됩니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 회로 차단기(circuit breaker)가 5번째(동일한 호출) 턴에서 작동하고, 카나리(canary)가 9번째 턴(가짜 키 읽기)에서 작동하는지 확인하세요.

2. 통계적 감지기 추가: 도구 호출률에 대한 EWMA(지수 가중 이동 평균) z-점수. 천천히 드리프트하는 궤적을 입력하고 감지기가 작동하지 않음을 보여주세요. 이제 하드 제한(10분 내 50회 이상의 도구 호출 금지)을 추가하고 동일한 궤적에서 하드 제한이 작동하는지 확인하세요.

3. 브라우저 에이전트(레슨 11)를 위한 카나리 토큰 세트를 설계하세요. 최소 3개의 카나리와 각각이 감지할 수 있는 사항을 나열하세요.

4. Cilium 네트워크 정책 문서를 읽으세요. 이그레스 리다이렉트(egress-redirect) 격리 흐름을 구체적으로 설명하세요: 어떤 정책 선택자, 어떤 파드, 어떤 이그레스 재작성, 어떤 알림이 포함되는지. "격리 결정"부터 "첫 리다이렉트 패킷"까지의 벽시계 지연 시간을 결정하는 요소는 무엇인가요?

5. 킬 스위치(kill-switch)된 에이전트의 재활성화 절차를 정의하세요. 누가 재활성화할 수 있나요? 문서화해야 할 사항은 무엇인가요? 재활성화 전에 에이전트에서 변경되어야 할 사항은 무엇인가요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|---|---|---|
| 킬 스위치(Kill switch) | "오프 버튼" | 에이전트의 편집 영역 밖에 있는 부울 값; 모든 중요한 동작 시 확인됨 |
| 서킷 브레이커(Circuit breaker) | "패턴 일시 정지" | 반복, 실패율 또는 속도 제한에 따른 동작별 트립 |
| 카나리 토큰(Canary token) | "허니토큰" | 에이전트가 정당하게 접근할 이유가 없는 유인 장치; 접근 시 경고 발생 |
| 허니팟(Honeypot) | "포렌식 샌드박스" | 격리된 에이전트를 관찰하는 리디렉션된 트래픽/작업 공간 |
| EWMA(지수 가중 이동 평균) | "이동 평균" | 지수 가중치 적용; 드리프트(기능 + 버그)에 적응 |
| CUSUM(누적 합) | "누적 합계" | 기준선 대비 지속적 편차 감지 |
| 하드 리미트(Hard limit) | "헌법적 규칙" | 적응되지 않음; 이력과 무관하게 상수 유지 |
| 컨스티튜셔널 리미트(Constitutional limit) | "항상 참인 규칙" | 레슨 17의 헌법에 연결됨; 에이전트가 편집 불가 |
