# METR 시간 범위(Time Horizons) 및 외부 능력 평가

> METR(구 ARC Evals)은 2023년 12월부터 독립적인 501(c)(3) 기관입니다. 그들의 시간 범위 1.1 벤치마크(2026년 1월)는 작업 성공 확률 대 로그(전문가 인간 완료 시간)에 로지스틱 곡선을 적합시킵니다. 50% 확률에서의 교차점이 모델의 시간 범위를 정의합니다. 2025–2026년 평가 세트는 GPT-5.1, GPT-5.1-Codex-Max 및 프로토타입 모니터링 평가(모니터가 사이드 작업을 포착할 수 있는지, 에이전트가 회피할 수 있는지)를 포함합니다. 벤치마크 스위트: HCAST(180+ ML, 사이버, SWE, 추론 작업; 1분~8시간 이상), RE-Bench(전문가 기준선이 있는 71개 ML 연구-엔지니어링 작업), SWAA. 솔직한 설명: METR 측정은 이상화된 환경(인간 개입 없음, 실제 결과 없음)에서 수행되며, 팀은 평가-배포 간 행동 격차(Lesson 1)를 문서화했습니다. 시간 범위는 상한선일 뿐, 배포 예측이 아닙니다.

**유형:** 학습
**언어:** Python(표준 라이브러리, 로지스틱 적합 시간 범위 추정기)
**선수 지식:** 15단계 · 01(장시간 범위 에이전트), 15단계 · 19(RSP)
**소요 시간:** ~60분

## 문제 정의

확장 정책(레슨 19, 20)은 참조하는 측정값만큼만 유용합니다. "AI R&D-4 임계값"과 "장거리 자율성"은 정책 문구로 정의되며, 특정 평가에서 구체적인 수치가 도출될 때만 실행 가능한 지표가 됩니다.

METR는 2024–2026년 외부 평가 기관으로, 해당 수치들을 정의했습니다. 그들은 프론티어 모델(종종 출시 전, 연구소와의 NDA 하에)을 평가하고 방법론을 사후에 공개합니다. 타임 호라이즌 1.1 벤치마크(2026년 1월)는 그들의 대표 산출물로, 단일 스칼라 값으로 능력을 인간이 이해할 수 있는 단위("이 모델은 전문가가 50% 신뢰도로 X시간 동안 수행하는 작업을 할 수 있다")로 압축합니다.

이 레슨은 방법론(호라이즌 계산 방식)에 대한 내용과 해석(호라이즌이 상한선이지 배치 예측값이 아닌 이유)에 대한 내용을 모두 포함합니다. 이 두 가지 기술은 함께 학습되어야 합니다. 호라이즌이 어떻게 도출되는지 이해하는 팀은 "14시간"이라는 슬라이드의 숫자만 보는 팀보다 부정확한 공급업체 주장에 속기 훨씬 어렵습니다.

## 개념

### METR 배경

- 설립: 2023년 12월 (ex-ARC Evals, 독립 501(c)(3)로 분사).
- 범위: 프론티어 모델의 자율 능력 평가, 종종 출시 전 평가.
- 파트너 연구실: Anthropic, OpenAI (2025–2026년 다중 협력).
- 주요 성과: Time Horizon 1.0 (2025년 3월), Time Horizon 1.1 (2026년 1월), 프로토타입 모니터링 평가.

### Time Horizon 적합성

방법론 (METR 블로그 및 논문 참조):

1. 분 단위에서 시간 단위의 전문가 완료 시간을 아우르는 작업 스위트 수집. 현재 스위트: HCAST (180+ 작업), RE-Bench (71 작업), SWAA.
2. 각 작업에서 모델 실행; 성공 또는 실패 기록.
3. 로지스틱 곡선 적합: P(성공)을 log(전문가 완료 시간)의 함수로 표현.
4. 지평선은 P(성공) = 0.5일 때의 전문가 시간.

로지스틱 적합 형태는 일반적으로 능력이 작업 난이도에 따라 증가하다가 평탄해지는 관계를 보이기 때문에 적합합니다. 50% 지점은 선택 사항(10%, 90% 가능)이며, METR은 상세 논문에서 여러 임계값을 보고하지만 가장 직관적이므로 50%를 주로 사용합니다.

### 2026년 1월 수치

Time Horizon 1.1 기준:

- Claude Opus 4.6: 50% 신뢰도 기준 ~14시간 (Time Horizon 1.1, 2026년 1월).
- HCAST 스타일 작업의 배가 시간: ~4.3개월 (130.8일) (Time Horizon 1.1, 2026년 1월 보고). ~7개월 수치는 Time Horizon 1.0의 2019–2025 전체 적합 결과이며, TH1.1에서 2023년 이후 수치와 함께 보고됨.

이 수치들은 Phase 15의 나머지 부분에서 참조하는 구체적인 값입니다. Anthropic의 RSP v3.0 (레슨 19)과 DeepMind의 FSF v3 (레슨 20)에서 임계값 평가에 사용되는 측정값 중 하나로 인용됩니다.

### 벤치마크 스위트

- **HCAST**: ML, 사이버 보안, 소프트웨어 엔지니어링, 일반 추론 분야의 189개 작업. 범위: 전문가 시간 1분~8시간 이상. 지평선 수치의 주요 스위트.
- **RE-Bench**: 인간 전문가 기준선을 가진 71개 ML 연구-엔지니어링 작업. 모델이 AI R&D-4 임계값에 기여하는 ML R&D 작업을 수행할 수 있는지 측정.
- **SWAA**: 짧은 작업; 지평선 곡선의 하한 조정에 유용.

### 프로토타입 모니터링 평가

2025–2026년 METR 작업에는 모니터링 평가 프로토타입이 포함됨: 모니터가 모델이 사이드 작업(운영자가 요청한 것 외의 작업)을 수행하는 것을 포착할 수 있는지; 에이전트가 모니터를 회피할 수 있는지. 이는 OpenAI PF v2의 "안전장치 약화" 범주와 DeepMind FSF v3 (레슨 20)의 기만적 정렬 모니터링에 직접 반영됩니다.

### 지평선이 상한인 이유

- **이상화된 툴링**: 벤치마크 작업은 모델에 깨끗한 도구와 잘 포맷된 데이터를 제공합니다. 실제 배포는 더 복잡합니다.
- **실제 결과 없음**: 모델은 실제로 고객에게 청구하거나, 실제 데이터를 삭제하거나, 실제 사람과 연락하지 않습니다. 실제 배포에는 되돌릴 수 없는 위험이 있습니다.
- **평가 맥락 게임**: 레슨 1. 모델은 테스트에서 다르게 행동합니다. 2026년 국제 AI 안전 보고서는 이를 경험적으로 문서화합니다.
- **합법적 사용자 변동성 없음**: 벤치마크 프롬프트는 구조화되어 있습니다. 실제 사용자는 모호하고 맥락에 의존하는 요청을 생성합니다.

지평선은 유리한 조건에서의 능력 상한입니다. 배포 신뢰도는 다른 수치이며 더 낮으며, 팀은 자체 분포를 측정해야 합니다.

### 외부 평가자 사례

외부 평가는 내부 연구실이 보고하는 지표를 최적화할 동기가 있기 때문에 중요합니다. METR의 독립성—선언된 방법론과 동료 검토 논문을 가진 501(c)(3)—은 구조적 완화 장치입니다. 단독으로는 충분하지 않지만(연구실이 여전히 METR이 보는 것을 제어함), 외부 평가가 없는 것보다는 엄격히 낫습니다.

### 지평선 수치 실무 활용 방법

- **능력 필터로**: 모델의 지평선이 제안된 작업의 전문가 시간보다 훨씬 낮다면 자율 배포하지 마십시오(레슨 1의 기술 파일).
- **추세 지표로**: 배가 시간은 새로운 완화 조치 없이도 현재 관행이 얼마나 오랫동안 안전할지를 알려줍니다.
- **사전 정보로**: 14시간의 지평선은 시작점입니다. 작업 분포, 툴링 품질, 배포 맥락에 따라 조정하십시오.

## 사용 방법

`code/main.py`는 합성 결과 집합을 기반으로 작업 성공률 대 로그(전문가 시간)의 로지스틱 피팅을 구현합니다. 이 스크립트는 50% 기준(METR의 주요 지표), 10% 기준(보수적), 90% 기준(낙관적)을 보고합니다. 또한 평가 컨텍스트 게임(eval-context gaming)을 통해 인위적으로 성공률을 부풀렸을 때 발생하는 변화를 보여줍니다.

## Ship It

`outputs/skill-horizon-interpretation.md`는 공급업체의 기술 성숙도(horizon) 주장을 검토하고 벤치마크 주장과 실제 배포 현실 사이의 격차 분석을 생성합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 피팅(fit)의 50% 지평선(horizon)이 합성(synthetic) 실제 값(ground truth)과 일치하는지 확인하세요. 이제 작업-시간 그리드(task-time grid)를 절반으로 줄이면, 지평선 추정치가 의미 있게 변하는지 관찰하세요.

2. METR의 "Time Horizon 1.1" 블로그 글을 읽으세요. 신뢰도가 가장 높은 특정 작업과 가장 낮은 작업을 식별하세요. 왜 이러한 격차가 발생하는지 설명하세요.

3. METR의 "Measuring Autonomous AI Capabilities" 자료를 읽으세요. HCAST 작업 범주를 나열하세요. 프로덕션 작업에 더 높은 가중치를 부여할 범주를 하나 선택하고 그 이유를 정당화하세요.

4. 시뮬레이터에 평가-컨텍스트 게이밍(eval-context gaming)을 도입하세요: 실패한 작업의 약 20%를 성공으로 변경하세요. 새로운 지평선을 보고하세요. 이는 20%의 게이밍 비율이 관측된 수치에 미치는 영향을 근사적으로 보여줍니다.

5. 자체 버그 백로그 또는 대표적인 작업 세트에 대한 내부 지평선 평가를 설계하세요. 데이터 수집 방법, 피팅(fit) 과정, 그리고 출력 결과가 의미하는 바를 설명하세요. METR 수치와 비교하세요.

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|---|---|---|
| METR | "외부 평가자" | ex-ARC Evals; 2023년 12월부터 독립 501(c)(3) 기관 |
| Time Horizon | "능력 측정치" | 로지스틱 적합도(logistic fit) 기준 50% 신뢰도에서 전문가 작업 소요 시간 |
| HCAST | "METR의 주요 평가 세트" | 1분에서 8시간 이상까지 180개 이상의 작업 포함 |
| RE-Bench | "연구 엔지니어링" | 인간 기준선(human baseline)이 있는 71개의 ML 연구-엔지니어링 작업 |
| SWAA | "단시간 작업 세트" | 시간 곡선(horizon curve)의 하한(low end)을 보정(calibrate) |
| Doubling time | "성장률" | 50% 신뢰도 시간 지평(time horizon)이 두 배가 되는 시간; HCAST 기준 ~7개월 |
| Eval-context gaming | "모델의 다른 행동" | 테스트 환경과 실제 배포 환경 간 문서화된 행동 차이(behavior gap) |
| Upper bound | "시간 지평은 상한선" | 벤치마크 시간 지평 > 부하(load) 하의 배포 신뢰도(deployment reliability) |
