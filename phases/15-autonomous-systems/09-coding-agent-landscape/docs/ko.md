# 자율 코딩 에이전트 현황 (2026)

> SWE-bench Verified는 3년 만에 4%에서 80.9%로 성장했습니다. 동일한 Claude Sonnet 4.5 모델이 SWE-agent v1에서 43.2%, Cline 자율에서 59.8% 점수를 기록했습니다 — 모델 자체만큼이나 주변 인프라(scaffolding)가 중요해졌습니다. OpenHands(이전 OpenDevin)는 가장 활발한 MIT 라이선스 플랫폼이며, CodeAct 루프는 JSON 도구 호출 대신 샌드박스에서 직접 Python 액션을 실행합니다. 헤드라인 수치 뒤에는 방법론적 문제가 숨겨져 있습니다: 500개 SWE-bench Verified 작업 중 161개는 1–2줄 변경만 필요로 하며, SWE-bench Pro(10+줄 작업)는 동일 프론티어 모델에서 23–59% 수준에 머뭅니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, CodeAct vs JSON 도구 호출 비교)
**선수 지식:** 14단계 · 07(도구 사용), 15단계 · 01(장시간 호라이즌 에이전트)
**소요 시간:** ~45분

## 문제 정의

"어떤 코딩 에이전트가 가장 좋은가"는 잘못된 질문입니다. 올바른 질문은 다음과 같습니다: 내 작업과 일치하는 작업 분포에서, 프로덕션 환경에서 실행할 스캐폴딩(scaffolding)을 사용할 때, 내가 얻는 종단간(end-to-end) 신뢰도는 얼마인가?

2022년부터 2026년 사이에 이 분야는 스캐폴딩(검색 계층, 플래너, 샌드박스, 편집-검증 루프, 피드백 형식 등)이 구조적 하중을 견딘다는 것을 배웠습니다. SWE-agent v1에서 Claude Sonnet 4.5는 SWE-bench Verified에서 43.2% 점수를 기록했지만, 동일한 모델이 Cline의 자율 스캐폴딩 내에서 실행되었을 때는 59.8%를 달성했습니다. 동일한 가중치임에도 16.6%의 절대적 점수 차이가 발생했습니다. 기본 모델은 구성 요소일 뿐이며, 루프가 제품입니다.

동반되는 문제는 벤치마크 포화 상태가 회귀(regression)를 숨긴다는 점입니다. SWE-bench Verified는 포화 상태에 가깝고, 쉬운 작업 꼬리(500개 작업 중 161개 작업으로 ≤2줄 변경 필요)가 상위 점수를 끌어올립니다. 실제 품질은 SWE-bench Pro(10+줄 변경 필요)와 같은 분포에서 측정하는 것이 더 적합하며, 여기서 동일한 리더들은 여전히 23~59% 사이에 머물러 있습니다.

## 개념

### SWE-bench, 한 단락

SWE-bench(Jimenez et al.)는 실제 GitHub 이슈와 정답 패치를 사용하여 테스트 스위트를 통과시키는 패치를 생성하도록 에이전트에게 요청합니다. SWE-bench Verified(OpenAI, 2024)는 모호하거나 오류가 있는 작업을 제거한 인간이 선별한 500개 작업 하위 집합입니다. SWE-bench Pro는 더 어려운 후속 평가—10줄 이상의 변경이 필요한 작업—로, 현재 최첨단 에이전트들은 23–59% 성능을 보입니다.

### 2022 → 2026 곡선이 실제로 보여주는 것

- **2022**: 연구용 모델이 원시 SWE-bench에서 ~4% 성능.
- **2024**: GPT-4 + Devin 스타일 스캐폴딩 ~14%; SWE-agent ~12%.
- **2025**: Aider와 SWE-agent 내부의 Claude 3.5/3.7 Sonnet이 40–55% 범위로 진입.
- **2026**: Claude Sonnet 4.5 및 최첨단 경쟁자들이 SWE-bench Verified에서 70–80%+ 성능. Epoch AI의 리더보드가 이를 실시간으로 추적합니다.

이 기울기는 세 가지 복합 소스에서 비롯되었습니다: 더 나은 기본 모델, 더 나은 스캐폴딩(CodeAct, 반성, 검증자 루프), 더 나은 벤치마크(Verified의 노이즈 제거).

### CodeAct vs JSON 도구 호출

OpenHands(All-Hands-AI, arXiv:2407.16741, 이전 OpenDevin)는 특정 아키텍처 베팅을 했습니다: 모델이 JSON 도구 호출을 생성하고 호스트가 이를 디코딩/실행하는 대신, 모델이 Python 코드를 생성하고 Jupyter 스타일 커널이 샌드박스에서 이를 실행합니다. 에이전트는 한 액션 내에서 파일 반복, 도구 체이닝, 자체 예외 처리가 가능합니다.

트레이드오프:

- **JSON 도구 호출**: 모든 액션은 한 턴; 감사 용이; 제한된 구성 가능성; 각 호출이 명시적 검증기를 통과하므로 기본적으로 안전.
- **CodeAct**: 한 액션이 전체 프로그램; 구성 가능; 강화된 샌드박스 필요(OpenHands는 Docker 격리 사용); 샌드박스 런타임이 허용하는 모든 실패 모드 가능.

두 아키텍처 모두 프로덕션에 사용 중입니다. CodeAct는 오픈 플랫폼(OpenHands, smolagents)에서 우세합니다. JSON 도구 호출은 관리형 서비스(Anthropic Managed Agents, OpenAI Assistants)에서 여전히 우세하며, 제공자가 실행기를 제어합니다.

### 2026 환경의 스캐폴딩

| 스캐폴드 | 라이선스 | 실행 모델 | 주목할 만한 특성 |
|---|---|---|---|
| OpenHands (OpenDevin) | MIT | Docker 내 CodeAct | 가장 활발한 오픈 플랫폼; 이벤트 스트림 재생 가능 |
| SWE-agent | MIT | 에이전트-컴퓨터 인터페이스(ACI) | 최초의 엔드투엔드 SWE-bench 스캐폴드 |
| Aider | Apache-2 | 로컬 리포지토리에서 diff를 통한 편집 | 최소 스캐폴드, 강력한 회귀 안정성 |
| Cline | Apache-2 | 도구 정책을 가진 VS Code 에이전트 | Sonnet 4.5에서 가장 높은 점수를 기록한 오픈 스캐폴드 |
| Devin (Cognition) | 프로프라이어터리 | 관리형 VM + 플래너 | 최초의 "AI 소프트웨어 엔지니어" 제품 범주 |
| Claude Code | 프로프라이어터리 | 권한 모드 + 루틴 | 레슨 10에서 에이전트 루프를 상세히 다룸 |

### 스캐폴딩이 지배적인 이유

코딩 실행은 장기 궤적(Lesson 1)입니다. 신뢰성은 단계마다 누적됩니다. 스캐폴딩이 점수를 높이는 세 가지 영역:

1. **검색**: 읽을 적절한 파일을 찾는 것이 숨겨진 병목입니다. SWE-agent의 ACI, OpenHands의 파일 인덱스, Aider의 리포지토리 맵이 모두 이를 해결합니다.
2. **검증자 루프**: 테스트 실행, 스택 추적 읽기, 재시도는 SWE-bench에서 10점 이상의 차이를 만듭니다.
3. **실패 격리**: 오류 시 롤백하는 샌드박스는 누적 손상을 방지합니다. 검증자 루프 유무에 따른 동일 모델은 완전히 다른 제품처럼 보입니다.

### 벤치마크 포화 상태와 실제 분포

OpenHands 저자와 Epoch AI는 SWE-bench Verified에 쉬운 꼬리 부분이 있음을 지적합니다: 500개 작업 중 161개는 1–2줄 변경만 필요합니다. 높은 점수는 부분적으로 이 꼬리 부분에 의해 주도됩니다. SWE-bench Pro는 10줄 이상의 변경으로 제한하며, 최첨단 시스템에서도 23–59% 범위를 반환합니다. 실제 프로덕션 분포는 Verified보다 Pro에 훨씬 가까울 것입니다.

에이전트 선택에 대한 시사점: 자체 버그 백로그의 Pro 유사 하위 집합을 실행하세요. 중요한 점수는 출시하는 작업과 유사한 작업에서의 점수입니다.

## 사용 방법

`code/main.py`는 고정된 미니 작업 분포에서 두 가지 장난감 에이전트 스캐폴드를 비교합니다:

1. **JSON 도구 호출** 스캐폴드: 한 턴에 하나의 액션을 수행합니다.
2. **CodeAct** 스캐폴드: 액션마다 작은 Python 코드 조각을 생성할 수 있습니다.

둘 다 "모델" 스텁(결정적 규칙)을 사용하므로, 이 비교는 모델 품질과 무관하게 스캐폴드 자체의 성능만 분리합니다. 출력 결과는 CodeAct 스캐폴드가 더 적은 턴 수로 더 많은 작업을 해결하지만, 액션당 영향 범위가 더 크다는 단점이 있음을 보여줍니다.

## Ship It

`outputs/skill-scaffold-audit.md`는 채택 전에 제안된 코딩 에이전트 스캐폴드를 감사하는 데 도움을 줍니다: 검색 품질, 검증기 존재 여부, 샌드박스 격리, 벤치마크-분포 적합성.

## 연습 문제

1. `code/main.py`를 실행해 보세요. 동일한 작업 세트에서 각 스캐폴드(scaffold)가 몇 턴(turn)을 소요하는지 확인하세요. 각 스캐폴드의 액션당 블라스트 반경(blast radius)은 어떻게 되나요?

2. OpenHands 논문(arXiv:2407.16741)을 읽어보세요. 논문에서는 복잡한 작업에서 CodeAct가 JSON 툴 호출보다 우수하다고 주장합니다. 논문이 인정하는 하나의 실패 모드(failure mode)를 식별하고, 해당 모드가 프로덕션 환경에서 언제 지배적으로 나타날지 한 문장으로 설명하세요.

3. 버그 백로그에서 두 파일에 걸쳐 10줄 이상의 변경이 필요한 작업 하나를 선택하세요. (a) JSON 툴 호출과 (b) CodeAct 환경에서 프론티어 모델(frontier model)의 종단간(end-to-end) 성공 확률을 추정하고, 그 차이를 정당화하세요.

4. SWE-bench Verified에는 161개의 단일 파일, 1–2줄 작업이 포함되어 있습니다. 이들을 제외하는 평가 지표를 구성하세요. 리더보드는 어떻게 변화하나요?

5. "Introducing SWE-bench Verified"(OpenAI)를 읽어보세요. 모호한 작업을 제거하기 위해 사용된 구체적인 방법론을 설명하고, 큐레이션 과정에서 놓칠 수 있는 하나의 범주를 언급하세요.

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|---|---|---|
| SWE-bench | "코딩 벤치마크" | 실제 GitHub 이슈와 정답 패치 및 테스트 스위트 |
| SWE-bench Verified | "정제된 부분 집합" | 500개의 인간 선별 작업, 쉬운 꼬리 분포 존재 |
| SWE-bench Pro | "더 어려운 부분 집합" | 10+ 줄 변경; 최전선 성능은 23–59% |
| CodeAct | "코드-액션" | 에이전트가 Python 코드 생성; Jupyter 스타일 커널이 샌드박스에서 실행 |
| JSON 툴 호출 | "함수 호출" | 각 액션은 구조화된 JSON 페이로드이며 실행 전 검증됨 |
| Scaffold | "에이전트 프레임워크" | 검색 + 계획자 + 실행기 + 검증기 루프가 기본 모델 주변에 구성 |
| ACI (에이전트-컴퓨터 인터페이스) | "SWE-에이전트 포맷" | 인간 셸이 아닌 LLM 인체공학에 맞춰 설계된 명령어 집합 |
| 검증기 루프 | "테스트-재시도" | 테스트 실행, 출력 읽기, 패치 수정; 모델 외 가장 큰 신뢰성 향상 요소 |
