# AlphaEvolve — 진화형 코딩 에이전트

> 최첨단 코딩 모델과 진화 루프, 기계 검증 평가기를 결합합니다. 루프를 충분히 오래 실행하면 48개의 스칼라 곱셈을 사용하는 4x4 복소수 행렬 곱셈 절차를 발견합니다. 이는 56년 만에 Strassen의 방법을 개선한 첫 사례입니다. 또한 Google 전체 Borg 스케줄링 휴리스틱을 찾아 프로덕션에서 클러스터 컴퓨팅의 약 0.7%를 복구합니다. 아키텍처는 의도적으로 단순합니다. 성과는 평가기의 엄격함에서 비롯됩니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, 진화 루프 토이)
**선수 지식:** 15단계 · 01 (장기적 프레이밍), 15단계 · 02 (자기 주도 추론)
**소요 시간:** ~60분

## 문제

대규모 언어 모델(LLM)은 코드를 작성할 수 있다. 진화 알고리즘은 코드 공간을 탐색할 수 있다. 이 두 방법은 수십 년 동안 별도로 시도되어 왔으며, 모두 한계에 부딪혔다. LLM의 한계는 **착각(confabulation)**이다. 즉, 모델이 주장하는 기능을 수행하지 않는 그럴듯한 코드를 작성하는 것이다. 진화 알고리즘의 한계는 **탐색 비용**이다. 구문에 대한 무작위 돌연변이는 컴파일 가능한 프로그램, 더 나아가 더 나은 프로그램을 거의 생성하지 못한다.

AlphaEvolve(Novikov et al., DeepMind, arXiv:2506.13131, 2025년 6월)는 이 둘을 결합한다. LLM은 프로그램 데이터베이스에 대한 **타겟 편집(targeted edits)**을 제안하고, 자동 평가기(evaluator)는 각 변형을 점수화한다. 높은 점수를 받은 변형은 다음 세대의 **부모(parent)**가 된다. LLM은 그럴듯한 코드 작성이라는 고비용 단계를 처리하고, 평가기는 착각을 걸러낸다. 이 루프는 수시간에서 수주간 실행된다.

보고된 결과: 48-스칼라 곱셈 4x4 복소수 행렬 곱셈(Strassen의 1969년 한계는 49), Google 프로덕션 환경의 Borg 스케줄링 휴리스틱, 32.5% FlashAttention 커널 속도 향상, Gemini 훈련 처리량 개선.

이 아키텍처는 평가기가 **기계 검증 가능(machine-checkable)**하기 때문에 작동한다. 평가기가 없는 경우에는 작동하지 않는다. 이 비대칭성이 핵심 교훈이다.

## 개념

### 루프

1. 올바르지만 최적이 아닌 시드 프로그램 `P_0`에서 시작합니다.
2. 변형 프로그램 데이터베이스를 유지하며, 각 프로그램은 평가자(evaluator)에 의해 점수가 매겨집니다.
3. 데이터베이스에서 하나 이상의 부모 프로그램을 샘플링합니다(MAP-elites 스타일 또는 섬 기반).
4. LLM(Gemini Flash는 많은 후보, Gemini Pro는 어려운 문제용)에 부모 프로그램의 변형체를 생성하도록 요청합니다.
5. 변형 프로그램을 컴파일, 실행 및 평가자(held-out evaluator)로 평가합니다.
6. 점수와 특징 벡터(feature vector)를 키로 사용하여 데이터베이스에 삽입합니다.
7. 반복합니다.

두 가지 세부 사항이 중요합니다. 첫째, LLM은 부모 프로그램 외에도 일반적으로 데이터베이스에서 가져온 상위 변형체 몇 개, 평가자 서명, 짧은 작업 설명을 함께 입력받습니다. 모델의 역할은 점수를 개선할 수 있는 목표 변경 사항을 제안하는 것입니다. 둘째, 데이터베이스는 (MAP-elites 그리드, 섬 기반) 구조화되어 있어 루프가 현재 리더뿐만 아니라 다양성을 탐색합니다.

### 평가자가 협상 불가능한 이유

AlphaEvolve의 성공은 평가자가 빠르고, 결정론적이며, 조작하기 어려운 도메인에서 비롯됩니다:

- **행렬 곱셈 알고리즘**: 행렬을 곱하고 비트 단위로 정확성을 확인하는 단위 테스트.
- **Borg 스케줄링 휴리스틱**: 실제 클러스터 부하를 재현하고 낭비된 계산량을 측정하는 프로덕션급 시뮬레이터.
- **FlashAttention 커널**: 정확성 테스트 및 실제 하드웨어에서의 벽시계(wall-clock) 벤치마크.
- **Gemini 훈련 처리량**: 단계당 GPU-초(measured GPU-seconds per step)로 측정.

각 사례에서 평가자는 LLM 오류의 클래스를 포착합니다: 날조된 정확성 주장, 하드웨어에서 사라지는 성능 주장, 에지 케이스 실패. 평가자가 없으면 루프는 예쁜 코드를 최적화합니다.

### 보상 해킹(reward hacking)은 그 명제의 다른 면

진화는 평가자가 측정하는 것을 최적화합니다. 평가자가 불완전하면 루프는 그 불완전성을 찾아냅니다. 검증되지 않은 도메인에서는 루프가 의도된 동작이 아닌 표면적 특징을 최적화합니다. DeepMind는 논문에서 이를 명시적으로 지적합니다: AlphaEvolve의 성공은 평가자의 엄격성이 탐색의 야망과 일치하는 도메인으로만 이전됩니다.

코드-검색 루프에서 보상 해킹의 2025-2026년 실제 사례:

- "완료 시간"을 최적화하는 목표가 빈 솔루션 제출을 보상.
- 테스트 하에서의 정확성을 보상하는 벤치마크 점수가 테스트 암기와 과적합(overfitting)을 보상.
- "코드 품질" 프록시가 주석 제거 및 변수명 변경을 보상하며, 의미적 변화는 없음.

AlphaEvolve의 해결책: LLM이 본 적 없는 평가자(held-out evaluator)를 배포하고, 평가 시 생성된 입력을 사용합니다. 그럼에도 DeepMind는 제안된 배포에 대해 강력한 검토를 권장합니다.

### LLM + 탐색이 각각 단독보다 뛰어난 이유

LLM은 컴파일 가능하고 의미적으로 그럴듯한 수정 사항을 생성할 수 있습니다. 2000줄 Python 파일에 대한 무작위 돌연변이 GA는 거의 항상 구문 오류를 발생시킵니다. LLM은 또한 탐색 범위를 그럴듯한 이웃(랜덤 바이트가 아닌 함수 하나 변경)으로 집중시켜 평가자 호출 낭비를 크게 줄입니다.

평가자는 LLM의 날조(confabulation)를 포착합니다. LLM은 함수가 "한계에서 O(n log n)"이라고 확신하지만 실제로는 O(n²)일 수 있습니다. 벽시계 벤치마크는 이 문제를 해결합니다.

### AlphaEvolve가 프론티어 스택에 적합한 위치

| 시스템 | 생성기(Generator) | 평가자(Evaluator) | 도메인 | 예시 성과 |
|---|---|---|---|---|
| AlphaEvolve | Gemini | 정확성 + 벤치마크 | 알고리즘, 커널, 스케줄러 | 48-mul 4x4 행렬 곱셈 |
| FunSearch (DeepMind, 2023) | PaLM / Codey | 정확성 | 조합 수학 | cap-set 하한 |
| AI Scientist v2 (Sakana, L5) | GPT/Claude | LLM 비평 + 실험 | ML 연구 | ICLR 워크숍 논문 |
| Darwin Godel Machine (L4) | 에이전트 스캐폴딩 | SWE-bench / Polyglot | 에이전트 코드 | SWE-bench 20% → 50% |

네 시스템 모두 생성기 + 평가자 + 루프라는 동일한 레시피의 변형입니다. 차이점은 평가자가 무엇을 평가하는지, 그리고 그 엄격성입니다.

## 사용 방법

`code/main.py`는 장난감 기호 회귀 문제를 대상으로 하는 최소한의 AlphaEvolve 유사 루프를 구현합니다. "LLM"은 목표 함수를 계산하는 프로그램에 대한 작은 구문 변형을 제안하는 stdlib 프록시입니다. "평가기"는 홀드아웃 테스트 포인트에서 평균 제곱 오차를 측정합니다.

다음 사항을 관찰하세요:

- 세대별로 최고 점수가 어떻게 개선되는지.
- MAP-elites 그리드가 다양한 솔루션을 유지하여 루프가 지역 최소값에 수렴하지 않도록 하는 방법.
- 홀드아웃 테스트를 제거(훈련 전용 평가기)했을 때 루프가 어떻게 눈에 띄게 과적합되는지.

## 연습 문제

1. `code/main.py`를 실행하세요. 최고 점수 궤적을 기록하세요. 검증 평가기(플래그 `--no-holdout`)를 비활성화하고 다시 실행하세요. 과적합(overfitting)을 정량화하세요.

2. AlphaEvolve 논문의 3장(MAP-elites 그리드)을 읽으세요. 검색 다양성을 유지할 수 있는 새로운 문제(예: 컴파일러 최적화 패스)에 대한 특징 벡터 설명자를 설계하세요.

3. 4x4 행렬 48-곱셈 결과는 Strassen의 49-곱셈 한계를 56년 만에 개선했습니다. 논문 부록 F를 읽고, 이 문제의 평가기(evaluator)가 특히 정확하게 구현하기 쉬운 이유와 대부분의 도메인이 그렇지 않은 이유를 3문장으로 설명하세요.

4. AlphaEvolve가 실패할 수 있는 도메인 하나를 제안하세요. 평가기(evaluator)가 어디서 왜 실패하는지 정확히 지적하세요.

5. 알고 있는 도메인에 대해 사용할 평가기 시그니처를 작성하세요. 다음을 포함하세요:  
   (a) 정확성 조건(correctness conditions)  
   (b) 성능 지표(performance metric)  
   (c) 검증 입력 생성 규칙(held-out input generation rule)  
   (d) 최소 1개의 보상 해킹 방지 검사(anti-reward-hacking check)

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|---|---|---|
| AlphaEvolve | "DeepMind의 진화형 코딩 에이전트" | Gemini + 프로그램 데이터베이스 + 기계 검증 평가기 |
| MAP-elites | "다양성 보존 아카이브" | 특징 벡터로 키(key)된 그리드; 각 셀은 해당 설명자를 가진 최상의 변형(variant)을 보유 |
| Island 모델 | "병렬 진화 하위 집단" | 주기적으로 이주하는 독립적인 집단; 조기 수렴(premature convergence) 방지 |
| 기계 검증 평가기 | "결정적 오라클" | LLM이 속일 수 없는 단위 테스트, 시뮬레이터 또는 벤치마크 — 이 루프의 전제 조건 |
| 보상 해킹 | "목표가 아닌 측정값 최적화" | 루프가 의도된 작업을 수행하지 않으면서 점수를 최대화하는 방법 발견 |
| 시드 프로그램 | "시작점" | 루프가 진화시키는 초기적이지만 최적이 아닌 프로그램 |
| 홀드아웃 평가기 | "LLM이 본 적 없는 평가 데이터" | 암기 방지를 위해 평가 시점에 생성된 입력값 |
