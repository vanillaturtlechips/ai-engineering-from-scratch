# 액션 예산, 반복 캡, 비용 제어 장치

> 한 중견 전자상거래 에이전트의 월간 LLM 비용이 "주문 추적" 스킬 활성화 후 $1,200에서 $4,800로 급증했습니다. 이는 가격 책정 오류가 아닙니다. 새로운 루프를 발견한 에이전트가 그 안에서 계속 비용을 소모한 것입니다. Microsoft의 에이전트 거버넌스 툴킷(2026년 4월 2일)은 이러한 문제에 대한 방어 체계를 정립했습니다: 요청당 `max_tokens`, 작업당 토큰 및 달러 예산, 일일/월간 캡, 반복 캡, 계층적 모델 라우팅, 프롬프트 캐싱, 컨텍스트 윈도우링, 고가 액션에 대한 HITL(Human-in-the-Loop) 체크포인트, 예산 초과 시 킬 스위치. Anthropic의 Claude Code Agent SDK도 동일한 기본 기능을 다른 이름으로 제공합니다. 금융 속도 제한(예: 10분 내 $50 초과 시 접근 차단)은 월간 캡보다 빠르게 루프를 감지합니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리, 계층적 비용 제어 시뮬레이터)  
**선수 지식:** 15단계 · 10 (권한 모드), 15단계 · 12 (지속 실행)  
**소요 시간:** ~60분

## 문제 정의

자율 에이전트는 매 턴마다 실제 돈을 소비합니다. 챗봇의 나쁜 출력은 단순히 잘못된 답변이지만, 에이전트의 나쁜 루프는 청구서로 이어집니다. 업계에서 문서화된 이 실패 모드의 용어는 "지갑 거부(Denial of Wallet)"입니다. 에이전트가 계속 추론을 수행하고, 계속 도구 호출을 하며, 계속 비용을 발생시키지만, 이를 막을 수 있는 설계가 전혀 없기 때문입니다.

해결책은 단일 숫자가 아닙니다. 다양한 시간 규모와 세부 수준에서 적용되는 제한 사항의 계층적 구조(스택)입니다: 요청당, 작업당, 시간당, 일일, 주간, 월간 제한. 잘 설계된 스택은 몇 분 안에 폭주하는 루프를, 몇 시간 안에 서서히 발생하는 누수를, 하루 안에 잘못된 릴리스를 감지합니다. 동일한 스택은 에이전트가 장기적이고 자율적인 작업을 수행할 때도 예산을 안전하게 유지합니다.

이것은 엔지니어링 교훈입니다: 수학적 계산은 간단하지만, 팀이 실패하는 지점은 바로 이 규율(Discipline)에 있습니다. 아래 제한 사항 목록은 Microsoft 에이전트 거버넌스 툴킷(Microsoft Agent Governance Toolkit) 또는 Anthropic Claude 코드 에이전트 SDK 문서에서 명명된 항목들입니다.

## 개념

### 비용-거버넌스 스택

1. **요청당 `max_tokens`.** 간단합니다. 단일 호출이 무제한 완료를 생성하지 않도록 방지합니다.
2. **작업당 토큰 예산.** 전체 실행 동안 N 토큰을 초과하지 않습니다. 한도 도달 시 강제 종료됩니다.
3. **작업당 달러 예산.** 토큰과 동일하지만 통화 기준입니다. Claude Code의 `max_budget_usd`.
4. **도구 호출당 제한.** `WebFetch` 호출 N회, `shell_exec` 호출 N회 등으로 제한됩니다.
5. **반복 제한(`max_turns`).** 총 에이전트 루프 반복 횟수; 무한 추론 루프를 방지합니다.
6. **분당/시간당/일당/월당 제한.** 롤링 윈도우. 다양한 시간 규모에서 누수를 감지합니다.
7. **재정적 속도 제한.** 예: "10분 내 지출이 $50을 초과하면 접근 차단." 월간 한도 발동 전 루프 기반 과소비 감지.
8. **계층적 모델 라우팅.** 기본적으로 작은 모델 사용; 분류기가 작업이 큰 모델을 필요로 한다고 판단할 때만 업그레이드.
9. **프롬프트 캐싱.** 시스템 프롬프트와 안정된 컨텍스트는 공급자 캐시에 저장; 재전송 토큰 비용이 거의 없음.
10. **컨텍스트 윈도우링.** 압축/요약을 통해 활성 컨텍스트를 임계값 이하로 유지; 직접적인 토큰 비용 절감.
11. **고가 행동에 대한 HITL(Human-in-the-Loop) 체크포인트.** 비용이 많이 드는 행동(긴 도구 호출, 대용량 다운로드, 비용이 큰 모델 업그레이드) 전에 인간 승인 필요.
12. **예산 초과 시 킬 스위치.** 어떤 한도가 발동되면 세션 중단. 한도는 기록되며 별도의 재활성화 경로가 필요합니다.

### 단일 한도가 아닌 스택이 필요한 이유

단일 월간 한도는 지갑이 비어버린 후에야 폭주하는 에이전트를 감지합니다. 단일 요청당 한도는 세션 수준에서 아무것도 감지하지 못합니다. 다양한 실패 모드에는 서로 다른 시간 규모가 필요합니다:

- **폭주 루프** (5초 재시도에 갇힌 에이전트): 속도 제한으로 감지.
- **느린 누수** (작업당 예상 작업량의 ~2배 수행): 일일 한도로 감지.
- **불량 릴리스** (새 버전이 5배 토큰 사용): 주간/월간 한도로 감지.
- **합법적 급증** (버그가 아닌 실제 수요): 명확한 로그와 함께 시간/일 한도로 감지.

### Claude Code의 예산 표면

Claude Code 에이전트 SDK는 (공개 문서) 다음을 노출합니다:

- `max_turns` — 반복 제한.
- `max_budget_usd` — 달러 한도; 초과 시 세션 중단.
- `allowed_tools` / `disallowed_tools` — 도구 허용 목록 및 거부 목록.
- 도구 사용 전 커스텀 비용 계산을위한 후크 포인트.

권한 모드 사다리(레슨 10)와 결합합니다. `max_budget_usd`가 없는 `autoMode` 세션은 통제되지 않은 자율성입니다. Anthropic은 Auto Mode에 예산 제어가 필요하다고 명시적으로 언급; 분류기는 비용과 직교합니다.

### EU AI Act, OWASP 에이전트 Top 10

Microsoft의 에이전트 거버넌스 툴킷은 OWASP 에이전트 Top 10과 EU AI Act 제14조(인간 감독) 요구사항을 다룹니다. EU에서 프로덕션을 위해 로깅과 한도 강제 적용은 선택 사항이 아닙니다.

### 관찰된 $1,200 → $4,800 사례

Microsoft 문서에 있는 실제 사례: 새 도구 추가 후 월간 비용이 3배 증가한 전자상거래 에이전트. 이 도구는 에이전트가 모든 세션 동안 주문 상태를 폴링할 수 있게 했습니다. 루프 감지 없음. 도구당 한도 없음. 주간 성장률 경고 없음. 해결책은 도구당 한도와 일일 성장률 경고 추가였습니다. 이는 템플릿입니다: 모든 새 도구 표면은 새로운 잠재적 루프; 모든 새 도구는 자체 한도와 경고가 필요합니다.

## 사용 방법

`code/main.py`는 계층적 비용-거버넌스 스택을 사용한 경우와 사용하지 않은 경우의 에이전트 실행을 시뮬레이션합니다. 시뮬레이션된 에이전트는 일정 턴 이후에 폴링 루프로 표류하게 되며, 계층적 스택은 속도 제한 창 내에서 이를 감지하는 반면 단일 월간 캡은 며칠 후까지 작동하지 않습니다.

## Ship It

`outputs/skill-agent-budget-audit.md`는 제안된 에이전트 배포의 비용-거버넌스 스택을 감사하고 누락된 레이어를 표시합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 폴링-루프 궤적에서 반복 횟수 제한에 도달하기 전에 속도 제한이 작동하는지 확인하세요. 이제 속도 제한을 비활성화하고 반복 횟수 제한이 발동되기 전까지 에이전트가 "소비"하는 양을 측정하세요.

2. 브라우저 에이전트(레슨 11)를 위한 도구별 제한 세트를 설계하세요. 어떤 도구가 가장 엄격한 제한을 필요로 하나요? 어떤 도구는 위험 없이 무제한 실행될 수 있나요?

3. Microsoft 에이전트 거버넌스 툴킷 문서를 읽으세요. 툴킷이 명명한 모든 제한 유형을 나열하세요. 각각을 실패 모드(무한 루프, 느린 누수, 잘못된 해제, 급증) 중 하나에 매핑하세요.

4. 현실적인 작업(예: "리포지토리의 50개 이슈 분류")에 대해 무인 야간 실행 비용을 산정하세요. `max_budget_usd`를 추정치의 2배로 설정하세요. 2배 설정을 정당화하세요.

5. Claude Code의 `max_budget_usd`는 세션 누적 비용에서 발동됩니다. 외부에서 강제할 보완적 속도 제한을 설계하세요. 어떤 조건에서 차단이 트리거되며, 재활성화는 어떻게 구현되나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|---|---|---|
| 지갑 거부 | "폭주하는 청구서" | 에이전트가 지출 상한 없이 생성하는 루프 |
| max_tokens | "요청당 상한" | 단일 완성의 크기에 대한 상한선 |
| max_turns | "반복 상한" | 세션 내 에이전트 루프 반복 횟수의 상한선 |
| max_budget_usd | "달러 킬 스위치" | 세션 비용 상한선; 초과 시 중단 |
| 속도 제한 | "속도 상한" | 짧은 시간 창(예: 10분 내 $50)당 지출 제한 |
| 계층적 라우팅 | "소형 모델 우선" | 저렴한 모델 기본 사용; 분류기가 필요할 때만 업그레이드 |
| 프롬프트 캐싱 | "캐시된 시스템 프롬프트" | 공급자 측 캐시로 재전송 토큰 비용을 거의 제로로 감소 |
| HITL 체크포인트 | "인간 승인 게이트" | 고비용 작업 전 인간 개입 필요 |
