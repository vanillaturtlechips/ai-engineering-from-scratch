# 잠재 확산 & 스테이블 디퓨전

> 512×512 이미지의 픽셀 공간 확산은 계산적 전쟁 범죄입니다. Rombach et al. (2022)는 이미지 생성에 786k 차원이 모두 필요하지 않다는 것을 알아냈습니다 — 의미론적 구조를 포착할 수 있을 정도면 충분하고, 나머지는 별도의 디코더로 처리하면 됩니다. VAE의 잠재 공간 내에서 확산을 실행하세요. 이 하나의 아이디어가 스테이블 디퓨전입니다.

**유형:** 구축
**언어:** Python
**사전 요구 사항:** 8단계 · 02 (VAE), 8단계 · 06 (DDPM), 7단계 · 09 (ViT)
**소요 시간:** ~75분

## 문제 정의

512² 픽셀 공간에서의 디퓨전은 U-Net이 `[B, 3, 512, 512]` 형태의 텐서에서 실행됨을 의미합니다. 5억 개 파라미터를 가진 U-Net의 경우 각 샘플링 단계는 약 100 GFLOPS가 소요됩니다. 50단계를 수행하면 이미지당 5 TFLOPS가 필요합니다. 10억 개의 이미지로 학습하면 계산 비용이 엄청나게 증가합니다.

이러한 FLOPs의 대부분은 인식적으로 중요하지 않은 세부 사항—손실 압축 VAE로 제거할 수 있는 고주파 텍스처—을 처리하는 데 사용됩니다. 롬바흐의 아이디어: VAE를 한 번 학습시킨 후(*첫 번째 단계*), 이를 고정하고 4채널 64×64 잠재 공간에서 디퓨전을 완전히 실행합니다(*두 번째 단계*). 동일한 U-Net을 사용하지만 픽셀 수는 1/16로 줄어듭니다. 비교 가능한 품질 대비 약 64배 적은 FLOPs가 소요됩니다.

이것이 Stable Diffusion의 레시피입니다. SD 1.x/2.x는 `64×64×4` 잠재 공간에 8억 6천만 개 파라미터 U-Net을 사용했고, SDXL은 `128×128×4`에 26억 개 파라미터 U-Net을 사용했습니다. SD3는 U-Net 대신 흐름 매칭을 적용한 디퓨전 트랜스포머(DiT)를 도입했습니다. Flux.1-dev(Black Forest Labs, 2024)는 120억 개 파라미터 DiT-MMDiT를 탑재했습니다. 모두 동일한 2단계 구조에서 실행됩니다.

## 개념

![Latent diffusion: VAE 압축 + 잠재 공간에서의 확산](../assets/latent-diffusion.svg)

**두 단계, 별도로 훈련됨.**

1. **1단계 — VAE.** 인코더 `E(x) → z`, 디코더 `D(z) → x`. 목표 압축: 각 공간 축에서 8× 다운샘플링 + 채널 조정으로 총 잠재 크기를 픽셀 수의 ~1/16로 설정. 손실 = 재구성(L1 + LPIPS 지각) + KL(`z`가 너무 가우시안으로 강제되지 않도록 작은 가중치 적용, `z`에서 정확한 샘플링이 필요하지 않기 때문). 종종 생성된 이미지가 선명하도록 적대적 손실로 훈련됨.

2. **2단계 — `z`에서의 확산.** `z = E(x_real)`을 데이터로 처리. U-Net(또는 DiT)을 훈련시켜 `z_t`의 노이즈를 제거. 추론 시: 확산을 통해 `z_0`를 샘플링한 후 `x = D(z_0)`로 이미지 생성.

**텍스트 조건화.** 두 가지 추가 구성 요소. 고정된 텍스트 인코더(SD 1.x의 경우 CLIP-L, SD 2/XL의 경우 CLIP-L+OpenCLIP-G, SD3 및 Flux의 경우 T5-XXL). 교차 어텐션 주입: 모든 U-Net 블록은 `[Q = 이미지 특징, K = V = 텍스트 토큰]`을 받아 혼합. 토큰은 텍스트가 이미지에 영향을 주는 유일한 경로.

**손실 함수는 Lesson 06과 동일.** 노이즈에 대한 DDPM/플로우 매칭 MSE. 데이터 도메인만 변경.

| SD3 | 2024 | MMDiT(다중모달 DiT) | 128×128×16 | T5-XXL + CLIP-L + CLIP-G | 2B / 8B |

## 구축 방법

`code/main.py`는 Lesson 06의 DDPM 위에 장난감 1-D "VAE"(데모용 항등 인코더 + 디코더; 실제 VAE는 컨브 넷임)를 쌓고 분류자-자유 가이던스를 활용한 클래스 조건화를 추가합니다. 이는 동일한 확산 손실이 원시 1-D 값 또는 인코딩된 값에서 실행될 때 작동함을 보여줍니다 — 핵심 통찰력입니다.

### 단계 1: 인코더/디코더

```python
def encode(x):    return x * 0.5          # 장난감 "압축"을 더 작은 스케일로
def decode(z):    return z * 2.0
```

실제 VAE는 학습된 가중치를 가집니다. 교육적 목적으로 이 선형 맵은 확산이 원본 데이터 공간을 신경 쓰지 않고 `z`에서 작동함을 보여주기 충분합니다.

### 단계 2: `z`-공간에서의 확산

Lesson 06과 동일한 DDPM입니다. 네트워크가 보는 데이터는 `z = E(x)`입니다. `z_0` 샘플링 후 `D(z_0)`로 디코딩합니다.

### 단계 3: 분류자-자유 가이던스

훈련 중 10% 확률로 클래스 레이블을 드롭(널 토큰으로 대체)합니다. 추론 시 `ε_cond`와 `ε_uncond`를 모두 계산한 후:

```python
eps_cfg = (1 + w) * eps_cond - w * eps_uncond
```

`w = 0` = 가이던스 없음(완전한 다양성), `w = 3` = 기본값, `w = 7+` = 포화/과도한 선명도.

### 단계 4: 텍스트 조건화(개념, 코드 아님)

클래스 레이블을 동결된 텍스트 인코더 출력으로 대체합니다. 텍스트 임베딩을 크로스-어텐션을 통해 U-Net에 전달합니다:

```python
h = h + CrossAttention(Q=h, K=text_embed, V=text_embed)
```

이것은 클래스-조건부 확산 모델과 Stable Diffusion 사이의 유일한 실질적 차이점입니다.

## 함정

- **VAE-스케일 불일치.** SD 1.x VAE는 인코딩 후 스케일링 상수(`scaling_factor ≈ 0.18215`)를 적용합니다. 이를 잊으면 U-Net이 분산이 완전히 잘못된 잠재 공간에서 훈련됩니다. 모든 체크포인트에는 이 값이 포함됩니다.
- **텍스트 인코더 오류.** SD3는 128개 이상의 토큰을 가진 T5-XXL이 필요하며, CLIP 전용으로의 대체는 정보 손실이 발생합니다. 항상 `use_t5=True`를 확인하거나 프롬프트 정확도가 급격히 떨어집니다.
- **잠재 공간 혼용.** SDXL, SD3, Flux는 모두 다른 VAE를 사용합니다. SDXL 잠재 공간에서 훈련된 LoRA는 SD3에서 작동하지 않습니다. Hugging Face diffusers 0.30+는 불일치하는 체크포인트 로드를 거부합니다.
- **CFG 값 과도.** `w > 10`은 포화되고 기름진 이미지를 생성하며 다양성 대신 프롬프트에 과적합됩니다. 적정 값은 `w = 3-7`입니다.
- **부정적 프롬프트 누출.** 빈 부정적 프롬프트는 널 토큰(null token)이 되고, 채워진 부정적 프롬프트는 `ε_uncond`가 됩니다. 이 둘은 동일하지 않으며, 일부 파이프라인은 널 토큰을 기본으로 사용합니다.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|---------------------|-----------|
| 첫 번째 단계 | "VAE" | 훈련된 인코더/디코더 쌍; 512²를 64²로 압축합니다. |
| 두 번째 단계 | "U-Net" | 잠재 공간(latent space) 상의 확산 모델(diffusion model). |
| CFG | "유도 척도(guidance scale)" | `(1+w)·ε_cond - w·ε_uncond`; 조건부(conditioning) 강도를 조정합니다. |
| 널 토큰 | "빈 프롬프트 임베딩" | `ε_uncond`에 사용되는 무조건부(unconditional) 임베딩. |
| 크로스 어텐션 | "텍스트가 입력되는 방식" | 각 U-Net 블록은 텍스트 토큰을 K(키)와 V(값)로 어텐션합니다. |
| DiT | "확산 트랜스포머(Diffusion Transformer)" | U-Net을 잠재 패치(latent patches) 상의 트랜스포머로 대체; 더 잘 확장됩니다. |
| MMDiT | "멀티모달 DiT" | SD3의 아키텍처: 텍스트 및 이미지 스트림과 공동 어텐션(joint attention). |
| VAE 스케일링 팩터 | "매직 넘버" | 잠재 변수를 ~5.4로 나누어 확산 모델이 단위 분산 공간에서 작동하도록 합니다. |

## 8GB 소비자 GPU에서 Flux-12B 실행 프로덕션 노트

참조 Flux 통합은 "소비자 GPU를 가지고 있는데 이걸로 배포할 수 있을까?"라는 질문에 대한 정설 레시피입니다. 이 트릭은 디퓨전 DiT에 적용된 프로덕션 추론 문헌에서 제시하는 세 가지 핵심 기법을 조합한 것입니다:

1. **단계적 로딩.** Flux에는 VRAM에 동시에 존재할 필요가 없는 세 가지 네트워크가 있습니다: T5-XXL 텍스트 인코더(~10GB, fp32 기준), CLIP-L(소형), 12B MMDiT, VAE. 프롬프트를 먼저 인코딩한 후 인코더를 *삭제*하고, DiT를 로드하여 디노이징한 다음 DiT를 *삭제*하고, VAE를 로드하여 디코딩합니다. 소비자용 8GB GPU는 한 번에 한 단계만 처리할 수 있습니다.
2. **bitsandbytes를 통한 4비트 양자화.** `BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_compute_dtype=torch.bfloat16)`를 T5 인코더와 DiT 모두에 적용합니다. 메모리를 8배 절약하며, Aritra의 벤치마크(노트북에 링크됨)에 따르면 텍스트-이미지 생성 시 품질 저하는 거의 없습니다.
3. **CPU 오프로딩.** `pipe.enable_model_cpu_offload()`는 각 순전파 단계가 진행될 때마다 모듈들을 CPU와 GPU 간에 자동으로 스왑합니다. 10-20% 지연 시간이 추가되지만 파이프라인이 실행되도록 합니다.

메모리 계산: `10GB T5 / 8 = 1.25GB` 양자화된 T5, `12B 파라미터 × 0.5바이트 = ~6GB` 양자화된 DiT, 그리고 활성화 메모리. stas00의 표현으로는 TP=1 추론의 극단적인 사례입니다 — 모델 병렬화 없이 최대 양자화를 적용한 것입니다. 프로덕션 환경에서는 H100에서 TP=2 또는 TP=4로 실행하지만, 단일 개발자 노트북에서는 이 레시피가 적합합니다.
