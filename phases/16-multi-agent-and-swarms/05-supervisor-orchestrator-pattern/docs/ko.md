# 감독자 / 오케스트레이터-워커 패턴

> 하나의 리드 에이전트가 계획하고 위임하며, 전문화된 워커들이 병렬 컨텍스트에서 실행하고 보고합니다. 이는 Anthropic의 연구 시스템(Claude Opus 4를 리드로, Sonnet 4를 서브 에이전트로 사용)의 기반이 되는 패턴이며, 내부 연구 평가에서 단일 에이전트 Opus 4 대비 +90.2% 성능 향상을 보였습니다. Anthropic의 엔지니어링 포스트에 따르면 BrowseComp에서 분산의 80%는 토큰 사용량만으로 설명되는데, 멀티 에이전트가 큰 승리를 거두는 이유는 각 서브 에이전트가 새로운 컨텍스트 윈도우를 얻기 때문입니다. 이 레슨은 감독자 패턴을 기본 요소부터 구축하고 2026년 프로덕션 배포에서의 엔지니어링 교훈을 다룹니다.

**유형:** 학습 + 구현  
**언어:** Python (표준 라이브러리, `threading`)  
**선수 지식:** 16단계 · 04 (기본 모델)  
**소요 시간:** ~75분

## 문제

연구는 단일 에이전트 시스템이 실패하는 대표적인 작업입니다. "2023년과 2026년 사이에 다중 에이전트 시스템에서 무엇이 변화했는가?"라는 질문을 던지면, 단일 에이전트는 5개의 논문을 순차적으로 읽고 컨텍스트의 절반을 해당 텍스트로 채운 다음, 모든 논문을 함께 추론해야 합니다. 그러나 다섯 번째 논문에 도달할 때쯤에는 첫 번째 논문을 잊어버립니다. 또한 병렬화가 불가능합니다.

감독자(supervisor) 패턴은 이 문제를 해결합니다: 하나의 리드 에이전트가 검색 계획을 수립하고, 각 하위 질문을 작업자(worker)에게 위임한 후 결과를 종합합니다. 각 작업자는 좁은 범위의 질문에 대해 200k 토큰의 전용 컨텍스트 창을 가집니다. 리드 에이전트는 원본 논문을 직접 보지 않으며, 작업자들의 요약만 확인합니다.

Anthropic의 프로덕션 연구 시스템은 내부 연구 평가에서 단일 Opus 4 대비 +90.2% 성능을 보고했습니다. 동일한 게시물은 BrowseComp 변동성의 80%가 *토큰 사용량만으로 설명된다*고 언급합니다. 하위 에이전트별 신선한 컨텍스트가 주요 메커니즘입니다.

## 개념

### 패턴

```
                 ┌──────────────┐
                 │   리드       │  계획, 분해,
                 │  (Opus 4)    │  종합
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 워커1    │  │ 워커2    │  │ 워커3    │
      │(소네트)  │  │(소네트)  │  │(소네트)  │
      └─────────┘  └─────────┘  └─────────┘
         신선한      신선한       신선한
         컨텍스트    컨텍스트     컨텍스트
```

리드는 원시 자료를 절대 읽지 않습니다. 워커들은 리드가 종합할 때까지 서로의 작업을 볼 수 없습니다. 각 화살표는 좁은 아티팩트를 전달하는 핸드오프입니다.

### 승리하는 이유

세 가지 메커니즘:

1. **하위 에이전트별 신선한 컨텍스트.** "FIPA-ACL 유산"을 탐색하는 워커는 리드가 계획에 소비한 40k 토큰을 전달받지 않습니다. 하나의 질문에 대해 200k 토큰 창을 제공받습니다.
2. **프롬프트 기반 전문화.** 리드의 프롬프트는 "분해 및 종합"이지 "연구"가 아닙니다. 각 워커의 프롬프트는 좁습니다: "X에서 변경된 사항을 찾아라." 집중된 프롬프트는 집중된 출력을 생성합니다.
3. **병렬성.** 워커들은 동시에 실행됩니다. 벽시계 시간은 대략 `max(워커_시간) + 계획 + 종합`이며, `sum(워커_시간)`이 아닙니다.

### 엔지니어링 교훈 (Anthropic 2025)

Anthropic 게시물은 2026년에도 여전히 유효한 여러 프로덕션 교훈을 나열합니다:

- **쿼리 복잡도에 맞춰 규모 조정.** 단순 쿼리: 1개 에이전트, 3-10개 도구 호출. 복잡한 쿼리: 10+ 에이전트. 리드가 이를 추정해야 하며, 호출자가 아닙니다.
- **넓게 시작한 후 좁히기.** 먼저 넓은 하위 질문으로 분해한 후, 답변이 깊이를 요구하면 하위 질문당 더 많은 워커를 생성합니다.
- **무지개 배포.** 에이전트는 장기간 실행되고 상태를 유지합니다. 전통적인 블루-그린 배포는 작동하지 않습니다. Anthropic은 무지개를 사용합니다: 새 버전을 점진적으로 롤아웃하면서 이전 버전이 자연스럽게 종료되도록 합니다.
- **토큰 사용량 지배.** 멀티 에이전트는 단일 에이전트 대비 ~15× 토큰을 사용합니다. 작업 가치가 비용을 정당화할 때만 실행해야 합니다.

### LangGraph 전환

LangGraph는 원래 `langgraph-supervisor` 라이브러리와 고수준 `create_supervisor` 헬퍼를 제공했습니다. 2025년 LangChain은 도구 호출을 통해 직접 감독자 패턴을 구현할 것을 권장하기 시작했습니다. 도구 호출은 *감독자가 보는 내용*(컨텍스트 엔지니어링)에 대한 더 많은 제어권을 제공하기 때문입니다. 라이브러리는 여전히 작동하며, 문서에서는 이제 도구 호출 형태를 권장합니다.

### 실패 모드

- **리드가 계획을 환각합니다.** 리드가 실제 질문을 분해하지 않는 하위 질문을 생성하면, 워커들은 잘못된 대상에 대해 정확한 연구를 수행합니다.
- **워커가 과도하게 탐색합니다.** 명시적인 범위 경계가 없으면 워커들은 할당된 하위 질문을 벗어나 종합 단계를 오염시킵니다.
- **종합 충돌.** 두 워커가 상충되는 사실을 반환합니다. 리드는 재질문(라운드 추가)하거나 불일치를 명시적으로 기록해야 합니다. 한쪽을 조용히 선택하는 것이 최악의 실패입니다: 사용자는 불일치가 발생했는지 알 수 없습니다.

### 감독자가 잘못된 경우

- **순차적 작업.** 2단계가 1단계의 출력을 문자 그대로 필요로 한다면, 병렬성은 아무런 이점이 없습니다. 파이프라인(CrewAI 순차적, LangGraph 선형 그래프)을 사용하세요.
- **단순 쿼리.** 단일 에이전트가 더 빠르고 저렴하게 처리합니다. 워커 생성 전 리드의 "규모 조정" 검사를 사용하세요.
- **엄격한 결정론.** 감독자는 LLM이 선택한 위임을 사용합니다. 감사/재생성이 적응성보다 중요할 때는 정적 그래프가 더 적합합니다.

## 빌드하기

`code/main.py`는 `threading`을 사용하여 세 개의 병렬 작업자(worker)를 감독하는 리드(lead)를 구현합니다. 리드는 쿼리를 하위 질문으로 분해하고, 작업자들은 각 하위 질문에 대해 동시에 실행되며, 리드는 결과를 종합합니다. 실제 LLM은 사용되지 않으며, 작업자들은 스크립트 기반으로 페치 및 요약 동작을 시뮬레이션합니다.

주요 구조:

- `Lead.plan(query)`는 쿼리를 3개의 하위 질문으로 분할합니다.
- `Worker.run(sub_q)`는 가짜 요약 결과를 반환합니다(실제 운영 환경에서는 도구 사용 에이전트로 대체 가능).
- `Lead.run(query)`는 스레드로 작업자들을 시작하고, 조인한 후 결과를 종합합니다.

실행 방법:

```
python3 code/main.py
```

출력은 계획(plan), 시작/종료 타임스탬프가 포함된 병렬 작업자 추적 기록, 그리고 최종 종합 결과를 보여줍니다. 벽시계 시간(wall-clock time) 측면에서 이점을 확인할 수 있습니다: 0.3초씩 소요되는 3개의 작업자가 0.9초가 아닌 약 0.35초 만에 완료됩니다.

## 사용 방법

`outputs/skill-supervisor-designer.md`는 사용자 질의를 입력받아 감독자 패턴 설계를 생성합니다: 리드 시스템 프롬프트, 작업자 역할, 하위 질문 분해 규칙, 통합 템플릿을 출력합니다. 새로운 연구 스타일 에이전트 시스템을 구축하기 전에 이 도구를 활용하세요.

## Ship It

감독자 패턴 배포 전 체크리스트:

- **모델 페어링.** 추론 계층 모델(Opus 클래스, `o3` 클래스)을 리드로 사용. 더 빠르고 저렴한 모델(Sonnet, `o4-mini`)을 워커로 사용.
- **워커 타임아웃.** 중앙값 실행 시간의 2배를 초과하는 워커는 종료; 리드는 범위를 좁혀 재시작하거나 해당 워커 없이 진행.
- **워커당 토큰 한도.** 하드 제한(예상 합성 입력의 10배)을 설정하여 예산 초과 워커 방지.
- **관측 가능성.** 리드의 계획, 각 워커의 도구 호출, 합성 과정을 추적. 이는 사후 디버깅의 기초가 됨.
- **레인보우 롤아웃.** 상태 유지가 필요한 장기 실행 에이전트는 점진적 버전 전환이 필요하며, 핫 스왑은 금지.

## 연습 문제

1. `code/main.py`를 실행한 후, 리드가 3개 대신 5개의 워커를 생성하도록 수정하세요. 벽시계 시간 효과를 관찰하세요. 이 데모에서 워커 수가 얼마일 때 생성 오버헤드가 병렬 처리 이점을 초과하나요?

2. 워커 타임아웃을 구현하세요: 0.5초 이상 실행되는 워커는 종료하고 리드가 나머지 결과를 합성하도록 하세요. 워커가 중단되었음을 알기 위해 필요한 관측 가능성은 무엇인가요?

3. 리드의 합성 단계에 충돌 감지 기능을 추가하세요: 두 워커가 상반된 답변을 반환하면 리드는 하나를 선택하는 대신 불일치를 기록합니다. LLM을 호출하지 않고 충돌을 어떻게 감지할 수 있나요?

4. Anthropic의 Research-system 엔지니어링 포스트를 읽으세요. 이 간단한 데모가 프로덕션 환경에서 실행되려면 채택해야 할 세 가지 실천 방법을 나열하세요.

5. LangGraph의 `create_supervisor` (레거시)와 새로운 도구 호출 권장 사항을 비교하세요. 어떤 것이 감독자가 볼 수 있는 내용을 더 잘 제어할 수 있나요? Anthropic이 왜 원시 워커 컨텍스트가 아닌 하위 답변만 명시적으로 합성에 전달하나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 감독자(Supervisor) | "리드 에이전트(Lead agent)" | 작업 계획, 위임, 종합을 담당하는 오케스트레이터 에이전트. 직접 작업을 수행하지는 않음. |
| 작업자(Worker) | "서브에이전트(Subagent)" | 감독자가 호출하는 좁은 범위와 자체 컨텍스트 윈도우를 가진 집중형 에이전트. |
| 오케스트레이터-작업자(Orchestrator-worker) | "감독자 패턴(Supervisor pattern)" | 같은 개념, 다른 이름. 2026년 문헌에서는 둘 다 사용됨. |
| 신선한 컨텍스트(Fresh context) | "클린 윈도우(Clean window)" | 작업자의 컨텍스트는 리드 에이전트(lead agent)의 히스토리가 아닌 시스템 프롬프트와 할당된 질문부터 시작함. |
| 레인보우 배포(Rainbow deployment) | "점진적 롤아웃(Gradual rollout)" | 장기간 실행되는 상태 기반 에이전트는 블루-그린(blue-green)이 아닌 버전 기반 드레인-앤-리플레이스(drain-and-replace)가 필요함. |
| 토큰 지배력(Token dominance) | "컨텍스트가 변수다(Context is the variable)" | Anthropic에 따르면 연구-평가 변동성의 80%는 모델 선택이 아닌 사용된 총 토큰 수에서 비롯됨. |
| 확장 노력(Scale effort) | "복잡도에 맞춰 에이전트 수 조정" | 리드 에이전트가 쿼리 난이도를 추정하고 1개 또는 10개 이상의 작업자를 생성함. |
| 종합 충돌(Synthesis conflict) | "작업자 간 불일치" | 두 작업자가 상충되는 사실을 반환할 경우, 리드 에이전트는 불일치를 표면화해야 하며 조용히 하나를 선택해서는 안 됨. |
