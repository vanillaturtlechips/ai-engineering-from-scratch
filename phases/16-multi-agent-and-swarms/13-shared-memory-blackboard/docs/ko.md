# 공유 메모리 및 블랙보드 패턴

> 2026년 다중 에이전트 시스템에는 두 가지 접근 방식이 공존합니다: **메시지 풀**(AutoGen GroupChat 또는 MetaGPT처럼 모든 에이전트가 서로의 메시지를 보는 방식)과 **구독 기반 블랙보드**(Context-Aware MCP 또는 Matrix 프레임워크처럼 에이전트가 관련 이벤트에 구독하는 방식). 둘 다 다중 에이전트 시스템의 유일한 상태 저장 부분 — 즉 흥미로운 버그가 발생하는 곳입니다. 대표적인 실패 모드는 **메모리 오염**입니다: 한 에이전트가 "사실"을 환각(hallucinate)하면 다른 에이전트들이 이를 검증된 것으로 처리하고, 정확도가 점진적으로 저하되어 즉각적인 충돌보다 디버깅이 훨씬 어려워집니다. 이 레슨에서는 stdlib로 두 구조를 구축하고, 오염 공격을 주입한 후 실제 프로덕션에서 효과가 있는 세 가지 완화 기법을 보여줍니다.

**유형:** 학습 + 실습  
**언어:** Python (stdlib, `threading`)  
**선수 지식:** 16단계 · 04 (기본 모델), 16단계 · 09 (병렬 스웜 네트워크)  
**소요 시간:** ~75분

## 문제

다중 에이전트 시스템은 에이전트들이 사실을 공유할 장소가 필요합니다. 문자 그대로의 옵션은 "메시지에 모든 것을 전달"하는 것이지만, 이는 불필요한 복사와 함께 공유 상태를 재창조하는 것입니다. 또 다른 옵션은 "모든 에이전트에게 글로벌 로그를 제공"하는 것이지만, 글로벌 로그는 무한히 성장하며 쉽게 오염됩니다. 세 번째 옵션은 "에이전트별 뷰를 투영"하는 것인데, 확장성이 있지만 스키마 중심적입니다.

에이전트 중 하나가 환각(hallucination)을 일으키고 이를 공유 상태에 기록하면, 해당 상태를 읽는 모든 하위 에이전트들은 환각을 사실로 받아들입니다. 인간이 이를 알아차릴 때쯤이면 추론 체인은 5단계 깊이까지 진행되고 근본 원인은 처음에 기록된 세 번째 메시지가 됩니다. 다중 에이전트 정확도 감소를 디버깅하는 것은 충돌 디버깅보다 더 어렵습니다.

이것은 메모리 오염(memory poisoning)입니다. 이는 MAST 분류(Cemri et al., arXiv:2503.13657)에서 두 번째로 가장 많이 기록된 실패 유형이며, 구조적인 문제입니다: 출처(provenance) 추적과 수정 불가능한 검증자(verifier) 없이 공유 메모리 설계를 사용하는 경우 결국 이 문제가 발생합니다.

## 개념

### 두 가지 주요 토폴로지

**전체 메시지 풀.** 모든 에이전트가 모든 메시지를 읽습니다. AutoGen GroupChat과 MetaGPT가 이 방식을 사용합니다. 단순하고 투명하며 검사 가능하지만, 각 에이전트의 컨텍스트가 다른 에이전트의 작업으로 채워지기 때문에 ~10개 이상의 에이전트로는 확장되지 않습니다.

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**구독 기능이 있는 블랙보드.** 에이전트가 관심 있는 주제를 선언하면, 기반 시스템이 관련 메시지만 라우팅합니다. CA-MCP(arXiv:2601.11595)와 Matrix 분산 프레임워크(arXiv:2511.21686)가 이 방식을 사용합니다. 더 확장 가능하지만, 구독을 의미 있게 만들기 위해 사전 스키마 설계가 필요합니다.

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 각 방식이 유리한 경우

- **전체 풀**은 에이전트 수가 적을 때(< 10), 이질적일 때, 그리고 대화가 단기적일 때 유리합니다. 모든 것을 볼 때 누가 무엇을 말했는지 추론하는 것이 간단합니다.
- **블랙보드**는 에이전트 수가 많을 때, 역할은 동질적이지만 인스턴스 수가 많을 때(군집), 그리고 대화가 장기적일 때 유리합니다. 라우팅은 토큰 비용과 컨텍스트 오염을 줄입니다.

프로덕션 시스템은 종종 혼합합니다: 상위 계층(계획 계층)에는 소규모 전체 풀, 하위 계층(작업자 계층)에는 블랙보드를 사용합니다.

### 메모리 오염, 한 시나리오

세 에이전트가 연구 작업을 수행합니다. 에이전트 A는 검색 에이전트입니다. 에이전트 B는 요약기입니다. 에이전트 C는 분석가입니다.

1. A는 페이지를 가져와 공유 상태에 메시지를 작성합니다: "연구는 42% 정확도 향상을 보고합니다."
2. 가져온 페이지는 실제로 "4.2% 향상"이라고 했습니다. A는 소수점을 환각했습니다.
3. B는 공유 상태를 읽고 작성합니다: "큰 42% 정확도 향상 보고됨(출처: A)."
4. C는 공유 상태를 읽고 작성합니다: "채택 권장 — 42% 상승은 혁신적입니다."
5. 최종 보고서는 존재하지 않는 42% 수치를 인용합니다.

에이전트가 충돌하지 않았습니다. 테스트도 실패하지 않았습니다. 시스템은 "작동"했습니다. 환각은 공유 상태를 통해 한 에이전트의 컨텍스트에서 모든 다운스트림 에이전트의 추론으로 전달되었습니다.

### 이것이 구조적인 이유

공유 상태가 없으면 에이전트 A의 환각은 A의 컨텍스트에 머뭅니다. 다운스트림 에이전트는 다시 가져오거나 재유도하여 오류를 발견할 수 있습니다. 순진한 공유 상태에서는 A의 컨텍스트가 모든 사람의 컨텍스트가 되고, 환각은 사실로 세탁됩니다.

문제는 공유 상태 자체가 아니라, **출처(provenance) 없이 독립적인 검증자 없이** 공유 상태가 사용되는 것입니다. 세 가지 완화 방법이 있습니다:

1. **모든 쓰기 시 출처(provenance)를 기록합니다.** 공유 상태의 모든 항목은 누가, 언제, 어떤 프롬프트로 작성했는지, 그리고 (해당되는 경우) 에이전트가 인용한 소스를 기록합니다. 다운스트림 에이전트는 출처에 따라 회의적으로 읽습니다.
2. **버전을 관리하고, 추가 전용(append-only)으로 처리합니다.** 수정은 이전 항목을 대체하는 새 항목이지, 인플레이스 업데이트가 아닙니다. 감사 추적이 보존됩니다.
3. **공유 상태에 쓸 수 없는 에이전트 최소 한 대를 유지합니다.** 읽기 전용 검증자 에이전트는 항목을 샘플링하고, 소스를 다시 가져오며, 불일치를 표시합니다. 풀에 쓸 수 없기 때문에 풀에 의해 오염될 수 없습니다.

### 블랙보드 선례(Hayes-Roth, 1985)

블랙보드 패턴은 LLM 에이전트보다 40년 앞서 등장했습니다. Hayes-Roth(1985, "A Blackboard Architecture for Control")는 글로벌 블랙보드를 관찰하고, 부분 솔루션을 기여하며, 다른 소스를 트리거하는 전문 지식 소스(Knowledge Sources)를 설명했습니다. 2026년 블랙보드(CA-MCP, Matrix)는 LLM 에이전트를 지식 소스로, JSON 블롭을 부분 솔루션으로 하는 동일한 패턴입니다. 오래된 문헌에는 현대 시스템이 재발견하는 쓰기 경쟁, 기회적 제어, 일관성에 대한 해결책이 문서화되어 있습니다.

### 프로젝션 vs 전체 뷰

순수한 블랙보드는 모든 구독자에게 동일한 프로젝션(주제 범위)을 제공합니다. 더 공격적인 설계는 **에이전트별 프로젝션**입니다: 각 에이전트는 역할에 맞게 커스터마이즈된 뷰를 얻습니다. LangGraph의 상태 리듀서는 2026년의 표준 구현입니다 — 리듀서 함수는 글로벌 상태를 역할별 슬라이스로 접습니다.

에이전트별 프로젝션은 더 확장 가능하지만 스키마가 필요합니다. 스키마 없이는 모든 에이전트의 프롬프트에서 임시 프로젝션을 재구성해야 합니다.

### 쓰기 경쟁 패턴

여러 에이전트가 동시에 쓰는 것은 LLM 문제뿐 아니라 동시성 문제입니다. 세 가지 패턴이 작동합니다:

- **순차적 작성자(단일 생산자).** 모든 쓰기는 직렬화하는 하나의 코디네이터 에이전트를 통과합니다. 간단하지만 병목 현상이 발생합니다.
- **버전 관리를 통한 낙관적 동시성.** 각 항목에는 버전이 있으며, 작성자는 버전 불일치 시 실패하고 재시도합니다. 고전적인 데이터베이스 기법입니다.
- **주제 파티셔닝.** 다른 에이전트가 다른 주제를 소유합니다. 주제 간 경쟁이 없습니다. 설계된 파티션 경계가 필요합니다.

대부분의 2026년 프레임워크는 LLM 호출이 충분히 느려 경쟁이 드물고 병목이 발생하지 않기 때문에 순차적 작성자를 기본값으로 합니다.

### 쓸 수 없는 검증자

가장 중요한 완화 방법은 읽기 전용 검증자입니다. 구현 규칙:

- 검증자는 팀과 상태를 공유합니다(블랙보드 또는 풀을 읽습니다).
- 검증자는 공유 상태에 대한 쓰기 핸들이 없으며, 별도의 검증 채널만 가집니다.
- 검증자는 쓰기에서 인용된 소스를 독립적으로 가져옵니다. 불일치를 표시합니다.
- 검증자의 출력은 인간이나 별도의 결정 에이전트로 라우팅되며, 풀에 다시 피드백되지 않습니다.

이 분리가 없으면 검증자의 출력은 풀의 새 항목이 되어, 오염된 풀이 검증자를 오염시키고, 검증이 오염됩니다.

## 빌드하기

`code/main.py`는 표준 라이브러리 Python으로 구현된 두 가지 토폴로지와 장난감 독극물 공격(toy poisoning attack) 및 세 가지 완화 기법(mitigations)을 포함합니다.

- `MessagePool` — 스레드-안전(thread-safe) 추가 전용 로그(append-only log)로 전체 읽기 가능.
- `Blackboard` — 주제-키 기반(topic-keyed) 발행/구독(pub/sub) 시스템으로 에이전트별 구독 기능 제공.
- `ProvenanceEntry` — 모든 쓰기 작업은 (작성자, 타임스탬프, 프롬프트 해시, 소스 URI)를 기록.
- `PoisoningScenario` — 에이전트 A가 소수점을 허구로 생성하는 3개 에이전트 연구 작업을 실행. 최종 보고서 출력.
- `Verifier` — 소스를 다시 가져와 불일치(inconsistencies)를 표시하는 읽기 전용 에이전트. 검증자(verifier)가 있는 동일한 시나리오 실행.

실행:

```
python3 code/main.py
```

예상 출력:
- 실행 1 (검증자 없음): 허구로 생성된 42%가 최종 보고서에 전파됨.
- 실행 2 (검증자 있음): 검증자가 불일치를 표시하고, 풀(pool)이 "flagged"로 표시되며, 최종 보고서에 철회(retraction) 포함.

## 사용 방법

`outputs/skill-memory-auditor.md`는 다중 에이전트 시스템의 공유 메모리 설계를 출처(provenance), 버전 관리(versioning), 검증자 분리(verifier separation) 측면에서 감사하는 스킬입니다. 프로덕션 환경 적용 전 새로운 다중 에이전트 아키텍처에 대해 실행해 보세요.

## Ship It

모든 공유 메모리 설계에 대해:

- 모든 쓰기 작업 시 출처(provenance)를 기록: `(작성자(writer), 타임스탬프(timestamp), 프롬프트 해시(prompt_hash), 인용된 도구 호출(tool_calls_cited), 소스 URI(source_uri))`.
- 로그를 추가 전용(append-only)으로 구성. 수정 사항은 기존 항목을 참조하는 새 항목으로 추가.
- 독립적인 소스 접근 권한을 가진 최소 1개의 읽기 전용 검증 에이전트(verifier agent)를 배치.
- 검증 에이전트 출력을 공유 풀이 아닌 별도의 채널로 라우팅.
- 대체(supersession)되는 쓰기 작업 비율을 로깅 — 이 비율 상승은 환각(hallucination) 패턴의 초기 증거.

| 블랙보드 | "공유 작업 공간" | 주제 기반 발행/구독. 에이전트는 관련 주제에 구독. 더 나은 확장성. |
| 출처(Provenance) | "누가 무엇을 작성했는가" | 각 작성 작업에 대한 메타데이터: 작성자, 타임스탬프, 프롬프트, 출처. |
| 메모리 오염(Memory poisoning) | "환각(hallucination) 확산" | 한 에이전트의 오류가 공유 상태에 들어가, 후속 에이전트들이 이를 사실로 받아들임. |
| 추가 전용(Append-only) | "인플레이스 업데이트 불가" | 수정 사항은 기존 항목을 대체하는 새 항목. 감사 추적 보존. |
| 읽기 전용 검증자(Unwritable verifier) | "독립 감사자" | 소스를 재조회하고 불일치를 표시하는 읽기 전용 에이전트. |
| 투영(Projection) | "범위 지정 뷰" | 전역 상태에서 계산된 에이전트별 뷰. LangGraph 리듀서가 대표적인 사례. |
| 지식 소스(Knowledge Source) | "전문가 에이전트" | 1985년 헤이즈-로스(Hayes-Roth)가 제안한 블랙보드 참여자.
