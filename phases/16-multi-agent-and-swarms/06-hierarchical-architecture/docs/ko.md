# 계층적 아키텍처와 그 실패 모드

> 계층적 구조는 감독자가 중첩된 형태입니다. 관리자 에이전트가 하위 관리자 위에, 하위 관리자가 작업자 위에 위치합니다. CrewAI의 `Process.hierarchical`는 교과서적인 버전입니다: `manager_llm`이 동적으로 작업을 위임하고 출력을 검증합니다. LangGraph의 동등한 구현은 `create_supervisor(create_supervisor(...))`입니다. 이는 실제 조직도와 같은 구조가 필요한 작업일 때 자연스러운 패턴입니다. 또한 이 패턴은 관리적 루프(managerial looping)로 붕괴될 가능성이 가장 높습니다 — 관리자 에이전트가 작업을 잘못 할당하거나, 하위 출력을 오해하거나, 합의에 도달하지 못하는 경우입니다. 순차적 접근이 종종 더 효과적입니다.

**유형:** 학습 + 실습  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 16단계 · 05 (감독자 패턴)  
**소요 시간:** ~60분

## 문제

감독자 패턴이 이해되면 자연스럽게 다음 질문은 "만약 작업자들도 감독자라면 어떻게 될까?"입니다. 팀에는 하위 팀이 있고, 회사에는 부서 내 부서가 있습니다. 계층적 아키텍처는 이를 반영합니다.

문제점: LLM 관리자는 인간 관리자와 동일하지 않습니다. 인간 관리자는 부하 직원이 알고 있는 것에 대한 안정적인 사전 지식을 가지고 있습니다. 반면 LLM 관리자는 매 턴마다 컨텍스트에 있는 내용을 기반으로 조직 구조를 재추론합니다. 컨텍스트에 작은 변화가 발생하면 전체 트리가 작업을 잘못 할당하게 됩니다.

## 개념

### 구조

```
                 매니저
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       서브-매니저 A         서브-매니저 B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

모든 내부 노드는 계획, 위임, 종합을 수행합니다. 리프 노드만 실제 작업을 수행합니다.

### 강점이 드러나는 경우

- **명확한 조직 매핑.** 실제 작업이 부서별("법무 검토, 재무 검토, 엔지니어링 검토 후 경영진 요약")로 이루어질 때 계층 구조가 명시적으로 표현됩니다.
- **지역적 요약.** 각 서브-매니저는 팀의 출력을 종합한 후 최상위 매니저에게 전달합니다. 최상위 매니저는 15개의 작업자 출력이 아닌 3개의 서브-매니저 요약본을 확인합니다.

### 실패하는 경우

2026년 사후 분석에서 지속적으로 발견되는 3가지 실패 모드:

1. **작업 할당 오류.** 매니저가 목표를 읽고 잘못된 분해를 상상하여 잘못된 서브-매니저에게 위임합니다. 서브-매니저는 주어진 작업을 충실히 수행하므로 오류는 최상위 종합 단계에서만 발견되며, 이는 인간이 발견할 수 있었던 지점에서 한 단계 떨어진 곳입니다.
2. **출력 오해.** 서브-매니저가 "주장 X 확인 불가"를 반환합니다. 최상위 매니저는 이를 "주장 X 확인되지 않음"으로 요약합니다. 의미가 각 레벨에서 점점 변합니다.
3. **합의 루프.** 두 서브-매니저가 의견을 달리하면 최상위 매니저가 조정을 요청합니다. 이들은 다시 하위 작업자에게 위임하고, 작업자는 재실행하며, 서브-매니저는 약간 다른 답변을 반환합니다. 이 과정이 반복됩니다. CrewAI의 `Process.hierarchical`는 단계 제한으로 이를 방지하지만, 제한 자체가 하이퍼파라미터가 됩니다.

### 결정적 질문

순차적(선형 파이프라인) vs 계층적: 작업에 실제로 독립적인 하위 팀이 존재하나요, 아니면 트리인 척하는 하나의 선형 흐름인가요? 후자라면 순차적을, 전자라면 계층적을 사용하되 명시적인 조정 규칙을 예산에 포함하세요.

### CrewAI의 구현

`Process.hierarchical`는 전문가 크루 위에 매니저 LLM을 배치합니다. 매니저는 다음을 수행합니다:

- 최상위 작업 수신,
- 크루에 하위 작업 할당,
- 크루 출력 평가,
- 수락, 재위임, 반복 결정.

문서: https://docs.crewai.com/en/introduction (핵심 개념 섹션에서 "계층적 프로세스" 참조).

### LangGraph의 구현

LangGraph는 중첩된 `create_supervisor` 호출을 사용합니다. 내부 감독자는 자체 그래프를 가지며, 외부 감독자는 내부 그래프를 불투명 노드로 취급합니다. 이는 디버깅 측면에서 CrewAI보다 더 깔끔하지만(그래프를 개별적으로 단계별로 확인 가능), 트리의 동적 재구성을 표현하기는 더 어렵습니다.

참조: https://reference.langchain.com/python/langgraph-supervisor.

## 빌드하기

`code/main.py`는 3단계 계층 구조를 실행합니다:

- 최상위 관리자: 작업을 "엔지니어링"과 "법률" 브랜치로 분할,
- 엔지니어링 하위 관리자: "프론트엔드"와 "백엔드" 작업자로 분할,
- 법률 하위 관리자: 단일 작업자.

데모는 모든 참여자가 동의하는 **정상 경로**와 최상위 관리자의 분할 과정에서 "법률"이 "재무"로 잘못 레이블링되는 **변형 경로**를 대조합니다. 이 오류는 하위 관리자가 재무 작업을 수행하고, 최상위 합성기가 재무 결과를 보고하며, 원래의 법률 질문은 무시되는 연쇄적 오류를 유발합니다.

실행 방법:

```
python3 code/main.py
```

출력은 "요청된 내용" 대 "전달된 내용"을 명확하게 나란히 비교한 두 경로를 보여줍니다.

## 사용 방법

`outputs/skill-hierarchy-fitness.md`는 주어진 작업에 계층적(hierarchical), 순차적(sequential), 또는 평탄한(flat) 감독자(supervisor)를 사용해야 하는지 평가합니다. 입력값: 작업 설명, 조직 구조, 조정 예산. 출력값: 특정 실패 모드(failure modes)를 방지할 수 있는 패턴 추천.

## Ship It

계층적 구조를 배포할 경우:

- **트리 깊이를 2로 제한.** 3단계만으로도 관측 가능성에서 대부분의 오류를 숨길 수 있습니다.
- **명시적 조정 예산 설정.** 최상위 관리자가 커밋해야 하는 최대 라운드 수를 지정(일반적으로 2).
- **모든 합성에 출처 기록.** 각 노드의 요약 정보는 어떤 리프 출력이 이를 생성했는지 반드시 인용해야 합니다.
- **분해 드리프트에 대한 경고.** 단계별 관리자의 분해 내용을 기록하고 사용자 쿼리와 비교합니다. 분해가 더 이상 쿼리를 커버하지 않으면 경고를 발생시킵니다.

## 연습 문제

1. `code/main.py`를 실행하고 happy vs perturbed를 비교하세요. 최상위 출력이 사용자의 질문과 완전히 달라질 때까지 몇 단계의 관리자 핸드오프가 발생하나요?
2. 세 번째 레벨(최상위 → 하위 → 하위-하위 → 작업자)을 추가하세요. 깊이가 증가함에 따라 perturbed 경로가 스스로 수정하는 빈도와 완전히 달라지는 빈도를 측정하세요.
3. 각 하위 관리자 아래에 항상 원본 사용자 질문을 변경 없이 받는 "카나리(canary)" 작업자를 구현하세요. 카나리 답변을 사용해 분해 드리프트(decomposition drift)를 감지하세요. 카나리가 합성된 답변과 불일치할 때 관리자는 어떻게 반응해야 할까요?
4. CrewAI의 `Process.hierarchical` 문서를 읽으세요. CrewAI가 적용하는 구체적인 가드레일(예: 단계 제한, manager_llm 제약) 하나를 식별하고, 해당 가드레일이 어떤 실패 모드를 방지하는지 설명하세요.
5. 중첩된 LangGraph 감독자와 CrewAI 계층 구조를 비교하세요. 어떤 방식이 조정 루프(reconciliation loop) 감지를 더 저렴하게 만드나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 계층적(Hierarchical) | "조직도 패턴" | 관리자 위의 관리자; 실제 작업은 말단만 수행. |
| 관리자 LLM(Manager LLM) | "보스" | 내부 노드에서 분해, 할당, 검증을 담당하는 LLM. |
| 분해 드리프트(Decomposition drift) | "보스가 길을 잃음" | 최상위 관리자의 분할이 더 이상 원본 질문을 커버하지 못함. |
| 조정 루프(Reconciliation loop) | "끝없는 회의" | 하위 관리자들이 의견 불일치; 최상위 관리자가 재할당; 작업자 재실행; 예산 소진까지 반복. |
| 깊이-2 한계(Depth-2 ceiling) | "2단계 이상 가지 않기" | 경험적 안전장치: 3단계 이상은 관측 가능성 붕괴. |
| 캐니리 질문(Canary question) | "모든 단계의 정답" | 원본 질문을 변경 없이 항상 수행하는 작업자; 드리프트 감지용. |
| 출처 체인(Provenance chain) | "누가 무엇을 말했는지" | 각 합성 결과를 생성한 말단 출력까지 역추적 가능한 기록. |
