# 다중 에이전트 프리미티브 모델

> 2026년에 출시되는 모든 다중 에이전트 프레임워크 — AutoGen, LangGraph, CrewAI, OpenAI Agents SDK, Microsoft Agent Framework — 는 4차원 설계 공간의 한 지점입니다. 네 가지 프리미티브, 그 이상도 이하도 없습니다: 에이전트, 핸드오프, 공유 상태, 오케스트레이터. 이 레슨은 이 네 가지를 처음부터 구축하고, 모든 프리미티브에 대해 장난감 시스템을 실행한 다음, 모든 주요 프레임워크를 동일한 축에 매핑하여 새로운 릴리스를 한 단락으로 읽을 수 있도록 합니다.

**유형:** 학습
**언어:** Python (표준 라이브러리)
**선수 지식:** 14단계 (에이전트 엔지니어링), 16단계·01 (다중 에이전트의 이유)
**소요 시간:** ~60분

## 문제

6개월마다 새로운 멀티에이전트 프레임워크가 출시됩니다. 2023년에는 AutoGen, 2024년에는 CrewAI, LangGraph, OpenAI Swarm이 등장했습니다. 2025년 4월에는 Google ADK, 2026년 2월에는 Microsoft Agent Framework RC가 출시되었습니다. 각 발표 자료에서는 모두 "올바른 추상화"라고 주장합니다.

하나씩 배우려고 하면 지칠 것입니다. API는 서로 다르게 생겼습니다. 문서들은 "에이전트"가 무엇인지에 대해 의견이 분분합니다. 한 프레임워크는 공유 메모리를 "블랙보드"라고 부르고, 다른 프레임워크는 "메시지 풀", 세 번째 프레임워크는 "StateGraph"라고 부릅니다. 이 분야가 단순히 혼란만 가중시키고 있다는 의심이 들기 시작합니다.

하지만 그렇지 않습니다. 마케팅을 걷어내면 네 가지 기본 요소는 안정적입니다. 이를 한 번만 배우면 새로운 프레임워크를 단 한 문단으로 이해할 수 있습니다.

## 개념

### 네 가지 기본 요소

1. **에이전트(Agent)** — 시스템 프롬프트와 도구 목록으로 구성된 시스템. 상태 없음; 모든 실행은 시스템 프롬프트와 현재 메시지 기록에서 시작.
2. **핸도프(Handoff)** — 한 에이전트에서 다른 에이전트로의 구조화된 제어 전달. 기계적으로, 새 에이전트를 반환하는 도구 호출 또는 조건을 따르는 그래프 엣지.
3. **공유 상태(Shared state)** — 하나 이상의 에이전트가 읽을 수(때로는 쓸 수) 있는 모든 데이터 구조. 메시지 풀, 블랙보드, 키-값 저장소, 벡터 메모리.
4. **오케스트레이터(Orchestrator)** — 다음에 말할 주체를 결정하는 존재. 옵션: 명시적 그래프(결정적), LLM 스피커 선택기(소프트), 마지막 스피커의 핸도프 호출(OpenAI Swarm), 큐 기반 스케줄러(스웜 아키텍처).

이것이 전체 설계 공간입니다. 모든 프레임워크는 각 축에 대한 기본값을 선택하며, 나머지는 표면적인 구문입니다.

### 모든 2026 프레임워크의 매핑

| 프레임워크 | 에이전트 | 핸도프 | 공유 상태 | 오케스트레이터 |
|-----------|-------|---------|--------------|--------------|
| OpenAI Swarm / Agents SDK | `Agent(instructions, tools)` | 도구 반환 시 Agent | 호출자의 문제 | LLM의 다음 핸도프 호출 |
| AutoGen v0.4 / AG2 | `ConversableAgent` | GroupChat의 스피커 선택기 | 메시지 풀 | 선택기 함수(LLM 또는 라운드 로빈) |
| CrewAI | `Agent(role, goal, backstory)` | `Process.Sequential / Hierarchical` | 체이닝된 작업 출력 | 관리자 LLM 또는 정적 순서 |
| LangGraph | 노드 함수 | 그래프 엣지 + 조건 | `StateGraph` 리듀서 | 그래프, 결정적 |
| Microsoft Agent Framework | 에이전트 + 오케스트레이션 패턴 | 패턴별 | 스레드/컨텍스트 | 패턴별 |
| Google ADK | 에이전트 + A2A 카드 | A2A 작업 | A2A 아티팩트 | 호스트 결정 |

표면적인 차이는 매우 커 보입니다. 그러나 그 아래에는 동일한 네 가지 조절기가 있습니다.

### 이것이 중요한 이유

기본 요소를 이해하면 프레임워크 비교가 간단한 체크리스트가 됩니다:

- 오케스트레이터가 LLM에 라우팅을 위임하는가(Swarm) 아니면 코드로 라우팅을 고정하는가(LangGraph)?
- 공유 상태가 전체 기록(GroupChat)인가 아니면 투영된 상태(StateGraph 리듀서)인가?
- 에이전트가 서로의 프롬프트를 수정할 수 있는가(CrewAI 관리자) 아니면 핸도프만 가능한가(Swarm)?

이 세 가지 질문이 주어진 문제에 맞는 프레임워크를 80% 결정합니다. "최고의 멀티에이전트 프레임워크"를 찾는 대신 실제로 중요한 축을 기준으로 설계하게 됩니다.

### 상태 없음의 통찰

공유 상태를 제외한 모든 기본 요소는 상태 없음입니다. 에이전트는 (프롬프트, 도구)의 함수입니다. 핸도프는 함수 호출입니다. 오케스트레이터는 스케줄러입니다. **시스템에서 유일한 상태 있는 요소는 공유 상태입니다.** 여기에 모든 흥미로운 버그(메모리 오염(레슨 15), 메시지 순서, 버전 관리, 쓰기 경합)가 숨어 있습니다.

공유 상태를 숨기는 프레임워크(Swarm)는 문제를 호출자에게 넘깁니다. 공유 상태를 중앙 집중화하는 프레임워크(LangGraph 체크포인트, AutoGen 풀)는 검사를 가능하게 하지만 조정 비용을 공유 상태 구현으로 전환합니다.

### 단일 기본 요소의 해부학


#### 에이전트

```
Agent = (system_prompt, tools, model, optional_name)
```

메모리 없음. 상태 없음. 동일한 시스템 프롬프트와 도구를 가진 두 에이전트는 상호 교환 가능합니다. 에이전트별 상태처럼 보이는 모든 것은 실제로 공유 상태 또는 핸도프 프로토콜에 있습니다.


#### 핸도프

```
Handoff = (from_agent, to_agent, reason, payload)
```

세 가지 구현이 주류입니다:

- **함수 반환** — 도구가 다음 에이전트를 반환합니다. OpenAI Swarm 패턴입니다. 에이전트는 도구 스키마에 라우팅 정보를 포함합니다.
- **그래프 엣지** — LangGraph. 엣지는 선언적입니다. LLM이 값을 생성하면 조건이 다음 노드를 선택합니다.
- **스피커 선택** — AutoGen GroupChat. 선택기 함수(때로는 LLM 호출)가 풀을 읽고 다음 스피커를 선택합니다.


#### 공유 상태

```
SharedState = { messages: [], artifacts: {}, context: {} }
```

최소한 메시지 목록. 종종 더 많은 것: 구조화된 아티팩트(CrewAI 작업 출력), 타입화된 컨텍스트(LangGraph 리듀서), 외부 메모리(MCP, 벡터 DB).

두 가지 토폴로지: **풀 풀**(모든 에이전트가 모든 메시지를 봄)과 **투영된 풀**(에이전트가 역할 범위 뷰를 봄). 풀 풀은 간단하지만 확장성이 떨어집니다. 투영된 풀은 확장 가능하지만 사전 스키마 설계가 필요합니다.


#### 오케스트레이터

```
Orchestrator = ({state, last_speaker}) -> next_agent
```

네 가지 유형:

- **정적** — 그래프는 빌드 시 고정됨(LangGraph 결정적, CrewAI Sequential).
- **LLM 선택** — LLM이 풀을 읽고 다음 스피커를 선택(AutoGen, CrewAI Hierarchical).
- **핸도프 주도** — 현재 에이전트가 핸도프 도구를 호출하여 결정(Swarm).
- **큐 주도** — 작업자가 공유 큐에서 작업을 가져옴; 명시적 다음 스피커 없음(스웜 아키텍처, Matrix). 

### 프레임워크 간 차이점

기본 요소가 고정되면 남은 설계 결정은 다음과 같습니다:

- **메모리 전략** — 일시적 vs 지속적 체크포인팅(LangGraph 체크포인터).
- **안전 경계** — 누가 핸도프를 승인할 수 있는가(인간 개입).
- **비용 계산** — 에이전트별 토큰 예산.
- **관측 가능성** — 핸도프 추적, 재생을 위한 상태 지속.

이 모든 것은 기본 요소 위에 구현 가능하며, 새로운 기본 요소가 아닙니다.

## 빌드하기

`code/main.py`는 표준 라이브러리 Python으로 150줄 내외의 네 가지 기본 요소를 구현합니다. 실제 LLM은 사용하지 않으며, 각 에이전트는 스크립트 정책이므로 조정 구조에 집중할 수 있습니다.

이 파일은 다음을 내보냅니다:

- `Agent` — 이름, 시스템 프롬프트, 도구, 정책 함수로 구성된 데이터 클래스.
- `Handoff` — 새 에이전트를 반환하는 함수.
- `SharedState` — 스레드 안전한 메시지 풀.
- `Orchestrator` — 세 가지 변형: `StaticOrchestrator`, `HandoffOrchestrator`, `LLMSelectorOrchestrator` (시뮬레이션).

데모는 동일한 3개 에이전트 파이프라인(연구 → 작성 → 검토)을 세 가지 오케스트레이터 유형에 적용하고 마지막에 메시지 풀을 출력합니다. 출력은 *누가 다음에 선택되는지*만 다르며, 에이전트와 공유 상태는 실행 간에 동일합니다.

실행 방법:

```
python3 code/main.py
```

예상 출력: 세 가지 오케스트레이터 실행 결과가 패턴별로 하나씩 출력됩니다. 각각 최종 메시지 풀을 표시합니다. 핸드오프 기반 실행에서는 연구자가 조기에 작업을 완료했다고 판단하면 도달하는 에이전트 수가 줄어듭니다. 이는 LLM 라우팅 트레이드오프의 축소 버전입니다.

## 사용 방법

`outputs/skill-primitive-mapper.md`는 모든 멀티에이전트 코드베이스나 프레임워크 문서를 읽고 4가지 기본 동작 매핑을 반환하는 스킬입니다. 새 프레임워크 릴리스에 적용하여 문서를 깊이 있게 읽기 전에 한 단락으로 된 개요를 얻을 수 있습니다.

## Ship It

새로운 프레임워크를 도입하기 전에 해당 프레임워크에 대한 원시 연산 매핑(primitive mapping)을 작성하세요. 만약 작성할 수 없다면, 문서가 불완전하거나 프레임워크가 다섯 번째 원시 연산(rare — 이전에 본 적 없는 공유 상태(shared-state) 유형인지 확인)을 발명하고 있는 것입니다.

아키텍처 문서에 매핑 내용을 고정(pin)하세요. 새로운 팀원이 합류할 때는 API 문서보다 먼저 매핑 내용을 제공하세요. 프레임워크 버전이 변경될 때는 변경 로그(changelog)가 아닌 매핑 내용을 비교(diff)하세요.

## 연습 문제

1. 서로 다른 에이전트 정책(agent policy)으로 `code/main.py`를 세 번 실행해 보세요. 오케스트레이터 선택(orchestrator choice)이 어떤 에이전트들을 실행하는지 어떻게 바꾸는지 관찰하세요.
2. 네 번째 오케스트레이터 유형을 구현하세요: 에이전트들이 작업을 위해 공유 상태(shared state)를 폴링하는 큐 기반(queue-driven) 방식. 어떤 데드락(deadlock)이 발생할 수 있으며, 어떻게 감지할 수 있나요?
3. LangGraph 퀵스타트(https://docs.langchain.com/oss/python/langgraph/workflows-agents)를 가져와서 네 가지 기본 요소(primitives)로 다시 작성해 보세요. LangGraph의 추상화(abstractions) 중 1:1로 매핑되는 것과 편의 래퍼(convenience wrappers)는 무엇인가요?
4. OpenAI Swarm 쿡북(https://developers.openai.com/cookbook/examples/orchestrating_agents)을 읽어보세요. Swarm이 가장 사용하기 쉽게 만든 네 가지 기본 요소(primitives)는 무엇이며, 어떤 요소를 호출자(caller)에게 맡기는지 식별하세요.
5. 이 표에 있는 프레임워크 중 공유 상태(shared state)를 완전히 숨기는 것을 찾아보세요. 에이전트들이 히스토리(history)를 다시 읽지 않고 핸드오프(handoff) 간 조율(coordination)이 필요할 때 무엇이 깨지나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 에이전트(Agent) | "도구가 있는 LLM" | `(system_prompt, tools, model)` 트리플. 상태 비저장(stateless). |
| 핸드오프(Handoff) | "제어권 이전" | 다음 에이전트와 선택적 페이로드를 명시하는 구조화된 호출. 세 가지 구현: 함수 반환, 그래프 엣지, 화자 선택. |
| 공유 상태(Shared state) | "메모리" / "컨텍스트" | 멀티에이전트 시스템의 유일한 상태 저장(stateful) 부분. 메시지 풀 또는 블랙보드. |
| 오케스트레이터(Orchestrator) | "조정자" | 다음에 실행할 에이전트를 결정하는 주체. 정적 그래프, LLM 선택기, 핸드오프 기반, 또는 큐 기반. |
| 프리미티브(Primitive) | "추상화" | 모든 프레임워크가 매개변수화하는 네 가지 축 중 하나. 프레임워크 기능이 아님. |
| 메시지 풀(Message pool) | "공유 채팅 기록" | 전체 기록 공유 상태. 추론하기 쉬우나 확장성이 떨어짐. |
| 투영 상태(Projected state) | "범위 지정 뷰" | 공유 상태에 대한 역할별 뷰. 확장 가능하나 스키마 설계 필요. |
| 화자 선택(Speaker selection) | "다음에 말할 사람" | 함수(주로 LLM)가 그룹에서 다음 에이전트를 선택하는 오케스트레이터 패턴.
