# 생성형 에이전트와 에머전트 시뮬레이션

> Park et al. 2023 (UIST '23, arXiv:2304.03442)는 25개의 에이전트로 구성된 샌드박스 **Smallville**을 **메모리 스트림**(자연어 로그), **반성**(에이전트가 자신의 스트림에 대해 생성하는 고수준 종합), **계획**(일 단위 행동 및 하위 계획)이라는 3부 아키텍처로 채웠다. 이 연구의 획기적인 결과는 발렌타인 데이 파티의 에머전스였다: "발렌타인 데이 파티를 열고 싶다"는 초기 설정을 가진 한 에이전트가 추가 스크립팅 없이도 초대장을 생성하고, 이를 통해 인구 전체에 퍼뜨리며, 일정을 조정한 결과 파티가 발생했다 — 이는 해당 사실을 전혀 알지 못한 24명의 에이전트로부터 비롯되었다. 모든 세 구성 요소가 신뢰성 있는 시뮬레이션에 필수적임을 실험(ablation study)에서 확인했다. 문서화된 실패 사례는 공간 규범 오류(닫힌 매장 입장, 1인용 화장실 공유)였다. 이는 2026년 에이전트 시뮬레이션 및 다중 에이전트 사회 평가의 참조 아키텍처다.

**유형:** 학습 + 구현  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 16단계 · 04 (원시 모델), 16단계 · 13 (공유 메모리)  
**소요 시간:** ~75분

## 문제

대부분의 다중 에이전트 시스템은 엄격하게 스크립트화된 팀입니다: 플래너가 계획을 세우고, 코더가 코드를 작성하며, 리뷰어가 검토합니다. 이는 잘 정의된 작업에는 효과적이지만, 에이전트들이 메모리, 우선순위, 개방된 세계를 가질 때 발생하는 예측 불가능하고 스크립트화되지 않은 행동을 포착하지 못합니다. 연구, 사회 시뮬레이션, 그리고 점점 더 많은 게임 AI는 이러한 두 번째 유형의 접근이 필요합니다.

Smallville 아키텍처는 이를 위한 벤치마크입니다. Park 2023 이전에는 최고의 에이전트 시뮬레이션이 얕은 스크립트 추종자에 불과했지만, 이후에는 개방된 세계에서 생성형 에이전트의 기본 패턴으로 자리 잡았습니다. 2026년에 에이전트 시뮬레이션을 구축한다면, Smallville의 세 가지 구성 요소를 사용하거나 왜 사용하지 않는지 명시적으로 정당화해야 합니다.

## 개념

### 세 가지 구성 요소

**메모리 스트림.** 관찰, 행동, 성찰, 계획에 대한 추가 전용 로그. 각 항목에는 타임스탬프, 유형, 설명(자연어), 그리고 파생된 메타데이터가 포함됨: **최신성(recency)**, **중요도(importance)** (에이전트가 1-10점으로 자체 평가), **관련성(relevance)** (현재 쿼리와의 코사인 유사도).

```
[2026-02-14 09:12:03] observation: Isabella Rodriguez가 나에게 재즈를 좋아하는지 물어봄
[2026-02-14 09:14:22] reflection:   나는 음악에 대한 긴 대화를 즐김
[2026-02-14 10:05:00] plan:         오늘 밤 Isabella의 발렌타인 데이 파티에 참석
```

메모리 검색은 세 가지 점수를 결합함: `score = w_recency * e^(-decay * age) + w_importance * importance + w_relevance * cos_sim`. 상위 k개 항목이 현재 프롬프트에 들어감.

**성찰(Reflection).** 주기적으로(최근 N개 메모리 또는 중요한 이벤트 발생 시), 에이전트는 최근 메모리에서 고차원적 종합을 생성함. 성찰 항목은 스트림으로 다시 들어가 다른 메모리와 마찬가지로 검색 가능함. 이는 에이전트가 "이해"를 구축하는 방식—아키텍처 관점에서 장기 신념과 동등함.

**계획(Plan).** 상향식 분해. 먼저 하루 단위 계획("출근, Klaus와 저녁 식사")을 수립한 후 시간 단위, 행동 단위 계획을 세움. 계획은 수정 가능: 관찰이 계획과 모순될 때, 에이전트는 영향을 받은 부분만 재계획함.

### 세 가지 구성 요소가 중요한 이유 (제거 실험)

Park et al.은 관찰, 성찰, 계획을 각각 제거한 실험을 수행했으며, 각각이 신뢰성(believability)에 영향을 미침:

- **관찰**이 없으면 에이전트는 맥락을 놓치고 오래된 신념에 따라 행동함.
- **성찰**이 없으면 에이전트는 고차원적 신념을 형성할 수 없으며, 상호작용이 피상적으로 남음.
- **계획**이 없으면 행동이 반응적 잡음으로 변하고, 목표가 사라짐.

인간 평가자의 신뢰성 점수는 세 가지 모두 있을 때 가장 높으며, 하나라도 제거하면 측정 가능한 성능 저하가 발생함.

### 발렌타인 데이의 창발성

한 에이전트인 Isabella Rodriguez는 "2월 14일 오후 5시에 Hobbs Cafe에서 발렌타인 데이 파티를 열고 싶다"는 목표를 부여받음. 나머지 24개 에이전트는 이러한 시드 없이 시작함. 시뮬레이션된 날짜 동안:

1. Isabella의 계획에는 사람들을 초대하는 것이 포함됨.
2. 각 초대는 이웃 에이전트의 메모리 스트림에 관찰로 기록됨.
3. 그 이웃의 성찰은 "Isabella가 파티를 연다"는 신념을 생성함.
4. 이웃의 계획에는 "2월 14일 파티 참석"이 포함됨.
5. 이웃들이 다른 이웃들에게 알리며, 중앙 조정 없이 초대가 확산됨.
6. 2월 14일 오후 5시, 여러 에이전트가 Hobbs Cafe에 모임.

이는 기술적 의미의 창발성: 시스템 수준의 행동(파티)이 지역 상호작용(이진 초대 + 개별 계획)에서 발생했으며, 중앙 조율자가 없었음.

### 문서화된 실패 모드

Park et al.은 다음을 명시적으로 문서화함:

- **공간 규범 오류.** 에이전트가 닫힌 상점에 들어가거나, 한 명만 사용 가능한 화장실을 함께 사용하려 함. 식사 공간이 아닌 곳에서 식사를 시도함. 모델은 환경만으로 사회-물리적 규범을 추론하지 못함.
- **메모리 과부하.** 깊은 시뮬레이션 실행 시 메모리 검색 비용이 증가함. 실용적 해결책: 주기적 메모리 압축(요약 및 삭제)과 낮은 중요도 항목의 감쇠.
- **성찰 환각.** 성찰은 메모리 스트림에 존재하지 않는 관계를 발명할 수 있음. 완화 방법: 성찰 프롬프트에 소스 메모리 ID를 포함하고 검색 시 검증.

이들은 프로덕션 관련 실패 모드: 2026년 에이전트 시뮬레이션은 모두 이를 상속받음.

### 세 가지 구성 요소 구현 규칙

1. **메모리는 추가 전용.** 메모리 항목을 변경하지 않음. 수정은 새 항목으로 추가.
2. **중요도 점수는 저렴함.** LLM을 호출해 작성 시점에 1-10점 중요도를 평가하고 점수를 캐시.
3. **검색은 순위 기반, 필터링 아님.** 결합 점수 기준 상위 k개 선택; 하드 필터 사용 안 함(맥락 손실 방지).
4. **성찰은 주기적으로 실행.** 처리되지 않은 메모리의 중요도 합이 임계값(예: 150)을 초과할 때 트리거.
5. **계획은 수정 가능.** 새 관찰이 계획과 모순될 때, 전체 계획이 아닌 영향을 받은 부분만 재생성.

### Smallville을 넘어선 생성 에이전트

2024-2026년 후속 연구는 아키텍처를 확장함:

- **정책/시장 조사를 위한 다중 에이전트 사회 시뮬레이션.** Smallville과 유사한 인구가 기능 변화에 대한 사용자 행동을 시뮬레이션. A/B 테스트보다 빠르지만 정확도는 논쟁 중.
- **게임용 NPC AI.** Smallville 에이전트를 사용한 RPG는 스크립트된 퀘스트 대신 창발적 스토리라인을 생성.
- **생성 에이전트 평가 벤치마크.** 작업 정확도 대신, 장기 실행에서의 신뢰성 + 행동 일관성이 메트릭이 됨.

아키텍처는 참조 구현. 확장 시 구성 요소(메모리를 위한 벡터 저장소, 검색 증강 성찰, 신경기호적 계획)를 교체하지만 세 부분 구조는 유지.

### 다중 에이전트 엔지니어링에 중요한 이유

Smallville은 구성 요소가 적절할 때 다중 에이전트 창발성이 저렴함을 증명한 개념 증명. 이 아키텍처는 오픈 소스 모델에서도 재현됨(작은 LLM은 급격히가 아닌 부드럽게 신뢰성 감소). **창발적 사회 행동**이 필요한 모든 프로덕션 시스템은 이 형태를 사용함. **엄격한 작업 실행**이 필요한 시스템은 이 단계의 이전 감독자/역할/기본 요소 패턴을 사용함.

## 구축 방법

`code/main.py`는 스크립트 기반 에이전트 정책(실제 LLM 없음)으로 stdlib Python에 세 가지 구성 요소를 구현합니다. 이 데모는 발렌타인 파티의 출현을 축소된 형태로 재현합니다:

- `MemoryStream` — 최근성/중요도/관련성 검색이 가능한 추가 전용 로그.
- `reflect(stream)` — 최근 고중요도 메모리에 대한 스크립트 기반 반성(reflection).
- `plan(agent_state)` — 현재 믿음을 기반으로 한 일 단위 및 시간 단위 계획.
- 시나리오: 5명의 에이전트. 에이전트 1이 "오후 5시에 파티를 연다"로 시작합니다. 시뮬레이션 틱이 진행되면서 초대장이 퍼지고 에이전트들이 수렴합니다.

실행 방법:

```
python3 code/main.py
```

예상 출력: 틱별 추적 로그. 최종 틱까지 5명 중 최소 3명의 에이전트가 계획에 파티를 포함시키고, 파티 장소에서 수렴합니다. 단일 시드(seed)가 오케스트레이터 없이도 조정된 도착을 생성했습니다.

## 사용 방법

`outputs/skill-simulation-designer.md`는 생성형 에이전트 시뮬레이션을 설계합니다: 에이전트 수, 메모리 스키마, 반성 주기, 계획 범위, 평가 메트릭.

## Ship It

프로덕션 시뮬레이션 규칙:

- **메모리는 데이터베이스다.** 실제 저장소(벡터 DB, Postgres)를 규모에 맞게 선택하라. 인메모리 표준 라이브러리는 프로토타입용이다.
- **검색 추적을 로깅하라.** 모든 액션에 대해 이를 유발한 상위 k개 메모리를 기록하라. 이것이 디버깅 능력이다.
- **에이전트별 토큰 예산을 설정하라.** 각 에이전트의 검색 + 반성 + 계획 단계는 틱당 O(k) LLM 호출이 발생한다. N 에이전트 × T 틱 × 틱당 호출 횟수는 예산을 초과할 수 있다.
- **주기적으로 메모리를 압축하라.** 중요도가 낮은 항목을 요약하고 제거하라. 보존 정책은 세부사항이 아닌 설계 결정이다.
- **공간적/사회적 규범 위반을 명시적으로 탐지하라.** 아키텍처는 이를 학습하지 않는다.

## 연습 문제

1. `code/main.py`를 실행하세요. 3명 이상의 에이전트가 파티에서 수렴하는지 확인하세요. 에이전트 수를 10명으로 늘리면 — 여전히 출현(emergence) 현상이 발생하나요?
2. 반성(reflection) 단계를 제거하세요. 행동 양상은 어떻게 변하나요? Park 2023의 제거 실험(ablation) 결과와 매핑해 보세요.
3. 경쟁하는 시드 목표("Klaus는 오후 5시에 연구 발표를 하고 싶어함")를 도입하세요. 에이전트들이 목표를 분할(split)하나요, 아니면 하나의 목표가 지배하나요? 어떤 요소가 이를 결정하나요?
4. 공간적 제약 조건을 추가하세요: Hobbs Cafe는 최대 4명의 에이전트만 수용 가능합니다. 시뮬레이션이 오버플로우를 우아하게 처리하나요, 아니면 "한 사람 화장실" 실패 패턴에 부딪히나요?
5. Park et al. (arXiv:2304.03442) 6장(출현 행동 실험)을 읽으세요. 여러분의 미니 모델에서 재현할 수 없는 행동 하나를 식별하세요. 아키텍처의 어떤 구성 요소를 강화해야 하나요?

| 계획(Plan) | "일/시간/행동 분해" | 하향식 계획 트리. 관찰 결과가 모순될 때 수정 가능. |
| 스몰빌(Smallville) | "Park 2023의 샌드박스" | 발렌타인데이 출현을 생성한 25개 에이전트 시뮬레이션. |
| 믿음 가능성(Believability) | "품질 지표" | 행동이 그럴듯한 에이전트처럼 보이는지 인간 평가자 점수. |
