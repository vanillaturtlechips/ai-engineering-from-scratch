# 역할 전문화 — 계획자, 비평가, 실행자, 검증자

> 2026년 가장 일반적인 다중 에이전트 분해 방식: 한 에이전트는 계획을 수립하고, 한 에이전트는 실행하며, 한 에이전트는 비평 또는 검증을 담당합니다. MetaGPT(arXiv:2308.00352)는 이를 역할 프롬프트로 인코딩된 SOP(표준 운영 절차)로 공식화합니다 — 제품 관리자, 아키텍트, 프로젝트 관리자, 엔지니어, QA 엔지니어 — `Code = SOP(팀)`을 따릅니다. ChatDev(arXiv:2307.07924)는 "통신적 환각 제거"(에이전트가 명시적으로 누락된 세부 정보를 요청) 기능을 갖춘 "채팅 체인"을 통해 디자이너, 프로그래머, 검토자, 테스터를 연결합니다. 검증자는 핵심 역할을 수행합니다: Cemri 등(MAST, arXiv:2503.13657)은 모든 다중 에이전트 실패가 누락되거나 손상된 검증으로 추적될 수 있음을 보여줍니다. PwC는 CrewAI에서 구조화된 검증 루프를 통해 정확도가 7배 향상(10% → 70%)되었다고 보고했습니다.

**유형:** 학습 + 구축  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 16단계 · 04(원시 모델), 16단계 · 05(감독자)  
**소요 시간:** ~60분

## 문제

일반적인 다중 에이전트 시스템은 일반적인 출력을 생성합니다. 그룹 채팅에 있는 세 명의 코더가 동일한 평범한 코드의 세 가지 버전을 작성합니다. 더 많은 에이전트를 추가하거나 더 많은 라운드를 진행해도 여전히 품질 임계값을 넘지 못합니다.

해결책은 더 많은 에이전트가 아니라 *다른* 에이전트입니다. 고유한 역할을 할당하세요. 비평가에게 계획자가 갖지 못한 도구를 제공하세요. 검증자에게 객관적인 테스트 스위트를 제공하세요. 이제 시스템은 단순한 병렬 추측이 아닌 근거 있는 수정을 통한 내부 불일치를 갖게 됩니다.

## 개념

### 네 가지 표준 역할

**플래너(Planner).** 목표를 읽고 단계 목록 또는 사양을 생성합니다. 도구: 지식 검색, 문서. 출력: 구조화된 계획.

**실행자(Executor).** 한 번에 하나의 계획 단계를 읽고 아티팩트를 생성합니다. 도구: 실제 작업 도구(코드 컴파일러, 셸, API 클라이언트). 출력: 아티팩트.

**비평가(Critic).** 실행자의 출력을 플래너의 의도와 대조하여 읽습니다. 도구: 아티팩트에 대한 읽기 전용 접근, 정적 분석. 출력: 이유가 포함된 수락/거부.

**검증자(Verifier).** 아티팩트를 읽고 결정적 검사를 실행합니다. 도구: 테스트 러너, 타입 체커, 스키마 검증기. 출력: 증거가 포함된 통과/실패.

비평가는 주관적이고 의견을 반영하며 종종 LLM 기반입니다. 검증자는 객관적이고 결정적이며 종종 코드 기반입니다. 이 두 역할은 동일하지 않습니다.

### MetaGPT의 SOP 패턴

MetaGPT(arXiv:2308.00352)는 소프트웨어 엔지니어링 SOP(표준 운영 절차)를 역할 프롬프트로 인코딩합니다:

- **제품 관리자(Product Manager)**가 PRD(제품 요구 사항 문서)를 작성합니다.
- **아키텍트(Architect)**가 시스템 설계를 생성합니다.
- **프로젝트 관리자(Project Manager)**가 작업을 분할합니다.
- **엔지니어(Engineer)**가 구현합니다.
- **QA 엔지니어(QA Engineer)**가 테스트를 실행합니다.

각 역할은 엄격한 입력/출력 스키마를 가집니다. 역할 프롬프트는 해당 역할이 *무엇인지*와 *생성해야 할 것*을 명시합니다. `Code = SOP(Team)` 공식 — 결정적 SOP는 LLM 팀을 예측 가능한 파이프라인으로 변환합니다.

### ChatDev의 의사소통적 환각 방지

ChatDev는 핵심 기능을 추가합니다: 실행자가 계획에 없는 특정 세부 정보가 필요할 때, 계속하기 전에 설계자에게 명시적으로 요청합니다. 이는 LLM의 전형적인 실패 모드인 그럴듯한 세부 정보 생성을 방지합니다.

구현: 역할 프롬프트에 "제공되지 않은 특정 정보가 필요할 때, 출력을 생성하기 전에 관련 역할의 이름을 지정하여 질문합니다."가 포함됩니다.

### 검증자가 가장 중요한 이유

Cemri 등(MAST)은 1642건의 다중 에이전트 실행 실패를 추적했습니다. 21.3%는 검증 누락 — 아무도 확인하지 않은 답변을 시스템이 제공했습니다. 나머지 79%는 종종 "실패한 검사가 무음으로 발생했거나 전혀 실행되지 않음"으로 추적됩니다. 검증은 핵심 역할입니다.

PwC(2025, CrewAI 배포)는 구조화된 검증 루프 추가가 정확도를 10%에서 70%로 향상시켰다고 보고했습니다. 한 역할에서 7배 향상.

### 비평가 vs 검증자

- 비평가는 아티팩트의 품질을 검토하는 LLM입니다. 주관적입니다. 그럴듯한 문체에 속을 수 있습니다.
- 검증자는 아티팩트에서 실행되는 결정적 프로그램입니다. 객관적입니다. 증거와 함께 통과/실패를 제공합니다.

둘 다 사용하세요. 비평가는 검증자가 표현할 수 없는 취향 문제를 포착합니다. 검증자는 런타임에서만 나타나는 버그를 비평가는 볼 수 없기 때문에 포착합니다.

### 안티 패턴

시스템의 모든 역할이 LLM이고 모든 역할의 출력이 "좋아 보입니다"입니다. MAST의 전형적인 실패 모드입니다. LLM이 아닌 코드에 의해 통과/실패가 결정되는 검증자를 최소 한 명 추가하세요.

### 프레임워크 매핑

- **CrewAI** — `Agent(역할, 목표, 백스토리)`는 교과서적인 전문화 표면입니다.
- **LangGraph** — 노드는 특수화된 프롬프트를 가질 수 있으며, 엣지는 파이프라인을 강제합니다.
- **AutoGen** — 그룹 채팅에서 한 단어 이름을 가진 역할별 ConversableAgents.
- **OpenAI Agents SDK** — 역할별 에이전트 간 핸드오프 도구.

## 빌드하기

`code/main.py`는 간단한 Python 함수를 구축하는 4단계 파이프라인을 구현합니다:

- **플래너(Planner)**는 스펙(spec)을 생성합니다.
- **실행기(Executor)**는 코드 문자열을 생성합니다.
- **비평가(Critic)**(LLM 시뮬레이션)는 명백한 문제를 표시합니다.
- **검증기(Verifier)**는 샌드박스(`exec`)에서 생성된 코드를 테스트 케이스와 비교합니다.

데모는 두 번 실행됩니다:  
1. 실행기가 올바른 코드를 생성하는 경우(비평가 + 검증기 모두 통과)  
2. 실행기가 스펙과 다른 코드를 생성하는 경우(비평가가 버그를 놓치지만 테스트 실패로 인해 검증기가 버그를 발견)

실행 방법:

```
python3 code/main.py
```

## 사용 방법

`outputs/skill-role-designer.md`는 작업(task)을 입력받아 역할 목록(3-5개 역할), 각 역할의 입출력 스키마, 검증자(verifier) 확인 사항을 생성합니다. 에이전트들을 프레임워크에 연결하기 전에 이 도구를 사용하세요.

## Ship It

체크리스트:

- **최소 한 개의 결정적 검증기(deterministic verifier).** 모든 역할을 LLM으로 구성하지 말 것.
- **역할별 명시적 입출력 스키마.** 기획자(planner)는 산문(prose)이 아닌 스펙(spec)을 반환해야 하며, 실행기(executor)는 해당 스키마를 읽어야 함.
- **의사소통 기반 환각 방지(communicative dehallucination).** 실행기는 정보가 누락되었을 때 반드시 기획자에게 요청해야 하며, 절대 정보를 임의로 생성해서는 안 됨.
- **비평가/비검증기 순서.** 비평가(critic)를 먼저 실행(저비용, 설계 문제 감지)한 후, 검증기(verifier)를 실행(고비용, 버그 감지).
- **반복 예산(loop budget).** 인간에게 위임하기 전에 최대 2회의 비평가-실행기 수정 라운드로 제한.

## 연습 문제

1. `code/main.py`를 실행하고 검증자(verifier)가 비평자(critic)가 놓친 버그를 어떻게 잡는지 관찰하세요. 정적 분석 검사(`return` 발생 횟수 카운트)를 추가 검증자로 추가하세요. 런타임 테스트가 놓치는 것 중 이 검사가 잡아내는 것은 무엇인가요?

2. 5번째 역할인 "요구사항 분석가(requirements analyst)"를 추가하세요. 이 역할은 사용자의 요구사항을 플래너(planner)가 처리할 수 있는 명세(spec)로 변환합니다. 이 역할로 어떤 의사소통적 환각 방지(communicative dehallucination) 요청이 전달되어야 할까요?

3. MetaGPT 3장("에이전트(Agents)")을 읽으세요. MetaGPT의 5가지 역할별 입력/출력 스키마를 나열하세요.

4. ChatDev의 채팅 체인 다이어그램(arXiv:2307.07924 Figure 3)을 읽으세요. 의사소통적 환각 방지가 무한 루프를 끊는 지점을 식별하세요.

5. PwC의 7배 정확도 향상은 검증 루프에서 비롯되었습니다. 검증자 추가가 도움이 되지 않는 세 가지 작업을 가설로 세우세요. 여기서 정확성 결정론적 검증이 불가능하거나 비용이 지나치게 높은 경우는 어떤 경우인가요?

## 핵심 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 역할 전문화 | "다른 에이전트, 다른 작업" | 플래너/실행자/비평가/검증자 역할에 맞춰 조정된 별도의 시스템 프롬프트. |
| SOP 패턴 | "인코딩된 표준 운영 절차" | MetaGPT의 프레임워크: 역할별 엄격한 입출력 스키마가 팀을 파이프라인으로 변환. |
| 의사소통적 환각 방지 | "꾸미기 전에 물어보기" | ChatDev 패턴: 실행자는 세부사항이 누락되었을 때 추측하지 않고 플래너에게 질문. |
| 비평가 | "LLM 리뷰어" | 주관적이고 의견을 제시하는 리뷰어. 취향 문제를 포착. 그럴듯한 문체에 속을 수 있음. |
| 검증자 | "결정론적 검사" | 코드 기반 합격/불합격. 테스트 실행기, 타입 체커, 스키마 검증기. 속을 수 없음. |
| 검증 격차 | "아무도 확인하지 않음" | MAST 실패의 21.3%. 버그를 잡을 수 있었던 검사 없이 답변 전달. |
| 수정 루프 | "비평가가 다시 보냄" | 비평가 거부는 피드백과 함께 실행자 재실행을 트리거. 예산 필요. |
| 올-LLM 안티패턴 | "나 보기에는 좋아" | 모든 역할이 LLM, 결정론적 검사 없음. 전형적인 MAST 실패 사례. |
