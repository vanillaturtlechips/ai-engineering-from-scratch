# 병렬 / 군집 / 네트워크 아키텍처

> 감독자(supervisor)와 대조: 중앙 결정자 없음. 에이전트들은 공유 이벤트 버스를 읽고, 비동기적으로 작업을 선택한 후 결과를 다시 씀. LangGraph는 분산형 동적 환경을 위한 "군집 아키텍처(Swarm Architecture)"를 명시적으로 지원. Matrix(arXiv:2511.21686)는 제어 및 데이터 흐름을 분산 큐를 통해 전달되는 직렬화된 메시지로 표현하여 오케스트레이터 병목 현상을 제거. 트레이드오프는 명확함: 확장성을 위한 결정성(determinism)과 추적 가능성(traceability) 희생. 군집 아키텍처는 많은 독립적인 하위 문제가 있는 작업에 적합; 단일 일관된 계획이 필요한 작업에는 부적합.

**유형:** 학습 + 구현  
**언어:** Python (표준 라이브러리, `threading`, `queue`)  
**선수 지식:** 16단계 · 05 (감독자 패턴), 16단계 · 04 (기본 모델)  
**소요 시간:** ~75분

## 문제

Supervisor는 몇 개의 작업자(worker)까지만 확장됩니다. 수백 개의 작업자는 어떻게 될까요? Supervisor 자체가 병목 현상이 됩니다: 누가 어떤 작업을 수행할지에 대한 모든 결정이 하나의 에이전트(agent)를 통해 집중되기 때문입니다. 하나의 느린 계획 단계(plan step)가 전체 시스템을 정체시킵니다.

Swarm 아키텍처는 설계를 뒤집습니다. 중앙 계획자(central planner)가 작업을 할당하는 대신, 작업자들이 공유 큐(shared queue)에서 작업을 직접 선택합니다. "조정(coordination)"은 이벤트 버스(event bus) 시맨틱스에 내장되어 있습니다. 오케스트레이터(orchestrator)가 없으며, 시스템은 큐가 허용하는 한 확장됩니다.

## 개념

### 형태

```
                ┌──── 공유 큐 ────┐
                │                │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
           결과 풀
```

오케스트레이터 없음. 각 워커는 반복 작업: 작업 가져오기, 처리, 결과 기록 (및 선택적으로 후속 작업 추가).

### 스웜이 적합한 경우

- **많은 독립 작업.** 스크래핑, 변환, 분류. 작업 간 의존성 없음.
- **변동성 있는 작업 시간.** 일부 작업이 100ms, 다른 작업이 10초 걸릴 경우, 스웜은 자동으로 부하 분산 — 빠른 워커가 다음 작업 가져감. 감독자는 작업 시간을 예측해야 함.
- **결정성보다 처리량.** 총 완료 시간에 관심, 엄격한 순서 요구 없음.

### 스웜이 실패하는 경우

- **순차적 워크플로우.** 3단계가 2단계 출력을 필요로 할 때, 스웜은 2단계 완료 전 3단계 실행 위험.
- **전역 계획 작업.** 복잡한 연구 질문은 플래너 필요. 스웜 연구자들은 독립적 사실 생성, 일관된 보고서 생성 불가.
- **디버깅.** 중앙 로그 없고 비동기 작업 시, 버그 재현 비용 큼.

### 매트릭스 (arXiv:2511.21686)

매트릭스는 2025년 논문으로, 스웜을 극한까지 발전시킴: 제어 흐름과 데이터 흐름 모두 분산 큐의 직렬화된 메시지. 중앙 조정자 없음. 장애 허용은 메시지 내구성에서 옴. 확장성은 메시지 브로커 문제, 시스템 문제 아님.

기여: 다중 에이전트 조정이 "감독자가 다음에 어떤 에이전트를 선택할지"가 아닌 "이 에이전트가 어떤 메시지 토픽을 구독하는가?"인 프로그래밍 모델. 시스템을 pub/sub 이벤트 메시처럼 보이게 함.

### 랭그래프 스웜 아키텍처

LangGraph 2025 문서는 "스웜 아키텍처"를 다중 에이전트 패턴 중 하나로 명시: 에이전트는 노드, 에지는 사이클이 있는 방향 그래프 형성. 어떤 노드든 풀에서 활성화 가능. 워커는 감독자 할당이 아닌 조건에 따라 작업 선택.

### 실패 모드: 기아 상태와 핫스팟

모든 워커가 가장 빠른 작업을 가져오면, 장기 실행 작업은 마지막 남은 작업일 때만 선택됨. 전형적인 큐 기아 상태.

완화 방안:
- 명시적 에이징(기다린 시간에 따라 우선순위 증가)이 있는 우선순위 큐.
- 워커 전문화: 일부 워커는 "긴" 작업만 처리.
- 백프레셔: 큐에 들어오는 빠른 작업 수 제한.

### 콘텐츠 기반 라우팅 링크

스웜은 콘텐츠 기반 라우팅(레슨 22)과 자연스럽게 결합. 일반 큐 대신 메시지 유형별 큐 사용. 전문 워커는 자신의 유형만 구독. 이는 수천 에이전트까지 확장 가능한 메시지 버스 아키텍처의 기초.

## 빌드 방법

`code/main.py`는 공유 `queue.Queue`에서 작업을 가져오는 4개의 작업자 스레드로 구성된 스웜을 구현합니다. 작업에는 가변적인 지속 시간(빠른 작업, 느린 작업)이 있습니다. 이 데모는 다음을 비교합니다:

- **순차적 기준선:** 단일 작업자가 모든 작업을 직렬로 처리합니다.
- **고정 할당:** 각 작업이 특정 작업자(감독자 스타일)에 미리 할당됩니다.
- **스웜:** 작업자가 공유 큐에서 작업을 가져옵니다.

스웜은 자동으로 부하를 분산시키는 반면, 고정 할당은 할당된 작업이 느릴 때 빠른 작업자를 유휴 상태로 둡니다.

실행 방법:

```
python3 code/main.py
```

출력은 작업자별 작업 수(스웜은 불균형하지만 최적으로 분배)와 벽시계 시간을 보여줍니다.

## 사용 방법

`outputs/skill-swarm-fit.md`는 작업에 스웜(swarm) 방식을 사용할지 감독자(supervisor) 방식을 사용할지 평가합니다. 입력값: 작업 독립성, 지속 시간 변동성, 순서 요구사항, 디버깅 가능성 요구사항.

## Ship It

체크리스트:

- **에이징(aging)이 적용된 우선순위 큐.** 장기 작업 기아 현상 방지.
- **워커 멱등성(idempotency).** 워커가 실행 중 충돌하면 작업이 여러 번 풀링될 수 있음. 워커는 멱등성(idempotent)을 가져야 함.
- **내구성 있는 큐(durable queue).** 프로덕션 환경에서는 Kafka, Redis Streams 또는 데이터베이스 기반 큐 사용. `queue.Queue`는 메모리 전용.
- **작업별 관측 가능성(observability).** 모든 작업에는 트레이스 ID(trace ID)가 있으며, 모든 워커는 시작/종료 시 이를 로깅해야 함.
- **백프레셔(back-pressure).** 큐가 워커의 처리 속도보다 빠르게 증가하면 생산자(producer)를 느리게 함.

## 연습 문제

1. `code/main.py`를 실행하세요. 가변 지속 시간 작업 부하에서 스웜(swarm)이 순차(sequential) 방식보다 얼마나 빠른가요? 고정 할당(fixed assignment) 방식보다 얼마나 빠른가요?
2. 우선순위 큐(priority queue) 변형 버전을 추가하세요(`queue.PriorityQueue` 사용). 작업의 "importance" 필드로 우선순위를 할당하세요. 지속적인 부하 하에서 저우선순위 작업이 영원히 실행되지 않는(starve) 현상이 발생하는지 관찰하세요.
3. 핫스팟 감지기 구현: 어떤 작업자가 가장 느린 작업자보다 3배 더 많은 작업을 처리할 때 로깅하세요. 이는 작업 지속 시간 분포에 대해 무엇을 나타내나요?
4. Matrix 논문(arXiv:2511.21686)의 초록과 3절을 읽으세요. Matrix가 수용한 확장성(scalability) 향상과 포기한 트레이서빌리티(traceability), 결정성(determinism) 중 각각 하나의 구체적인 트레이드오프를 식별하세요.
5. 스웜 데모를 (작업 유형, 페이로드) 튜플의 `queue.Queue`를 사용하도록 변환하세요. 작업자가 특정 유형에만 구독하도록 구현하세요. 작업이 이질적(heterogeneous)일 때 어떤 라우팅 규칙이 합리적인가요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 스웜 아키텍처(Swarm architecture) | "분산된 에이전트" | 작업자(Worker)가 공유 큐에서 작업을 가져옴; 중앙 오케스트레이터 없음. |
| 이벤트 버스(Event bus) | "에이전트가 토픽을 구독" | 메시지 브로커가 작업 유형 또는 내용에 따라 작업자에게 작업을 라우팅. |
| 스타베이션(Starvation) | "작업이 실행되지 않음" | 고우선순위 작업이 지속적으로 도착하여 저우선순위 작업이 선택되지 않는 현상. |
| 핫스폿팅(Hot-spotting) | "한 작업자가 과부하" | 한 작업자가 대부분의 작업을 받는 부하 불균형 현상. |
| 백프레셔(Back-pressure) | "생산자 속도 저하" | 큐가 가득 찼을 때 상위 시스템에 생산 중지를 알리는 메커니즘. |
| 멱등 작업자(Idempotent worker) | "재실행 안전" | 동일한 작업을 두 번 처리해도 결과가 동일함. 작업자가 실행 중 충돌할 수 있으므로 필요. |
| 내구성 큐(Durable queue) | "충돌 시 생존" | 디스크 또는 복제 저장소에 백업된 큐; 작업자 충돌 시 작업이 손실되지 않음. |
| 매트릭스 프레임워크(Matrix framework) | "완전한 메시지 전달 스웜" | 데이터 및 제어 흐름이 분산 큐의 직렬화된 메시지로 구성됨. |
