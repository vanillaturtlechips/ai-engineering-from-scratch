# 실패 모드 — MAST, 집단사고, 단일문화, 연쇄 오류

> 2026년 기준 분류 체계는 **MAST**(Cemri et al., NeurIPS 2025, arXiv:2503.13657)로, 7개의 최신 오픈소스 MAS(Multi-Agent System)에서 수집된 1642개의 실행 트레이스를 분석하여 도출되었으며 **41–86.7% 실패율**을 보입니다. 세 가지 주요 범주: **명세 문제**(41.77%) — 역할 모호성, 불명확한 작업 정의; **협조 실패**(36.94%) — 통신 장애, 상태 불일치; **검증 결함**(21.30%) — 검증 누락, 품질 검사 부재. **집단사고**(arXiv:2508.05687) 계열은 다음을 추가합니다: 단일문화 붕괴(동일 기반 모델 → 상관 오류), 순응 편향(에이전트 간 오류 강화), 부족한 마음 이론, 혼합 동기 역학, 연쇄적 신뢰성 실패. 연쇄 오류 사례: 결제 실패가 주문 재시도를 유발하고, 이는 재고 재시도를 트리거하여 재고 서비스를 마비시킴(초 단위 10배 부하 — 회로 차단기 필요). 메모리 오염: 한 에이전트의 환각이 공유 메모리에 유입되면 하위 에이전트가 이를 사실로 처리; 정확도가 점진적으로 저하되어 근본 원인 진단이 어려움. **STRATUS**(NeurIPS 2025)는 전용 탐지/진단/검증 에이전트를 통해 1.5배 완화 성공률 향상을 보고합니다. 이 레슨은 실패 모드를 1급 엔지니어링 대상으로 취급합니다.

**유형:** 학습  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 16단계 · 13(공유 메모리), 16단계 · 14(합의 및 BFT), 16단계 · 15(투표 및 토론 토폴로지)  
**소요 시간:** ~75분

## 문제

멀티에이전트 시스템은 실제 작업에서 41-86.7%의 실패율을 보입니다(Cemri et al. 2025가 7개의 오픈소스 MAS에서 측정한 결과). 이는 "에이전트를 더 추가하기만" 해서는 디버깅할 수 없습니다. 실패 원인은 구조적 문제에 있습니다. MAST 분류 체계는 이러한 범주들을 제공합니다. 이 강의에서는 각 범주를 구체적인 탐지, 진단, 완화 패턴과 매핑하여 실패율이 더 이상 임의적으로 보이지 않도록 합니다.

2026년 프로덕션 모범 사례는 실패 모드를 설계 입력으로 취급하는 것입니다. 각 MAST 범주를 지목하고 배포한 완화 전략을 명시할 수 있을 때까지 아키텍처는 "충분히 좋은" 상태가 아닙니다.

## 개념

### MAST 범주

**명세 문제 (실패의 41.77%).** 에이전트의 작업이 충분히 엄격하게 정의되지 않았습니다. 예시:

- 역할 모호성: 두 에이전트가 모두 자신이 검토자라고 생각합니다.
- 작업 미정의: 사용자가 특정 관점을 원했을 때 "이것을 요약하라"고 요청합니다.
- 성공 기준 암묵적: 에이전트가 성공 여부를 알 수 없습니다.

완화 방법:
- 명시적 역할 계약 작성. 각 에이전트의 프롬프트에 수행하는 작업과 수행하지 않는 작업을 명시합니다.
- 작업별 수용 테스트. 에이전트가 시작하기 전에 "완료는 X와 같다"고 정의합니다.
- 사전 비행 명세 검사: 별도의 에이전트가 작업 정의를 검토한 후 디스패치합니다.

**조정 실패 (36.94%).** 통신 또는 상태 붕괴.

예시:
- 두 에이전트가 동기화 없이 공유 상태를 업데이트합니다.
- 에이전트 간 메시지 손실 (큐 실패, 타임아웃).
- 상태 드리프트: 에이전트 A는 작업이 완료되었다고 생각하지만, 에이전트 B는 여전히 실행 중입니다.

완화 방법:
- 낙관적 동시성을 갖춘 버전 관리 공유 상태.
- 중요 메시지에 대한 명시적 확인 (확인될 때까지 재시도).
- 주기적 상태 동기화 체크포인트; 드리프트를 조기에 감지합니다.

**검증 누락 (21.30%).** 출력에 대한 독립적인 검사가 없습니다.

예시:
- 한 에이전트가 성공을 주장하지만, 아무도 검증하지 않습니다.
- 에이전트 체인이 각각 이전 출력을 신뢰합니다.
- 테스트 범위가 누락된 복합 동작.

완화 방법:
- 독립적인 검증 에이전트 (레슨 13). 읽기 전용, 독립적인 소스 접근.
- 명시적 핸드오프 계약: "A의 출력은 B가 시작하기 전에 체커 C를 통과해야 합니다."
- 사후 분석을 위한 결과 로깅.

### 집단사고 계열 (arXiv:2508.05687)

에이전트가 동질화되거나 서로를 모방할 때 발생하는 5가지 관련 실패:

**단일 문화 붕괴.** 동일한 기본 모델 또는 학습 데이터 → 상관 오류. 세 에이전트가 LLM을 공유할 때, 그들은 환각도 공유합니다.

**순응 편향.** 에이전트가 틀렸을 때도 가장 큰 소리나 가장 확신하는 동료를 향해 조정합니다.

**불충분한 마음 이론.** 에이전트가 서로의 믿음을 모델링하지 못해 조정이 실패합니다 (레슨 18).

**혼합 동기 역학.** 부분적으로 일치하는 인센티브를 가진 에이전트가 타협 지점으로 표류하며, 이는 아무도 만족시키지 못합니다.

**연쇄적 신뢰성 실패.** 한 구성 요소의 오류 패턴이 종속 구성 요소의 오류 패턴을 유발합니다.

### 연쇄적 예시 — 재시도 폭풍

2026년 사고 패턴의 전형:

```
결제 서비스 실패 10% 요청
   ↓
주문 에이전트가 결제 재시도 (지수 백오프지만 순진함)
   ↓
각 재시도는 새로운 주문-재고 확인
   ↓
재고 서비스가 정상 부하 2배 감지
   ↓
재고 서비스 타임아웃 시작
   ↓
모든 주문이 재고 확인 재시도
   ↓
재고 서비스가 정상 부하 10배 감지
   ↓
클러스터 다운
```

해결책은 고전적입니다: **서킷 브레이커.** 다운스트림 오류율이 임계값을 초과하면 캐시된 또는 기본 결과로 단락합니다. 요청당 재시도 예산도 제한합니다.

서킷 브레이커는 수정 없이 분산 시스템에서 직접 차용한 몇 안 되는 다중 에이전트 실패 완화 방법 중 하나입니다.

### 메모리 오염 (재방문)

레슨 13에서: 한 에이전트의 환각이 공유 메모리 사실이 되고, 다운스트림 에이전트가 오염된 사실을 기반으로 추론합니다. MAST 용어로는 공유 메모리 계층의 검증 누락입니다.

점진적 정확도 감소가 증상입니다. 충돌이 아닌, 근본 원인 파악이 어려운 느린 드리프트가 발생합니다.

완화 방법: 추가 전용 로그, 출처, 쓰기 불가 검증자. 레슨 13에서 이미 다뤘습니다.

### STRATUS — 실패 감지 전용 에이전트

STRATUS (NeurIPS 2025)는 다음 배포 시 1.5배 완화 성공률 향상을 보고합니다:

- **감지 에이전트.** 증상 패턴(높은 불일치, 재시도 급증, 정확도 드리프트)을 감시합니다.
- **진단 에이전트.** 증상을 바탕으로 MAST 분류에서 가능한 근본 원인을 추론합니다.
- **검증 에이전트.** 완화 조치 적용 후 증상이 해결되었는지 확인합니다.

이것은 SRE 스타일의 사고 대응을 에이전트 시스템에 적용한 것입니다. 세 역할은 모두 특수 프롬프트가 있는 LLM 에이전트일 수 있습니다.

### 실패 모드 감사

2026년 모범 사례는 연간(또는 주요 릴리스별) 실패 모드 감사입니다:

1. **트레이스 샘플.** 실제 실행 트레이스 ~1000개 수집.
2. **분류.** 각 트레이스의 실패를 MAST + 집단사고 범주에 매핑.
3. **범주별 실패율 계산.** 어떤 범주가 시스템을 지배하는가?
4. **완화 방법 순위.** 어떤 수정이 가장 많은 실패를 제거하는가?
5. **2-3개 완화 방법 선택.** 구현; 다음 분기에 재감사.

구체적인 선택보다 규율이 더 중요합니다. 감사가 없으면 실패가 잡음으로 섞여 체계적으로 해결되지 않습니다.

### 시스템이 조용히 실패할 때

가장 위험한 실패 범주는 침묵하는 정확성 실패입니다. 시끄럽게 실패하는 시스템(충돌, 예외, 경고)은 모니터링할 수 있습니다. 그럴듯하지만 잘못된 출력을 생성하는 시스템은 예외 로그로 감지할 수 없습니다. 이것이 검증 누락이 건수는 21.30%이지만 실패당 비용이 가장 비싼 이유입니다.

투자 대상:
- 샘플 기반 인간 검토.
- 골든 데이터셋 회귀 테스트.
- 중요 출력에 대한 에이전트 간 상호 검증.

### 실패 vs 느린 실패

일부 실패는 즉각적이고, 일부는 느립니다. 즉각적 실패(타임아웃, 스키마 불일치, 인증 오류)는 감지하기 쉽습니다. 느린 실패(메모리 오염, 단일 문화 드리프트, 역할 모호성)는 감지 및 예방이 어렵습니다.

2026년 엔지니어링 기법: 느린 실패 프록시를 계측하여 드리프트를 가시적 오류 전에 포착합니다. 합의율, 재시도율, 출력 길이 분포, 연속 에이전트 버전 간 편집 거리 등이 유용한 프록시입니다.


## Ship It

프로덕션 환경에서의 실패 모드 규율:

- **분기별 MAST(MAST) 감사.** 연간이 아닌 분기별로 수행. 시스템 성장에 따라 카테고리 변경.
- **모든 곳에 서킷 브레이커(circuit breaker).** 모든 종속 서비스로의 아웃바운드 호출에 적용. 기본 개방 임계값은 5-10% 오류율.
- **골든 데이터셋(golden dataset).** 소규모 고품질 수동 검증 데이터셋. 매주 회귀 테스트 수행.
- **STRATUS(STRATUS) 트리오.** 프로덕션 모니터링용 감지(detection) + 진단(diagnosis) + 검증(validation) 에이전트. 초기에는 감지 에이전트만 사용; 증상이 복잡해질 때 진단 에이전트 추가.
- **실패 예산(failure budget).** 카테고리별 실패율에 대한 명시적 SLO(Service Level Objective). 예산 초과 시 출하 중지 논의 트리거.

## 연습 문제

1. `code/main.py`를 실행하세요. 서킷 브레이커가 재시도 폭주를 제한하는지 확인하세요. 실패 임계값을 다양하게 변경하며 트레이드오프를 관찰하세요.
2. **지연 실패 프록시**를 구현하세요: 3개의 병렬 에이전트 간 합의율. 합의율이 급격히 떨어지면 경고를 트리거하세요. 에이전트 출력을 점진적으로 상관시켜 단일 문화 드리프트를 시뮬레이션하세요.
3. Cemri et al. (arXiv:2503.13657)을 읽으세요. 그들의 7개 MAS 시스템 중 하나를 선택하고 상위 3개 실패 범주를 매핑하세요. 이들이 MAST가 예측하는 것과 어떻게 다른지 설명하세요.
4. Groupthink 논문(arXiv:2508.05687)을 읽으세요. 5가지 패턴 중 프로덕션 환경에서 감지하기 가장 어려운 패턴을 식별하세요. 프록시 메트릭을 제안하세요.
5. 특정 다중 에이전트 시스템(예: STRATUS 스타일)에 대해 감지-진단-검증 트리오를 설계하세요. 감지는 어떤 증상을 모니터링하나요? 진단은 어떤 완화 조치를 권장하나요? 검증은 어떻게 효과를 확인하나요?

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| MAST | "2026년 분류 체계" | Cemri 2025; 3개의 근본 범주 + 14가지 실패 하위 유형. |
| 명세 문제(Specification Problem) | "역할 모호성" | 작업 또는 역할이 충분히 정의되지 않음; 에이전트가 무엇을 해야 할지 모름. |
| 조정 실패(Coordination Failure) | "상태 표류" | 에이전트 간 통신 또는 동기화 장애. |
| 검증 격차(Verification Gap) | "아무도 확인하지 않음" | 독립적인 검증 없이 출력이 수용됨. |
| 집단사고 계열(Groupthink family) | "동질성 실패" | 단일문화, 순응, 부족한 마음 이론(ToM), 혼합 동기, 연쇄적 실패. |
| 단일문화 붕괴(Monoculture collapse) | "같은 모델, 같은 환각" | 공유된 기본 모델 또는 학습 데이터로 인한 상관 오류. |
| 재시도 폭풍(Retry storm) | "오류 증폭의 연쇄" | 하나의 실패가 재시도를 유발하여 다운스트림 부하를 증폭시킴. |
| 서킷 브레이커(Circuit breaker) | "오류율 기준 실패 신속 대응" | 오류율이 임계값을 초과하면 개방; 기본값으로 단락 처리. |
| STRATUS | "사고 대응 트리오" | 탐지 + 진단 + 검증 에이전트. 1.5배 완화 성공률. |
| 메모리 오염(Memory poisoning) | "환각이 전파됨" | 공유 메모리의 사실 정보가 오염됨; 다운스트림 에이전트가 오염된 정보로 추론. |
