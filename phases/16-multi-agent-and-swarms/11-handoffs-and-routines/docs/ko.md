# 핸드오프 및 루틴 — 상태 비저장 오케스트레이션

> OpenAI의 Swarm(2024년 10월)은 다중 에이전트 오케스트레이션을 두 가지 기본 요소로 단순화했습니다: **루틴**(시스템 프롬프트로서의 지시사항 + 도구)과 **핸드오프**(다른 에이전트를 반환하는 도구). 상태 머신도, 분기 DSL도 없이 LLM이 올바른 핸드오프 도구를 호출하여 라우팅합니다. OpenAI 에이전트 SDK(2025년 3월)는 프로덕션용 후속 버전입니다. Swarm 자체는 가장 깔끔한 개념적 참조로 남아 있으며, 전체 소스는 수백 줄에 불과합니다. 이 패턴은 API 표면이 대략 "에이전트 = 프롬프트 + 도구; 핸드오프 = 에이전트를 반환하는 함수"이기 때문에 널리 퍼졌습니다. 한계: 상태 비저장이므로 메모리는 호출자의 문제입니다.

**유형:** 학습 + 구축  
**언어:** Python (표준 라이브러리)  
**선수 지식:** 16단계 · 04 (기본 모델)  
**소요 시간:** ~60분

## 문제

모든 멀티에이전트 프레임워크는 자체 DSL(도메인 특화 언어)을 학습하도록 요구합니다: LangGraph의 노드와 엣지, CrewAI의 크루와 태스크, AutoGen의 GroupChat과 매니저 등. 이러한 DSL은 실제 추상화 계층이지만, 시스템을 필요 이상으로 무겁게 만듭니다.

Swarm은 반대 방향으로 접근합니다: 모델이 이미 보유한 **툴 호출(tool-calling)** 기능을 활용합니다. 작업 인계(hand-off)는 툴 호출로 처리되며, 오케스트레이터는 현재 대화를 주도하는 에이전트입니다. 상태 머신(state machine)은 에이전트들의 **시스템 프롬프트(system prompt)** 에 암시적으로 구현됩니다.

## 개념

### 두 가지 기본 요소

**루틴(Routine).** 에이전트의 역할과 사용 가능한 도구를 정의하는 시스템 프롬프트입니다. "당신은 트리아지 에이전트입니다. 사용자가 환불을 요청하면 환불 에이전트로 전달합니다."와 같은 범위 기반 지시문이라고 생각하면 됩니다.

**핸도프(Handoff).** 에이전트가 호출할 수 있는 도구로, 새로운 에이전트 객체를 반환합니다. Swarm 런타임은 에이전트 반환 값을 감지하고 다음 턴에 활성 에이전트를 전환합니다.

이것이 전체 추상화입니다.

```
def transfer_to_refunds():
    return refund_agent  # Swarm이 Agent 반환 → 활성 에이전트 전환

triage_agent = Agent(
    name="triage",
    instructions="사용자를 적절한 전문가에게 라우팅하세요.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

트리아지 에이전트의 시스템 프롬프트는 사용자 메시지에 따라 적절한 핸도프를 선택하도록 합니다. LLM의 도구 호출이 라우팅을 수행합니다.

### 왜 바이럴인가

- **작은 API.** 배울 개념이 두 가지뿐입니다.
- **모델이 이미 하는 일을 활용.** 도구 호출은 이미 모든 제공업체에서 프로덕션 등급입니다.
- **상태 머신 부담 없음.** 그래프를 설명할 필요 없이, 에이전트의 프롬프트가 누구에게 전달할지 설명합니다.

### 상태 비저장의 트레이드오프

Swarm은 실행 간에 명시적으로 상태 비저장입니다. 프레임워크는 실행 중 메시지 기록을 유지하지만, 아무것도 영구 저장하지 않습니다. 메모리, 연속성, 장기 작업 — 모두 호출자의 문제입니다.

프로덕션(OpenAI Agents SDK, 2025년 3월)에서는 이것이 주요 변경 사항 중 하나였습니다: SDK는 내장 세션 관리, 가드레일, 추적을 추가하면서 핸도프 기본 요소를 유지합니다.

### Swarm/핸도프가 적합한 경우

- **트리아지 패턴.** 일선 에이전트가 사용자를 전문가에게 라우팅합니다.
- **기술 기반 핸도프.** "작업에 코드가 필요하면 코더에게 호출, 연구가 필요하면 연구원에게 호출."
- **짧고 경계가 있는 대화.** 고객 지원, FAQ-to-티켓, 간단한 워크플로우.

### Swarm이 어려움을 겪는 경우

- **공유 메모리를 가진 장기 세션.** 핸도프는 대화 상태를 새 에이전트의 프롬프트와 기록으로 재설정합니다. 호출자 관리 메모리 없이는 에이전트 간 영구 상태가 없습니다.
- **병렬 실행.** 핸도프는 한 번에 하나씩 — 활성 에이전트가 전환됩니다. 병렬성은 호출자가 여러 Swarm 실행을 조율해야 합니다.
- **감사 및 재생.** 상태 비저장 실행은 정확히 재생하기 어렵습니다. LLM의 핸도프 선택은 결정적이지 않습니다.

### OpenAI Agents SDK (2025년 3월)

프로덕션 후속 버전은 다음을 추가합니다:

- **세션 상태.** 실행 간 영구 스레드.
- **가드레일.** 입력/출력 검증 훅.
- **추적.** 모든 도구 호출과 핸도프가 기록됩니다.
- **핸도프 필터.** 핸도프 시 어떤 컨텍스트가 전송되는지 제어.

핸도프 기본 요소는 유지되며, 프로덕션 편의 기능이 추가됩니다.

### Swarm vs GroupChat

둘 다 LLM 기반 라우팅을 사용하지만, **누가 다음을 선택하는지**에서 차이가 있습니다:

- GroupChat: 외부(함수 또는 LLM)의 선택기가 다음 발화자를 선택합니다.
- Swarm: 현재 에이전트가 핸도프 도구를 호출하여 후임자를 선택합니다.

Swarm은 "에이전트가 다음을 결정", GroupChat은 "관리자가 다음을 결정"입니다. Swarm의 결정은 활성 에이전트의 도구 호출에 있고, GroupChat의 결정은 `GroupChatManager`에 있습니다.

## 빌드하기

`code/main.py`는 Swarm을 처음부터 구현합니다: `Agent` 데이터 클래스, 핸드오프 메커니즘(툴이 `Agent`를 반환), 그리고 에이전트 전환을 감지하는 실행 루프.

데모: 트리아지 에이전트가 환불, 영업, 또는 지원 전문가로 라우팅합니다. 각 전문가는 자체 툴을 보유합니다. 실행 루프는 각 핸드오프를 출력합니다.

실행:

```
python3 code/main.py
```

## 사용 방법

`outputs/skill-handoff-designer.md`는 주어진 작업에 대한 핸드오프 토폴로지를 설계합니다: 어떤 에이전트가 존재하는지, 어떤 핸드오프를 호출할 수 있는지, 어떤 컨텍스트가 전송되는지.

## Ship It

체크리스트:

- **핸도프 로깅.** 모든 핸도프는 from-agent, to-agent, 컨텍스트 스냅샷을 포함한 트레이스 이벤트를 기록합니다.
- **컨텍스트 전송 규칙.** 핸도프 시 이동할 데이터 결정: 전체 히스토리(비용 높음), 최근 N개 메시지, 또는 요약.
- **핸도프 가드레일.** 다른 툴 권한을 가진 전문가로의 핸도프는 인증되어야 합니다 — 그렇지 않으면 프롬프트 주입으로 원치 않는 핸도프가 발생할 수 있습니다.
- **루프 감지.** 두 에이전트가 서로 반복적으로 전달하는 것은 일반적인 실패 사례입니다. 최근 K개 링 체크로 감지합니다.
- **대체 에이전트.** 핸도프 대상이 존재하지 않을 경우 안전한 기본값으로 폴백합니다.

## 연습 문제

1. `code/main.py`를 실행하고 환불 에이전트(refund agent)로 트리아지(triage)합니다. 두 번째 턴(turn)의 활성 에이전트가 환불(refund)인지 확인합니다.
2. 루프 감지 규칙(loop-detection rule)을 추가합니다: 동일한 두 에이전트가 연속으로 3번 핸드오프한 경우, 강제 종료(exit)를 유발합니다. 폴백(fallback) 방식을 설계하세요.
3. OpenAI Agents SDK 문서에서 핸드오프 필터(handoff filters) 관련 내용을 읽고, "핸드오프 시 요약(summarize-on-handoff)" 버전을 구현합니다: 아웃고잉 에이전트(outgoing agent)가 인커밍 에이전트(incoming agent)가 인계받기 전에 컨텍스트를 불릿 요약(bullet summary)으로 압축합니다.
4. Swarm 핸드오프와 GroupChatManager 선택기(selector)를 비교합니다. 어떤 패턴이 프롬프트 인젝션(prompt injection)을 더 악화시키며, 그 이유는 무엇인가요?
5. Swarm 쿡북(https://developers.openai.com/cookbook/examples/orchestrating_agents)을 읽고, Swarm이 명시적으로 내린 설계 결정 중 OpenAI Agents SDK가 변경하거나 유지한 사례를 하나 식별합니다.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 루틴(Routine) | "에이전트 프롬프트" | 시스템 프롬프트 + 도구 목록. 역할과 사용 가능한 핸드오프(hand-off)를 정의합니다. |
| 핸드오프(Handoff) | "다른 에이전트로 전달" | 활성 에이전트가 호출할 수 있는 도구로, 새로운 에이전트를 반환합니다. 런타임이 활성 에이전트를 전환합니다. |
| 상태 비저장(Stateless) | "실행 간 메모리 없음" | 스웜(Swarm)은 아무것도 지속하지 않습니다. 메모리는 호출자의 책임입니다. |
| 활성 에이전트(Active agent) | "지금 말하는 주체" | 현재 대화를 진행 중인 에이전트. 핸드오프가 이를 변경합니다. |
| 컨텍스트 전달(Context transfer) | "핸드오프 시 이동되는 것" | 수신 에이전트가 볼 수 있는 기록 정책: 전체, 최근 N개, 또는 요약본. |
| 핸드오프 루프(Handoff loop) | "에이전트들이 핑퐁" | 두 에이전트가 서로 계속 핸드오프를 반복하는 실패 모드. |
| OpenAI 에이전트 SDK | "프로덕션 스웜" | 2025년 3월 출시 예정 후속 버전; 핸드오프 기본 기능 위에 세션, 가드레일, 추적 기능을 추가합니다. |
| 핸드오프 필터(Handoff filter) | "전달 게이트" | SDK 기능으로, 핸드오프 경계에서 컨텍스트를 검사하고 수정합니다.
