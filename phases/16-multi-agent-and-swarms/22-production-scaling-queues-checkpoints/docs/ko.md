# 프로덕션 확장 — 큐, 체크포인트, 내구성

> 수천 개의 동시 실행으로 멀티 에이전트 시스템을 확장하려면 **내구성 있는 실행**이 필요합니다. LangGraph 런타임은 각 슈퍼스텝 후 `thread_id`를 키로 체크포인트를 기록합니다(기본값: Postgres). 작업자 충돌 시 임대권이 해제되고 다른 작업자가 재개합니다. 에이전트는 인간 입력을 기다리며 무기한 대기할 수 있습니다. **MegaAgent**(arXiv:2408.09955)는 에이전트별 생산자-소비자 큐를 Idle/Processing/Response 3가지 상태와 2계층 협업(그룹 내 채팅 + 그룹 간 관리자 채팅)으로 구현했습니다. **Fiber/async**는 LLM 스트리밍에서 작업당 스레드보다 우수합니다: 스레드는 토큰 대기 시 99% 유휴 상태이지만, 파이버는 I/O 시 협력적으로 양보합니다. 반론: Ashpreet Bedi의 "Scaling Agentic Software"는 부하가 증명될 때까지 **FastAPI + Postgres + 기타 없음**을 주장합니다 — 단순한 아키텍처는 예상보다 더 멀리 갑니다. 이 레슨에서는 내구성 있는 체크포인트 로그, 상태 전이가 있는 에이전트별 작업 큐, async-vs-thread 데모, 그리고 실용적인 "간단하게 시작" 규칙을 구현합니다.

**유형:** 학습 + 실습  
**언어:** Python (표준 라이브러리, `asyncio`, `sqlite3`)  
**선수 지식:** 16단계 · 09 (병렬 스웜 네트워크), 16단계 · 13 (공유 메모리)  
**소요 시간:** ~75분

## 문제

프로토타입 다중 에이전트 시스템은 인메모리 이벤트 루프로 한 대의 노트북에서 3개의 에이전트와 함께 작동합니다. 이를 프로덕션 환경으로 이전할 때 다음과 같은 문제가 발생합니다:

- 에이전트가 몇 시간 동안 실행되기도 합니다(긴 연구 작업, 인간 개입 대기 시간).
- 워커 프로세스가 충돌합니다. 재시작 시 상태가 손실됩니다.
- 최대 부하는 평균의 10배입니다. 수평 확장이 필요합니다.
- 사용자는 에이전트 실행 횟수로 비용을 지불합니다. 과금에는 정확히 한 번(exactly-once) 시맨틱스가 필요합니다.

인메모리 이벤트 루프는 이 중 어떤 것도 해결하지 못합니다. 내구성 있는 실행 계층이 필요합니다. 2026년 표준 옵션은 다음과 같습니다:

1. 체크포인트를 지원하는 워크플로 엔진(Temporal, LangGraph 런타임)
2. 상태 저장소를 갖춘 메시지 큐(Postgres + SQS/RabbitMQ)
3. 액터 모델 프레임워크(에이전트별 MegaAgent의 생산자-소비자 모델)
4. 직접 구현한 FastAPI + Postgres(Bedi의 주장)

이 강의에서는 각각의 미니어처 버전을 구축합니다.

## 개념

### 지속적 실행(durable execution), 패턴

지속적 실행 엔진은 각 "단계"(LangGraph 용어로는 슈퍼-단계) 후 전체 프로그램 상태를 영구 저장합니다. 충돌 시:

```
worker crashes mid-step
  -> lease timeout
  -> another worker picks up the thread_id
  -> resumes from last checkpoint
  -> no duplicate side effects
```

이 작동 요구사항:

- **직렬화 가능한 상태.** 모든 에이전트 상태는 영구 저장 가능해야 합니다. 활성 데이터베이스 연결을 가진 함수 클로저는 생존하지 못합니다.
- **결정적 재개.** 동일한 상태와 동일한 입력이 주어지면 에이전트는 동일한 액션을 생성합니다(또는 LLM 호출을 위해 외부 결정적 오라클에 위임).
- **멱등적 부작용.** 외부 호출(툴 호출, 결제)은 멱등적이어야 하거나 중복 제거 키를 사용해야 합니다.

LangGraph는 각 슈퍼-단계 후 체크포인트를 기록하며, Temporal은 각 활동 후 기록하고, Restate는 이벤트 소싱 저널을 사용합니다. 세 가지 모두 동일한 패턴을 구현합니다.

### LangGraph의 런타임

각 에이전트는 `thread_id`를 가지며, 상태는 타입이 지정된 딕셔너리입니다. 각 슈퍼-단계는 체크포인트 테이블에 행을 기록합니다. 재개 시 런타임은 처음부터가 아닌 마지막 체크포인트부터 재생합니다. 에이전트는 인간 입력을 기다리는 동안 `interrupt()`할 수 있으며, 런타임은 상태를 영구 저장하고 워커를 해제합니다. 입력이 도착하면 어떤 워커든 재개할 수 있습니다.

이것은 2026년 4월 기준 참조 생산 설계입니다.

### MegaAgent의 에이전트별 큐

arXiv:2408.09955는 확장 실험을 설명합니다: 하나의 클러스터에서 수천 개의 동시 에이전트. 아키텍처:

```
agent i:
  state ∈ {Idle, Processing, Response}
  in_queue   <- 에이전트 i로 주소 지정된 메시지
  out_queue  -> 응답 + 부작용

coordinators:
  intra-group chat  (동일 그룹 내 에이전트)
  inter-group admin chat  (고수준 라우팅)
```

두 계층 조정은 그룹 내 대화가 밀집하게 일어나도록 하면서 그룹 간 통신은 희소하게 유지합니다 — 수천 에이전트에서 비용을 선형으로 유지하는 패턴입니다.

### 비동기 vs 작업당 스레드

LLM 호출은 I/O 바운드입니다. 다음 토큰을 기다리는 스레드는 99% 시간 동안 유휴 상태입니다. 각 스레드는 ~1MB RAM을 소모하며, 10,000개의 동시 호출 시 스택만 10GB가 필요합니다.

파이버(Python `asyncio`, Go 고루틴, Rust `tokio`)는 I/O에서 협력적으로 양보합니다. 동일한 10,000개 호출은 프로세스 내에서 편안하게 실행됩니다. LLM-에이전트 규모에서 비동기는 최적화가 아닌 아키텍처입니다.

예외: CPU 바운드 후처리(임베딩, 토크나이저 트릭)는 여전히 스레드나 프로세스를 원합니다. I/O 계층과 CPU 계층을 분리하세요.

### 베디의 반론

"Scaling Agentic Software"(Ashpreet Bedi, 2026)는 대부분의 팀이 부하를 측정하기 전에 과도하게 엔지니어링한다고 주장합니다. 실용적인 기본값:

- FastAPI + Postgres.
- 각 에이전트 실행은 행이며, 낙관적 동시성으로 상태를 인플레이스 업데이트.
- `pg_notify` 또는 간단한 Celery 워커를 통한 백그라운드 작업.
- 애플리케이션 코드에 재시도 정책.

관리 가능한 작업에서 ~100개 이하의 동시 에이전트 실행 부하에는 이 정도면 충분합니다. 실패 측정 시 업그레이드하세요.

규칙: 단순 아키텍처로 해결할 수 없는 구체적인 문제가 발생했을 때 지속적 실행 프레임워크를 도입하세요. 조기 도입은 효과가 없는 의식에 시간을 낭비합니다.

### 정확히 한 번(exactly-once) 의미

유료 에이전트 실행에는 "정확히 한 번 효과적"(최소 한 번 전달 + 멱등적 소비자)이 필요합니다. 엔지니어링 기법:

- **실행별 중복 제거 키.** 모든 부작용 호출에 포함.
- **아웃박스 패턴.** 부작용은 먼저 테이블에 기록한 후 별도 프로세스가 실행합니다. 두 단계 모두 멱등적.
- **보상 트랜잭션.** 부작용 성공 시 추적 기록 실패 시 보상 작업 예약.

이들은 LLM-특정이 아닌 데이터베이스 엔지니어링 패턴입니다. LLM 비용은 LLM 호출이 느리다는 점뿐이며, 나머지는 표준 분산 시스템입니다.

### 레인보우 배포

Anthropic의 다중 에이전트 연구 시스템은 "레인보우 배포"를 사용합니다: 장기 실행 에이전트를 코드 배포 시마다 종료하지 않도록 여러 버전의 에이전트 런타임을 동시에 실행합니다. 트래픽 일부에 새 버전을 카나리 배포하고, 에이전트가 종료되면 이전 버전을 폐기합니다.

이는 장기 상태 저장 시스템의 표준이며, 2026년 적응은 에이전트가 몇 시간 동안 실행될 수 있으므로 배포 주기가 이를 수용해야 한다는 점입니다.

### 표준 생산 체크리스트

- 지속적 상태(체크포인트, 스냅샷 또는 아웃박스 + 재생 가능한 로그).
- 멱등적 부작용.
- LLM 호출을 위한 비동기 I/O 계층.
- 중복 제거와 함께 최소 한 번 전달.
- 상태 저장 작업을 위한 레인보우/카나리 배포.
- 관측 가능성: 에이전트별 추적, 슈퍼-단계 감사, 재시도 카운터.

## 빌드 및 실행

`code/main.py`는 다음을 구현합니다:

- `CheckpointStore` — 스레드 ID 키를 사용하는 SQLite 기반 체크포인트 로그. 각 슈퍼-스텝마다 행을 추가합니다.
- `run_with_checkpoint(agent, thread_id)` — 실행 중 충돌을 시뮬레이션합니다. 두 번째 작업자가 마지막 체크포인트부터 재개합니다.
- `AgentQueue` — 작은 작업 큐를 갖춘 에이전트별 Idle / Processing / Response 상태 머신.
- `demo_async_vs_threads()` — asyncio와 스레드를 통해 500개의 동시 시뮬레이션 "LLM 호출"을 실행합니다. 벽시계 시간과 피크 메모리(추정치)를 보고합니다.

실행:

```
python3 code/main.py
```

예상 출력: 시뮬레이션된 충돌 후 체크포인트 재개 성공; 비동기 버전은 500개의 동시 호출을 1초 미만에 처리; 스레드 버전은 수 초가 소요되며 동시 단위당 훨씬 더 많은 메모리를 사용합니다.

## 사용 방법

`outputs/skill-scaling-advisor.md`는 **내구성 있는 실행(durable-execution) 선택**에 대한 조언을 제공합니다: FastAPI + Postgres, LangGraph 런타임, Temporal, 또는 커스텀 솔루션. 부하(load), 상태 유지(state-retention) 요구사항, 배포 빈도(deploy frequency)에 따라 조정됩니다.

## Ship It

캐노니컬 프로덕션 강화:

- **간단하게 시작 (Bedi의 규칙).** 실패 측정이 필요할 때까지 FastAPI + Postgres 사용.
- **최적화 전 모든 것에 계측 적용.** 실행당 지연 시간 히스토그램, 단계별 시간, 재시도 횟수, 실패 분류.
- **부작용에 대한 아웃박스 패턴.** 특히 결제 및 외부 API 호출 시.
- **레인보우 배포.** 배포 중 진행 중인 에이전트 실행을 절대 중단하지 않음.
- **특정 문제 발생 시 내구성 실행 엔진(Temporal / LangGraph / Restate) 채택:** 1시간 길이의 인간 개입 대기, 지역 간 조정, 복잡한 재시도/보상 정책.
- **I/O 계층에는 비동기 사용.** CPU 바운드 후처리에는 스레드만 사용.

## 연습 문제

1. `code/main.py`를 실행하세요. 체크포인트 재개(checkpoint resume)가 작동하는지 확인하고, 비동기(async) vs 스레드 동시성(thread concurrency) 차이를 측정하세요.
2. **아웃박스(outbox)** 테이블을 구현하세요: 모든 도구 호출(tool call)은 먼저 아웃박스에 쓴 후, 별도의 고루틴(goroutine)/태스크(task)가 실행하도록 합니다. 도구 호출을 두 번 실행하여 멱등성(idempotency)을 검증하세요.
3. **레인보우 배포(rainbow deploy)**를 시뮬레이션하세요: 두 개의 동시 런타임 버전을 실행하고, 새 스레드 ID(thread_id)의 절반을 각각에 라우팅합니다. 이전 버전에서 진행 중인 스레드가 중단되지 않음을 확인하세요.
4. LangGraph의 런타임 문서(아래 링크 참조)를 읽으세요. 런타임 기능 중 FastAPI + Postgres로 직접 구현할 때 가장 오래 걸릴 기능을 식별하세요. 이것이 채택 이유가 될 수 있는지, 아니면 연기할 수 있는지 판단하세요.
5. MegaAgent(arXiv:2408.09955) 3절을 읽으세요. 두 계층 조정(그룹 내 + 그룹 간 관리자 채팅) 방식이 명시되어 있습니다. 이를 두 개의 큐 패밀리(queue family)를 가진 메시지 큐에 매핑하는 방법을 스케치하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| Durable execution | "프로그램 상태 지속" | 엔진이 각 슈퍼스텝 후 상태를 기록; 충돌 복구가 결정론적. |
| Super-step | "트랜잭션 경계" | 체크포인트 사이의 작업 단위. LangGraph 용어. |
| thread_id | "에이전트 실행 식별자" | 체크포인트와 재개 로직을 바인딩하는 키. |
| Idempotency | "재시도 안전" | 부작용 반복 시 한 번 시도한 것과 동일한 결과 생성. |
| Outbox 패턴 | "부작용 분리" | 테이블에 의도 기록; 별도 실행기가 수행 후 완료 표시. |
| At-least-once delivery | "중복 가능성" | 메시지 큐 의미론; 중복 방지 키로 소비자가 효과적 한 번 처리. |
| Rainbow 배포 | "버전 겹침" | 장기 실행 작업 중 여러 런타임 버전 동시 실행. |
| Async fiber | "협력적 양보" | 사용자 모드 동시성; I/O 바운드 부하에 스레드 대비 저렴. |
| Checkpoint | "상태 스냅샷" | 슈퍼스텝 경계의 직렬화된 상태; 재개 핵심 요소.
