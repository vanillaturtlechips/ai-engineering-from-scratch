# 도구 인터페이스 — 에이전트가 구조화된 입출력(I/O)이 필요한 이유

> 언어 모델은 토큰을 생성합니다. 프로그램은 액션을 수행합니다. 이 둘 사이의 간극을 메우는 것이 도구 인터페이스입니다: 모델이 액션을 요청하고 호스트가 이를 실행할 수 있도록 하는 계약입니다. 모든 2026년 스택 — OpenAI, Anthropic, Gemini의 함수 호출; MCP의 `tools/call`; A2A의 태스크 파트 — 은 동일한 4단계 루프의 다른 인코딩 방식입니다. 이 강의에서는 해당 루프의 이름을 명명하고 이를 실행하는 데 필요한 최소 장치를 보여줍니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, LLM 없음)
**선수 조건:** 11단계 (LLM 완성 API)
**소요 시간:** ~45분

## 학습 목표

- 텍스트만 생성할 수 있는 LLM이 왜 단독으로 현실 세계에 대한 행동을 취할 수 없는지 설명.
- 4단계 도구 호출 루프(설명 → 결정 → 실행 → 관찰)를 그리고 각 단계를 누가 소유하는지 명시.
- 도구 설명을 세 부분으로 작성: 이름, JSON 스키마 입력, 결정적 실행기 함수.
- 순수 도구(pure tool)와 부작용 도구(side-effecting tool)를 구분하고, 안전성 측면에서 이 분류가 중요한 이유 설명.

## 문제 정의

LLM은 다음 토큰에 대한 확률 분포를 출력합니다. 이것이 전체 출력 표면입니다. 만약 채팅 모델에 "방갈로르 현재 날씨는 어때?"라고 질문하면, 모델은 그럴듯한 문장을 생성할 수 있지만 날씨 API를 호출할 수는 없습니다. 생성된 문장은 우연히 맞을 수도 있고 3일 전 정보일 수도 있습니다.

이러한 격차를 해소하는 것이 도구 인터페이스의 목적입니다. 호스트 프로그램(에이전트 런타임, Claude Desktop, ChatGPT, Cursor 또는 사용자 정의 스크립트)은 모델에게 호출 가능한 도구 목록을 광고합니다. 모델은 특정 작업이 필요하다고 판단하면 도구 이름과 인수를 지정하는 구조화된 페이로드를 출력합니다. 호스트는 이 페이로드를 파싱하고 실제 도구를 실행한 후 결과를 다시 모델에 제공합니다. 이 루프는 모델이 더 이상 도구 호출이 필요하지 않다고 판단할 때까지 계속됩니다.

이 계약의 첫 번째 버전은 2023년 6월 OpenAI의 "functions" 파라미터로 출시되었습니다. Anthropic은 Claude 2.1에서 `tool_use` 블록을 도입했고, Gemini는 몇 달 후 `functionDeclarations`를 추가했습니다. 모든 제공업체는 이제 동일한 형태를 노출합니다: JSON 스키마로 타입이 지정된 도구 목록을 입력으로 받아 JSON 페이로드 형태의 도구 호출을 출력합니다. 모델 컨텍스트 프로토콜(2024년 11월)은 이 계약을 일반화하여 하나의 도구 레지스트리가 모든 모델을 지원하도록 했습니다. A2A(2026년 4월, v1.0)는 에이전트 간 위임을 위해 동일한 기본 요소를 계층화했습니다.

4단계 루프는 이 모든 것의 불변의 기본 구조입니다. 13단계의 나머지 모든 내용은 이 구조의 확장입니다.

## 개념

### 1단계: 설명

호스트는 각 도구를 세 가지 필드로 선언합니다.

- **이름.** 안정적이고 기계가 읽을 수 있는 식별자. `get_weather`와 같이 사용하며, "weather thing"은 사용하지 않습니다.
- **설명.** 한 단락으로 된 자연어 설명. "사용자가 특정 도시의 현재 상태에 대해 물을 때 사용합니다. 과거 데이터에는 사용하지 마세요."
- **입력 스키마.** 도구의 인수를 설명하는 JSON 스키마 객체(2020-12 초안).

모델은 이 목록을 받습니다. 현대 제공업체는 공급자별 템플릿을 사용하여 시스템 프롬프트에 이 선언을 직렬화하므로, 호출자는 구조화된 형식만 처리하면 됩니다.

### 2단계: 결정

사용자의 메시지와 사용 가능한 도구를 기반으로 모델은 세 가지 동작 중 하나를 선택합니다.

1. **텍스트로 직접 답변.** 도구 호출 없음.
2. **하나 이상의 도구 호출.** 구조화된 호출 객체를 출력합니다. `parallel_tool_calls: true`(OpenAI 및 Gemini 기본값, Anthropic은 선택적) 설정 시 모델은 한 턴에 여러 호출을 출력할 수 있습니다.
3. **거부.** 엄격한 모드 구조화된 출력은 호출 대신 타입이 지정된 `refusal` 블록을 생성할 수 있습니다.

도구 호출 페이로드에는 세 가지 안정적인 필드가 있습니다: 호출 `id`, 도구 `name`, JSON `arguments` 객체. id는 병렬 호출이 순서대로 반환되지 않을 때 호스트가 나중에 결과와 특정 호출을 연관시킬 수 있도록 존재합니다.

### 3단계: 실행

호스트는 호출을 수신하고 선언된 스키마에 대해 인수를 검증한 후 실행기를 실행합니다. 잘못된 인수는 모델이 필드를 허구로 생성했거나 잘못된 타입을 사용했음을 의미합니다. 이는 약한 모델에서 매우 흔한 실패 모드입니다. 프로덕션 호스트는 잘못된 인수에 대해 다음 세 가지 중 하나를 수행합니다: 빠르게 실패하고 오류를 모델에 표시, 제약 조건이 있는 파서로 JSON 복구, 또는 검증 오류를 프롬프트에 포함하여 모델 재시도.

실행기 자체는 일반 코드입니다. Python, TypeScript, 셸 명령어, 데이터베이스 쿼리 등. 일반적으로 문자열이지만 모든 JSON 값 또는 구조화된 콘텐츠 블록(MCP의 텍스트, 이미지 또는 리소스 참조)을 생성할 수 있습니다. 결과는 직렬화 가능해야 합니다.

### 4단계: 관찰

호스트는 도구 결과를 대화에 추가(일치하는 `id`를 가진 `tool` 역할 메시지로)하고 모델을 다시 호출합니다. 이제 모델은 도구 출력을 컨텍스트로 가지고 최종 답변을 생성하거나 추가 호출을 요청할 수 있습니다. 이는 모델이 호출을 중단하거나 호스트가 반복 횟수에 대한 안전 한계에 도달할 때까지 계속됩니다.

### 신뢰 분할

도구는 안전에 중요한 두 가지 유형으로 나뉩니다.

- **순수(Pure).** 읽기 전용, 결정적, 부작용 없음. `get_weather`, `search_docs`, `get_current_time`. 추측적으로 호출해도 안전합니다.
- **중대한(Consequential).** 상태 변경, 비용 발생, 사용자 데이터 접근. `send_email`, `delete_file`, `execute_trade`. 게이트가 필요합니다.

Meta의 2026년 "Rule of Two" 에이전트 보안 규칙은 단일 턴에서 다음 세 가지 중 최대 두 가지만 결합할 수 있다고 명시합니다: 신뢰할 수 없는 입력, 민감한 데이터, 중대한 동작. 도구 인터페이스는 이 규칙을 강제하는 곳입니다. 호출 거부, 사용자 확인 요구 또는 범위 승격으로 강제할 수 있습니다. 전체 보안 장은 Phase 13 · 15, 에이전트 수준 권한 정책은 Phase 14 · 09를 참조하세요.

### 루프의 위치

| 컨텍스트 | 설명 주체 | 결정 주체 | 실행 주체 |
|---------|---------------|-------------|--------------|
| 단일 턴 함수 호출 (OpenAI/Anthropic/Gemini) | 앱 개발자 | LLM | 앱 개발자 |
| MCP | MCP 서버 | MCP 클라이언트를 통한 LLM | MCP 서버 |
| A2A | 에이전트 카드 게시자 | 호출 에이전트 | 호출된 에이전트 |
| 웹 브라우저 (함수 호출 에이전트) | 브라우저 확장 / WebMCP | LLM | 브라우저 런타임 |

모든 곳에서 동일한 4단계. 열 이름은 변경되지만 구조는 동일합니다.

### 왜 모델에 JSON을 출력하도록 프롬프트하지 않는가?

"모델에 JSON으로 답하도록 요청"은 함수 호출 이전의 패턴입니다. 프론티어 모델에서는 약 5~15%, 더 작은 모델에서는 훨씬 더 높은 실패율을 보입니다. 실패 모드에는 누락된 중괄호, 후행 쉼표, 허구로 생성된 필드, 잘못된 타입 등이 포함됩니다. 이후 JSON 복구 패스, 재시도 또는 제약 조건이 있는 디코더가 필요합니다.

네이티브 함수 호출은 세 가지 이유로 더 낫습니다. 첫째, 제공업체는 정확한 호출 형태에 대해 모델을 종단 간 훈련시켜 엄격한 모드에서 유효한 JSON 비율을 98~99%까지 높입니다. 둘째, 호출 페이로드는 자유 텍스트가 아닌 자체 프로토콜 슬롯에 위치하므로 도구 호출이 사용자 표시 답변으로 유출되지 않습니다. 셋째, 제공업체는 제약 조건이 있는 디코딩(OpenAI의 엄격한 모드, Anthropic의 `tool_use`, Gemini의 `responseSchema`)으로 스키마 준수를 강제합니다. 출력은 검증이 보장됩니다.

Phase 13 · 02는 세 제공업체 API를 나란히 비교합니다. Phase 13 · 04는 구조화된 출력을 심층적으로 다룹니다.

### 회로 차단기

루프는 모델이 호출을 중단하거나 호스트가 최대 턴 수에 도달할 때 종료됩니다. 프로덕션 호스트는 이를 5~20턴으로 설정합니다. 그 이상은 모델이 종료할 수 없는 루프에 갇혔을 가능성이 큽니다. Claude Code는 기본값을 20, OpenAI Assistants는 10, Cursor의 에이전트 모드는 25로 설정합니다.

대안인 무제한 루프는 "에이전트가 밤새 API 호출로 $400를 소비했다"는 사후 분석으로 6개월마다 나타납니다. 제한 없이 출시하지 마세요.

Phase 14 · 12는 오류 복구 및 자가 치유를 심층적으로 다룹니다. Phase 17은 프로덕션 속도 제한을 다룹니다.

### Phase 13의 진행 방향

- 레슨 02~05는 제공업체 수준의 도구 호출 표면을 다듬습니다.
- 레슨 06~14는 루프를 MCP로 일반화합니다.
- 레슨 15~18은 적대적 서버, 적대적 사용자, 인증되지 않은 원격 인증 표면에 대해 루프를 방어합니다.
- 레슨 19~22는 에이전트 간 협업, 관측 가능성, 라우팅 및 패키징으로 패턴을 확장합니다.
- 레슨 23은 모든 프리미티브를 사용하는 완전한 생태계를 출시합니다.

남은 모든 레슨은 이 4단계 루프의 확장입니다. 불변으로 유지하세요.

## 사용 방법

`code/main.py`는 LLM 없이 4단계 루프를 실행합니다. 가짜 "결정자(decider)" 함수는 사용자 메시지에 대한 패턴 매칭으로 모델을 시뮬레이션합니다. 실행기(executor), 스키마 검증기(schema validator), 관찰 단계(observe-step) 하네스는 실제 구현입니다. 실행 시 중간 상태를 출력 가능한 전체 요청/응답 흐름을 확인한 후, 이후 레슨에서 가짜 결정자를 실제 제공자로 교체할 수 있습니다.

확인 사항:

- 도구 레지스트리(tool registry)는 각 도구당 이름(name), 설명(description), 스키마(schema), 실행기 참조(executor reference) 4개 필드를 보유합니다.
- 검증기(validator)는 표준 라이브러리(stdlib)로 작성된 최소 JSON 스키마 하위 집합(타입, 필수 필드, enum, 최소/최대값)입니다. 13단계 · 04에서는 더 완전한 버전이 제공됩니다.
- 루프는 반복 횟수를 5회로 제한합니다. 프로덕션 에이전트는 이러한 회로 차단기(circuit breaker)가 반드시 필요합니다.

## Ship It

이 레슨은 `outputs/skill-tool-interface-reviewer.md`를 생성합니다. 초안 도구 정의(이름 + 설명 + 스키마 + 실행기 개요)가 주어졌을 때, 스킬은 루프 적합성(loop fitness)을 감사합니다: 이름이 기계-안정적인지, 설명이 완전한 사용 요약인지, 스키마가 JSON Schema 2020-12를 올바르게 사용하는지, 순수(pure)-대-결과적(consequential) 분류가 명시적으로 되어 있는지를 검토합니다.

## 연습 문제

1. `code/main.py`에 `get_stock_price(ticker)`라는 네 번째 도구를 추가하세요. 설명을 "사용자가 티커로 현재 주가를 요청할 때 사용합니다. 과거 가격 또는 시장 요약에는 사용하지 마세요."로 작성하세요. 하네스를 실행하고 티커를 언급한 쿼리가 새 도구로 라우팅되는지 확인하세요.

2. 스키마 검증기를 깨트리세요. 필수 필드가 누락된 `arguments` 객체를 가진 호출을 전달하고, 실행 전에 호스트가 이를 거부하는지 확인하세요. 그런 다음 알 수 없는 추가 필드가 있는 호출을 전달하세요. 호스트가 거부해야 할까요, 무시해야 할까요? 안전성으로 선택을 정당화하세요.

3. 하네스의 각 도구를 순수(pure) 또는 결과적(consequential)으로 분류하세요. 필요한 레지스트리 항목에 `consequential: true` 플래그를 추가하고, 루프를 수정하여 결과적 도구가 선택될 때마다 "사용자와 확인" 라인을 출력하세요. 이는 모든 프로덕션 호스트에 필요한 확인 게이트의 형태입니다.

4. 좋아하는 클라이언트(Claude Desktop, Cursor, ChatGPT 또는 사용자 정의 스택)에 대해 위의 공급자 열 테이블을 채운 4단계 루프를 종이에 그려보세요. 13단계 · 06의 MCP 특정 변형과 상호 참조하세요.

5. OpenAI의 함수 호출 가이드를 처음부터 끝까지 읽으세요. 요청에는 있지만 여기에 제시된 4단계 루프에는 없는 하나의 필드를 식별하세요. 이 필드가 추가하는 것과 필수적이기보다는 편리한 이유를 설명하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| 도구(Tool) | "모델이 호출할 수 있는 것" | 이름 + JSON-스키마 타입 입력 + 실행기(executor) 함수의 삼중항 |
| 함수 호출(Function calling) | "네이티브 도구 사용" | 산문 대신 구조화된 도구 호출을 생성하기 위한 공급자 수준의 API 지원 |
| 도구 호출(Tool call) | "모델의 실행 요청" | 모델이 생성한 `id`, `name`, `arguments` 필드를 포함한 JSON 페이로드 |
| 도구 결과(Tool result) | "도구가 반환한 것" | 실행기(executor)의 출력을 매칭된 id와 함께 `tool` 역할 메시지로 감싼 것 |
| 병렬 도구 호출(Parallel tool calls) | "한 번에 여러 호출" | 하나의 모델 턴에서 여러 호출 객체를 포함하며, id로 독립적/순서 지정 가능 |
| 엄격 모드(Strict mode) | "보장된 JSON" | 선언된 스키마에 대해 모델 출력을 강제 검증하는 제약된 디코딩 |
| 순수 도구(Pure tool) | "읽기 전용 도구" | 부작용 없음; 재실행 안전 |
| 결과적 도구(Consequential tool) | "액션 도구" | 외부 상태 변경; 게이트, 감사 또는 사용자 확인 필요 |
| 4단계 루프(Four-step loop) | "도구 호출 사이클" | 설명(describe) → 결정(decide) → 실행(execute) → 관찰(observe) |
| 호스트(Host) | "에이전트 런타임" | 도구 레지스트리를 보유하고 모델을 호출하며 실행기를 실행하는 프로그램 |
