# MCP 클라이언트 구축 — 발견, 호출, 세션 관리

> 대부분의 MCP 콘텐츠는 서버 튜토리얼을 제공하며 클라이언트에 대해서는 간략히 언급합니다. 클라이언트 코드는 프로세스 생성, 기능 협상, 여러 서버에 걸친 도구 목록 병합, 샘플링 콜백, 재연결, 네임스페이스 충돌 해결 등 복잡한 오케스트레이션이 이루어지는 곳입니다. 이 강의에서는 세 가지 다른 MCP 서버를 하나의 평면적인 도구 네임스페이스로 통합하는 멀티서버 클라이언트를 구축합니다.

**유형:** 실습
**언어:** Python (표준 라이브러리, 멀티서버 MCP 클라이언트)
**선수 조건:** 13단계 · 07 (MCP 서버 구축)
**소요 시간:** ~75분

## 학습 목표

- MCP 서버를 자식 프로세스로 생성하고, `initialize`를 완료한 후 `notifications/initialized`를 전송.
- 서버별 세션 상태(기능(capabilities), 도구 목록(tool list), 마지막으로 확인한 알림 ID) 유지.
- 여러 서버의 도구 목록을 하나의 네임스페이스로 병합하고 충돌 처리.
- 도구 호출을 해당 서버로 라우팅하고 응답을 재조립.

## 문제 정의

실제 에이전트 호스트(Claude Desktop, Cursor, Goose, Gemini CLI)는 여러 MCP 서버를 동시에 로드합니다. 사용자는 파일 시스템 서버, Postgres 서버, GitHub 서버를 동시에 실행할 수 있습니다. 클라이언트의 작업:

1. 각 서버를 생성(spawn)합니다.
2. 각각 독립적으로 핸드셰이크를 수행합니다.
3. 각 서버에 대해 `tools/list`를 호출하고 결과를 평탄화(flatten)합니다.
4. 모델이 `notes_search`를 방출(emit)할 때, 병합된 네임스페이스에서 이를 조회하고 적절한 서버로 라우팅합니다.
5. 어떤 서버에서의 알림(`tools/list_changed`)도 차단 없이 처리합니다.
6. 전송 실패 시 재연결을 수행합니다.

이 모든 것을 직접 구현하는 것이 "장난감" 수준의 시스템과 "서비스 가능한" 시스템을 구분하는 기준입니다. 공식 SDK는 이를 래핑하지만, 정신 모델(mental model)은 반드시 자신의 것이어야 합니다.

## 개념

### 자식 프로세스 생성

`subprocess.Popen`을 `stdin=PIPE, stdout=PIPE, stderr=PIPE`로 사용. `bufsize=1`로 설정하고 텍스트 모드를 사용하여 라인 단위로 읽기. 각 서버는 하나의 프로세스이며, 클라이언트는 서버당 하나의 `Popen` 핸들을 보유.

### 서버별 세션 상태

서버당 하나의 `Session` 객체가 다음을 보유:

- `process` — Popen 핸들.
- `capabilities` — 서버가 `initialize` 시 선언한 기능.
- `tools` — 마지막 `tools/list` 결과.
- `pending` — 요청 ID를 응답 대기 중인 프로미스/퓨처에 매핑하는 맵.

요청은 본질적으로 비동기적. 서버 A가 호출 중일 때 서버 B로 전송된 `tools/call`은 차단되지 않아야 함. 큐를 사용하는 스레드 또는 asyncio를 사용.

### 병합된 네임스페이스

클라이언트가 집계된 도구 목록을 볼 때 이름이 충돌할 수 있음. 두 서버가 모두 `search`를 노출할 수 있음. 클라이언트는 다음 세 가지 옵션 중 선택 가능:

1. **서버 이름 접두사 추가.** `notes/search`, `files/search`. 명확하지만 미관상 좋지 않음.
2. **묵묵히 덮어쓰기.** 나중에 로드된 서버의 `search`가 이전 것을 덮어씀. 위험함; 충돌을 숨김.
3. **충돌 거부.** 두 번째 서버 로드를 거부하고 사용자에게 알림. 보안 민감 호스트에 가장 안전.

Claude Desktop은 서버 이름 접두사 방식을 사용. Cursor는 명확한 오류와 함께 충돌 거부 방식을 채택. VS Code MCP도 서버 이름 접두사 방식을 사용.

### 라우팅

병합 후, 디스패치 테이블이 `tool_name -> session`을 매핑. 모델이 이름으로 호출을 생성하면 클라이언트는 세션을 찾아 해당 서버의 stdin에 `tools/call` 메시지를 쓴 후 응답을 대기.

### 샘플링 콜백

서버가 `initialize` 시 `sampling` 기능을 선언했다면, `sampling/createMessage`를 보내 클라이언트에게 LLM 실행을 요청할 수 있음. 클라이언트는 다음을 수행해야 함:

1. 샘플이 해결될 때까지 해당 서버에 대한 추가 요청을 차단하거나, 구현이 동시성을 지원하면 파이프라인 처리.
2. LLM 공급자 호출.
3. 응답을 서버로 다시 전송.

레슨 11에서 샘플링 종단 간 처리를 다룸. 이 레슨에서는 완성도를 위해 스텁 처리.

### 알림 처리

`notifications/tools/list_changed`는 `tools/list`를 다시 호출해야 함을 의미. `notifications/resources/updated`는 사용 중인 리소스라면 다시 읽어야 함을 의미. 알림은 응답을 생성해서는 안 됨 — 확인 응답(ack)을 시도하지 말 것.

흔한 클라이언트 버그: 알림 메시지가 스트림에 남아 있는 동안 `tools/call`에서 읽기 루프를 차단. 모든 메시지를 큐에 푸시하는 백그라운드 리더 스레드를 사용; 메인 스레드가 큐에서 메시지를 꺼내 디스패치.

### 재연결

전송 실패 가능성: 서버 충돌, OS가 프로세스 종료, stdio 파이프 손상. 클라이언트는 stdout에서 EOF를 감지하고 세션을 죽은 상태로 처리. 옵션:

- 서버를 조용히 재시작하고 재핸드셰이크. 순수 읽기 전용 서버에 적합.
- 사용자에게 실패를 알림. 사용자 세션이 보이는 상태 유지 서버에 적합.

13단계 · 09에서 Streamable HTTP 재연결 시맨틱스를 다룸; stdio는 더 간단.

### Keepalive 및 세션 ID

Streamable HTTP는 `Mcp-Session-Id` 헤더를 사용. stdio에는 세션 ID가 없음 — 프로세스 정체성 자체가 세션. Keepalive 핑은 선택 사항; stdio 파이프는 비활성 상태에서 끊기지 않음.

## 사용 방법

`code/main.py`는 세 개의 시뮬레이션된 MCP 서버를 서브프로세스로 생성하고, 각각과 핸드셰이크를 수행한 후, 도구 목록을 병합하고, 도구 호출을 적절한 서버로 라우팅합니다. "서버"는 실제로 장난감 응답기(실제 LLM 없음)를 실행하는 다른 Python 프로세스입니다. 실행하면 다음을 확인할 수 있습니다:

- 각각 고유한 기능 집합을 가진 세 개의 초기화.
- 세 개의 `tools/list` 결과가 7개 도구의 네임스페이스로 병합됨.
- 도구 이름 기반의 라우팅 결정.
- 네임스페이스 접두사를 통한 충돌 방지.

주목할 부분:

- `Session` 데이터 클래스는 서버별 상태를 깔끔하게 보관합니다.
- 백그라운드 리더 스레드는 메인 스레드를 차단하지 않고 stdout의 모든 줄을 비동기로 처리합니다.
- 디스패치 테이블은 단순한 `dict[str, Session]`입니다.
- 충돌 처리는 명시적으로 이루어집니다: 두 서버가 동일한 이름을 선언할 경우, 나중에 선언된 서버에 접두사가 추가되어 이름이 변경됩니다.

## Ship It

이 레슨은 `outputs/skill-mcp-client-harness.md`를 생성합니다. MCP 서버(선언적 목록: 이름, 명령어, 인수)가 주어지면, 스킬은 해당 서버들을 실행하는 하네스를 생성하고, 도구 목록을 병합하며, 충돌 해결 기능을 갖춘 라우팅 함수를 제공합니다.

## 연습 문제

1. `code/main.py`를 실행하고 서버 생성 로그를 확인하세요. 시뮬레이션된 서버 프로세스 중 하나를 SIGTERM으로 종료한 후 클라이언트가 EOF를 감지하고 해당 세션을 죽은 상태로 표시하는 방식을 관찰하세요.

2. 네임스페이스 접두사(prefixing)를 구현하세요. 두 서버가 `search`를 노출할 때 두 번째 서버의 엔드포인트를 `<server>/search`로 이름 변경하세요. 디스패치 테이블을 업데이트하고 도구 호출이 올바르게 라우팅되는지 확인하세요.

3. 서버 재시작을 위한 연결 풀 스타일의 백오프(backoff)를 추가하세요: 연속 실패 시 지수 백오프 적용, 최대 30초로 제한, 3회 실패 후 사용자에게 알림 전송.

4. 100개의 동시 MCP 서버를 지원하는 클라이언트 설계를 스케치하세요. 간단한 디스패치 딕셔너리를 대체할 데이터 구조는 무엇인가요? (힌트: 접두사 네임스페이싱을 위한 트라이(trie) 구조, 서버당 도구 수 메트릭 추가)

5. 공식 MCP Python SDK로 클라이언트를 포팅하세요. SDK는 `stdio_client`와 `ClientSession`을 래핑합니다. 다중 서버 라우팅을 유지하면서 코드 라인을 ~200줄에서 ~40줄로 축소해야 합니다.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|----------------|------------------------|
| MCP 클라이언트 | "에이전트 호스트" | 서버를 생성하고 도구 호출을 조율하는 프로세스 |
| 세션 | "서버별 상태" | 기능, 도구 목록, 보류 중인 요청 관리 정보 |
| 병합된 네임스페이스 | "하나의 도구 목록" | 모든 활성 서버에 걸친 평탄화된 도구 이름 집합 |
| 네임스페이스 충돌 | "두 서버가 같은 도구 사용" | 클라이언트는 중복 도구에 접두사 추가, 거부, 또는 선착순으로 처리해야 함 |
| 라우팅 | "이 호출을 누가 처리하나?" | 도구 이름에서 소유 서버로의 디스패치 |
| 백그라운드 리더 | "비차단 stdout" | 서버 stdout을 큐로 비우는 스레드 또는 작업 |
| 샘플링 콜백 | "LLM-as-a-service" | 서버로부터 `sampling/createMessage`를 처리하는 클라이언트 핸들러 |
| `notifications/*_changed` | "원시 데이터 변경" | 클라이언트가 다시 발견하거나 다시 읽어야 함을 알리는 신호 |
| 재연결 정책 | "서버가 죽었을 때" | 전송 실패 시 재시작 시맨틱스 |
| Stdio 세션 | "프로세스 = 세션" | 세션 ID 없음; 자식 프로세스 수명이 세션과 동일 |
