# LLM 라우팅 레이어 — LiteLLM, OpenRouter, Portkey

> 공급자 종속은 비용이 많이 듭니다. 서로 다른 도구 호출 작업 부하는 서로 다른 모델에 적합합니다. 라우팅 게이트웨이는 하나의 API 표면, 재시도, 장애 조치, 비용 추적, 가드레일을 제공합니다. 2026년을 주도하는 세 가지 아키타입은 다음과 같습니다: LiteLLM(오픈소스 자체 호스팅), OpenRouter(관리형 SaaS), Portkey(프로덕션 등급, 2026년 3월 오픈소스화). 이 강의에서는 결정 기준을 설명하고 stdlib 라우팅 게이트웨이를 직접 구성해 봅니다.

**유형:** 학습
**언어:** Python (stdlib, 라우팅 + 장애 조치 + 비용 추적기)
**선수 조건:** 13단계 · 02(함수 호출), 13단계 · 17(게이트웨이)
**소요 시간:** ~45분

## 학습 목표

- 자체 호스팅(self-hosted), 관리형(managed), 프로덕션 등급(production-grade) 라우팅 옵션을 구분합니다.
- 정의된 우선순위 순서에 따라 공급자 실패 시 재시도하는 폴백 체인(fallback chain)을 구현합니다.
- 공급자 간 요청별 비용(cost)과 토큰 사용량(token usage)을 추적합니다.
- 주어진 프로덕션 제약 조건에 따라 LiteLLM, OpenRouter, Portkey 중 적절한 솔루션을 선택합니다.

## 문제 상황

프로바이더 라우팅이 중요한 시나리오:

1. **비용.** Claude Sonnet은 Haiku보다 3배 비쌉니다. 트리아지(triage) 작업에는 Haiku로 충분하지만, 합성(synthesis) 작업에는 Sonnet이 가치가 있습니다. 요청별로 라우팅합니다.

2. **장애 조치.** OpenAI에 장애가 발생한 시간대가 있습니다. 모든 요청이 실패합니다. 재배포 없이 Anthropic으로 자동 폴백(fallback)하기를 원합니다.

3. **지연 시간.** 실시간 채팅 UI는 첫 토큰까지의 빠른 응답 시간이 필요합니다. 배치 요약기(batch summarizer)는 그렇지 않습니다. 지연 시간 SLA(서비스 수준 계약)에 따라 라우팅합니다.

4. **규정 준수.** EU 사용자는 EU 리전에 머물러야 합니다. 지역별로 라우팅합니다.

5. **실험.** 동일한 워크로드에서 두 모델을 A/B 테스트합니다. 테스트 버킷별로 라우팅합니다.

모든 통합을 위해 이 모든 것을 직접 코딩하는 것은 반복적입니다. 라우팅 게이트웨이는 OpenAI 호환 API를 하나 제공하고 나머지는 처리합니다.

## 개념

### OpenAI 호환 프록시 형태

모두가 OpenAI 형태를 사용합니다. 라우팅 게이트웨이는 `/v1/chat/completions`를 노출하고 OpenAI 스키마를 수락한 후 내부적으로 Anthropic / Gemini / Cohere / Ollama / 기타 서비스로 프록시합니다. 클라이언트는 신경 쓸 필요가 없습니다.

### 모델 별칭

`claude-3-5-sonnet-20251022` 대신 코드에서는 `our_smart_model`이라고 작성합니다. 게이트웨이는 별칭을 실제 모델에 매핑합니다. Anthropic이 Claude 4를 출시하면 서버 측에서 별칭을 변경합니다. 코드는 변경할 필요가 없습니다.

### 폴백 체인

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

게이트웨이는 이 설정을 구성 파일에서 정의합니다. 재시도 횟수는 예산에 따라 계산되므로 폴백 캐스케이드가 비용을 폭발적으로 증가시키지 않습니다.

### 시맨틱 캐싱

동일하거나 거의 동일한 프롬프트는 공급자 대신 캐시를 히트합니다. 반복되는 에이전트 루프에서 30~60%의 절감 효과를 얻을 수 있습니다. 키는 임베딩 기반이며, 거의 동일한 프롬프트는 동일한 캐시 슬롯을 공유합니다.

### 가드레일

게이트웨이 수준:

- **PII 삭제.** 프롬프트 전송 전 정규식 또는 ML 기반 패스 적용.
- **정책 위반.** 금지된 콘텐츠가 포함된 프롬프트 거부.
- **출력 필터.** 유출 가능성이 있는 완성본 제거.

Portkey와 Kong은 의견 기반 가드레일을 제공합니다. LiteLLM은 선택적으로 제공합니다.

### 키별 속도 제한

하나의 API 키 = 하나의 팀. 키별 예산을 통해 한 팀이 공유 할당량을 소모하지 못하도록 합니다. 대부분의 게이트웨이가 이를 지원합니다.

### 자체 호스팅 vs 관리형 트레이드오프

| 요소 | LiteLLM (자체 호스팅) | OpenRouter (관리형) | Portkey (프로덕션) |
|------|----------------------|----------------------|----------------------|
| 코드 | 오픈소스, Python | 관리형 SaaS | 오픈소스 (2026년 3월) + 관리형 |
| 설정 | 프록시 배포 | 가입 | 둘 중 하나 |
| 공급자 | 100+ | 300+ | 100+ |
| 청구 | 자체 키 | OpenRouter 크레딧 | 자체 키 |
| 관측 가능성 | OpenTelemetry | 대시보드 | 전체 OTel + PII 삭제 |
| 최적 대상 | 완전한 제어가 필요한 팀 | 빠른 프로토타이핑 | 규정 준수 프로덕션 |

LiteLLM은 SRE 팀이 있고 데이터 주권을 원할 때 유리합니다. OpenRouter는 단일 구독과 인프라 없이 빠르게 시작할 때 유리합니다. Portkey는 가드레일과 규정 준수를 기본으로 필요로 할 때 유리합니다.

### 비용 추적

모든 요청에는 `provider`, `model`, `input_tokens`, `output_tokens`가 포함됩니다. 게이트웨이가 유지하는 가격표에서 모델별 토큰당 가격을 가져와 곱합니다. 사용자/팀/프로젝트별 집계 가능.

### MCP + 라우팅

게이트웨이는 LLM 호출과 MCP 샘플링 요청을 모두 라우팅할 수 있습니다. 샘플링 요청의 `modelPreferences`가 특정 모델을 선호하면 게이트웨이는 적절한 백엔드로 변환합니다. Phase 13 · 17 (MCP 게이트웨이)과 이 레슨의 라우팅 게이트웨이가 하나의 서비스로 통합되는 지점입니다.

### 라우팅 전략

- **정적 우선순위.** 목록의 첫 번째 모델 사용; 오류 시 폴백.
- **로드 밸런싱.** 라운드 로빈 또는 가중치 기반.
- **비용 인식.** 지연 시간/품질을 충족하는 가장 저렴한 모델 선택.
- **지연 시간 인식.** 최근 N분 동안 가장 빠른 모델 선택.
- **작업 인식.** 프롬프트 분류기가 코딩은 한 모델로, 요약은 다른 모델로 라우팅.

## 사용 방법

`code/main.py`는 약 150줄로 라우팅 게이트웨이를 구현합니다: OpenAI 형식의 요청을 수락하고, 공급업체별 스텁으로 변환하며, 우선순위 폴백 체인을 실행하고, 요청별 비용을 추적하며, 입력에 대해 PII(개인 식별 정보) 삭제 필터를 적용합니다. 정상 요청, 주 공급업체 장애 시 폴백 트리거, 삭제 필터로 포착된 PII 유출의 세 가지 시나리오로 실행할 수 있습니다.

주목할 부분:

- `ROUTES` 딕셔너리: 별칭 -> 우선순위 정렬된 구체 공급업체 목록.
- 폴백 루프는 5xx 오류 발생 시 재시도합니다.
- 비용 추적기는 토큰 사용량을 모델별 요금으로 곱해 계산합니다.
- PII 삭제기는 전달 전 SSN(사회보장번호) 형식의 패턴을 제거합니다.

## Ship It

이 레슨은 `outputs/skill-routing-config-designer.md`를 생성합니다. 워크로드 프로필(지연 시간, 비용, 규정 준수)이 주어지면, 스킬은 LiteLLM / OpenRouter / Portkey를 선택하고 라우팅 구성을 생성합니다.

## 연습 문제

1. `code/main.py`를 실행하세요. 정전 시나리오를 트리거하고, 폴백이 두 번째 공급자로 전환되는지 확인하며 비용이 정확히 할당되는지 검증하세요.

2. 시맨틱 캐싱 추가: 프롬프트의 SHA256 해시를 조회 키로 사용; 캐시 적중 시 즉시 반환. 반복 호출 시 비용 절감 효과를 측정하세요.

3. 프롬프트 분류기 추가: "code ..." 프롬프트는 지능성을 우선시하는 별칭으로, "summarize ..." 프롬프트는 속도를 우선시하는 별칭으로 라우팅하세요.

4. 팀별 예산 설계: 각 팀에 월별 지출 한도 설정; 한도 초과 시 게이트웨이가 요청 거부. 적용 단위(요청별 또는 윈도우 기반)를 선택하세요.

5. LiteLLM, OpenRouter, Portkey 문서를 나란히 비교하세요. 각 플랫폼이 제공하는 다른 두 플랫폼에 없는 고유 기능 하나씩을 명시하세요.
