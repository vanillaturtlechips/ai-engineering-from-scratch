# 비동기 및 호그와일드! 추론

> 추측 디코딩(Phase 10 · 15)은 하나의 시퀀스 내에서 토큰을 병렬화합니다. 다중 에이전트 프레임워크는 전체 시퀀스에 걸쳐 병렬화를 수행하지만 명시적인 조정(투표, 하위 작업 분할)을 강제합니다. 호그와일드! 추론(Rodionov et al., arXiv:2504.06261)은 다른 방식을 사용합니다: 동일한 LLM의 N개 인스턴스를 공유 키-값 캐시에 대해 병렬로 실행합니다. 각 작업자는 다른 작업자가 생성한 토큰을 즉시 확인할 수 있습니다. QwQ, DeepSeek-R1과 같은 현대 추론 모델은 미세 조정 없이도 이 공유 캐시를 통해 자체적으로 조정할 수 있습니다. 이 접근법은 실험적이지만, 추측 디코딩과 직교하는 완전히 새로운 추론 병렬화 축을 열어줍니다. 이 강의에서는 표준 Python 라이브러리로 두 작업자 호그와일드! 시뮬레이터를 구현하고, 공유 캐시 협력이 기존 모델의 추론 능력에서 어떻게 나타나는지 설명합니다.

**유형:** 구축
**언어:** Python (표준 라이브러리)
**선수 조건:** Phase 10 · 12 (추론 최적화), Phase 10 · 15 (추측 디코딩)
**소요 시간:** ~60분

## 학습 목표

- 세 가지 일반적인 병렬 LLM 토폴로지(투표(voting), 하위 작업(sub-task), Hogwild!)를 설명하고, 각각이 대상으로 하는 문제 유형을 명시한다.
- Hogwild!의 핵심 설정: 여러 작업자(multiple workers), 하나의 공유 KV 캐시(shared KV cache), 자기 프롬프트(self-prompting)를 통한 발생적 조정(emergent coordination)을 기술한다.
- 작업자 수 `N`, 작업 수준 병렬성 `p`, 조정 오버헤드 `c`의 함수로 Hogwild!의 벽시계 시간(wall-time) 속도 향상(speedup)을 계산한다.
- 장난감 문제(toy problem)에서 두 작업자(two-worker) Hogwild! 시뮬레이터를 구현하고 발생적 작업 분할(emergent task division)을 관찰한다.

## 문제 정의

현대 LLM(Large Language Model)은 긴 추론 체인을 생성하여 어려운 문제를 해결합니다. 5000토큰의 단계별 논리는 일반적이며, 깊은 수학 문제에서는 수만 토큰이 생성되기도 합니다. 70B 모델에서 35토큰/초의 디코딩 속도를 기준으로 50,000토큰은 24분이 소요됩니다. 이는 대화형 모델과는 거리가 먼 성능입니다.

추측 디코딩(Speculative Decoding, Phase 10 · 15)은 단일 시퀀스 내에서 병렬화를 통해 3-5배의 속도 향상을 제공합니다. 그러나 그 이상의 성능 향상은 자기회귀적 디코딩의 순차적 의존성이라는 근본적인 한계에 부딪힙니다. 각 새 토큰은 이전 모든 토큰에 의존하기 때문입니다.

명백한 질문: 시퀀스 간 병렬화가 가능한가? 동일한 모델의 여러 복사본을 동일한 문제에 대해 실행하고, 이들이 협력하며 작업을 분할할 수 있을까?

기존 연구: 투표 앙상블(N개의 모델 실행 후 다수결로 답변 선택), 사고 트리(분기된 추론 경로 재결합), 다중 에이전트 프레임워크(각 에이전트에게 하위 작업 할당 및 조정자 사용) 등이 있습니다. 이들은 특정 작업 영역에서 유용하지만, 투표 규칙, 분기-프루닝 논리, 에이전트 간 메시징 프로토콜과 같은 명시적 조정 메커니즘을 도입해야 합니다.

Hogwild! 추론은 다른 접근법을 취합니다. N개의 작업자가 단일 KV 캐시를 공유합니다. 각 작업자는 다른 작업자가 생성한 토큰을 즉시 자신의 컨텍스트처럼 볼 수 있습니다. 작업자들은 훈련이나 파인튜닝 없이도 작업을 분할하는 방법을 스스로 파악합니다. 현대 추론 모델(QwQ, DeepSeek-R1, Claude-family 추론 모드)은 공유 캐시를 읽고 "작업자 2가 이미 기본 사례를 처리했으므로 귀납적 단계를 작업하겠다"와 같은 판단을 내릴 수 있습니다.

속도 향상은 작업 부하에 따라 다르며 2026년 4월 기준으로 실험적 단계입니다. 그러나 이 아이디어는 추론 병렬화의 새로운 축을 열었다는 점에서 주목할 가치가 있습니다.

## 개념

### 설정

N개의 워커 프로세스를 초기화하고, 모두 동일한 LLM을 실행합니다. 워커별 KV 캐시 대신 **하나의 공유 캐시**를 유지합니다. 워커 `i`가 토큰 `t_j`를 생성하면, 해당 토큰은 공유 캐시의 다음 위치에 기록됩니다. 워커 `k`가 다음 단계를 수행할 때, 캐시의 현재 상태(모든 N개 워커가 지금까지 생성한 내용 포함)를 읽습니다.

단계 실행 시, 워커들은 토큰 기록을 위해 경쟁합니다. 워커별 위치 인덱스는 없으며, 캐시는 단일로 성장하는 시퀀스입니다. 순서는 기록 도착 시간에 따라 결정됩니다.

### 조정(coordination)이 발생하는 이유

워커들은 공유 프롬프트를 가집니다. 일반적으로 다음과 같은 내용입니다: "당신은 이 문제를 함께 해결하는 N개의 인스턴스 중 하나입니다. 각 인스턴스는 공유 메모리를 읽고 다른 인스턴스가 작성한 내용을 볼 수 있습니다. 중복 작업을 피하세요." 프롬프트와 공유 캐시만으로도 충분합니다. 추론 모델은 캐시를 읽고, 문제의 어떤 부분이 이미 시도되었는지 인지하며, (항상 그렇지는 않지만) 미탐구 부분으로 전환합니다.

Hogwild! 논문(Rodionov et al., 2025)은 다음과 같은 관찰을 보고합니다:

- 워커들은 계획을 수립하고 캐시를 통해 다른 워커들과 통신합니다.
- 워커들은 다른 워커들의 추론 오류를 발견하고 지적합니다.
- 워커들은 계획이 실패하면 적응하여 대안을 제안합니다.
- 중복 확인을 요청받으면, 워커들은 중복을 감지하고 전환합니다.

이 모든 것은 파인튜닝이 필요하지 않습니다. 모델의 기존 추론 능력에서 발생하는 현상입니다.

### 명명법

논문 이름은 비동기 업데이트 옵티마이저인 Hogwild! SGD(Recht et al., 2011)를 변형한 것입니다. 유사점: SGD의 비동기 워커들은 공유 파라미터 벡터에 기록하며, Hogwild! 추론의 워커들은 공유 KV 캐시에 기록합니다. 둘 다 동기화 보장 대신 경험적 수렴에 의존합니다.

### RoPE가 이를 가능하게 함

로터리 위치 임베딩(RoPE, Su et al. 2021)은 Q와 K 벡터의 회전을 통해 위치 정보를 인코딩합니다. 위치가 회전으로 표현되고 고정된 오프셋이 아니기 때문에, 토큰의 위치가 이동해도 KV 캐시 항목을 재계산할 필요가 없습니다. 워커 `i`가 위치 `p`에 공유 캐시에 기록하면, 다른 워커들은 해당 위치를 읽을 때 캐시된 항목을 직접 사용할 수 있습니다 — 재회전 불필요.

학습된 위치 또는 절대 위치 모델에서는 Hogwild!가 모든 동시 기록 시 캐시 무효화가 필요합니다. RoPE는 캐시를 안정적으로 유지합니다.

### 벽시계 시간 계산

`T_serial`을 단일 워커가 문제를 해결하는 데 걸리는 시간이라고 합시다. `p`를 작업 수준 병렬화 가능 비율, `c`를 단계별 조정 오버헤드(확장된 캐시 읽기, 기록할 내용 결정)로 정의합니다.

- 단일 워커 시간: `T_serial`.
- 조정이 무료일 때 N-워커 Hogwild! 시간: `T_serial * ((1 - p) + p / N)`. 클래식 암달(Amdahl) 법칙.
- 조정 오버헤드 포함 시: `T_serial * ((1 - p) + p / N) + c * steps_per_worker`.

워커가 생산적이려면, `c`가 단계별 디코딩 시간에 비해 작아야 합니다. 5k+ 토큰을 생성하는 추론 모델에서는 수백 토큰의 조정 오버헤드를 감수하고도 이득을 볼 수 있습니다. 짧은 채팅 작업에서는 조정 오버헤드가 지배적이며, Hogwild!는 직렬 처리보다 나쁩니다.

### 구체적 예시

추론 문제: 10k 토큰의 사고 사슬(chain-of-thought). 문제의 `p = 0.7`이 병렬화 가능 내용(다른 증명 전략, 다른 사례 분석)이고, `c = 200` 토큰의 조정 오버헤드가 워커당 발생한다고 가정합니다. `N = 4` 워커일 때:

- 직렬 시간: 10,000 디코딩 단계.
- Hogwild! 시간: 10,000 * (0.3 + 0.7 / 4) + 200 * 4 = 10,000 * 0.475 + 800 = 5,550 디코딩 단계.
- 속도 향상: 10,000 / 5,550 = 1.8x.

이는 적은 수치입니다. 더 긴 추론 문제(50k 토큰)에서는 조정 오버헤드가 상쇄되고 속도 향상이 2.5-3x까지 증가합니다. Hogwild!는 자연스럽게 멀티스레드 코드를 작성할 수 있는 언어의 스레드 수준 병렬 처리와 동등한 추론 기법입니다.

### Hogwild!를 사용할 때

- 수천 토큰의 긴 추론 문제 중 독립적인 하위 목표로 병렬화 가능한 경우.
- 단계별로 생각하도록 훈련된 추론 모델. 비추론 모델은 자체 조정이 잘 되지 않습니다.
- 공유 캐시와 N개 워커 프로세스를 수용할 충분한 VRAM이 있는 단일 노드 배포. 캐시는 공유되지만, 각 워커는 자체 활성화 메모리를 가집니다.

### 사용하지 말아야 할 때

- 짧은 대화형 채팅. 조정 오버헤드가 지배적입니다.
- 병렬화되지 않는 작업(단일 선형 증명, 단일 컴파일). N=1이 최대입니다.
- 비추론 모델. 조정이 발생하지 않습니다.
- 멀티 노드 배포. 공유 캐시는 매우 빠른 워커 간 동기화가 필요합니다. 노드 내부는 괜찮지만, 노드 간은 지연 문제가 발생합니다.

### 실험적 현황

2026년 4월 기준, Hogwild!는 오픈소스 PyTorch 구현체가 있는 연구 방법입니다. 프로덕션 채택은 이루어지지 않았습니다. 세 가지 장애물:

1. 동시 프로세스 간 공유 KV 캐시 관리는 복잡한 엔지니어링 문제입니다.
2. 발생적 조정은 작업 의존적이며, 벤치마크는 아직 구축 중입니다.
3. 속도 향상은 이미 존재하는 추측 디코딩(speculative decoding)에 비해 적으며, 둘을 결합할 수 있지만 결합 엔지니어링은 또 다른 계층입니다.

알아둘 가치가 있습니다. 실험해 볼 가치가 있습니다. 아직 제품에 베팅할 가치는 없습니다.

## 빌드하기

`code/main.py`는 장난감 Hogwild! 시뮬레이터를 구현합니다:

- 두 개의 작업자 프로세스, 각각 결정론적 "LLM"으로 여러 토큰 범주(work-token, observe-token, coordinate-token) 중 하나를 알려진 확률로 생성합니다.
- 두 작업자가 읽고 쓸 수 있는 공유 캐시(토큰 목록).
- 간단한 조정 논리: 작업자가 다른 작업자가 이미 특정 범주에서 충분한 work-token을 생성했음을 확인하면 다른 범주를 선택합니다.

시뮬레이터는 고정된 단계 예산으로 실행되며 다음을 보고합니다:

- 생성된 총 work-token 수.
- 총 벽 시계 시간(작업자 단계 수).
- 단일 작업자 대비 유효 속도 향상.
- 어떤 작업자가 어떤 토큰을 작성했는지에 대한 추적.

### 단계 1: 공유 캐시

두 작업자가 추가하는 목록. 실제 구현에서는 간단한 잠금(Python `threading.Lock`)을 사용하지만, 시뮬레이션에서는 카운터로 대체합니다.

### 단계 2: 작업자 루프

각 작업자는 매 단계마다 다음을 수행합니다:

- 현재 공유 캐시를 읽습니다.
- 캐시에 이미 있는 내용을 기반으로 작성할 토큰 범주를 결정합니다.
- 하나의 토큰을 작성합니다.

### 단계 3: 조정 휴리스틱

범주 X에 이미 K개의 토큰이 캐시에 있고 작업자의 의도된 범주가 X인 경우, 작업자는 범주 Y로 전환합니다. 이는 "이미 처리된 내용을 인지하고 다른 작업을 수행한다"는 추론 모델 동작을 간단히 모방한 것입니다.

### 단계 4: 측정된 속도 향상

시뮬레이터를 N=1 작업자와 N=2 작업자로 실행하고 동일한 총 단계 예산을 사용합니다. 생성된 work-token 수를 계산합니다. N=2는 조정 기반 작업 분할로 인해 약 1.5-1.8배 더 많은 work-token을 생성해야 합니다.

### 단계 5: 조정 기능 테스트

조정 휴리스틱의 민감도를 낮춥니다. 다시 실행하고 좋은 조정이 없을 때 N=2가 중복된 토큰을 생성하여 속도 향상이 1 미만으로 떨어지는 것을 관찰합니다. 이는 논문의 관찰과 일치합니다: 작업자가 자체 조정 능력을 갖춘 추론 모델일 때만 이 트릭이 작동합니다.

## 사용 방법

2026년 4월 기준 Hogwild!의 프로덕션 통합은 연구 등급입니다. Yandex/HSE/IST의 레퍼런스 구현은 PyTorch 기반이며 DeepSeek-R1 및 QwQ 모델에서 단일 노드 멀티프로세스 설정을 대상으로 합니다.

실용적 도입 경로:

1. 추론 작업 워크로드를 프로파일링합니다. 탐색적(여러 전략, 사례 분석, 검색) 토큰과 선형 토큰의 비율을 측정합니다.
2. 탐색이 우세하면 두 작업자 Hogwild! 실험을 실행합니다. 벽시계 시간 개선도를 측정합니다.
3. 개선도가 1.3배 미만이면 조정 중심 영역에 있습니다. 단일 작업자로 되돌립니다.
4. 개선도가 1.5배 초과면 N=4로 확장하고 다시 측정합니다. 일반적으로 N=4-8에서 한계 수익이 발생합니다.

추측 디코딩과 결합: 각 Hogwild! 작업자는 독립적으로 추측 디코딩을 사용할 수 있습니다. 두 속도 향상은 대략 곱해져, 3배 추측 디코딩과 1.8배 Hogwild!가 순진(naive) 단일 작업자 디코딩 대비 효과적인 5.4배를 제공합니다.

## Ship It

이 레슨은 `outputs/skill-parallel-inference-router.md`를 생성합니다. 추론 작업 부하 프로필(토큰 예산, 작업 병렬화 프로필, 모델 패밀리, 배포 대상)이 주어졌을 때, 투표(voting), 사고 트리(tree-of-thought), 다중 에이전트(multi-agent), Hogwild!, 추측 디코딩(speculative decoding) 전략들 사이에서 라우팅을 수행합니다.

## 연습 문제

1. 기본 설정으로 `code/main.py`를 실행하세요. 동일한 벽 시간(wall time)에서 N=2 Hogwild! 구성이 N=1 기준선보다 더 많은 작업 토큰(work-token)을 생성하는지 확인하세요.

2. 조정 휴리스틱(coordination heuristic)의 강도를 줄이세요(`coordination_weight=0.1`로 설정). 다시 실행한 후 속도 향상이 사라지는 것을 보여주세요. 이유를 설명하세요: 작업자(workers)가 조정하지 못할 때 노력을 중복하기 때문입니다.

3. `p=0.8, c=500` 및 N=4 작업자(workers)로 50k-토큰 추론 작업에 대한 Hogwild! 예상 속도 향상을 계산하세요. `p=0.3, c=200` 및 N=4로 1k-토큰 채팅 작업도 동일하게 계산하세요. 왜 하나는 성공이고 다른 하나는 실패인지 설명하세요.

4. Hogwild! 논문 4절(예비 평가)을 읽으세요. 저자들이 보고한 두 가지 실패 모드(failure modes)를 식별하세요. 더 나은 조정 프롬프트(coordination prompt)가 각각을 어떻게 완화할 수 있는지 설명하세요.

5. 장난감 모델에서 Hogwild!과 추측 디코딩(speculative decoding)을 결합하세요: 각 작업자는 내부적으로 2-토큰 추측 디코딩을 사용합니다. 곱셈적 속도 향상(multiplicative speedup)을 보고하세요. 두 작업자가 동일한 공유 캐시 접두사(shared-cache prefix)를 확장하려고 할 때 어떤 장부 관리(bookkeeping) 문제가 발생하는지 설명하세요.

## 주요 용어

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| Hogwild! | "병렬 작업자, 공유 캐시" | 동일한 LLM의 N개 인스턴스가 하나의 공유 KV 캐시로 동시에 실행; 자기 프롬프트(self-prompting)를 통한 발생적 조정 |
| 공유 KV 캐시 | "조정 매체" | 모든 작업자가 읽고 쓰는 단일 확장 KV 버퍼; 작업자 간 즉시 토큰 가시성 제공 |
| 발생적 조정 | "학습 불필요" | 추론 가능 LLM은 공유 캐시를 읽고 미세 조정이나 명시적 프로토콜 없이 작업을 분할할 수 있음 |
| 조정 오버헤드(c) | "방향 설정에 소요된 토큰" | 확장된 캐시 읽기 및 작업 결정 시 작업자당 비용; 총 디코딩 시간 대비 작아야 함 |
| 병렬화 가능 비율(p) | "병렬 실행 가능 작업" | 작업 수준 병렬성: 전체 작업 중 본질적으로 순차적이지 않은 부분의 비율 |
| RoPE는 Hogwild!를 가능하게 함 | "회전식 위치는 이동 불변" | 위치가 회전으로 표현되므로 공유 캐시에 쓸 때 이전 토큰 재계산이 필요 없음 |
| 투표 앙상블 | "N회 실행, 다수 선택" | 가장 간단한 병렬 추론 토폴로지; 분류 작업에 유용, 장문 추론에는 덜 적합 |
| 사고 트리 | "분기 및 가지치기" | 여러 분기를 탐색하고 가지치기하는 추론 전략; 명시적 조정 로직 필요 |
| 다중 에이전트 프레임워크 | "하위 작업 할당" | 각 에이전트에게 역할 부여; 조정자가 오케스트레이션; 프로토콜 오버헤드 큼 |
