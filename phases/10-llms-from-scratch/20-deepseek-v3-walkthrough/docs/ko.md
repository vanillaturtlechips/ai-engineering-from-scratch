# DeepSeek-V3 아키텍처 분석

> Phase 10 · Lesson 14에서는 모든 오픈 모델이 조정하는 6가지 아키텍처 조절 장치를 명명했습니다. DeepSeek-V3(2024년 12월 출시, 총 6710억 개 파라미터, 활성 370억 개)는 이 6가지 모두를 조정하고 여기에 4가지를 추가로 도입했습니다: Multi-Head Latent Attention(MLA), 보조 손실 없는 부하 분산, Multi-Token Prediction(MTP), DualPipe 훈련. 이 레슨은 DeepSeek-V3의 아키텍처를 위에서 아래로 분석하며 공개된 설정(config)으로부터 모든 파라미터 수를 유도합니다. 끝까지 학습하면 6710억/370억 비율이 왜 최적의 선택인지, 그리고 MLA와 MoE가 함께 사용될 때 최전선(frontier)에서 각각 단독으로 사용하는 것보다 왜 더 효과적인지 설명할 수 있게 됩니다.

**유형:** 학습
**언어:** Python (표준 라이브러리, 파라미터 계산기)
**선수 지식:** Phase 10 · 14(오픈 모델 워크스루), Phase 10 · 17(NSA), Phase 10 · 18(MTP), Phase 10 · 19(DualPipe)
**소요 시간:** ~75분

## 학습 목표

- DeepSeek-V3 설정(config)을 위에서 아래로 읽고, 6개의 GPT-2 조절 매개변수(knob)와 4개의 DeepSeek 전용 추가 기능을 기준으로 각 필드를 설명하라.
- 총 파라미터 수(671B), 활성 파라미터 수(37B), 그리고 각각에 기여하는 구성 요소를 유도하라.
- 128k 컨텍스트에서 MLA(Mixture of Experts Layer)의 KV 캐시 풋프린트(footprint)를 계산하고, 동일한 활성 파라미터 수를 가진 GQA(Gather-Query-Attention)를 사용하는 밀집 모델(dense model)과 비교하라.
- 4개의 DeepSeek 전용 혁신 기술(MLA, MTP, 보조 손실 없는 라우팅(auxiliary-loss-free routing), DualPipe)을 명시하고, 각각이 아키텍처/학습 스택의 어떤 부분을 대상으로 하는지 설명하라.

## 문제 정의

DeepSeek-V3는 Llama 계열과 의미 있게 다른 아키텍처를 가진 최초의 프론티어 오픈 모델입니다. Llama 3 405B는 "6개의 노브가 조정된 GPT-2"인 반면, DeepSeek-V3는 6개의 노브에 추가로 4개의 노브가 더 있는 GPT-2입니다. Llama 3 구성(config)을 읽는 것은 DeepSeek 구성 읽기를 위한 워밍업이지만, 깊은 구조 — 어텐션 블록의 형태, 라우팅 로직, 학습 시 목표 함수 — 는 충분히 달라서 별도의 설명이 필요합니다.

이를 학습하는 보상: DeepSeek-V3의 오픈 웨이트 공개는 오픈 모델에서 "프론티어 능력"의 정의를 바꿨습니다. 이 아키텍처는 2026년 많은 학습 실행에서 복사하는 청사진입니다. 프론티어 LLM 학습 또는 추론과 관련된 모든 역할에서 이를 이해하는 것은 기본 요건입니다.

## 개념

### 다시 한번 불변 코어

DeepSeek-V3는 여전히 자기회귀적(autoregressive)입니다. 여전히 디코더 블록을 쌓습니다. 각 블록은 여전히 어텐션(attention) + MLP + 두 개의 RMSNorm을 포함합니다. 여전히 MLP에서 SwiGLU를 사용합니다. 여전히 RoPE를 사용합니다. 사전 정규화(pre-norm) 방식입니다. 가중치 공유 임베딩(weight-tied embeddings)을 사용합니다. 모든 Llama 또는 Mistral과 동일한 베이스라인입니다.

### 변화: GQA 대신 MLA

Phase 10 · 14에서 GQA는 Q 헤드 그룹 간 K와 V를 공유하여 KV 캐시를 축소한다는 것을 알고 있습니다. Multi-Head Latent Attention(MLA)은 더 나아갑니다: K와 V는 공유 저랭크 잠재 표현(`kv_lora_rank`)으로 압축된 후, 헤드별로 실시간으로 압축 해제됩니다. KV 캐시는 잠재 표현만 저장합니다. 일반적으로 토큰당 레이어당 512개의 부동소수점(1024개가 아닌 512개)을 사용합니다.

128k 컨텍스트에서 MLA를 사용하는 DeepSeek-V3(레이어당 토큰당 하나의 공유 잠재 `c^{KV}`; K와 V는 이 잠재에서 업-투영을 통해 유도되며, 이는 후속 행렬 곱에 흡수될 수 있음):

```
kv_cache = num_layers * kv_lora_rank * max_seq_len * bytes_per_element
         = 61 * 512 * 131072 * 2
         = 7.6 GB
```

가상의 GQA 베이스라인(Llama 3 70B 형태, 8개의 KV 헤드, 헤드 차원 128)은 다음과 같습니다:

```
kv_cache = 2 * 61 * 8 * 128 * 131072 * 2
         = 30.5 GB
```

MLA는 128k 컨텍스트에서 Llama-3-70B 스타일 GQA 캐시보다 4배 더 작습니다.

트레이드오프: MLA는 어텐션 계산(헤드별)마다 압축 해제 단계를 추가합니다. 추가 계산량은 대역폭 절약에 비해 작습니다. 긴 컨텍스트 추론에 순수익입니다.

### 라우팅: 보조 손실 없는 부하 분산

MoE 라우터는 각 토큰을 처리할 상위-k 전문가를 결정합니다. 순진한 라우터는 너무 많은 작업을 소수의 전문가에 집중시켜 다른 전문가를 유휴 상태로 만듭니다. 표준 해결책: 부하 불균형을 페널티로 주는 보조 손실 항을 추가합니다. 이는 작동하지만 주 작업 성능을 약간 저하시킵니다.

DeepSeek-V3는 보조 손실 없는 방식을 도입합니다. 전문가별 편향 항이 라우터 로짓에 추가되고, 훈련 중에 간단한 규칙에 따라 조정됩니다: 전문가 `e`가 과부하 상태면 `bias_e`를 감소시키고, 저부하 상태면 증가시킵니다. 추가 손실 항이 없습니다. 훈련은 깨끗하게 유지됩니다. 전문가 부하는 균형을 유지합니다.

주 손실에 미치는 영향: 측정 가능한 영향 없음. MoE 아키텍처에 미치는 영향: 더 깨끗하며, 튜닝할 보조 손실 하이퍼파라미터가 없습니다.

### MTP: 더 밀도 높은 훈련 + 무료 초안

Phase 10 · 18에서 DeepSeek-V3는 D=1 MTP 모듈을 추가하여 두 위치 앞의 토큰을 예측한다는 것을 알고 있습니다. 추론 시, 훈련된 모듈은 80% 이상의 수용률을 가진 추측적 디코딩(speculative decoding) 초안으로 재활용됩니다. 훈련 시, 각 은닉 상태는 D+1=2개의 타겟에 대해 감독되어 더 밀도 높은 신호를 제공합니다.

파라미터: 671B 메인 모델 위에 14B. 오버헤드: 2.1%.

### 훈련: DualPipe

Phase 10 · 19에서 DualPipe는 순방향 및 역방향 청크를 노드 간 all-to-all 통신과 겹치게 하는 양방향 파이프라인임을 알고 있습니다. DeepSeek-V3의 2,048-H800 규모에서 1F1B가 파이프라인 버블로 인해 잃었을 약 245k GPU-시간을 회수합니다.

### 구성, 필드별 설명

다음은 단순화된 DeepSeek-V3 구성입니다:

```
hidden_size: 7168
intermediate_size: 18432   (밀집 MLP 은닉 크기, 처음 몇 레이어에 사용)
moe_intermediate_size: 2048 (전문가 MLP 은닉 크기)
num_hidden_layers: 61
first_k_dense_layers: 3    (처음 3 레이어는 밀집 MLP 사용)
num_attention_heads: 128
num_key_value_heads: 128   (MLA 하에서 num_heads와 형식적으로 동일하지만,
                           실제 압축은 kv_lora_rank에 있음)
kv_lora_rank: 512          (MLA 잠재 차원)
num_experts: 256            (MoE 전문가 수/블록)
num_experts_per_tok: 8      (상위-8 라우팅)
shared_experts: 1           (블록당 항상 켜진 공유 전문가)
max_position_embeddings: 163840
rope_theta: 10000.0
vocab_size: 129280
mtp_module: 1               (깊이 1에 1개의 MTP 모듈)
```

해석:

- `hidden_size=7168`: 임베딩 차원.
- `num_hidden_layers=61`: 총 블록 깊이.
- `first_k_dense_layers=3`: 처음 3개 블록은 18432 크기의 밀집 MLP를 사용합니다. 나머지 58개는 MoE를 사용합니다.
- `num_attention_heads=128`: 128개의 쿼리 헤드.
- `kv_lora_rank=512`: K와 V는 이 잠재 차원으로 압축된 후 헤드별로 압축 해제됩니다.
- `num_experts=256, num_experts_per_tok=8`: 각 MoE 블록은 256개의 전문가를 가지며, 상위 8개를 라우팅합니다.
- `shared_experts=1`: 256개의 라우팅된 전문가 외에, 1개의 항상 켜진 전문가가 모든 토큰에 기여합니다. 모든 토큰이 신뢰할 수 있는 기본값을 얻도록 하는 "밀집 바닥"으로 생각하십시오.
- `moe_intermediate_size=2048`: 각 전문가의 MLP 은닉 크기. 밀집 MLP보다 작습니다. 왜냐하면 256개의 전문가가 있기 때문입니다.

### 파라미터 계산

전체 계산은 `code/main.py`에 있습니다. 주요 내용:

- 임베딩: `vocab * hidden = 129280 * 7168 = ~0.93B`.
- 처음 3개 밀집 블록: MLA를 사용한 어텐션(블록당 ~144M) + 밀집 MLP(블록당 ~260M) + 정규화. 총 약 1.2B.
- 58개 MoE 블록: MLA를 사용한 어텐션(~144M) + 256개 전문가(각각 30M) + 1개 공유 전문가(30M) + 정규화. 모든 전문가 포함 시 블록당 총 ~7.95B. 58개 MoE 블록 총 461B.
- MTP 모듈: 14B.

총계: 코어 아키텍처 ~476B + MTP 14B + 발표된 671B 숫자는 추가 구조적 파라미터(편향 텐서, 전문가별 구성 요소, 공유 전문가 스케일링 등)를 포함합니다. 계산기에서 재현한 숫자는 발표된 수치와 3-5% 이내입니다. 이 차이는 DeepSeek 보고서의 2절 부록에 문서화된 세부 계산에서 비롯됩니다.

순방향 시 활성 파라미터:

- 어텐션: 레이어당 144M * 61 = 8.8B(모든 레이어 작동).
- MLP 활성: 처음 3개 레이어는 밀집(3 * 260M = 780M), 58개 MoE 레이어는 각각 8개 라우팅 + 1개 공유 + 라우팅 오버헤드. 레이어당 활성 MLP: ~260M. 총: 3 * 260M + 58 * 260M = ~15.9B.
- 임베딩 + 정규화: 1.2B.
- 총 활성: 대략 26B 코어 + 14B MTP(훈련되었으나 추론 시 항상 실행되지 않음) ≈ 37B.

### 671B / 37B 비율

18x 희소성 비율(활성 파라미터는 전체의 5.5%). DeepSeek-V3는 오픈 웨이트로 출시된 가장 희소한 프론티어 MoE 모델입니다. Mixtral 8x7B의 비율 13/47(28%)은 훨씬 더 밀집되어 있습니다. Llama 4 Maverick의 비율 17B/400B(4.25%)는 비슷합니다. DeepSeek의 베팅: 프론티어 규모에서 더 많은 전문가와 낮은 활성화 비율이 활성 FLOP당 더 나은 품질을 생성합니다.

### DeepSeek-V3의 위치

| 모델 | 총계 | 활성 | 비율 | 어텐션 | 새로운 아이디어 |
|-------|------|-------|-------|-----------|-------------|

### 후속: R1, V4

DeepSeek-R1(2025)은 V3 백본에서 추론 훈련을 진행한 것입니다. R1은 동일한 아키텍처를 사용합니다. 변경된 것은 사전 훈련 아키텍처가 아닌 사후 훈련 레시피(검증 가능한 작업에 대한 대규모 RL)입니다.

DeepSeek-V4(출시된다면)는 MLA + MoE + MTP를 유지하고 Phase 10 · 17의 NSA 후속인 DSA(DeepSeek Sparse Attention)를 추가할 것으로 예상됩니다. 계보는 안정적입니다: 아키텍처 수준의 혁신이 축적되며, 각 버전은 추가 조절 장치를 도입합니다.

## 사용 방법

`code/main.py`는 DeepSeek-V3의 형태에 특화된 파라미터 계산기입니다. 이를 실행하여 출력값을 논문의 수치와 비교하고, 가상의 변형 모델(256 experts vs 512, top-8 vs top-16, MLA rank 512 vs 1024)에 적용해 보세요.

확인해야 할 사항:

- 총 파라미터 수 vs 논문 발표 수치 671B(6710억).
- 활성 파라미터 수 vs 논문 발표 수치 37B(370억).
- 128k 컨텍스트에서의 KV 캐시 — MLA vs GQA 비교.
- 레이어별 세부 분석을 통해 실제 파라미터 예산이 어디에 할당되는지 확인.

## 연습 문제

1. `code/main.py`를 실행하세요. 계산기의 총 파라미터 추정치(671B)를 논문과 비교하고 차이가 발생하는 원인을 파악하세요. 논문의 2절에는 전체 항목별 세부 정보가 있습니다.

2. 설정을 수정하여 MLA 랭크(MLA rank)를 512 대신 256으로 변경하세요. 128k 컨텍스트에서 결과적인 KV 캐시 크기를 계산하세요. 이는 몇 %의 메모리 감소를 가져오며, 헤드당 표현력(per-head expressiveness)에는 어떤 비용이 발생하나요?

3. DeepSeek-V3(256 전문가, top-8)의 라우팅 방식을 가상의 (512 전문가, top-8) 변형 모델과 비교하세요. 총 파라미터는 증가하지만 활성 파라미터(active parameters)는 동일합니다. 이론적으로 추가 전문가 용량이 제공하는 이점은 무엇이며, 추론 시 어떤 비용이 발생하나요?

4. DeepSeek-V3 기술 보고서(arXiv:2412.19437)의 2.1절을 읽고 MLA에 대해 설명하세요. 추론 효율성(inference-time efficiency)을 위해 K와 V 디컴프레션 행렬이 후속 행렬 곱셈(matmul)에 "흡수"될 수 있는 이유를 3문장으로 설명하세요.

5. DeepSeek-V3는 대부분의 연산에 FP8 훈련을 사용합니다. 671B 가중치 저장 시 FP8과 BF16의 메모리 절감 효과를 계산하세요. 이는 14.8T-토큰 훈련 예산과 어떻게 연관되나요?

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| MLA | "Multi-Head Latent Attention" | K와 V를 공유 저랭크 잠재 공간(kv_lora_rank, 일반적으로 512)으로 압축하고, 헤드별로 실시간 복원; KV 캐시는 잠재 공간만 저장 |
| kv_lora_rank | "MLA 압축 차원" | K와 V에 대한 공유 잠재 공간의 크기; DeepSeek-V3는 512 사용 |
| First k dense layers | "초기 레이어는 밀집 유지" | 처음 몇 개의 MoE-모델 레이어는 MoE 라우터를 건너뛰고 안정성을 위해 밀집 MLP 실행 |
| num_experts_per_tok | "Top-k 라우팅" | 토큰당 활성화되는 라우팅 전문가 수; DeepSeek-V3는 8개 사용 |
| Shared experts | "항상 활성 전문가" | 라우팅과 무관하게 모든 토큰을 처리하는 전문가; DeepSeek-V3는 1개 사용 |
| Auxiliary-loss-free routing | "편향 조정 부하 균형" | 손실 항 추가 없이 전문가 부하 균형을 유지하기 위해 훈련 중 조정되는 전문가별 편향 항 |
| MTP module | "추가 예측 헤드" | h^(1)과 E(t+1)로부터 t+2를 예측하는 트랜스포머 블록; 더 밀집된 훈련, 무료 추론-디코딩 초안 |
| DualPipe | "양방향 파이프라인" | 순방향/역방향 계산을 노드 간 all-to-all과 겹쳐 처리하는 훈련 스케줄 |
| Active parameter ratio | "희소성" | active_params / total_params; DeepSeek-V3는 5.5% 달성 |
| FP8 training | "8비트 훈련" | FP8에서 훈련 저장 및 많은 계산 연산 수행; BF16 대비 메모리 약 절반 사용(작은 품질 손실 발생)
