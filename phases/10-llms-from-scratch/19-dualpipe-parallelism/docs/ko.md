# DualPipe 병렬화

> DeepSeek-V3는 2,048개의 H800 GPU에서 훈련되었으며, MoE 전문가(experts)가 노드 간에 분산되어 있습니다. 노드 간 전문가(all-to-all) 통신은 계산 1 GPU-시간당 통신 1 GPU-시간이 소요되었습니다. GPU는 절반의 시간 동안 유휴 상태였습니다. DualPipe(DeepSeek, 2024년 12월)는 순전파(forward)와 역전파(backward) 계산을 이들이 트리거하는 all-to-all 통신과 중첩시키는 양방향 파이프라인입니다. 버블(bubbles)이 감소하고 처리량(throughput)이 증가하며, 두 개의 모델 파라미터 복사본을 유지하는 것(이름의 "dual")은 이미 전문가 병렬화(Expert Parallelism)가 전문가들을 랭크(ranks) 간에 분산시키는 상황에서 비용이 적게 듭니다. 이 레슨은 DualPipe가 실제로 수행하는 작업과 Sea AI Lab의 DualPipeV 개선판이 2배의 파라미터 비용을 버블을 약간 더 타이트하게 만드는 대가로 어떻게 제거하는지에 대한 Learn-type 워크스루입니다.

**유형:** Learn  
**언어:** Python (stdlib, 스케줄 시뮬레이터)  
**선수 지식:** Phase 10 · 05 (분산 훈련, FSDP, DeepSpeed), Phase 10 · 14 (오픈 모델 아키텍처 및 MoE)  
**소요 시간:** ~60분

## 학습 목표

- **DualPipe 순방향-역방향 청크(forward-backward chunk)의 네 가지 구성 요소를 명명하고**, 각 구성 요소가 **왜 별도의 오버랩 윈도우(overlap window)를 가지는지** 설명하시오.
- **대규모에서의 파이프라인 버블 문제(pipeline bubble problem)**를 설명하고, **실제 적용 시 "버블 없는(bubble-free)"**이 마케팅에서 말하는 것과 어떻게 다른지 설명하시오.
- **8개의 PP(파이프라인 병렬) 랭크와 16개의 마이크로 배치(micro-batch)**에 대해 **DualPipe 스케줄링을 수작업으로 추적**하고, **순방향(forward) 및 역방향(reverse) 스트림이 서로의 유휴 슬롯(idle slot)을 채우는 것**을 확인하시오.
- **DualPipeV(Sea AI Lab, 2025)가 만드는 트레이드오프(tradeoff)**를 설명하시오: **2x 파라미터 복제(parameter replication)를 포기하는 대신**, **전문가 병렬화(Expert Parallelism)가 비활성화될 때 약간 더 큰 버블**이 발생한다는 점.

## 문제

2k H800 GPU에서 671B MoE 모델을 훈련할 때 세 가지 복합적인 병목 현상이 발생합니다:

1. **메모리 압박.** 각 GPU는 모델의 일부를 보유합니다. 128개 헤드에 걸쳐 61개 레이어에서 시퀀스 8k의 활성화 메모리는 매우 큽니다.
2. **파이프라인 버블.** 전통적인 파이프라인 병렬화(GPipe, 1F1B)는 GPU가 해당 단계의 입력 또는 그래디언트를 기다리는 동안 유휴 상태로 남아 있습니다. 8단계 구성에서 1F1B 스케줄링을 사용하더라도 GPU 시간의 약 12%가 버블로 소모될 수 있습니다.
3. **노드 간 all-to-all 통신.** 전문가 병렬화를 사용하는 MoE는 노드 전체에 전문가를 분산시킵니다. 매 순전파 시 토큰을 전문가에게 디스패치하기 위한 all-to-all 통신이 발생하고, 또 다른 통신이 결과를 결합합니다. 2k GPU에서는 이 작업이 1:1의 계산-통신 비율로 쉽게 이어집니다.

이러한 각 문제에는 별도의 해결책이 있습니다: 메모리 문제에는 그래디언트 체크포인팅, 파이프라인 버블에는 Zero Bubble(Sea AI Lab, 2023), all-to-all 통신에는 전문가 병렬 통신 커널이 있습니다. DualPipe는 이러한 해결책들이 함께 작동하도록 통합합니다. 이 스케줄은 단일 순전파-역전파 청크 내에서 계산과 통신을 오버랩시키고, 파이프라인의 양쪽 끝에서 마이크로 배치를 동시에 주입하며, 결과적인 스케줄을 사용하여 계산 창 내부에 all-to-all 통신을 숨깁니다.

보고된 결과: 파이프라인 버블이 거의 제거되었으며, DeepSeek-V3의 14.8T 토큰 훈련 실행에서 95% 이상의 GPU 활용률을 달성했습니다.

## 개념

### 파이프라인 병렬성 복습

N개의 레이어를 가진 모델을 P개의 장치에 분할합니다. 장치 `i`는 레이어 `i * N/P .. (i+1) * N/P - 1`을 보유합니다. 마이크로 배치(micro-batch)는 장치 0부터 P-1까지 순방향으로 흐르고, 역전파는 P-1부터 0까지 역방향으로 흐릅니다. 각 장치는 이전 장치가 출력을 보낸 후에만 순방향 단계를 시작할 수 있으며, 하위 장치가 상류 기울기(upstream gradient)를 보낸 후에만 역전파를 시작할 수 있습니다.

GPipe (Huang et al., 2019)는 한 번에 하나의 마이크로 배치만 예약하여 대부분의 GPU 시간을 낭비합니다. 1F1B (Narayanan et al., 2021)는 여러 마이크로 배치에 대해 순방향 및 역전파 패스를 교차시킵니다. Zero Bubble (Qi et al., 2023)은 역전파를 입력 역전파(B)와 가중치 역전파(W)로 분할하고 버블을 채우기 위해 예약합니다. Zero Bubble 이후에는 파이프라인이 거의 타이트해집니다.

DualPipe는 다음 단계로, 두 가지 아이디어를 추가합니다:

### 아이디어 1: 청크 분해(chunk decomposition)

각 순방향 청크는 다음 네 가지 구성 요소로 분할됩니다:

- **어텐션(attention).** Q/K/V 투영, 어텐션, 출력 투영.
- **올투올 디스패치(all-to-all dispatch).** 토큰을 해당 전문가(expert)로 전송하는 노드 간 통신.
- **MLP.** MoE 전문가 계산.
- **올투올 컴바인(all-to-all combine).** 전문가 출력을 다시 가져오는 노드 간 통신.

역방향 청크는 각 구성 요소의 기울기 버전을 추가합니다. DualPipe는 올투올 디스패치가 다음 청크의 어텐션 계산과 병렬로 실행되고, 올투올 컴바인이 다음 청크의 MLP 계산과 병렬로 실행되도록 예약합니다.

### 아이디어 2: 양방향 예약(bidirectional scheduling)

대부분의 파이프라인 예약은 단계 0에서 마이크로 배치를 주입하고 단계 P-1로 흐르게 합니다. DualPipe는 양쪽 끝에서 마이크로 배치를 주입합니다. 단계 0은 순방향 마이크로 배치가 시작되는 것을 보고, 단계 P-1도 순방향 마이크로 배치가 시작되는 것을 봅니다. 두 스트림은 중간에서 만납니다.

이를 위해 장치 `i`는 초기 파이프라인 레이어 `i`와 후기 파이프라인 레이어 `P - 1 - i`를 모두 보유해야 합니다. 이것이 DualPipe의 "듀얼(dual)" 부분입니다: 각 장치는 두 방향의 모델 레이어 복사본을 유지합니다. DeepSeek-V3 규모에서는 2배의 파라미터 복제 비용이 발생하지만, 전문가 병렬성(Expert Parallelism)이 이미 MoE 전문가를 매우 얇게 분산시키기 때문에 비전문가 레이어를 두 번 복제하는 것은 큰 부담이 아닙니다.

중요한 점은 한 방향의 순방향 스트림과 다른 방향의 역전파 스트림이 단일 방향 예약에서 버블이 발생하는 위치와 정확히 겹친다는 것입니다. 버블이 사라집니다.

### 수작업으로 추적한 예약

P = 4개의 랭크(ranks)와 8개의 마이크로 배치(4개 순방향/4개 역방향)를 고려해 봅시다. 시간은 왼쪽에서 오른쪽으로 흐르며, 행은 장치 랭크입니다.

```
           시간 →
랭크 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
랭크 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
랭크 2:        F1 F2  F3/F5R F4/F6R    B1 ...
랭크 3:           F1  F2/F5R F3/F6R    ...
```

"F4/F5R" 표기법 해석: 랭크 1은 동일한 시간 슬롯에서 마이크로 배치 4의 순방향(파이프라인 왼쪽에서 오른쪽)과 마이크로 배치 5의 순방향(오른쪽에서 왼쪽)을 실행합니다. 이것이 "양방향"의 운영적 의미입니다.

랭크 2에서는 교차 스트림이 더 빨리 겹치고, 랭크 0과 P-1에서는 가장 늦게 겹칩니다. 예약의 안정된 중간 단계에서는 모든 랭크가 X 방향의 순방향과 Y 방향의 역전파를 겹쳐서 실행합니다. 컴퓨팅은 바쁘게 작동합니다. 순방향 패스의 올투올 디스패치는 역전파 계산 내부에 숨겨지고, 올투올 컴바인은 순방향 계산 내부에 숨겨집니다. 버블이 완전히 제거됩니다.

### 버블 계산

표준 1F1B 파이프라인 버블(랭크당 낭비되는 시간):

```
bubble_1F1B = (P - 1) * 순방향_청크_시간
```

Zero Bubble 개선은 버블을 줄이지만 완전히 제로는 아닙니다. DualPipe는 안정된 단계에서 마이크로 배치 수가 파이프라인 깊이의 2배수로 나누어질 때 버블이 없습니다. 안정된 단계 외부(예: 워밍업 및 쿨다운)에서는 일부 버블이 있지만, 마이크로 배치 수에 따라 증가하지 않습니다 — 논문에서 강조하는 핵심 속성입니다.

마케팅 용어로는 "버블 프리(bubble-free)"입니다. 기술적 용어로는 버블이 마이크로 배치 수에 따라 증가하지 않습니다. Sea AI Lab의 후속 분석(DualPipeV / Cut-in-half)에 따르면 전문가 병렬성(Expert Parallelism)이 병목이 아닐 때만 완전한 제로 버블이 가능합니다. EP 기반 올투올(all-to-all)이 있는 경우 항상 일정 수준의 예약 타협이 존재합니다.

### DualPipeV — 개선판

Sea AI Lab (2025)는 EP 통신 겹침이 핵심이 아닐 때 2배 파라미터 복제가 낭비적이라고 지적했습니다. 그들의 DualPipeV 예약은 양방향 주입을 "V-형태" 예약으로 접어서 단일 파라미터 복사본에서 실행합니다. 버블은 DualPipe보다 약간 크지만 메모리 절약 효과는 상당합니다. DeepSeek은 EP-off 모드로 오픈소스 DualPipe 구현에서 DualPipeV를 채택했습니다.

트레이드오프:

| 기능 | DualPipe | DualPipeV | 1F1B | Zero Bubble |
|---------|---------|-----------|------|------------|
| 장치당 파라미터 복사본 | 2 | 1 | 1 | 1 |
| 마이크로 배치 대비 버블 | 상수 | 약간 증가 | 증가 | 증가 |
| 계산-통신 겹침 | 완전 | 부분 | 최소 | 부분 |
| 사용 시기 | EP-집약적 MoE | 밀집 또는 EP-경량 | 기준선 | 모든 파이프라인 |

### 14.8T-토큰 실행에 대한 의미

DeepSeek-V3의 사전 학습은 2,048개의 H800 GPU에서 약 2.8M GPU-시간으로 14.8T 토큰을 소비했습니다. 순진한 1F1B를 사용했다면 파이프라인 버블로 인해 12-15%인 340-420K GPU-시간을 낭비했을 것입니다. 이는 70B 모델 전체를 학습시킬 수 있는 양입니다. DualPipe는 이 중 대부분을 회복했습니다. 내부 로그 없이는 기여도를 직접 정량화하기 어렵지만, 논문의 주장은 훈련 전반에 걸쳐 평균 95% 이상의 GPU 사용률입니다.

1,000개 미만의 GPU를 사용하는 소규모 실행에서는 DualPipe가 과합니다 — 파이프라인 버블은 총 비용 대비 상대적으로 작으며, 밀집 모델 학습은 거의 올투올 병목 현상을 겪지 않습니다. 수천 개의 GPU 규모에서 프론티어 MoE 학습을 수행할 때는 사실상 필수적입니다.

### 스택 내 위치

- **FSDP**(Phase 10 · 05)와 상호 보완적. FSDP는 모델 파라미터를 랭크 간에 분할하고, DualPipe는 랭크 간 계산을 예약합니다. 둘을 결합할 수 있습니다.
- **ZeRO-3** 기울기 분할과 호환. 두 복사본 복제에 대한 장부 관리는 ZeRO의 분할 기울기와 협력해야 합니다.
- 특정 클러스터 토폴로지에 맞춰 조정된 **커스텀 올투올 커널**이 필요. DeepSeek의 오픈소스 커널이 참조 구현입니다.

## 사용 방법

`code/main.py`는 파이프라인 스케줄 시뮬레이터입니다. `(P, n_micro_batches, schedule)`을 입력으로 받아 1F1B, Zero Bubble, DualPipe, DualPipeV 각각에 대한 안정 단계 활용률(stable-phase utilization)을 출력합니다. 이는 교육용 도구이며, 논문에서의 정성적 주장과 일치하는 수치를 보여주지만 실제 측정된 속도 향상에 대한 주장은 아닙니다.

시뮬레이터의 가치: 다양한 P(파이프라인 병렬 깊이)와 마이크로 배치 수로 실행한 후 1F1B에서는 버블 비율이 증가하지만 DualPipe에서는 그렇지 않은 것을 관찰하세요.

실제 학습 실행을 위한 통합 고려사항:

- 마이크로 배치 수로 깔끔하게 나누어지는 파이프라인 병렬 깊이(P)를 선택하세요.
- 전문가 병렬 메시가 양방향(all-to-all) 통신을 지원하는지 확인하세요. DeepSeek의 커널이 참조 구현입니다.
- 처음 스케줄 구현 시 디버깅에 일주일 정도 소요될 것으로 예상하세요. 장부 관리가 까다롭습니다.
- 집계된 GPU 활용률뿐 아니라 각 랭크(rank)별 GPU 활용률을 모니터링하세요. DualPipe의 이점은 느린 작업자(stragglers)를 타이트하게 조정함으로써 발생합니다.

## Ship It

이 레슨은 `outputs/skill-dualpipe-planner.md`를 생성합니다. 훈련 클러스터 사양(GPU 수, 토폴로지, 인터커넥트, 모델 형태)이 주어졌을 때, 파이프라인 병렬화 전략, 사용할 스케줄링 알고리즘, 그리고 목표 규모에서의 예상 버블 비율(bubble fraction)을 추천합니다.

## 연습 문제

1. `(P=8, micro_batches=16, schedule=dualpipe)`와 `(P=8, micro_batches=16, schedule=1f1b)` 설정으로 `code/main.py`를 실행해 보세요. GPU 사용률 차이를 계산하고, 이를 백만 토큰당 복구된 GPU-시간 단위로 표현해 보세요.

2. `(P=4, micro_batches=8, schedule=dualpipe)`에 대한 스케줄 테이블을 직접 그려보세요. 각 시간 슬롯에 마이크로 배치 ID와 방향(forward/backward)을 표시하고, 버블이 없는 첫 번째 시간 슬롯을 식별해 보세요.

3. DeepSeek-V3 기술 보고서(arXiv:2412.19437)의 그림 5를 읽어보세요. DualPipe 순전파 청크 내에서 all-to-all 디스패치의 중첩 구간을 식별하고, 계산 스케줄이 이를 어떻게 숨기는지 설명해 보세요.

4. P=8 파이프라인 단계를 가진 70B 밀집 모델과 P=16 파이프라인 단계를 가진 671B MoE 모델에 대해 DualPipe의 2x 파라미터 오버헤드를 계산해 보세요. MoE 사례에서 오버헤드가 상대적으로 작은 이유(대부분의 파라미터가 전문가이며 대규모 EP 그룹에 샤딩됨)를 설명해 보세요.

5. DualPipe와 Chimera(2021년 경쟁 양방향 스케줄러)를 비교해 보세요. 논문 3.4절을 참고하여 Chimera에는 없지만 DualPipe에 추가된 두 가지 구체적인 속성을 식별해 보세요.

| 1F1B | "기본 파이프라인 스케줄" | 순전파 1회/역전파 1회 교차 스케줄링; DualPipe가 능가하는 기준 방식 |
| 제로 버블 | "Sea AI Lab 2023" | 역전파를 B(입력 그래디언트)와 W(가중치 그래디언트)로 분할; 파이프라인을 거의 완전히 조임 |
| 듀얼파이프 | "DeepSeek-V3 스케줄" | 양방향 파이프라인 + 연산-통신 중첩; 버블이 마이크로 배치 수에 따라 증가하지 않음 |
| 듀얼파이프V | "절반 절단" | 2배 파라미터 복제를 희생하여 버블이 약간 커지는 V-형태 개선판 |
| 청크 | "파이프라인 작업 단위" | 하나의 마이크로 배치가 하나의 파이프라인 단계를 통과하는 순전파 또는 역전파 |
| 올투올 디스패치 | "토큰을 전문가에게 전송" | 토큰을 할당된 MoE 전문가로 라우팅하는 노드 간 통신 |
| 올투올 컴바인 | "전문가 출력 수집" | MLP 이후 전문가 출력을 수집하는 노드 간 통신 |
| 전문가 병렬화(EP) | "GPU 간 전문가 분산" | MoE 전문가를 랭크에 분할하여 다른 GPU가 다른 전문가를 보유하도록 함 |
| 파이프라인 병렬화(PP) | "GPU 간 레이어 분할" | 모델 레이어를 랭크에 분할; DualPipe가 스케줄링하는 차원 |
| 버블 비율 | "낭비되는 GPU 시간" | (버블_시간 / 총_시간); DualPipe가 0에 가깝게 만드는 비율 |
