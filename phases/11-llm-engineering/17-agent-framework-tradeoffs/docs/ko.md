# 에이전트 프레임워크 트레이드오프 — LangGraph vs CrewAI vs AutoGen vs Agno

> 모든 프레임워크는 동일한 데모(연구 에이전트가 보고서 작성)를 판매하며 동일한 버그(상태 스키마와 오케스트레이션 계층의 충돌)를 숨깁니다. 문제의 형태와 일치하는 추상화를 가진 프레임워크를 선택하세요. 나머지는 두 번 작성해야 하는 접착제(glue) 코드입니다.

**유형:** 학습  
**언어:** Python  
**사전 요구 사항:** Phase 11 · 09 (함수 호출), Phase 11 · 16 (LangGraph)  
**소요 시간:** ~45분

## 문제 정의

여러 LLM 호출이 필요한 작업이 있습니다. 연구 워크플로우(계획, 검색, 요약, 인용)일 수도 있고, 코드 리뷰 파이프라인(변경 사항 파싱, 비평, 패치, 검증)일 수도 있으며, 항공편 예약, 이메일 작성, 경비 보고서 제출을 처리하는 다중 턴 어시스턴트일 수도 있습니다. 이때 프레임워크를 선택하게 됩니다.

3일 후, 해당 프레임워크의 추상화 계층이 누수된다는 것을 발견합니다. CrewAI는 역할을 제공하지만 "연구자"가 구조화된 계획을 "작가"에게 전달해야 할 때 문제가 발생합니다. AutoGen은 에이전트 간 채팅 기능을 제공하지만 체크포인트가 대화 로그의 피클(pickle) 파일로만 저장되는 1급 상태(first-class state) 개념이 없습니다. LangGraph는 상태 그래프를 제공하지만 에이전트가 수행할 작업을 알기 전에 모든 전환(transition) 이름을 지정해야 합니다. Agno는 단일 에이전트 기본 요소를 제공하지만 3개의 동시 작업자로 확장하려 할 때 오류가 발생합니다.

해결책은 "최고의 프레임워크를 선택하는 것"이 아닙니다. 프레임워크의 핵심 추상화 개념을 문제 구조에 맞추는 것입니다. 이 레슨에서는 그 방법을 제시합니다.

## 개념

![에이전트 프레임워크 매트릭스: 핵심 추상화 vs 문제 형태](../assets/framework-matrix.svg)

2026년 시장을 주도하는 4가지 프레임워크가 있습니다. 이들의 핵심 추상화는 서로 다릅니다.

| 프레임워크 | 핵심 추상화 | 최적 적합 | 최악 적합 |
|-----------|------------------|----------|-----------|
| **LangGraph** | `StateGraph` — 타입이 지정된 상태, 노드, 조건부 엣지, 체크포인터. | 명시적 상태와 인간 개입 중단점이 있는 워크플로우; 시간 여행 디버깅이 필요한 프로덕션 에이전트. | 토폴로지가 알려지지 않은 느슨한 역할 기반 브레인스토밍. |
| **CrewAI** | `Crew` — 역할(목표, 백스토리), 작업, 프로세스(순차 또는 계층적). | 짧은 선형/계층적 계획을 가진 역할극 또는 페르소나 기반 워크플로우. | 크루의 턴 기록을 넘어 상태(stateful)가 필요한 모든 것; 복잡한 분기. |
| **AutoGen** | `ConversableAgent` 쌍 — 종료 조건까지 번갈아 대화하는 2개 이상의 에이전트. | 교사-학생, 제안자-비판자, 행위자-검토자와 같이 채팅에서 사고가 발생하는 다중 에이전트 *대화*. | 알려진 DAG가 있는 결정적 워크플로우; 재시작 시 지속 상태가 필요한 모든 것. |
| **Agno** | `Agent` — 단일 LLM + 도구 + 메모리, 팀으로 조합 가능. | 빠르게 구축 가능한 단일 에이전트 및 경량 팀; 강력한 멀티모달리티 및 내장 스토리지 드라이버. | 사용자 정의 리듀서가 있는 깊고 명시적으로 분기된 그래프. |

### "추상화"의 실제 의미

프레임워크의 핵심 추상화는 아키텍처를 설명할 때 화이트보드에 그리는 것입니다.

- **LangGraph** → 그래프를 그립니다. 노드는 단계, 엣지는 전환, 모든 지점의 상태 객체는 타입이 지정됩니다. 정신 모델은 상태 머신입니다.
- **CrewAI** → 조직도를 그립니다. 각 역할은 직무 설명을 가지며 매니저가 작업을 라우팅합니다. 정신 모델은 전문가 소규모 팀입니다.
- **AutoGen** → Slack DM을 그립니다. 두 에이전트가 서로 메시지를 주고받으며, 중재자가 필요하면 세 번째 에이전트가 참여합니다. 정신 모델은 채팅입니다.
- **Agno** → 도구가 달린 단일 상자를 그립니다. 팀을 위해 상자를 나란히 배치합니다. 정신 모델은 "배터리 포함 에이전트"입니다.

### 상태 문제

상태는 대부분의 프레임워크 선택이 프로덕션에서 실패하는 지점입니다.

- **LangGraph.** 타입이 지정된 상태(`TypedDict` 또는 Pydantic 모델), 필드별 리듀서, 1급 체크포인터(SQLite/Postgres/Redis). 재개, 중단, 시간 여행이 무료입니다. *(11단계 · 16단계 참조)*
- **CrewAI.** 상태는 `context` 필드를 통해 작업 간 문자열로 흐르거나 `output_pydantic`을 통해 구조화됩니다. 기본적으로 크루별 지속 저장소가 없으며, 재시작 시 크루를 유지하려면 직접 추가해야 합니다.
- **AutoGen.** 상태는 채팅 기록과 사용자 정의 `context`입니다. 대화 기록은 지속되지만, 임의 워크플로우 상태는 어댑터를 작성하지 않으면 지속되지 않습니다.
- **Agno.** `storage=`를 통해 `Agent`에 연결된 내장 스토리지 드라이버(SQLite, Postgres, Mongo, Redis, DynamoDB) — 대화 세션과 사용자 메모리가 자동으로 지속됩니다. 전체 그래프 체크포인터는 아니며 세션 저장소입니다.

### 분기 문제

모든 비중요 에이전트는 분기합니다. 분기 결정자가 중요합니다.

- **LangGraph** — 조건부 엣지를 통해 개발자가 결정합니다. 라우팅은 명명된 분기를 가진 Python 함수입니다. 분기는 컴파일된 그래프에서 1급이며, 체크포인터는 어떤 분기가 선택되었는지 기록합니다.
- **CrewAI** — 계층 모드에서는 매니저가, 순차 모드에서는 빌드 타임에 개발자가 결정합니다. 라우팅은 작업 목록에 암시적이며, 매니저의 프롬프트 외에는 1급 "if"가 없습니다.
- **AutoGen** — 에이전트가 채팅을 통해 결정합니다. 분기는 다음 발화자에서 발생합니다. `GroupChatManager`가 다음 발화자를 선택하며, `speaker_selection_method`를 직접 작성할 수 있지만 기본값은 LLM 기반입니다.
- **Agno** — 에이전트가 다음에 호출할 도구를 결정합니다. 팀에는 조정자/라우터/협업자 모드가 있으며, 그 이상의 분기는 개발자의 책임입니다.

### 관측 가능성 문제

- **LangGraph** — LangSmith 또는 모든 OTel 내보내기를 통한 OpenTelemetry. 모든 노드 전환은 추적 스팬이며, 체크포인트는 재생 가능한 추적 역할을 합니다. LangSmith이 1차 옵션이며, Langfuse/Phoenix에도 어댑터가 있습니다.
- **CrewAI** — 2025년 말부터 1급 OpenTelemetry; Langfuse, Phoenix, Opik, AgentOps와 통합.
- **AutoGen** — `autogen-core`를 통한 OpenTelemetry 통합; AgentOps와 Opik에 커넥터. 추적 세분성은 노드별이 아닌 에이전트-메시지별입니다.
- **Agno** — 내장 `monitoring=True` 플래그 및 OpenTelemetry 내보내기; 세션 추적을 위한 Langfuse와 긴밀한 통합.

### 비용 및 지연 시간

4개 프레임워크 모두 호출당 오버헤드(프레임워크 로직, 검증, 직렬화)를 추가합니다. 오버헤드 증가 순서: Agno ≈ LangGraph < CrewAI ≈ AutoGen. 차이는 프레임워크가 수행하는 추가 LLM 라우팅 양에 의해 결정됩니다. CrewAI의 계층적 매니저는 다음 참가자를 결정하는 데 토큰을 소비하며, AutoGen의 `GroupChatManager`도 마찬가지입니다. LangGraph은 `llm.invoke`를 작성한 위치에서만 토큰을 소비합니다. Agno의 단일 에이전트 경로는 가볍습니다.

실행당 비용이 중요할 때는 LLM 선택 라우팅보다 명시적 라우팅(LangGraph 엣지, AutoGen `speaker_selection_method`)을 선호하세요.

### 상호 운용성

- **LangGraph** ↔ **LangChain** 도구, 리트리버, LLM. 1급 MCP 어댑터(도구로 MCP 서버 가져오기).
- **CrewAI** ↔ `BaseTool`에서 상속된 도구; LangChain 도구, LlamaIndex 도구, MCP 도구가 모두 적응됩니다. `allow_delegation=True`를 통한 크루 간 위임.
- **AutoGen** → `FunctionTool`은 모든 Python 호출 가능 객체를 래핑; MCP 어댑터 사용 가능. 에이전트 간 패턴을 위한 AG2 생태계와 긴밀한 결합.
- **Agno** → `@tool` 데코레이터 또는 BaseTool 서브클래스; MCP 어댑터; 도구는 에이전트 및 팀 간에 공유 가능.

## 기술

> 주어진 에이전트 문제에 특정 프레임워크가 적합한 이유를 한 문장으로 설명할 수 있다.

사전 점검 목록:

1. **형태 그리기.** 그래프(타입화된 상태, 명명된 전이)인가? 역할극(전문가 간 작업 전달)인가? 채팅(에이전트들이 완료될 때까지 대화)인가? 도구를 가진 단일 에이전트인가?
2. **분기 결정자 정하기.** 개발자 결정 분기 → LangGraph. 관리자-에이전트 결정 → CrewAI 계층적. 채팅-발생적 → AutoGen. 도구 호출 결정 → Agno.
3. **상태 예산 확인.** 체크포인트 재개, 시간 여행, 실행 중 인간 개입이 필요한가? 필요하다면 LangGraph가 기본 선택; Agno 세션은 대화 범위 상태를 커버.
4. **비용 예산 확인.** LLM 선택 라우팅은 매 턴 추가 토큰 비용이 발생. 에이전트가 하루에 수천 번 실행된다면 명시적 라우팅을 선호.
5. **프레임워크 오버헤드 예산.** 모든 프레임워크는 추가 의존성. 작업이 LLM 호출 2회와 도구 사용 1회라면, 프레임워크 없이 30줄 파이썬 코드를 작성하는 것이 가장 저렴.

그래프, 조직도, 채팅 흐름, 에이전트 박스를 그리기 전에는 프레임워크를 선택하지 마라. 실제 필요한 것과 프레임워크의 상태 모델이 충돌한다면 선택하지 마라.

## 결정 매트릭스

| 문제 형태 | 선호 프레임워크 | 이유 |
|---------------|---------------------|-----|
| 타입이 지정된 상태, 인간 승인, 장기 실행이 있는 워크플로우 DAG | LangGraph | 1급 상태, 체크포인팅, 인터럽트, 시간 이동 기능 지원. |
| 명확한 역할이 있는 연구/작성 파이프라인 | CrewAI(순차적) 또는 LangGraph 서브그래프 | CrewAI에서 역할-작업 할당이 간단하게 표현 가능; 분기 복잡도가 증가하면 LangGraph로 확장. |
| 제안자-평가자 또는 교사-학생 대화 | AutoGen | 2개 에이전트 채팅이 기본 형태. |
| 도구, 세션, 메모리를 가진 단일 에이전트 | Agno | 가장 가벼운 설정, 내장 저장소 및 메모리 지원. |
| 리듀서가 있는 수천 개의 병렬 팬아웃 | LangGraph + `Send` | 1급 병렬 디스패치 프리미티브를 가진 유일한 프레임워크. |
| 빠른 프로토타입, 프레임워크 제약 없음 | 순수 Python + 프로바이더 SDK | 프레임워크가 없는 것이 가장 빠른 프레임워크.>

## 연습 문제

1. **쉬움.** 동일한 작업 — "Anthropic의 본사 위치 조사, 200단어 분량의 보고서 작성, 출처 인용" — 을 LangGraph(네 개의 노드: 계획, 검색, 작성, 인용)와 CrewAI(세 개의 역할: 연구자, 작성자, 편집자)로 각각 구현해 보세요. 실행당 토큰 비용과 코드 라인 수를 보고하세요.
2. **중간.** 동일한 작업을 AutoGen(연구자 ↔ 작성자 채팅, `GroupChat`을 통해 편집자 참여)과 Agno(`search_tools` 및 `write_tools`를 가진 단일 에이전트, 세션 저장소 추가)로 구현해 보세요. 다음 기준으로 네 가지 구현을 평가하세요: (a) 실행당 비용, (b) 충돌 후 재개 가능성, (c) 작성 단계 전 인간 승인 주입 가능성.
3. **어려움.** 짧은 문제 설명(`{has_typed_state, has_roles, has_dialogue, has_parallel_fanout, needs_resume}` 형식의 JSON)을 입력받아 한 문장 설명과 함께 프레임워크 추천을 반환하는 결정 트리 스크립트 `pick_framework.py`를 작성하세요. 직접 설계한 6가지 케이스로 검증하세요.

## 주요 용어

| 용어 | 사람들이 말하는 표현 | 실제 의미 |
|------|---------------------|-----------|
| 오케스트레이션(Orchestration) | "에이전트들이 어떻게 협력하는가" | 다음에 실행할 노드/역할/에이전트를 결정하는 계층. |
| 내구성 있는 상태(Durable state) | "재시작 후 재개" | 프로세스 종료 후에도 유지되는 상태, 체크포인트 또는 세션 저장소에 연결됨. |
| LLM 선택 라우팅(LLM-selected routing) | "모델이 결정하게 하라" | 플래너 LLM이 매 턴마다 다음 단계를 선택; 유연하지만 모든 결정에 토큰 비용이 발생. |
| 명시적 라우팅(Explicit routing) | "개발자가 결정" | 파이썬 함수 또는 정적 엣지가 다음 단계를 선택; 저렴하고 감사 가능. |
| 크루(Crew) | "CrewAI 팀" | 역할 + 작업 + 프로세스(순차적 또는 계층적)가 단일 실행 가능한 단위로 결합됨. |
| 그룹 채팅(GroupChat) | "AutoGen의 다중 에이전트 채팅" | 화자 선택기가 있는 N명의 에이전트 간 관리되는 대화. |
| 팀(Team, Agno) | "다중 에이전트 Agno" | 에이전트 집합에 대한 라우팅/조정/협업 모드. |
| 상태 그래프(StateGraph) | "LangGraph의 그래프" | 타입이 지정된 상태, 노드, 조건부 엣지, 체크포인터 기본 요소.
